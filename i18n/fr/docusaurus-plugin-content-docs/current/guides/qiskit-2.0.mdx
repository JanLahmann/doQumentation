# Guide de migration vers Qiskit v2.0 {#qiskit-v20-migration-guide}

Qiskit v2.0 est la prochaine étape du cycle de vie nouvellement stabilisé du SDK Qiskit.
Cette version majeure rompt la compatibilité avec les versions précédentes de Qiskit, mais,
contrairement à la version v1.0, elle n'implique aucun changement dans la structure des paquets.

Ce guide décrit les chemins de migration pour les changements de fonctionnalités les plus importants dans Qiskit v2.0, organisés par module.
Utilise la table des matières à droite pour naviguer vers le module qui t'intéresse.

Pour en savoir plus sur les avantages de Qiskit v2.0, consulte le [blog IBM&reg;](https://www.ibm.com/quantum/blog/qiskit-2-0-release-summary).

<span id="qiskit.circuit"></span>
## qiskit.circuit

<span id="c_if"></span>
### Suppression des conditions `c_if` héritées {#removal-of-legacy-c_if-conditions}

Les conditions classiques dans Qiskit sont représentées de manière préférentielle par l'instruction [`IfElseOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.IfElseOp) depuis avant Qiskit v1.0.
Ces opérations sont généralement ajoutées à un [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit) en utilisant la méthode [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) comme gestionnaire de contexte.

Cela signifie également que l'objet [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) n'a plus d'attribut `condition`, et qu'il n'est plus possible d'enchaîner les appels `c_if` sur le résultat d'un appel de méthode de création de porte d'un [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit).

Les seules instructions qui peuvent désormais avoir une condition sont [`IfElseOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.IfElseOp) et [`WhileLoopOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.WhileLoopOp).
La façon la plus correcte de tester une condition est d'utiliser une vérification `isinstance` pour valider que tu as la bonne classe.
Si tu veux prendre en charge l'attribut `Instruction.condition` hérité pendant la transition de ta bibliothèque vers Qiskit v2.0 et versions ultérieures, tu peux utiliser `getattr(operation, "condition", None)`, ce qui amènera généralement les instances `Instruction` à se comporter comme dans Qiskit v1.x et antérieur.

La méthode d'origine, héritée, pour prendre en charge les conditions d'égalité de registre simples dans Qiskit consistait à utiliser la méthode `c_if` sur des instructions individuelles ou des ensembles d'instructions.
Cette approche était strictement moins riche en fonctionnalités que la forme de flot de contrôle complète, et n'a jamais été bien prise en charge sur le matériel.
Cette méthode héritée a été supprimée dans Qiskit v2.0 ; tu dois utiliser la méthode [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) avec une condition appropriée.
Tu peux continuer à utiliser la forme à 2-uplet `(register, integer)` pour représenter une égalité de registre, ou utiliser le [sous-système de calcul classique](https://docs.quantum.ibm.com/api/qiskit/circuit_classical) plus expressif pour représenter des conditions plus complexes.

Par exemple, tu aurais pu écrire :

```python
from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
qc.x(0).c_if(cr, 3)
```

La traduction la plus directe de ceci est d'utiliser la forme à 2-uplet avec le gestionnaire de contexte [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) :

```python
from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
with qc.if_test((cr, 3)):
    qc.x(0)
```

Note que dans cette forme, la porte conditionnelle peut également accéder à plusieurs opérations (il suffit d'en ajouter dans le bloc indenté) sans réévaluer les conditions.
Tu peux aussi fournir une instruction ``else`` et utiliser une logique conditionnelle plus complexe :

```python
from qiskit.circuit import QuantumCircuit
from qiskit.circuit.classical import expr

qc = QuantumCircuit(2, 2)
qc.h(0)
qc.h(1)
qc.measure(qc.qubits, qc.clbits)

# Apply an X to qubit 0 if both measurements were the same:
with qc.if_test(expr.equal(qc.clbits[0], qc.clbits[1])) as else_:
    qc.x(0)
# ... or apply a Y to both qubits otherwise:
with else_:
    qc.y(0)
    qc.y(1)
```

<span id="instruction_durations"></span>
### Suppression de `Instruction.duration` et `Instruction.unit` {#removal-of-instructionduration-and-instructionunit}

Les objets [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) du Circuit avaient auparavant des attributs appelés `duration` et `unit`.
Ceux-ci n'existent plus dans Qiskit v2.0.
En effet, ce champ ne faisait généralement que dupliquer des informations qui constituaient une propriété fixe du matériel, et qui étaient stockées explicitement dans le [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target).
Le fait de stocker ces informations séparément pour chaque instruction individuelle augmentait considérablement les besoins en mémoire de Qiskit et ralentissait même les opérations de Circuit qui n'avaient pas directement besoin d'y accéder.

L'instruction [`Delay`](https://docs.quantum.ibm.com/api/qiskit/circuit#delay) contient toujours ces champs, tout comme le nouveau [`BoxOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#box) introduit dans v2.0, car ces instructions ont un délai _variable_.

Tu peux récupérer la durée spécifiée par le matériel d'une [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) particulière prise en charge par le matériel en interrogeant le [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target).
Pour ce faire, indexe d'abord le [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) avec le nom de l'opération, puis les indices des qubits matériels sous forme de tuple.
Cela retournera soit `None` (si l'instruction est prise en charge, mais qu'aucune propriété spécifique ne lui a été assignée), soit une instance [`InstructionProperties`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties), dont le champ [`duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties#qiskit.transpiler.InstructionProperties.duration) peut fournir la durée en secondes (ou `None`, si non renseignée).

```python
target = "<some Target from a hardware vendor>"

# Query the duration of a CX instruction on hardware qubits (2, 4),
# returning `None` if the duration is not specified.
properties = target["cx"][2, 4]
duration = None if properties is None else properties.duration

# You can also do this in one step, using `getattr`:
duration = getattr(target["cx"][2, 4], "duration", None)
```

En général, à mesure que le flot de contrôle classique devient plus courant dans les exécutions de Circuit, la « durée » d'un Circuit sera de moins en moins bien définie.
Bien que Qiskit v2.x inclue encore l'attribut [`QuantumCircuit.duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.duration), il est recommandé de s'en éloigner, car il sera probablement supprimé à l'avenir ; les hypothèses sous lesquelles il est valide seront de plus en plus souvent violées.
Tu peux plutôt utiliser la nouvelle méthode [`QuantumCircuit.estimate_duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.estimate_duration), qui est plus sûre et plus claire quant aux hypothèses nécessaires.

## qiskit.transpiler

<span id="generate_preset_pass_manager"></span>
### Changements dans l'interface `generate_preset_pass_manager` {#changes-to-generate_preset_pass_manager-interface}

Qiskit v2.0.0 inclut une série de mises à jour de l'interface du gestionnaire de passes préconfiguré (c'est-à-dire les fonctions
[`generate_preset_pass_manager`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.generate_preset_pass_manager) et
[`transpile`](https://docs.quantum.ibm.com/api/qiskit/compiler#transpile)) qui affectent la
gestion des contraintes libres telles que `coupling_map` ou `basis_gates`. Le nombre de
cas d'utilisation de la transpilation où des contraintes libres sont acceptées a été réduit en faveur des
entrées `target` et `backend` (où `backend` est un [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) qui
contient une instance [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target)),
qui permettent une communication plus expressive de ces contraintes.

Les exemples suivants montrent les principaux changements et les alternatives suggérées pour la migration. Note que tous
les conseils de migration ci-dessous s'appliquent à la fois à `transpile` et à `generate_preset_pass_manager` :

1. L'argument d'entrée `backend_properties` n'est plus pris en charge, et la classe `BackendProperties` a
   été supprimée. Tu dois utiliser les arguments `target` ou `backend` à la place pour communiquer
   les durées et erreurs de porte. Il existe plusieurs façons de construire un [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) personnalisé :

  a. Construire un target à partir de zéro et ajouter les erreurs et durées pertinentes aux portes correspondantes :

    ```python
    from qiskit.transpiler import Target, InstructionProperties, generate_preset_pass_manager
    from qiskit.circuit.library import XGate, CXGate

    target = Target(num_qubits=2)
    target.add_instruction(CXGate(), {(0, 1): InstructionProperties(error=.0001, duration=5e-7)})
    target.add_instruction(
        XGate(),
        {
            (0,): InstructionProperties(error=.00001, duration=5e-8),
            (1,): InstructionProperties(error=.00002, duration=6e-8)
        }
    )

    pm = generate_preset_pass_manager(target=target)
    ```

  b. Utiliser [`GenericBackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.fake_provider.GenericBackendV2) pour construire rapidement un
     target générique avec des erreurs et durées préremplies. Celles-ci peuvent également être mises à jour après la construction :

    ```python
    from qiskit.providers.fake_provider import GenericBackendV2
    from qiskit.transpiler import generate_preset_pass_manager

    target = GenericBackendV2(num_qubits=2, basis_gates=["x", "cx"]).target
    target.update_instruction_properties("x", (0,), InstructionProperties(0.00001))
    target.update_instruction_properties("x", (1,), InstructionProperties(0.00002))
    target.update_instruction_properties("cx", (0, 1), InstructionProperties(0.0001))

    pm = generate_preset_pass_manager(target=target)
    ```

2. Les arguments d'entrée `instruction_durations` et `timing_constraints` ne sont plus pris en charge. Tu dois utiliser
  les arguments `target` ou `backend` à la place. Les alternatives sont la construction d'un [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) personnalisé à partir de
  zéro, la construction d'un backend générique, et, en plus de celles-ci, il est possible de construire un target personnalisé avec
  `instruction_durations` et/ou `timing_constraints` en utilisant la méthode `Target.from_configuration` :

    ```python
    from qiskit.transpiler import Target, generate_preset_pass_manager
    from qiskit.transpiler.timing_constraints import TimingConstraints
    from qiskit.transpiler.instruction_durations import InstructionDurations

    instruction_durations = InstructionDurations([("x", None, 200), ("cx", None, 1000)], dt=1e-7)

    timing_constraints = TimingConstraints(
            granularity=1, min_length=1, pulse_alignment=1, acquire_alignment=1
        )

    target = Target.from_configuration(num_qubits=2,
                                        basis_gates=["x", "cx"],
                                        instruction_durations=instruction_durations,
                                        timing_constraints=timing_constraints)

    pm = generate_preset_pass_manager(target=target)
    ```

3. Fournir `coupling_map` et/ou `basis_gates` conjointement avec un `backend` est déconseillé, et à partir de
  v2.0 un `UserWarning` sera déclenché. Dans ces cas, il y a plusieurs sources de vérité,
  et des informations telles que les erreurs et durées de porte du backend ne peuvent pas être résolues sur la base des
  informations fournies.
  L'alternative suggérée est de définir un target personnalisé qui combine les contraintes choisies soit
  à partir de zéro, en utilisant [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#from_configuration) ou [`GenericBackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.fake_provider.GenericBackendV2).

4. L'argument `basis_gates` n'accepte plus de portes de base personnalisées. L'alternative suggérée
  est l'argument `target`, qui peut être construit à partir de zéro ou via [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#from_configuration)
  comme suit :

    ```python
    from qiskit.circuit.library import XGate
    from qiskit.transpiler.target import Target

    basis_gates = ["my_x", "cx"]
    custom_name_mapping = {"my_x": XGate()}
    target = Target.from_configuration(
        basis_gates=basis_gates, num_qubits=2, custom_name_mapping=custom_name_mapping
    )
    pm = generate_preset_pass_manager(target=target)
    ```

5. La combinaison d'entrée où un `coupling_map` personnalisé est fourni avec soit un `backend` soit des `basis_gates`
  contenant des portes à trois qubits ou plus n'est plus autorisée. La coupling map
  ne fournit pas les détails de connectivité nécessaires pour déterminer l'action de la porte.
  Dans ces cas, le chemin de migration recommandé est la création d'un target personnalisé à partir de zéro.

6. L'argument `inst_map` a été supprimé sans alternative. Il était utilisé pour fournir des informations
  de pulse au transpiler, et le module complet `qiskit.pulse` a été supprimé.

## qiskit.pulse

Le module `qiskit.pulse` a été complètement supprimé de Qiskit, sans remplacement.
Cela inclut les éléments suivants :

- la propriété `calibrations` de [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit), [`DAGCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGCircuit) et [`DAGDependency`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGDependency).
- les méthodes `has_calibration_for` et `add_calibration` de [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit).
- la méthode `has_calibration_for` de [`DAGCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGCircuit).
- les passes du transpiler `PulseGates`, `RXCalibrationBuilder`, `RZXCalibrationBuilder`, `EchoRZXWeylDecomposition`, `ValidatePulseGates`.
- le module `qiskit.scheduler` et tout son contenu.
- les méthodes `instruction_schedule_map`, `drive_channel`, `measure_channel`, `acquire_channel` et `control_channel` de [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2).
- les méthodes `has_calibration`, `get_calibration`, `instruction_schedule_map` et `update_from_instruction_schedule_map` de [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target).
- l'argument `inst_map` de [`generate_preset_pass_manager`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.generate_preset_pass_manager), [`transpile`](https://docs.quantum.ibm.com/api/qiskit/compiler#transpile), et [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#qiskit.transpiler.Target.from_configuration).
- le champ `calibration` de [`InstructionProperties`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties).

Cela s'est produit parce que les QPU IBM Quantum&reg; ont cessé d'offrir un accès au niveau du pulse, et que l'équipe Qiskit n'avait plus les ressources pour maintenir ce paquet peu utilisé.
Tu peux continuer à utiliser le module `qiskit.pulse` avec la série v1.x de Qiskit, qui bénéficie d'un support de correction de bugs jusqu'en septembre 2024 et d'un support de sécurité jusqu'en mars 2025, mais le module est supprimé de la série v2.x.

La plupart des utilisations de `qiskit.pulse` peuvent être remplacées par un accès direct aux portes fractionnaires sur les QPU IBM Quantum Heron (et les générations ultérieures, le cas échéant).
Tu peux en savoir plus dans [le guide de migration pour la conversion de l'usage du pulse vers les portes fractionnaires](/guides/pulse-migration).

Tu ne peux plus charger des fichiers QPY contenant des objets `ScheduleBlock` dans Qiskit v2.0 en raison de la suppression de Qiskit Pulse.
Tu peux continuer à charger ces fichiers dans la série Qiskit v1.x.
