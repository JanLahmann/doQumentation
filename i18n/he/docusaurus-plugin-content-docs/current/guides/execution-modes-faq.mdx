# שאלות נפוצות על מצבי הרצה {#execution-modes-faqs}

<Accordion>
  <AccordionItem title="Does Qiskit Runtime local testing mode support different execution modes?">

מצב הבדיקה המקומי תומך בתחביר של מצבי ההרצה השונים, אך מכיוון שאין תזמון כלשהו בבדיקה מקומית, המצבים עצמם מתעלמים.
  </AccordionItem>

  <span id="parallel"></span>
  <AccordionItem title="How many jobs can run in parallel for a specific backend?">

מספר המשימות הרצות במקביל מבוסס על רמת המקביליות המוגדרת עבור ה-Backend, שהיא חמש עבור רוב ה-Backends כיום.
  </AccordionItem>

  <AccordionItem title="How is usage reported for failed or canceled jobs?">

ראה את הסעיף [משימות שנכשלו ובוטלו](/guides/estimate-job-run-time#failed-job) בדף מצבי ההרצה.
  </AccordionItem>

</Accordion>

## סשנים {#sessions}

<Accordion>
  <AccordionItem title="What happens to my jobs if a session is closed?">

אם אתה משתמש בקלאס `Session` ב-`qiskit-ibm-runtime`:

  - `Session.close()` פירושו שהסשן אינו מקבל עוד משימות חדשות, אך המשימות הקיימות ממשיכות עד השלמתן.
  - `Session.cancel()` מבטל את כל משימות הסשן הממתינות.

אם אתה משתמש ב-REST API ישירות:

  - `PATCH /sessions/{id}` עם `accepting_jobs=False` פירושו שהסשן אינו מקבל עוד משימות חדשות, אך המשימות הקיימות ממשיכות עד השלמתן.
  - `DELETE /sessions/{id}/close` מבטל את כל משימות הסשן הממתינות.
  </AccordionItem>

  <AccordionItem title="If I am using session mode and expect my experiment to take many hours, is there a way to ask for calibrations to happen?">

לא. כיול לפי דרישה אינו זמין.
  </AccordionItem>

  <AccordionItem title="Is there an interactive timeout (interactive TTL) with session mode?">

כן. הדבר מפחית עלויות בלתי רצויות אם משתמש שוכח לסגור את הסשן שלו.
  </AccordionItem>

  <AccordionItem title="Can I change the interactive TTL or the maximum TTL of a session?">

לא ניתן לשנות את ערך ה-TTL האינטראקטיבי. ניתן לשנות את ערך ה-TTL המקסימלי של סשן (ראה [ציון אורך הסשן](/guides/run-jobs-session#specify-length)), אך הוא חייב להיות קטן מהמקסימום המוגדר במערכת. פנה למנהל המערכת שלך ליצור קשר עם תמיכת IBM אם אתה זקוק ל-TTL אינטראקטיבי שונה או ל-TTL מקסימלי שונה של המערכת.

  </AccordionItem>

  <AccordionItem title="How does session usage impact IBM Quantum Network members who are not billed by usage?">

חברי IBM Quantum Network מקבלים קיבולת שמורה ב-QPUs של IBM Quantum&reg;. השימוש מנוכה מקיבולת זו, ומופעים עם קיבולת נמוכה יותר נתונים לזמני תור ארוכים יותר.
  </AccordionItem>

  <AccordionItem title="Do I get the same parallelism in session mode that I get with batch mode?">

כן. אם אתה שולח מספר משימות בו-זמנית בסשן, משימות אלו ירוצו במקביל.
  </AccordionItem>

  <AccordionItem title="Can sessions be interrupted by QPU upgrades or calibrations?">

לא. סשנים פועלים במצב ייעודי, כלומר למשתמש יש גישה מלאה ל-Backend. סשנים אינם נקטעים לעולם על ידי כיולים או שדרוגי תוכנה.
  </AccordionItem>

  <AccordionItem title="Is compilation time counted as usage in session mode?">

כן. במצב סשן, השימוש הוא זמן השעון שה-QPU **מחויב לסשן**. הוא מתחיל כאשר משימת הסשן הראשונה מתחילה ומסתיים כאשר הסשן הופך ללא פעיל, נסגר, או כאשר המשימה האחרונה מסתיימת, לפי המאוחר שבהם. לכן, השימוש ממשיך להצטבר לאחר סיום הסשן אם ה-QPU עדיין מריץ משימה. בנוסף, הזמן שחולף לאחר סיום משימה בזמן שה-QPU ממתין למשימת סשן נוספת (ה-TTL האינטראקטיבי) נחשב כשימוש. זו הסיבה שעליך לוודא שהסשן נסגר ברגע שסיימת לשלוח אליו משימות.
  </AccordionItem>

</Accordion>

## אצווה {#batch}

<Accordion>
  <AccordionItem title="How many jobs run in parallel in batch mode?">

מספר המשימות הרצות במקביל מבוסס על רמת המקביליות המוגדרת עבור ה-Backend, שהיא חמש עבור רוב ה-Backends. עם זאת, מספר המשימות הרצות בו-זמנית באצווה פעילה עשוי להיות נמוך יותר, מכיוון שייתכן שיש משימות אחרות שכבר רצות כאשר האצווה הופכת לפעילה.
  </AccordionItem>

  <AccordionItem title="How is running _N_ PUBs in job mode different from running _N_ single-PUB jobs in batch mode?">

ההבדל העיקרי הוא בפשרה בין זמן ועלות:

מצב אצווה:

- זמן ההרצה הכולל קצר יותר מכיוון שהעיבוד הקלאסי עשוי לרוץ במקביל.
- קיים תקורה קלה עבור הרצת כל משימה, ולכן בסופו של דבר אתה משלם מעט יותר עבור משימות באצווה לעומת כאשר כל המעגלים מורצים במשימה בודדת.
- מכיוון שמצב אצווה אינו מעניק לך גישה בלעדית ל-Backend, משימות בתוך אצווה עשויות לרוץ עם משימות של משתמשים אחרים או משימות כיול.
- אם חלק מהמשימות נכשלות, אתה עדיין מקבל תוצאות מהמשימות שהושלמו.
- ניתן לנקוט פעולה באמצע עומס עבודה באצווה על סמך תוצאות המשימות שהושלמו. לדוגמה, ניתן לבטל את שאר המשימות אם התוצאות הראשוניות נראות שגויות.

מצב משימה:

- זמן ההרצה הכולל צפוי להיות גבוה יותר מכיוון שאין מקביליות.
- אינך משלם עבור התקורה הנוספת לכל משימה הקשורה לעומסי עבודה באצווה.
- כל המעגלים שלך ירוצו יחד.
- אם משימה בודדת זו נכשלת, אינך מקבל תוצאות חלקיות.
- המשימה שלך עשויה לפגוע במגבלה אם היא מכילה יותר מדי מעגלים או אם המעגלים גדולים מדי.

באופן כללי, אם כל אחת מהמשימות שלך צורכת פחות מדקה של זמן QPU, שקול לאחד אותן למשימה גדולה יותר (זה חל על כל מצבי ההרצה).
  </AccordionItem>

  <AccordionItem title="How many jobs I can submit in a batch?">

אמנם אין מגבלות על מספר המשימות שניתן לשלוח באצווה, אך קיים זמן מקסימלי הקשור לאצווה. כלומר, כאשר זמן השעון של האצווה (שמתחיל כאשר משימת האצווה הראשונה מתחילה לרוץ) חורג מהזמן המקסימלי המוגדר במערכת, האצווה לא תקבל משימות חדשות, וכל משימה שנמצאת בתור אך אינה רצה תבוטל. בנוסף, קיימות מגבלות על כמות השימוש שהמשימות שלך יכולות לצרוך בהתאם לתוכנית שלך. כדי לקבוע את הזמן המקסימלי הקשור לאצווה, השתמש ב[מתודה `batch.details()`](/guides/run-jobs-batch#batch-details) וחפש את הערך `max_time`.
  </AccordionItem>

  <AccordionItem title="When would my batch mode jobs run in parallel with other users' jobs?">

רמת המקביליות המוגדרת עבור Backend נקראת גם "נתיבי הרצה". אם ישנם נתיב הרצה אחד או יותר פנויים, והמשימות באצווה שלך הן הבאות בתור, המתזמן מתחיל מספיק משימות כדי למלא את הנתיבים. באופן דומה, אם האצווה שלך אינה מכילה מספיק משימות למילוי הנתיבים, המתזמן מתחיל משימות של משתמשים אחרים.

דוגמה: ל-Backend שבחרת יש חמישה נתיבי הרצה, ושניים מהם כרגע תפוסים על ידי משימות של משתמשים אחרים. האצווה שלך עם שש משימות היא הבאה בתור.

מכיוון שיש שלושה נתיבים פנויים, המתזמן מתחיל שלוש מתוך שש משימות האצווה שלך. הוא ממשיך להתחיל משימות באצווה שלך ככל שמשימות מסתיימות ונתיבי הרצה מתפנים. אם נתיב מתפנה ואין עוד משימות באצווה שלך, המתזמן מתחיל את המשימה הבאה בתור.
  </AccordionItem>

  <AccordionItem title="Do all of my batch jobs need to wait in the queue?">

מכיוון שה-QPUs הם משאבים מוגבלים ומשותפים, כל המשימות צריכות לחכות בתור. עם זאת, כאשר המשימה הראשונה באצווה שלך מתחילה לרוץ, כל שאר המשימות באצווה זו למעשה קופצות לראש התור ומקבלות עדיפות על ידי המתזמן.
  </AccordionItem>

  <AccordionItem title="Does a batch end automatically when the last associated job ends?">

כן. עם זאת, קיימת תקורה קלה הקשורה לאיתור אוטומטי זה, לכן עליך תמיד לסגור את האצווה והסשן שלך.
  </AccordionItem>

  <AccordionItem title="Can batches be interrupted by calibrations or software upgrades">

כן. עומסי עבודה באצווה עלולים להיות מופרעים על ידי כיולים או שדרוגי תוכנה.
  </AccordionItem>

  <AccordionItem title="Is compilation time counted as usage in batch mode?">

לא. במצב אצווה, רק הזמן שמושקע בחומרה הקוונטית נחשב כשימוש.
  </AccordionItem>

</Accordion>
