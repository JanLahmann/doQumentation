# Qiskit v2.0 Migrationsleitfaden {#qiskit-v20-migration-guide}

Qiskit v2.0 ist der nächste Schritt im neu stabilisierten Lebenszyklus des Qiskit SDK.
Diese Hauptversion bricht zwar die Kompatibilität mit früheren Versionen von Qiskit,
beinhaltet jedoch – im Gegensatz zum v1.0-Release – keine Änderungen an der Paketstruktur.

Dieser Leitfaden beschreibt Migrationspfade für die wichtigsten Feature-Änderungen in Qiskit v2.0, geordnet nach Modul.
Nutze das Inhaltsverzeichnis auf der rechten Seite, um zum gewünschten Modul zu navigieren.

Mehr über die Vorteile von Qiskit v2.0 erfährst du im [IBM&reg;-Blog](https://www.ibm.com/quantum/blog/qiskit-2-0-release-summary).

## qiskit.circuit

<span id="c_if"></span>
### Entfernung der veralteten `c_if`-Bedingungen {#removal-of-legacy-c_if-conditions}

Klassische Bedingungen werden in Qiskit bereits seit vor v1.0 bevorzugt durch die Instruktion [`IfElseOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.IfElseOp) dargestellt.
Diese Operationen werden typischerweise einem [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit) über die Methode [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) als Kontextmanager hinzugefügt.

Das bedeutet auch, dass das [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction)-Objekt kein `condition`-Attribut mehr besitzt und du keine `c_if`-Aufrufe an die Ausgabe einer Gate-erzeugenden Methode eines [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit) mehr verketten kannst.

Die einzigen Instruktionen, die noch eine Bedingung haben können, sind [`IfElseOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.IfElseOp) und [`WhileLoopOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.WhileLoopOp).
Die korrekteste Methode, eine Bedingung zu prüfen, ist eine `isinstance`-Prüfung, um sicherzustellen, dass du die richtige Klasse hast.
Wenn du das veraltete Attribut `Instruction.condition` während der Umstellung deiner Bibliothek auf Qiskit v2.0 und später unterstützen möchtest, kannst du `getattr(operation, "condition", None)` verwenden, was dazu führt, dass sich `Instruction`-Instanzen typischerweise so verhalten wie in Qiskit v1.x und früher.

Die ursprüngliche, veraltete Methode zur Unterstützung einfacher Register-Gleichheits-Bedingungen in Qiskit war die Verwendung der `c_if`-Methode auf einzelnen Instruktionen oder Instruktionsmengen.
Diese war ausdrucksarm im Vergleich zur vollständigen Control-Flow-Form und wurde auf Hardware nie gut unterstützt.
Diese veraltete Methode wurde in Qiskit v2.0 entfernt; du solltest die Methode [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) mit einer geeigneten Bedingung verwenden.
Du kannst weiterhin die 2-Tupel-Form `(register, integer)` verwenden, um eine Register-Gleichheit darzustellen, oder du nutzt das ausdrucksstärkere [klassische Berechnungssubsystem](https://docs.quantum.ibm.com/api/qiskit/circuit_classical) für komplexere Bedingungen.

Du hast zum Beispiel vielleicht Folgendes geschrieben:

```python
from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
qc.x(0).c_if(cr, 3)
```

Die direkteste Übersetzung davon ist die Verwendung der 2-Tupel-Form mit dem Kontextmanager [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test):

```python
from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
with qc.if_test((cr, 3)):
    qc.x(0)
```

Beachte, dass du in dieser Form auch mehrere Operationen innerhalb des bedingten Blocks ausführen kannst (füge einfach weitere Zeilen in den eingerückten Block ein), ohne die Bedingung erneut auszuwerten.
Du kannst außerdem eine ``else``-Verzweigung angeben und komplexere bedingte Logik verwenden:

```python
from qiskit.circuit import QuantumCircuit
from qiskit.circuit.classical import expr

qc = QuantumCircuit(2, 2)
qc.h(0)
qc.h(1)
qc.measure(qc.qubits, qc.clbits)

# Apply an X to qubit 0 if both measurements were the same:
with qc.if_test(expr.equal(qc.clbits[0], qc.clbits[1])) as else_:
    qc.x(0)
# ... or apply a Y to both qubits otherwise:
with else_:
    qc.y(0)
    qc.y(1)
```

<span id="instruction_durations"></span>
### Entfernung von `Instruction.duration` und `Instruction.unit` {#removal-of-instructionduration-and-instructionunit}

Circuit-[`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction)-Objekte hatten bisher Attribute namens `duration` und `unit`.
Diese existieren in Qiskit v2.0 nicht mehr.
Der Grund dafür ist, dass dieses Feld in der Regel nur Informationen doppelt gespeichert hat, die eine feste Eigenschaft der Hardware sind und bereits explizit im [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) abgelegt werden.
Die separate Speicherung dieser Information für jede einzelne Instruktion erhöhte den Speicherbedarf von Qiskit erheblich und verlangsamte auch Circuit-Operationen, die diese Information gar nicht direkt benötigten.

Die [`Delay`](https://docs.quantum.ibm.com/api/qiskit/circuit#delay)-Instruktion enthält diese Felder weiterhin, ebenso wie die in v2.0 neue Instruktion [`BoxOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#box), da diese Instruktionen eine _variable_ Verzögerung aufweisen.

Du kannst die hardware-seitig festgelegte Dauer einer bestimmten hardware-unterstützten [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) abfragen, indem du den [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) verwendest.
Dazu indizierst du den [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) zuerst mit dem Namen der Operation und dann mit den Indizes der Hardware-Qubits als Tupel.
Dies gibt entweder `None` zurück (wenn die Instruktion unterstützt wird, aber keine spezifischen Eigenschaften zugewiesen wurden) oder eine [`InstructionProperties`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties)-Instanz, aus der das Feld [`duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties#qiskit.transpiler.InstructionProperties.duration) die Dauer in Sekunden liefern kann (oder `None`, falls nicht angegeben).

```python
target = "<some Target from a hardware vendor>"

# Query the duration of a CX instruction on hardware qubits (2, 4),
# returning `None` if the duration is not specified.
properties = target["cx"][2, 4]
duration = None if properties is None else properties.duration

# You can also do this in one step, using `getattr`:
duration = getattr(target["cx"][2, 4], "duration", None)
```

Im Allgemeinen wird die „Dauer" eines Circuits immer schwieriger zu definieren sein, je häufiger klassischer Control-Flow in Circuit-Ausführungen eingesetzt wird.
Obwohl Qiskit v2.x das Attribut [`QuantumCircuit.duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.duration) noch enthält, wird empfohlen, davon wegzugehen, da es voraussichtlich in einer zukünftigen Version entfernt wird; die Annahmen, unter denen es gültig ist, werden immer häufiger verletzt.
Verwende stattdessen die neue Methode [`QuantumCircuit.estimate_duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.estimate_duration), die sicherer ist und die erforderlichen Annahmen klarer kommuniziert.

## qiskit.transpiler

<span id="generate_preset_pass_manager"></span>
### Änderungen an der `generate_preset_pass_manager`-Schnittstelle {#changes-to-generate_preset_pass_manager-interface}

Qiskit v2.0.0 enthält eine Reihe von Aktualisierungen der Preset-Pass-Manager-Schnittstelle (d. h. der Funktionen
[`generate_preset_pass_manager`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.generate_preset_pass_manager) und
[`transpile`](https://docs.quantum.ibm.com/api/qiskit/compiler#transpile)), die sich auf die
Behandlung loser Constraint-Eingaben wie `coupling_map` oder `basis_gates` auswirken. Die Anzahl der
Transpilations-Anwendungsfälle, bei denen lose Constraints akzeptiert werden, wurde zugunsten der
Eingaben `target` und `backend` reduziert (wobei `backend` ein [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) ist, das
eine [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target)-Instanz enthält),
die eine ausdrucksstärkere Kommunikation dieser Constraints ermöglichen.

Die folgenden Beispiele zeigen die wichtigsten Änderungen und empfohlene Alternativen für die Migration. Beachte, dass alle
nachfolgenden Migrationstipps sowohl für `transpile` als auch für `generate_preset_pass_manager` gelten:

1. Das Eingabeargument `backend_properties` wird nicht mehr unterstützt, und die Klasse `BackendProperties` wurde
   entfernt. Verwende stattdessen die Argumente `target` oder `backend`, um Gate-Dauern und Fehlerraten zu übermitteln.
   Es gibt mehrere Möglichkeiten, ein benutzerdefiniertes [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) zu erstellen:

  a. Erstelle ein Target von Grund auf und füge die relevanten Fehler und Dauern zu den entsprechenden Gates hinzu:

    ```python
    from qiskit.transpiler import Target, InstructionProperties, generate_preset_pass_manager
    from qiskit.circuit.library import XGate, CXGate

    target = Target(num_qubits=2)
    target.add_instruction(CXGate(), {(0, 1): InstructionProperties(error=.0001, duration=5e-7)})
    target.add_instruction(
        XGate(),
        {
            (0,): InstructionProperties(error=.00001, duration=5e-8),
            (1,): InstructionProperties(error=.00002, duration=6e-8)
        }
    )

    pm = generate_preset_pass_manager(target=target)
    ```

  b. Verwende [`GenericBackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.fake_provider.GenericBackendV2), um schnell ein
     generisches Target mit vorausgefüllten Fehlern und Dauern zu erstellen. Diese können auch nach der Erstellung aktualisiert werden:

    ```python
    from qiskit.providers.fake_provider import GenericBackendV2
    from qiskit.transpiler import generate_preset_pass_manager

    target = GenericBackendV2(num_qubits=2, basis_gates=["x", "cx"]).target
    target.update_instruction_properties("x", (0,), InstructionProperties(0.00001))
    target.update_instruction_properties("x", (1,), InstructionProperties(0.00002))
    target.update_instruction_properties("cx", (0, 1), InstructionProperties(0.0001))

    pm = generate_preset_pass_manager(target=target)
    ```

2. Die Eingabeargumente `instruction_durations` und `timing_constraints` werden nicht mehr unterstützt. Verwende
  stattdessen die Argumente `target` oder `backend`. Die Alternativen sind: ein benutzerdefiniertes [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) von
  Grund auf erstellen, ein generisches Backend erstellen, und – zusätzlich dazu – ist es möglich, ein benutzerdefiniertes Target mit
  `instruction_durations` und/oder `timing_constraints` über die Methode `Target.from_configuration` zu erstellen:

    ```python
    from qiskit.transpiler import Target, generate_preset_pass_manager
    from qiskit.transpiler.timing_constraints import TimingConstraints
    from qiskit.transpiler.instruction_durations import InstructionDurations

    instruction_durations = InstructionDurations([("x", None, 200), ("cx", None, 1000)], dt=1e-7)

    timing_constraints = TimingConstraints(
            granularity=1, min_length=1, pulse_alignment=1, acquire_alignment=1
        )

    target = Target.from_configuration(num_qubits=2,
                                        basis_gates=["x", "cx"],
                                        instruction_durations=instruction_durations,
                                        timing_constraints=timing_constraints)

    pm = generate_preset_pass_manager(target=target)
    ```

3. Die Kombination aus `coupling_map` und/oder `basis_gates` zusammen mit einem `backend` wird nicht empfohlen, und
  ab v2.0 wird eine `UserWarning` ausgegeben. In diesen Fällen gibt es mehrere Quellen der Wahrheit,
  und Informationen wie Gate-Fehler und -Dauern aus dem Backend können anhand der gegebenen Informationen nicht aufgelöst werden.
  Die empfohlene Alternative ist die Definition eines benutzerdefinierten Targets, das die gewählten Constraints entweder
  von Grund auf, über [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#from_configuration) oder [`GenericBackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.fake_provider.GenericBackendV2) kombiniert.

4. Das Argument `basis_gates` akzeptiert keine benutzerdefinierten Basis-Gates mehr. Die empfohlene Alternative
  ist das Argument `target`, das von Grund auf oder über [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#from_configuration) erstellt werden kann,
  zum Beispiel:

    ```python
    from qiskit.circuit.library import XGate
    from qiskit.transpiler.target import Target

    basis_gates = ["my_x", "cx"]
    custom_name_mapping = {"my_x": XGate()}
    target = Target.from_configuration(
        basis_gates=basis_gates, num_qubits=2, custom_name_mapping=custom_name_mapping
    )
    pm = generate_preset_pass_manager(target=target)
    ```

5. Die Eingabe-Kombination, bei der eine benutzerdefinierte `coupling_map` zusammen mit einem `backend` oder `basis_gates`
  angegeben wird, die Gates mit drei oder mehr Qubits enthalten, ist nicht mehr erlaubt. Die Coupling Map
  liefert nicht die notwendigen Konnektivitätsdetails, um die Wirkung des Gates bestimmen zu können.
  In diesen Fällen ist der empfohlene Migrationspfad die Erstellung eines benutzerdefinierten Targets von Grund auf.

6. Das Argument `inst_map` wurde ohne Alternative entfernt. Es diente dazu, dem Transpiler Pulse-Informationen
  bereitzustellen, und das vollständige Modul `qiskit.pulse` wurde entfernt.

## qiskit.pulse

Das Modul `qiskit.pulse` wurde vollständig aus Qiskit entfernt, ohne Ersatz.
Dazu gehören die folgenden verwandten Elemente:

- die Eigenschaft `calibrations` von [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit), [`DAGCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGCircuit) und [`DAGDependency`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGDependency).
- die Methoden `has_calibration_for` und `add_calibration` von [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit).
- die Methode `has_calibration_for` von [`DAGCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGCircuit).
- die Transpiler-Passes `PulseGates`, `RXCalibrationBuilder`, `RZXCalibrationBuilder`, `EchoRZXWeylDecomposition`, `ValidatePulseGates`.
- das Modul `qiskit.scheduler` und alle seine Inhalte.
- die Methoden `instruction_schedule_map`, `drive_channel`, `measure_channel`, `acquire_channel` und `control_channel` von [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2).
- die Methoden `has_calibration`, `get_calibration`, `instruction_schedule_map` und `update_from_instruction_schedule_map` von [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target).
- das Argument `inst_map` von [`generate_preset_pass_manager`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.generate_preset_pass_manager), [`transpile`](https://docs.quantum.ibm.com/api/qiskit/compiler#transpile) und [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#qiskit.transpiler.Target.from_configuration).
- das Feld `calibration` von [`InstructionProperties`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties).

Dies geschah, weil IBM Quantum&reg; QPUs keinen Pulse-Level-Zugriff mehr anbieten und dem Qiskit-Team keine Ressourcen mehr zur Verfügung standen, um das wenig genutzte Paket zu pflegen.
Du kannst das Modul `qiskit.pulse` weiterhin verwenden, solange du die v1.x-Serie von Qiskit nutzt, die Bugfix-Support bis September 2024 und Sicherheitssupport bis März 2025 hat, aber das Modul wurde aus der v2.x-Serie entfernt.

Die meisten Verwendungen von `qiskit.pulse` können durch direkten Zugriff auf fraktionale Gates auf IBM Quantum Heron QPUs (und späteren Generationen, falls zutreffend) ersetzt werden.
Mehr dazu erfährst du im [Migrationsleitfaden für die Umstellung von Pulse-Nutzung auf fraktionale Gates](/guides/pulse-migration).

Du kannst keine QPY-Dateien mehr laden, die `ScheduleBlock`-Objekte enthalten, da Qiskit Pulse in Qiskit v2.0 entfernt wurde.
Du kannst diese Dateien weiterhin in der Qiskit v1.x-Serie laden.
