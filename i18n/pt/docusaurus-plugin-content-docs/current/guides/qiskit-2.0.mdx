# Guia de migração para o Qiskit v2.0 {#qiskit-v20-migration-guide}

O Qiskit v2.0 é o próximo passo dentro do ciclo de vida recentemente estabilizado do Qiskit SDK.
Esta versão principal quebra a compatibilidade com versões anteriores do Qiskit, mas,
ao contrário do lançamento v1.0, não envolve nenhuma mudança na estrutura de empacotamento.

Este guia descreve os caminhos de migração para as mudanças de funcionalidades mais importantes no Qiskit v2.0, organizados por módulo.
Use o índice no lado direito para navegar até o módulo que você deseja.

Leia mais sobre os benefícios do Qiskit v2.0 no [blog da IBM&reg;](https://www.ibm.com/quantum/blog/qiskit-2-0-release-summary).

<span id="qiskit.circuit"></span>
## qiskit.circuit

<span id="c_if"></span>
### Remoção das condições legadas `c_if` {#removal-of-legacy-c_if-conditions}

As condições clássicas no Qiskit foram preferencialmente representadas pela instrução [`IfElseOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.IfElseOp) desde antes do Qiskit v1.0.
Essas operações são tipicamente adicionadas a um [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit) usando o método [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) como gerenciador de contexto.

Isso também significa que o objeto [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) não possui mais um atributo `condition`, e que você não pode encadear chamadas `c_if` na saída de um método de criação de Gate de um [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit).

As únicas instruções que podem ter uma condição agora são [`IfElseOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.IfElseOp) e [`WhileLoopOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.WhileLoopOp).
A maneira mais correta de verificar uma condição é usar uma verificação `isinstance` para validar que você tem a classe correta.
Se você quiser suportar o atributo legado `Instruction.condition` durante a transição da sua biblioteca para o Qiskit v2.0 e versões posteriores, você pode usar `getattr(operation, "condition", None)`, o que geralmente fará com que instâncias de `Instruction` se comportem como no Qiskit v1.x e anteriores.

O método original e legado de suporte a condições simples de igualdade de registradores no Qiskit era usar o método `c_if` em instruções individuais ou conjuntos de instruções.
Esse método era estritamente menos funcional do que a forma completa de controle de fluxo e nunca foi bem suportado em hardware.
Esse método legado foi removido no Qiskit v2.0; você deve usar o método [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) com uma condição adequada.
Você pode continuar a usar a forma de 2-tupla `(register, integer)` para representar uma igualdade de registrador, ou pode usar o mais expressivo [subsistema de computação clássica](https://docs.quantum.ibm.com/api/qiskit/circuit_classical) para representar condições mais complexas.

Por exemplo, você pode ter escrito:

```python
from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
qc.x(0).c_if(cr, 3)
```

A tradução mais direta disso é usar a forma de 2-tupla com o gerenciador de contexto [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test):

```python
from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
with qc.if_test((cr, 3)):
    qc.x(0)
```

Observe que nessa forma, você também pode fazer com que o Gate condicional acesse múltiplas operações (basta adicionar mais no bloco indentado) sem reavaliar as condições.
Você também pode fornecer uma instrução ``else`` e usar lógica condicional mais complexa:

```python
from qiskit.circuit import QuantumCircuit
from qiskit.circuit.classical import expr

qc = QuantumCircuit(2, 2)
qc.h(0)
qc.h(1)
qc.measure(qc.qubits, qc.clbits)

# Apply an X to qubit 0 if both measurements were the same:
with qc.if_test(expr.equal(qc.clbits[0], qc.clbits[1])) as else_:
    qc.x(0)
# ... or apply a Y to both qubits otherwise:
with else_:
    qc.y(0)
    qc.y(1)
```

<span id="instruction_durations"></span>
### Remoção de `Instruction.duration` e `Instruction.unit` {#removal-of-instructionduration-and-instructionunit}

Objetos [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) de Circuit tinham anteriormente atributos chamados `duration` e `unit`.
Eles não existem mais no Qiskit v2.0.
Isso ocorre porque o campo geralmente apenas duplicava informações que eram uma propriedade fixa do hardware e eram armazenadas explicitamente no [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target).
Armazenar as informações separadamente para cada instrução individual aumentava significativamente as demandas de memória do Qiskit e tornava mais lentas até mesmo as operações de Circuit que não precisavam tocar diretamente nas informações.

A instrução [`Delay`](https://docs.quantum.ibm.com/api/qiskit/circuit#delay) ainda contém esses campos, assim como o novo [`BoxOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#box) introduzido no v2.0, porque essas instruções têm um atraso _variável_.

Você pode recuperar a duração especificada pelo hardware de uma [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) suportada pelo hardware específico consultando o [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target).
Para fazer isso, indexe o [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) primeiro com o nome da operação e, em seguida, com os índices dos Qubits de hardware como uma tupla.
Isso retornará `None` (se a instrução for suportada, mas nenhuma propriedade específica foi atribuída a ela) ou uma instância de [`InstructionProperties`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties), da qual o campo [`duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties#qiskit.transpiler.InstructionProperties.duration) pode fornecer a duração em segundos (ou `None`, se não fornecida).

```python
target = "<some Target from a hardware vendor>"

# Query the duration of a CX instruction on hardware qubits (2, 4),
# returning `None` if the duration is not specified.
properties = target["cx"][2, 4]
duration = None if properties is None else properties.duration

# You can also do this in one step, using `getattr`:
duration = getattr(target["cx"][2, 4], "duration", None)
```

Em geral, à medida que o controle de fluxo clássico se torna mais comum nas execuções de Circuit, a "duração" de um Circuit ficará cada vez menos bem definida.
Embora o Qiskit v2.x ainda inclua o atributo [`QuantumCircuit.duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.duration), recomenda-se se afastar dele, pois provavelmente será removido no futuro; as suposições sob as quais ele é válido serão violadas com cada vez mais frequência.
Em vez disso, você pode usar o novo método [`QuantumCircuit.estimate_duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.estimate_duration), que é mais seguro e mais claro sobre as suposições necessárias.

## qiskit.transpiler

<span id="generate_preset_pass_manager"></span>
### Mudanças na interface de `generate_preset_pass_manager` {#changes-to-generate_preset_pass_manager-interface}

O Qiskit v2.0.0 inclui uma série de atualizações na interface do gerenciador de passes predefinido (ou seja, as funções
[`generate_preset_pass_manager`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.generate_preset_pass_manager) e
[`transpile`](https://docs.quantum.ibm.com/api/qiskit/compiler#transpile)) que afetam o
tratamento de entradas de restrições soltas, como `coupling_map` ou `basis_gates`. O número de
casos de uso de transpilação onde restrições soltas são aceitas foi reduzido em favor das
entradas `target` e `backend` (onde `backend` é um [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) que
contém uma instância de [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target)),
que permitem uma comunicação mais expressiva dessas restrições.

Os exemplos a seguir mostram as principais mudanças e alternativas sugeridas para migração. Observe que todas
as dicas de migração abaixo se aplicam tanto a `transpile` quanto a `generate_preset_pass_manager`:

1. O argumento de entrada `backend_properties` não é mais suportado e a classe `BackendProperties` foi
   removida. Você deve usar os argumentos `target` ou `backend` para comunicar
   durações e erros de Gate. Existem múltiplos caminhos para construir um [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) personalizado:

  a. Construa um target do zero e adicione os erros e durações relevantes aos Gates correspondentes:

    ```python
    from qiskit.transpiler import Target, InstructionProperties, generate_preset_pass_manager
    from qiskit.circuit.library import XGate, CXGate

    target = Target(num_qubits=2)
    target.add_instruction(CXGate(), {(0, 1): InstructionProperties(error=.0001, duration=5e-7)})
    target.add_instruction(
        XGate(),
        {
            (0,): InstructionProperties(error=.00001, duration=5e-8),
            (1,): InstructionProperties(error=.00002, duration=6e-8)
        }
    )

    pm = generate_preset_pass_manager(target=target)
    ```

  b. Use [`GenericBackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.fake_provider.GenericBackendV2) para construir rapidamente um
     target genérico com erros e durações pré-preenchidos. Estes também podem ser atualizados após a construção:

    ```python
    from qiskit.providers.fake_provider import GenericBackendV2
    from qiskit.transpiler import generate_preset_pass_manager

    target = GenericBackendV2(num_qubits=2, basis_gates=["x", "cx"]).target
    target.update_instruction_properties("x", (0,), InstructionProperties(0.00001))
    target.update_instruction_properties("x", (1,), InstructionProperties(0.00002))
    target.update_instruction_properties("cx", (0, 1), InstructionProperties(0.0001))

    pm = generate_preset_pass_manager(target=target)
    ```

2. Os argumentos de entrada `instruction_durations` e `timing_constraints` não são mais suportados. Você deve usar os
  argumentos `target` ou `backend`. As alternativas são construir um [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) personalizado do
  zero, construir um backend genérico e, além disso, é possível construir um target personalizado com
  `instruction_durations` e/ou `timing_constraints` usando o método `Target.from_configuration`:

    ```python
    from qiskit.transpiler import Target, generate_preset_pass_manager
    from qiskit.transpiler.timing_constraints import TimingConstraints
    from qiskit.transpiler.instruction_durations import InstructionDurations

    instruction_durations = InstructionDurations([("x", None, 200), ("cx", None, 1000)], dt=1e-7)

    timing_constraints = TimingConstraints(
            granularity=1, min_length=1, pulse_alignment=1, acquire_alignment=1
        )

    target = Target.from_configuration(num_qubits=2,
                                        basis_gates=["x", "cx"],
                                        instruction_durations=instruction_durations,
                                        timing_constraints=timing_constraints)

    pm = generate_preset_pass_manager(target=target)
    ```

3. Fornecer `coupling_map` e/ou `basis_gates` junto com um `backend` é desencorajado e, a partir do
  v2.0, um `UserWarning` será emitido. Nesses casos, há múltiplas fontes de verdade,
  e informações como erros e durações de Gate do backend não podem ser resolvidas com base nas
  informações fornecidas.
  A alternativa sugerida é definir um target personalizado que combine as restrições escolhidas, seja
  do zero, usando [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#from_configuration) ou [`GenericBackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.fake_provider.GenericBackendV2).

4. O argumento `basis_gates` não aceita mais Gates de base personalizados. A alternativa sugerida
  é o argumento `target`, que pode ser construído do zero ou através de [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#from_configuration)
  como:

    ```python
    from qiskit.circuit.library import XGate
    from qiskit.transpiler.target import Target

    basis_gates = ["my_x", "cx"]
    custom_name_mapping = {"my_x": XGate()}
    target = Target.from_configuration(
        basis_gates=basis_gates, num_qubits=2, custom_name_mapping=custom_name_mapping
    )
    pm = generate_preset_pass_manager(target=target)
    ```

5. A combinação de entradas onde um `coupling_map` personalizado é fornecido com um `backend` ou `basis_gates`
  que contêm Gates com três ou mais Qubits não é mais permitida. O mapa de conectividade
  não fornece os detalhes de conectividade necessários para determinar a ação do Gate.
  Nesses casos, o caminho de migração recomendado é a criação de um target personalizado do zero.

6. O argumento `inst_map` foi removido sem alternativa. Era usado para fornecer informações de pulso
  ao Transpiler, e o módulo completo `qiskit.pulse` foi removido.

## qiskit.pulse

O módulo `qiskit.pulse` foi completamente removido do Qiskit, sem substituição.
Isso inclui os itens relacionados:

- a propriedade `calibrations` de [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit), [`DAGCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGCircuit) e [`DAGDependency`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGDependency).
- os métodos `has_calibration_for` e `add_calibration` de [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit).
- o método `has_calibration_for` de [`DAGCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGCircuit).
- os passes do Transpiler `PulseGates`, `RXCalibrationBuilder`, `RZXCalibrationBuilder`, `EchoRZXWeylDecomposition`, `ValidatePulseGates`.
- o módulo `qiskit.scheduler` e todo o seu conteúdo.
- os métodos `instruction_schedule_map`, `drive_channel`, `measure_channel`, `acquire_channel` e `control_channel` de [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2).
- os métodos `has_calibration`, `get_calibration`, `instruction_schedule_map` e `update_from_instruction_schedule_map` de [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target).
- o argumento `inst_map` de [`generate_preset_pass_manager`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.generate_preset_pass_manager), [`transpile`](https://docs.quantum.ibm.com/api/qiskit/compiler#transpile) e [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#qiskit.transpiler.Target.from_configuration).
- o campo `calibration` de [`InstructionProperties`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties).

Isso aconteceu porque os QPUs do IBM Quantum&reg; deixaram de oferecer acesso em nível de pulso, e a equipe do Qiskit não tinha mais recursos para manter o pacote pouco utilizado.
Você pode continuar a usar o módulo `qiskit.pulse` enquanto usa a série v1.x do Qiskit, que tem suporte a correções de bugs até setembro de 2024 e suporte de segurança até março de 2025, mas o módulo foi removido da série v2.x.

A maioria dos usos de `qiskit.pulse` pode ser substituída com acesso direto a Gates fracionários nos QPUs IBM Quantum Heron (e gerações posteriores, se aplicável).
Você pode ler mais em [o guia de migração para uso de pulso convertido para Gates fracionários](/guides/pulse-migration).

Você não pode mais carregar arquivos QPY que contêm objetos `ScheduleBlock` no Qiskit v2.0 devido à remoção do Qiskit Pulse.
Você pode continuar a carregar esses arquivos na série Qiskit v1.x.
