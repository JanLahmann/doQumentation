# Посібник з міграції на Qiskit v2.0 {#qiskit-v20-migration-guide}

Qiskit v2.0 — це наступний крок у межах нового стабілізованого життєвого циклу Qiskit SDK.
Цей мажорний реліз порушує сумісність з попередніми версіями Qiskit, але,
на відміну від релізу v1.0, не передбачає жодних змін у структурі пакетів.

Цей посібник описує шляхи міграції для найважливіших змін функціональності у Qiskit v2.0, організованих за модулями.
Використовуй зміст праворуч для переходу до потрібного модуля.

Читай більше про переваги Qiskit v2.0 у [блозі IBM&reg;](https://www.ibm.com/quantum/blog/qiskit-2-0-release-summary).

<span id="qiskit.circuit"></span>
## qiskit.circuit

<span id="c_if"></span>
### Видалення застарілих умов `c_if` {#c_if}

Класичні умови в Qiskit переважно представляються інструкцією [`IfElseOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.IfElseOp) ще з часів до Qiskit v1.0.
Ці операції зазвичай додаються до [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit) за допомогою методу [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) як менеджера контексту.

Це також означає, що [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) більше не має атрибута `condition`, і ти не можеш ланцюжком викликати `c_if` на результаті методу [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit), що створює вентиль.

Єдині інструкції, які тепер можуть мати умову, — це [`IfElseOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.IfElseOp) та [`WhileLoopOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.WhileLoopOp).
Найкоректніший спосіб перевірити умову — використати перевірку `isinstance`, щоб переконатися, що маєш правильний клас.
Якщо хочеш підтримувати застарілий атрибут `Instruction.condition` під час переходу своєї бібліотеки на Qiskit v2.0 і пізніше, можна використати `getattr(operation, "condition", None)`, що зазвичай змусить екземпляри `Instruction` поводитись так само, як у Qiskit v1.x і раніше.

Оригінальний, застарілий метод підтримки простих умов рівності регістрів у Qiskit полягав у використанні методу `c_if` на окремих інструкціях або наборах інструкцій.
Це було суворо менш функціональним, ніж повна форма управляючого потоку, і ніколи не мало належної підтримки на залізі.
Цей застарілий метод видалено у Qiskit v2.0; слід використовувати метод [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) з відповідною умовою.
Можна продовжувати використовувати 2-tuple форму `(register, integer)` для представлення рівності регістрів, або використовувати більш виразну [підсистему класичних обчислень](https://docs.quantum.ibm.com/api/qiskit/circuit_classical) для більш складних умов.

Наприклад, раніше ти міг писати:

```python
from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
qc.x(0).c_if(cr, 3)
```

Найпряміший переклад цього — використати 2-tuple форму з менеджером контексту [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test):

```python
from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
with qc.if_test((cr, 3)):
    qc.x(0)
```

Зверни увагу, що в цій формі умовний вентиль може також мати доступ до кількох операцій (просто додай їх до блоку з відступом) без повторного обчислення умов.
Можна також надати оператор ``else`` і використовувати складнішу умовну логіку:

```python
from qiskit.circuit import QuantumCircuit
from qiskit.circuit.classical import expr

qc = QuantumCircuit(2, 2)
qc.h(0)
qc.h(1)
qc.measure(qc.qubits, qc.clbits)

# Apply an X to qubit 0 if both measurements were the same:
with qc.if_test(expr.equal(qc.clbits[0], qc.clbits[1])) as else_:
    qc.x(0)
# ... or apply a Y to both qubits otherwise:
with else_:
    qc.y(0)
    qc.y(1)
```

<span id="instruction_durations"></span>
### Видалення `Instruction.duration` та `Instruction.unit` {#instruction_durations}

Об'єкти [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) схеми раніше мали атрибути `duration` та `unit`.
Вони більше не існують у Qiskit v2.0.
Це пов'язано з тим, що це поле загалом лише дублювало інформацію, яка є фіксованою властивістю заліза, і зберігалась явно у [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target).
Окреме зберігання цієї інформації для кожної окремої інструкції суттєво збільшувало вимоги до пам'яті Qiskit і сповільнювало навіть операції зі схемами, яким ця інформація не була безпосередньо потрібна.

Інструкція [`Delay`](https://docs.quantum.ibm.com/api/qiskit/circuit#delay) все ще містить ці поля, як і нова у v2.0 інструкція [`BoxOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#box), оскільки ці інструкції мають _змінну_ затримку.

Тривалість конкретної підтримуваної залізом [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction), визначеної залізом, можна отримати, запитавши [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target).
Для цього слід спочатку проіндексувати [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) за назвою операції, а потім за індексами апаратних кубітів у вигляді tuple.
Це поверне або `None` (якщо інструкція підтримується, але жодних конкретних властивостей до неї не призначено), або екземпляр [`InstructionProperties`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties), з якого поле [`duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties#qiskit.transpiler.InstructionProperties.duration) може надати тривалість у секундах (або `None`, якщо не задано).

```python
target = "<some Target from a hardware vendor>"

# Query the duration of a CX instruction on hardware qubits (2, 4),
# returning `None` if the duration is not specified.
properties = target["cx"][2, 4]
duration = None if properties is None else properties.duration

# You can also do this in one step, using `getattr`:
duration = getattr(target["cx"][2, 4], "duration", None)
```

Загалом, оскільки класичне управління потоком стає дедалі поширенішим у виконаннях схем, «тривалість» схеми ставатиме дедалі менш визначеною.
Хоча Qiskit v2.x все ще включає атрибут [`QuantumCircuit.duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.duration), рекомендується відмовитись від нього, оскільки його, ймовірно, буде видалено в майбутньому; припущення, за яких він є дійсним, порушуватимуться дедалі частіше.
Натомість можна використовувати новий метод [`QuantumCircuit.estimate_duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.estimate_duration), який є безпечнішим і чіткіше описує необхідні припущення.

## qiskit.transpiler

<span id="generate_preset_pass_manager"></span>
### Зміни інтерфейсу `generate_preset_pass_manager` {#generate_preset_pass_manager}

Qiskit v2.0.0 включає низку оновлень інтерфейсу менеджера пресетних проходів (тобто функцій
[`generate_preset_pass_manager`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.generate_preset_pass_manager) та
[`transpile`](https://docs.quantum.ibm.com/api/qiskit/compiler#transpile)), які впливають на
обробку вільно заданих обмежень, таких як `coupling_map` або `basis_gates`. Кількість
варіантів використання транспіляції, де приймаються вільні обмеження, зменшено на користь
аргументів `target` та `backend` (де `backend` — це [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2), що
містить екземпляр [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target)),
які дозволяють більш виразно передавати ці обмеження.

Наступні приклади демонструють основні зміни та рекомендовані альтернативи для міграції. Зверни увагу, що всі
наведені нижче поради щодо міграції стосуються як `transpile`, так і `generate_preset_pass_manager`:

1. Вхідний аргумент `backend_properties` більше не підтримується, і клас `BackendProperties` було
   видалено. Для передачі тривалостей вентилів та похибок слід використовувати аргументи `target` або `backend`. Існує кілька способів побудови власного [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target):

  a. Побудувати target з нуля і додати відповідні похибки та тривалості до відповідних вентилів:

    ```python
    from qiskit.transpiler import Target, InstructionProperties, generate_preset_pass_manager
    from qiskit.circuit.library import XGate, CXGate

    target = Target(num_qubits=2)
    target.add_instruction(CXGate(), {(0, 1): InstructionProperties(error=.0001, duration=5e-7)})
    target.add_instruction(
        XGate(),
        {
            (0,): InstructionProperties(error=.00001, duration=5e-8),
            (1,): InstructionProperties(error=.00002, duration=6e-8)
        }
    )

    pm = generate_preset_pass_manager(target=target)
    ```

  b. Використати [`GenericBackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.fake_provider.GenericBackendV2) для швидкої побудови
     загального target із заздалегідь заповненими похибками та тривалостями. Їх можна також оновити після побудови:

    ```python
    from qiskit.providers.fake_provider import GenericBackendV2
    from qiskit.transpiler import generate_preset_pass_manager

    target = GenericBackendV2(num_qubits=2, basis_gates=["x", "cx"]).target
    target.update_instruction_properties("x", (0,), InstructionProperties(0.00001))
    target.update_instruction_properties("x", (1,), InstructionProperties(0.00002))
    target.update_instruction_properties("cx", (0, 1), InstructionProperties(0.0001))

    pm = generate_preset_pass_manager(target=target)
    ```

2. Вхідні аргументи `instruction_durations` та `timing_constraints` більше не підтримуються. Слід використовувати
  аргументи `target` або `backend`. Альтернативи — побудова власного [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) з
  нуля, побудова загального бекенду, а крім того, є можливість побудувати власний target за допомогою
  `instruction_durations` та/або `timing_constraints`, використовуючи метод `Target.from_configuration`:

    ```python
    from qiskit.transpiler import Target, generate_preset_pass_manager
    from qiskit.transpiler.timing_constraints import TimingConstraints
    from qiskit.transpiler.instruction_durations import InstructionDurations

    instruction_durations = InstructionDurations([("x", None, 200), ("cx", None, 1000)], dt=1e-7)

    timing_constraints = TimingConstraints(
            granularity=1, min_length=1, pulse_alignment=1, acquire_alignment=1
        )

    target = Target.from_configuration(num_qubits=2,
                                        basis_gates=["x", "cx"],
                                        instruction_durations=instruction_durations,
                                        timing_constraints=timing_constraints)

    pm = generate_preset_pass_manager(target=target)
    ```

3. Надання `coupling_map` та/або `basis_gates` разом із `backend` не рекомендується, і починаючи з
  v2.0 буде видаватися `UserWarning`. У таких випадках існує кілька джерел правди,
  і така інформація, як похибки вентилів та тривалості з бекенду, не може бути вирішена на основі
  наданої інформації.
  Рекомендована альтернатива — визначити власний target, що поєднує вибрані обмеження, або
  з нуля, або за допомогою [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#from_configuration) чи [`GenericBackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.fake_provider.GenericBackendV2).

4. Аргумент `basis_gates` більше не приймає власні базисні вентилі. Рекомендована альтернатива —
  аргумент `target`, який можна побудувати з нуля або через [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#from_configuration),
  наприклад:

    ```python
    from qiskit.circuit.library import XGate
    from qiskit.transpiler.target import Target

    basis_gates = ["my_x", "cx"]
    custom_name_mapping = {"my_x": XGate()}
    target = Target.from_configuration(
        basis_gates=basis_gates, num_qubits=2, custom_name_mapping=custom_name_mapping
    )
    pm = generate_preset_pass_manager(target=target)
    ```

5. Комбінація вхідних даних, де надається власний `coupling_map` разом із `backend` або `basis_gates`,
  що містять вентилі з трьома і більше кубітами, більше не дозволяється. Карта зв'язності
  не надає необхідних деталей підключення для визначення дії вентиля.
  У таких випадках рекомендований шлях міграції — створення власного target з нуля.

6. Аргумент `inst_map` було видалено без альтернативи. Він використовувався для надання інформації про пульси
  транспілятору, і весь модуль `qiskit.pulse` було видалено.

## qiskit.pulse

Модуль `qiskit.pulse` було повністю видалено з Qiskit без заміни.
Це включає пов'язані елементи:

- властивість `calibrations` [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit), [`DAGCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGCircuit) та [`DAGDependency`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGDependency).
- методи [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit) `has_calibration_for` та `add_calibration`.
- метод [`DAGCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGCircuit) `has_calibration_for`.
- проходи транспілятора `PulseGates`, `RXCalibrationBuilder`, `RZXCalibrationBuilder`, `EchoRZXWeylDecomposition`, `ValidatePulseGates`.
- модуль `qiskit.scheduler` та весь його вміст.
- методи [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) `instruction_schedule_map`, `drive_channel`, `measure_channel`, `acquire_channel` та `control_channel`.
- методи [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) `has_calibration`, `get_calibration`, `instruction_schedule_map` та `update_from_instruction_schedule_map`.
- аргумент `inst_map` функцій [`generate_preset_pass_manager`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.generate_preset_pass_manager), [`transpile`](https://docs.quantum.ibm.com/api/qiskit/compiler#transpile) та [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#qiskit.transpiler.Target.from_configuration).
- поле `calibration` [`InstructionProperties`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties).

Це сталося тому, що IBM Quantum&reg; QPU припинили пропонувати доступ на рівні пульсів, і команда Qiskit більше не мала ресурсів для підтримки маловикористовуваного пакету.
Ти можеш продовжувати використовувати модуль `qiskit.pulse`, використовуючи серію v1.x Qiskit, яка підтримується виправленнями помилок до вересня 2024 року та підтримкою безпеки до березня 2025 року, але модуль видалено із серії v2.x.

Більшість використань `qiskit.pulse` можна замінити прямим доступом до дробових вентилів на IBM Quantum Heron QPU (і пізніших поколіннях, якщо застосовно).
Детальніше можна прочитати у [посібнику з міграції з pulse на дробові вентилі](/guides/pulse-migration).

Ти більше не можеш завантажувати QPY-файли, що містять об'єкти `ScheduleBlock`, у Qiskit v2.0 через видалення Qiskit Pulse.
Ці файли можна продовжувати завантажувати у серії Qiskit v1.x.
