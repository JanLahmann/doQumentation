# Estructura de cuentas de IBM Cloud {#ibm-cloud-account-structure}

Antes de configurar IBM Quantum&reg; Platform, es importante entender la estructura de la [cuenta de gestión de identidad y acceso (Identity and Access Management, IAM)](https://cloud.ibm.com/docs/account?topic=account-overview) de IBM Cloud&reg;. Como se muestra en el siguiente diagrama de alto nivel, existe una **cuenta (account)** en el nivel más alto.  Solo hay un propietario de la cuenta, que tiene control exclusivo sobre la administración y gestión de facturación (billing).

![Se muestra un diagrama de alto nivel, ilustrando las relaciones entre la cuenta, los usuarios, las instancias y las políticas de acceso.](/docs/images/guides/cloud-account-structure/account-structure.svg 'Estructura de la cuenta en la nube')

Esta cuenta contiene múltiples **usuarios** e **instancias de servicio** (como IBM Qiskit Runtime).  Cada instancia de servicio existe en una **región** específica y tiene un **plan**, por lo que si deseas tener múltiples planes disponibles para tus usuarios, tendrás múltiples instancias de servicio, cada una asociada con un plan diferente.

A cada usuario se le asignan **políticas de acceso (access policies)**, que les otorgan varios niveles de acceso a las instancias del servicio.  Las políticas de acceso pueden agruparse como un **grupo de acceso (access group)**.

De forma predeterminada, todos los usuarios dentro de la cuenta pueden verse entre sí. En la configuración de la cuenta, puedes activar la opción para restringir la visibilidad, garantizando que solo los usuarios con los permisos para el servicio IAM puedan ver a los demás. En particular, IBM Cloud no admite grupos de usuarios ni administradores específicos para determinados grupos.

## Políticas y grupos de acceso {#access-policies-and-groups}

Como se describió anteriormente, a cada usuario se le puede asignar una o más [políticas de acceso:](https://cloud.ibm.com/docs/account?topic=account-iamusermanpol) de forma individual, como parte de un grupo de acceso, o de ambas formas.

Dentro de una política de acceso, puedes especificar los permisos al seleccionar los roles de la _plataforma_ y del _servicio_ o creando roles personalizados (custom roles). Los roles de plataforma definen acciones que se encuentran a nivel de plataforma, tales como el crear o administrar instancias. Mientras tanto, los roles de servicio otorgan acceso a realizar acciones dentro del servicio, como invocar un punto final de la API de "crear trabajos" (es decir, ejecutar un trabajo).

<Admonition type="note">
Esta guía describe una representación simplificada del modelo IAM. Para obtener los detalles completos, consulta la [documentación de IBM Cloud IAM](https://cloud.ibm.com/docs/account?topic=account-iamoverview).
</Admonition>

![Este diagrama ilustra las relaciones entre los grupos de acceso, las políticas de acceso y los permisos de usuario.](/docs/images/guides/cloud-account-structure/policies.svg 'Políticas de acceso y grupos de acceso')

## Roles {#roles}

Existen dos familias de roles: los roles gestión u administración de plataforma y los roles de acceso a servicios.

Los roles de **gestión de la plataforma (Platform management)** definen acciones permitidas, tales como asignar acceso de usuario y crear instancias de servicio para gestionar y administrar recursos a nivel de plataforma. Los roles de plataforma también se aplican a las acciones que se pueden tomar dentro del contexto de los servicios de gestión de cuentas (account management), como invitar y remover a otros usuarios, gestionar grupos de acceso y gestionar identificadores o IDs de servicios.

Los roles de **acceso a los servicios (Service access)** definen las acciones permitidas, tales como invocar o realizar llamados hacia las APIs del servicio o el acceder a una sección principal, el panel de control o tablero principal (dashboard). Cuando se crea una política de acceso (ya sea en un grupo de acceso o para un usuario), puedes verificar qué acciones forman parte del rol revisando la descripción. Por ejemplo `quantum-computing.job.create - Create a job to run a program` (Crear un trabajo para ejecutar un programa).

También puedes determinar las acciones permitidas por cada rol desde la página de [Roles de IAM (IAM Roles)](https://cloud.ibm.com/iam/roles). Selecciona Qiskit Runtime en el menú desplegable (dropdown) en la parte superior de la página. Luego, para una lista más detallada, haz clic en el número en la columna junto al nombre del rol. Por ejemplo, al visitar esa página y hacer clic en el número junto al rol de 'Administrador' (Manager), puedes ver que este rol incluye la capacidad de eliminar un trabajo (`quantum-computing.job.delete`).
