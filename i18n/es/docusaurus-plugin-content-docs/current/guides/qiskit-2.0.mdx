# Guía de migración a Qiskit v2.0 {#qiskit-v20-migration-guide}

Qiskit v2.0 es el siguiente paso dentro del ciclo de vida recientemente estabilizado del SDK de Qiskit.
Esta versión mayor rompe la compatibilidad con versiones anteriores de Qiskit, pero,
a diferencia de la versión v1.0, no implica ningún cambio en la estructura de empaquetado.

Esta guía describe las rutas de migración para los cambios de características más importantes en Qiskit v2.0, organizados por módulo.
Usa la tabla de contenidos en el lado derecho para navegar al módulo que te interesa.

Lee más sobre los beneficios de Qiskit v2.0 en el [blog de IBM&reg;](https://www.ibm.com/quantum/blog/qiskit-2-0-release-summary).

<span id="qiskit.circuit"></span>
## qiskit.circuit

<span id="c_if"></span>
### Eliminación de las condiciones heredadas `c_if` {#removal-of-legacy-c_if-conditions}

Las condiciones clásicas en Qiskit han sido representadas preferentemente por la instrucción [`IfElseOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.IfElseOp) desde antes de Qiskit v1.0.
Estas operaciones se agregan típicamente a un [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit) usando el método [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) como gestor de contexto.

Esto también significa que [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) ya no tiene un atributo `condition`, y que no puedes encadenar llamadas `c_if` en la salida de una llamada a un método creador de compuertas de [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit).

Las únicas instrucciones que podrían tener una condición ahora son [`IfElseOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.IfElseOp) y [`WhileLoopOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.WhileLoopOp).
La forma más correcta de verificar una condición es usar una comprobación `isinstance` para validar que tienes la clase correcta.
Si quieres dar soporte al atributo heredado `Instruction.condition` durante la transición de tu biblioteca a Qiskit v2.0 y versiones posteriores, puedes usar `getattr(operation, "condition", None)`, lo que generalmente hará que las instancias de `Instruction` se comporten como lo hacían en Qiskit v1.x y versiones anteriores.

El método original y heredado para dar soporte a condiciones simples de igualdad de registros en Qiskit era usar el método `c_if` en instrucciones individuales o conjuntos de instrucciones.
Este método era estrictamente menos funcional que la forma completa de flujo de control, y nunca tuvo un buen soporte en hardware.
Este método heredado ha sido eliminado en Qiskit v2.0; debes usar el método [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) con una condición apropiada.
Puedes continuar usando la forma de 2-tupla `(register, integer)` para representar una igualdad de registro, o puedes usar el más expresivo [subsistema de computación clásica](https://docs.quantum.ibm.com/api/qiskit/circuit_classical) para representar condiciones más complejas.

Por ejemplo, puede que hayas escrito:

```python
from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
qc.x(0).c_if(cr, 3)
```

La traducción más directa de esto es usar la forma de 2-tupla con el gestor de contexto [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test):

```python
from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
with qc.if_test((cr, 3)):
    qc.x(0)
```

Ten en cuenta que en esta forma, también puedes hacer que la compuerta condicional acceda a múltiples operaciones (simplemente agrega más en el bloque indentado) sin re-evaluar las condiciones.
También puedes proporcionar una sentencia ``else``, y usar lógica condicional más compleja:

```python
from qiskit.circuit import QuantumCircuit
from qiskit.circuit.classical import expr

qc = QuantumCircuit(2, 2)
qc.h(0)
qc.h(1)
qc.measure(qc.qubits, qc.clbits)

# Apply an X to qubit 0 if both measurements were the same:
with qc.if_test(expr.equal(qc.clbits[0], qc.clbits[1])) as else_:
    qc.x(0)
# ... or apply a Y to both qubits otherwise:
with else_:
    qc.y(0)
    qc.y(1)
```

<span id="instruction_durations"></span>
### Eliminación de `Instruction.duration` e `Instruction.unit` {#removal-of-instructionduration-and-instructionunit}

Los objetos [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) de circuito tenían anteriormente atributos llamados `duration` y `unit`.
Estos ya no existen en Qiskit v2.0.
Esto se debe a que el campo generalmente solo duplicaba información que era una propiedad fija del hardware, y se almacenaba explícitamente en el [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target).
Almacenar la información por separado para cada instrucción individual aumentaba considerablemente los requerimientos de memoria de Qiskit y ralentizaba incluso las operaciones de circuito que no necesitaban acceder directamente a esa información.

La instrucción [`Delay`](https://docs.quantum.ibm.com/api/qiskit/circuit#delay) aún contiene estos campos, al igual que el nuevo en v2.0 [`BoxOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#box), porque estas instrucciones tienen un retardo _variable_.

Puedes obtener la duración especificada por el hardware de una [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) particular soportada por hardware consultando el [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target).
Para hacerlo, indexa el [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) primero con el nombre de la operación, y luego los índices de los qubits de hardware como una tupla.
Esto retornará ya sea `None` (si la instrucción está soportada, pero no se le asignaron propiedades específicas), o una instancia de [`InstructionProperties`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties), desde la cual el campo [`duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties#qiskit.transpiler.InstructionProperties.duration) puede proveer la duración en segundos (o `None`, si no se proporciónó).

```python
target = "<some Target from a hardware vendor>"

# Query the duration of a CX instruction on hardware qubits (2, 4),
# returning `None` if the duration is not specified.
properties = target["cx"][2, 4]
duration = None if properties is None else properties.duration

# You can also do this in one step, using `getattr`:
duration = getattr(target["cx"][2, 4], "duration", None)
```

En general, a medida que el flujo de control clásico se vuelve más común dentro de las ejecuciones de circuitos, la "duración" de un circuito estará cada vez menos definida.
Aunque Qiskit v2.x aún incluye el atributo [`QuantumCircuit.duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.duration), se recomienda alejarse de él, ya que probablemente será eliminado en el futuro; los supuestos bajo los cuales es válido serán violados con mayor frecuencia.
En su lugar, puedes usar el nuevo método [`QuantumCircuit.estimate_duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.estimate_duration), que es más seguro y más claro sobre los supuestos necesarios.

## qiskit.transpiler

<span id="generate_preset_pass_manager"></span>
### Cambios en la interfaz de `generate_preset_pass_manager` {#changes-to-generate_preset_pass_manager-interface}

Qiskit v2.0.0 incluye una serie de actualizaciones en la interfaz del gestor de pasadas preestablecido (es decir, las funciones
[`generate_preset_pass_manager`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.generate_preset_pass_manager) y
[`transpile`](https://docs.quantum.ibm.com/api/qiskit/compiler#transpile)) que afectan el
manejo de entradas de restricciones sueltas, como `coupling_map` o `basis_gates`. El número de
casos de uso de transpilación donde se aceptan restricciones sueltas ha sido reducido en favor de las
entradas `target` y `backend` (donde `backend` es un [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) que
contiene una instancia de [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target)),
que permiten una comunicación más expresiva de estas restricciones.

Los siguientes ejemplos muestran los principales cambios y las alternativas sugeridas para migrar. Ten en cuenta que todos
los consejos de migración a continuación se aplican tanto a `transpile` como a `generate_preset_pass_manager`:

1. El argumento de entrada `backend_properties` ya no está soportado, y la clase `BackendProperties` ha sido
   eliminada. Debes usar los argumentos `target` o `backend` en su lugar para comunicar
   duraciones y errores de compuertas. Hay múltiples rutas para construir un [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) personalizado:

  a. Construye un target desde cero y agrega los errores y duraciones relevantes a las compuertas correspondientes:

    ```python
    from qiskit.transpiler import Target, InstructionProperties, generate_preset_pass_manager
    from qiskit.circuit.library import XGate, CXGate

    target = Target(num_qubits=2)
    target.add_instruction(CXGate(), {(0, 1): InstructionProperties(error=.0001, duration=5e-7)})
    target.add_instruction(
        XGate(),
        {
            (0,): InstructionProperties(error=.00001, duration=5e-8),
            (1,): InstructionProperties(error=.00002, duration=6e-8)
        }
    )

    pm = generate_preset_pass_manager(target=target)
    ```

  b. Usa [`GenericBackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.fake_provider.GenericBackendV2) para construir rápidamente un
     target genérico con errores y duraciones pre-llenados. Estos también pueden actualizarse después de la construcción:

    ```python
    from qiskit.providers.fake_provider import GenericBackendV2
    from qiskit.transpiler import generate_preset_pass_manager

    target = GenericBackendV2(num_qubits=2, basis_gates=["x", "cx"]).target
    target.update_instruction_properties("x", (0,), InstructionProperties(0.00001))
    target.update_instruction_properties("x", (1,), InstructionProperties(0.00002))
    target.update_instruction_properties("cx", (0, 1), InstructionProperties(0.0001))

    pm = generate_preset_pass_manager(target=target)
    ```

2. Los argumentos de entrada `instruction_durations` y `timing_constraints` ya no están soportados. Debes usar los
  argumentos `target` o `backend` en su lugar. Las alternativas son construir un [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) personalizado desde
  cero, construir un backend genérico, y, además de estas, es posible construir un target personalizado con
  `instruction_durations` y/o `timing_constraints` usando el método `Target.from_configuration`:

    ```python
    from qiskit.transpiler import Target, generate_preset_pass_manager
    from qiskit.transpiler.timing_constraints import TimingConstraints
    from qiskit.transpiler.instruction_durations import InstructionDurations

    instruction_durations = InstructionDurations([("x", None, 200), ("cx", None, 1000)], dt=1e-7)

    timing_constraints = TimingConstraints(
            granularity=1, min_length=1, pulse_alignment=1, acquire_alignment=1
        )

    target = Target.from_configuration(num_qubits=2,
                                        basis_gates=["x", "cx"],
                                        instruction_durations=instruction_durations,
                                        timing_constraints=timing_constraints)

    pm = generate_preset_pass_manager(target=target)
    ```

3. Se desaconseja proporcionar `coupling_map` y/o `basis_gates` junto con un `backend`, y a partir de
  v2.0 se generará un `UserWarning`. En estos casos hay múltiples fuentes de verdad,
  e información como los errores de compuerta y las duraciones del backend no se puede resolver en base a la
  información proporcionada.
  La alternativa sugerida es definir un target personalizado que combine las restricciones elegidas ya sea
  desde cero, usando [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#from_configuration) o [`GenericBackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.fake_provider.GenericBackendV2).

4. El argumento `basis_gates` ya no acepta compuertas base personalizadas. La alternativa sugerida
  es el argumento `target`, que puede construirse desde cero o a través de [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#from_configuration)
  como:

    ```python
    from qiskit.circuit.library import XGate
    from qiskit.transpiler.target import Target

    basis_gates = ["my_x", "cx"]
    custom_name_mapping = {"my_x": XGate()}
    target = Target.from_configuration(
        basis_gates=basis_gates, num_qubits=2, custom_name_mapping=custom_name_mapping
    )
    pm = generate_preset_pass_manager(target=target)
    ```

5. La combinación de entrada donde se proporciona un `coupling_map` personalizado junto con un `backend` o `basis_gates`
  que contienen compuertas con tres o más qubits ya no está permitida. El mapa de acoplamiento
  no proporciona los detalles de conectividad necesarios para poder determinar la acción de la compuerta.
  En estos casos, la ruta de migración recomendada es la creación de un target personalizado desde cero.

6. El argumento `inst_map` ha sido eliminado sin alternativa. Se utilizaba para proporcionar información de pulsos
  al transpilador, y el módulo completo `qiskit.pulse` ha sido eliminado.

## qiskit.pulse

El módulo `qiskit.pulse` ha sido completamente eliminado de Qiskit, sin reemplazo.
Esto incluye los siguientes elementos relacionados:

- la propiedad `calibrations` de [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit), [`DAGCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGCircuit) y [`DAGDependency`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGDependency).
- los métodos `has_calibration_for` y `add_calibration` de [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit).
- el método `has_calibration_for` de [`DAGCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGCircuit).
- las pasadas del transpilador `PulseGates`, `RXCalibrationBuilder`, `RZXCalibrationBuilder`, `EchoRZXWeylDecomposition`, `ValidatePulseGates`.
- el módulo `qiskit.scheduler` y todo su contenido.
- los métodos `instruction_schedule_map`, `drive_channel`, `measure_channel`, `acquire_channel` y `control_channel` de [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2).
- los métodos `has_calibration`, `get_calibration`, `instruction_schedule_map` y `update_from_instruction_schedule_map` de [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target).
- el argumento `inst_map` de [`generate_preset_pass_manager`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.generate_preset_pass_manager), [`transpile`](https://docs.quantum.ibm.com/api/qiskit/compiler#transpile), y [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#qiskit.transpiler.Target.from_configuration).
- el campo `calibration` de [`InstructionProperties`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties).

Esto ocurrió porque los QPUs de IBM Quantum&reg; dejaron de ofrecer acceso a nivel de pulsos, y el equipo de Qiskit ya no tenía recursos para mantener el paquete de poco uso.
Puedes continuar usando el módulo `qiskit.pulse` mientras usas la serie v1.x de Qiskit, que tiene soporte de corrección de errores hasta septiembre de 2024 y soporte de seguridad hasta marzo de 2025, pero el módulo ha sido eliminado de la serie v2.x.

La mayoría de los usos de `qiskit.pulse` pueden reemplazarse con acceso directo a compuertas fraccionales en los QPUs IBM Quantum Heron (y generaciones posteriores, si aplica).
Puedes leer más en [la guía de migración para el uso de pulsos convertidos a compuertas fraccionales](/guides/pulse-migration).

Ya no puedes cargar archivos QPY que contengan objetos `ScheduleBlock` en Qiskit v2.0 debido a la eliminación de Qiskit Pulse.
Puedes continuar cargando estos archivos en la serie Qiskit v1.x.
