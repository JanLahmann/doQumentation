# Qiskit v2.0 마이그레이션 가이드 {#qiskit-v2-0-migration-guide}

Qiskit v2.0은 Qiskit SDK의 새롭게 안정화된 수명 주기 내에서의 다음 단계입니다.
이 주요 버전은 이전 버전의 Qiskit과의 호환성을 깨뜨리지만,
v1.0 릴리스와 달리 패키징 구조의 변경은 포함되어 있지 않습니다.

이 가이드는 Qiskit v2.0의 가장 중요한 기능 변경 사항에 대한 마이그레이션 경로를 모듈별로 설명합니다.
오른쪽의 목차를 사용하여 관심 있는 모듈로 이동하세요.

Qiskit v2.0의 이점에 대한 자세한 내용은 [IBM&reg; 블로그](https://www.ibm.com/quantum/blog/qiskit-2-0-release-summary)에서 확인하세요.

<span id="qiskit.circuit"></span>
## qiskit.circuit {#qiskit-circuit}

<span id="c_if"></span>
### 레거시 `c_if` 조건 제거 {#removal-of-legacy-c-if-conditions}

Qiskit에서 고전적 조건은 Qiskit v1.0 이전부터 [`IfElseOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.IfElseOp) 명령어로 우선적으로 표현되어 왔습니다.
이러한 연산은 일반적으로 [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) 메서드를 컨텍스트 관리자로 사용하여 [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit)에 추가됩니다.

이는 또한 [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction)에 더 이상 `condition` 속성이 없으며, [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit) Gate 생성 메서드 호출의 출력에 `c_if` 호출을 연결할 수 없다는 것을 의미합니다.

이제 조건을 가질 수 있는 명령어는 [`IfElseOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.IfElseOp)와 [`WhileLoopOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.WhileLoopOp)뿐입니다.
조건을 테스트하는 가장 올바른 방법은 `isinstance` 검사를 사용하여 올바른 클래스를 확인하는 것입니다.
라이브러리가 Qiskit v2.0 이후로 전환하는 동안 레거시 `Instruction.condition` 속성을 지원하려면, `getattr(operation, "condition", None)`을 사용할 수 있습니다. 이는 일반적으로 `Instruction` 인스턴스가 Qiskit v1.x 및 이전 버전에서와 동일하게 동작하도록 합니다.

Qiskit에서 단순한 레지스터 동등 조건을 지원하는 원래의 레거시 방법은 개별 명령어 또는 명령어 세트에 `c_if` 메서드를 사용하는 것이었습니다.
이 방법은 완전한 제어 흐름 형식보다 기능이 엄격하게 제한되어 있었으며, 하드웨어에서 잘 지원되지 않았습니다.
이 레거시 방법은 Qiskit v2.0에서 제거되었습니다. 적절한 조건과 함께 [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) 메서드를 사용해야 합니다.
레지스터 동등을 표현하기 위해 `(register, integer)`의 2-튜플 형식을 계속 사용하거나, 더 복잡한 조건을 표현하기 위해 더 표현력 있는 [고전적 계산 서브시스템](https://docs.quantum.ibm.com/api/qiskit/circuit_classical)을 사용할 수 있습니다.

예를 들어, 다음과 같이 작성했을 수 있습니다:

```python
from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
qc.x(0).c_if(cr, 3)
```

이것의 가장 직접적인 변환은 [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) 컨텍스트 관리자와 함께 2-튜플 형식을 사용하는 것입니다:

```python
from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
with qc.if_test((cr, 3)):
    qc.x(0)
```

이 형식에서는 조건을 재평가하지 않고도 여러 연산에 조건부 Gate를 적용할 수 있습니다(들여쓰기된 블록에 더 많은 항목을 추가하면 됩니다).
또한 ``else`` 문을 제공하고 더 복잡한 조건부 로직을 사용할 수 있습니다:

```python
from qiskit.circuit import QuantumCircuit
from qiskit.circuit.classical import expr

qc = QuantumCircuit(2, 2)
qc.h(0)
qc.h(1)
qc.measure(qc.qubits, qc.clbits)

# Apply an X to qubit 0 if both measurements were the same:
with qc.if_test(expr.equal(qc.clbits[0], qc.clbits[1])) as else_:
    qc.x(0)
# ... or apply a Y to both qubits otherwise:
with else_:
    qc.y(0)
    qc.y(1)
```

### `Instruction.duration`과 `Instruction.unit` 제거 {#removal-of-instruction-duration-and-instruction-unit}

Circuit [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) 객체는 이전에 `duration`과 `unit`이라는 속성을 가지고 있었습니다.
이 속성들은 Qiskit v2.0에 더 이상 존재하지 않습니다.
이는 해당 필드가 일반적으로 하드웨어의 고정 속성인 정보를 중복 저장하고 있었으며, [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target)에 명시적으로 저장되어 있었기 때문입니다.
각 개별 명령어에 대해 정보를 별도로 저장하면 Qiskit의 메모리 요구량이 크게 증가하고, 해당 정보를 직접 사용하지 않는 Circuit 연산도 느려졌습니다.

[`Delay`](https://docs.quantum.ibm.com/api/qiskit/circuit#delay) 명령어는 이 필드를 여전히 포함하고 있으며, v2.0에서 새로 추가된 [`BoxOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#box)도 마찬가지입니다. 이 명령어들은 _가변_ 지연을 가지기 때문입니다.

특정 하드웨어 지원 [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction)의 하드웨어 지정 지속 시간은 [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target)을 쿼리하여 가져올 수 있습니다.
이를 위해 먼저 연산의 이름으로 [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target)을 인덱싱한 다음, 하드웨어 Qubit의 인덱스를 튜플로 인덱싱합니다.
이렇게 하면 `None`(명령어가 지원되지만 특정 속성이 할당되지 않은 경우) 또는 [`InstructionProperties`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties) 인스턴스가 반환됩니다. 후자의 경우 [`duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties#qiskit.transpiler.InstructionProperties.duration) 필드가 초 단위의 지속 시간을 제공하거나(제공되지 않은 경우 `None`) 합니다.

```python
target = "<some Target from a hardware vendor>"

# Query the duration of a CX instruction on hardware qubits (2, 4),
# returning `None` if the duration is not specified.
properties = target["cx"][2, 4]
duration = None if properties is None else properties.duration

# You can also do this in one step, using `getattr`:
duration = getattr(target["cx"][2, 4], "duration", None)
```

일반적으로 Circuit 실행 내에서 고전적 제어 흐름이 더 일반화됨에 따라 Circuit의 "지속 시간"은 점점 더 명확하게 정의되기 어려워질 것입니다.
Qiskit v2.x는 여전히 [`QuantumCircuit.duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.duration) 속성을 포함하고 있지만, 앞으로 제거될 가능성이 높으므로 이 속성에서 벗어나는 것이 권장됩니다. 이 속성이 유효한 가정들은 점점 더 자주 위반될 것입니다.
대신, 새로운 [`QuantumCircuit.estimate_duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.estimate_duration) 메서드를 사용할 수 있습니다. 이 메서드는 더 안전하고 필요한 가정에 대해 더 명확합니다.

## qiskit.transpiler {#qiskit-transpiler}

<span id="generate_preset_pass_manager"></span>
### `generate_preset_pass_manager` 인터페이스 변경 사항 {#changes-to-generate-preset-pass-manager-interface}

Qiskit v2.0.0에는 프리셋 패스 관리자 인터페이스(즉,
[`generate_preset_pass_manager`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.generate_preset_pass_manager)와
[`transpile`](https://docs.quantum.ibm.com/api/qiskit/compiler#transpile) 함수)에 대한 일련의 업데이트가 포함되어 있으며,
이는 `coupling_map` 또는 `basis_gates`와 같은 느슨한 제약 입력의 처리에 영향을 미칩니다. 느슨한 제약이 허용되는
Transpiler 사용 사례의 수가 `target` 및 `backend` 입력(여기서 `backend`는 [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) 인스턴스를 포함하는 [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2))을 선호하여 줄었습니다.
이 입력들은 이러한 제약을 더 표현력 있게 전달할 수 있도록 합니다.

다음 예제들은 주요 변경 사항과 마이그레이션을 위한 대안을 보여줍니다. 아래의 모든 마이그레이션 팁은 `transpile`과 `generate_preset_pass_manager` 모두에 적용된다는 점에 유의하세요:

1. `backend_properties` 입력 인수는 더 이상 지원되지 않으며, `BackendProperties` 클래스는 제거되었습니다. Gate 지속 시간과 오류를 전달하려면 `target` 또는 `backend` 인수를 사용해야 합니다. 커스텀 [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target)을 구성하는 방법은 여러 가지가 있습니다:

  a. 처음부터 Target을 빌드하고 해당 Gate에 관련 오류와 지속 시간을 추가합니다:

    ```python
    from qiskit.transpiler import Target, InstructionProperties, generate_preset_pass_manager
    from qiskit.circuit.library import XGate, CXGate

    target = Target(num_qubits=2)
    target.add_instruction(CXGate(), {(0, 1): InstructionProperties(error=.0001, duration=5e-7)})
    target.add_instruction(
        XGate(),
        {
            (0,): InstructionProperties(error=.00001, duration=5e-8),
            (1,): InstructionProperties(error=.00002, duration=6e-8)
        }
    )

    pm = generate_preset_pass_manager(target=target)
    ```

  b. [`GenericBackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.fake_provider.GenericBackendV2)를 사용하여 사전에 채워진 오류와 지속 시간이 있는 일반 Target을 빠르게 빌드합니다. 구성 후에도 업데이트할 수 있습니다:

    ```python
    from qiskit.providers.fake_provider import GenericBackendV2
    from qiskit.transpiler import generate_preset_pass_manager

    target = GenericBackendV2(num_qubits=2, basis_gates=["x", "cx"]).target
    target.update_instruction_properties("x", (0,), InstructionProperties(0.00001))
    target.update_instruction_properties("x", (1,), InstructionProperties(0.00002))
    target.update_instruction_properties("cx", (0, 1), InstructionProperties(0.0001))

    pm = generate_preset_pass_manager(target=target)
    ```

2. `instruction_durations`와 `timing_constraints` 입력 인수는 더 이상 지원되지 않습니다. 대신 `target` 또는 `backend` 인수를 사용해야 합니다. 대안으로는 처음부터 커스텀 [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target)을 구성하거나, 일반 Backend를 구성하는 것이 있으며, 이에 더해 `Target.from_configuration` 메서드를 사용하여 `instruction_durations` 및/또는 `timing_constraints`로 커스텀 Target을 빌드하는 것도 가능합니다:

    ```python
    from qiskit.transpiler import Target, generate_preset_pass_manager
    from qiskit.transpiler.timing_constraints import TimingConstraints
    from qiskit.transpiler.instruction_durations import InstructionDurations

    instruction_durations = InstructionDurations([("x", None, 200), ("cx", None, 1000)], dt=1e-7)

    timing_constraints = TimingConstraints(
            granularity=1, min_length=1, pulse_alignment=1, acquire_alignment=1
        )

    target = Target.from_configuration(num_qubits=2,
                                        basis_gates=["x", "cx"],
                                        instruction_durations=instruction_durations,
                                        timing_constraints=timing_constraints)

    pm = generate_preset_pass_manager(target=target)
    ```

3. `backend`와 함께 `coupling_map` 및/또는 `basis_gates`를 제공하는 것은 권장되지 않으며,
  v2.0부터 `UserWarning`이 발생합니다. 이러한 경우 여러 가지 신뢰할 수 있는 소스가 있으며,
  제공된 정보를 기반으로 Backend의 Gate 오류 및 지속 시간과 같은 정보를 해결할 수 없습니다.
  권장되는 대안은 처음부터, [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#from_configuration) 또는 [`GenericBackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.fake_provider.GenericBackendV2)를 사용하여 선택한 제약을 결합하는 커스텀 Target을 정의하는 것입니다.

4. `basis_gates` 인수는 더 이상 커스텀 기저 Gate를 허용하지 않습니다. 권장되는 대안은
  `target` 인수입니다. 이는 다음과 같이 처음부터 또는 [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#from_configuration)을 통해 빌드할 수 있습니다:

    ```python
    from qiskit.circuit.library import XGate
    from qiskit.transpiler.target import Target

    basis_gates = ["my_x", "cx"]
    custom_name_mapping = {"my_x": XGate()}
    target = Target.from_configuration(
        basis_gates=basis_gates, num_qubits=2, custom_name_mapping=custom_name_mapping
    )
    pm = generate_preset_pass_manager(target=target)
    ```

5. 세 개 이상의 Qubit을 가진 Gate를 포함하는 `backend` 또는 `basis_gates`와 함께 커스텀 `coupling_map`을 제공하는 입력 조합은 더 이상 허용되지 않습니다. coupling map은
  Gate의 동작을 결정하는 데 필요한 연결 세부 정보를 제공하지 않습니다.
  이러한 경우 권장되는 마이그레이션 경로는 처음부터 커스텀 Target을 생성하는 것입니다.

6. `inst_map` 인수는 대안 없이 제거되었습니다. 이 인수는 Transpiler에 펄스 정보를 제공하는 데 사용되었으며, 전체 `qiskit.pulse` 모듈이 제거되었습니다.

<span id="qiskit.pulse"></span>
## qiskit.pulse {#qiskit-pulse}

`qiskit.pulse` 모듈은 대체 없이 Qiskit에서 완전히 제거되었습니다.
여기에는 관련 항목들이 포함됩니다:

- [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit), [`DAGCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGCircuit) 및 [`DAGDependency`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGDependency) 속성 `calibrations`.
- [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit) 메서드 `has_calibration_for` 및 `add_calibration`.
- [`DAGCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGCircuit) 메서드 `has_calibration_for`.
- Transpiler 패스 `PulseGates`, `RXCalibrationBuilder`, `RZXCalibrationBuilder`, `EchoRZXWeylDecomposition`, `ValidatePulseGates`.
- `qiskit.scheduler` 모듈과 그 모든 내용.
- [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) 메서드 `instruction_schedule_map`, `drive_channel`, `measure_channel`, `acquire_channel` 및 `control_channel`.
- [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) 메서드 `has_calibration`, `get_calibration`, `instruction_schedule_map` 및 `update_from_instruction_schedule_map`.
- [`generate_preset_pass_manager`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.generate_preset_pass_manager), [`transpile`](https://docs.quantum.ibm.com/api/qiskit/compiler#transpile) 및 [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#qiskit.transpiler.Target.from_configuration)의 `inst_map` 인수.
- [`InstructionProperties`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties)의 `calibration` 필드.

이 변경은 IBM Quantum&reg; QPU가 펄스 수준 접근을 중단했고, Qiskit 팀이 거의 사용되지 않는 패키지를 유지 관리할 리소스가 더 이상 없었기 때문입니다.
2024년 9월까지 버그 수정 지원이 제공되고 2025년 3월까지 보안 지원이 제공되는 Qiskit v1.x 시리즈를 사용하면서 `qiskit.pulse` 모듈을 계속 사용할 수 있지만, 이 모듈은 v2.x 시리즈에서 제거되었습니다.

`qiskit.pulse`의 대부분의 사용 사례는 IBM Quantum Heron QPU(및 해당되는 경우 이후 세대)의 분수 Gate에 대한 직접 접근으로 대체할 수 있습니다.
[펄스 사용법을 분수 Gate로 변환하는 마이그레이션 가이드](/guides/pulse-migration)에서 자세한 내용을 확인하세요.

Qiskit Pulse 제거로 인해 Qiskit v2.0에서는 `ScheduleBlock` 객체를 포함하는 QPY 파일을 더 이상 로드할 수 없습니다.
Qiskit v1.x 시리즈에서는 이러한 파일을 계속 로드할 수 있습니다.
