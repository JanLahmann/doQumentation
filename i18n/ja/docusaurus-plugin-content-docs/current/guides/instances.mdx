# インスタンスの作成と管理 {#create-and-manage-instances}

IBM Quantum Platform サービスへのアクセスは、あなたが割り当てられている**インスタンス**によって管理されます。ユーザーは複数のインスタンスに所属することができます。

インスタンスは仮想サーバー（Qiskit Runtime のデプロイ）であり、量子プログラムの実行やクラシカルコンピュートタスク（エラー緩和の処理など）を含む、ワークロードの実行を管理します。インスタンスは*クラウドリソース名*（CRN）によって識別されます。この CRN は、Qiskit プログラムからワークロードを送信する際に使用できます。

ダッシュボードからアクセス可能なインスタンスを確認するか、ダッシュボードの「Instances」タブをクリックすることで確認できます。各インスタンスはその CRN 識別子とともに一覧表示されます。

使用するリージョンごと、使用するサービスプラン（Open や Pay-As-You-Go など）ごとに異なるインスタンスが必要です。アカウントマネージャーおよび十分な権限を持つユーザーは、インスタンスを作成し、割り当て、使用制限、量子コンピューター、ユーザー権限などの設定を定義できます。

IBM Quantum Platform の[ダッシュボード](https://quantum.cloud.ibm.com/)には、アクセス可能なインスタンスが表示されます。アクセス権限に応じて、[Instances ページ](https://quantum.cloud.ibm.com/instances)でインスタンスの詳細を確認、編集、削除することができます。

Open Plan に関連付けられていないすべてのインスタンスにはコストが発生するため、使用しているインスタンスを把握しておくことが重要です。

<span id="open-plan"></span>
## Open Plan インスタンス {#open-plan}

Open Plan ユーザーは、世界最高峰の QPU 上で量子回路を無料で実行できます（28 日間のローリングウィンドウあたり最大 10 分の量子時間）。QPU の使用状況は、[Platform ダッシュボード](https://quantum.cloud.ibm.com/)および[Workloads](https://quantum.cloud.ibm.com/workloads) ページで追跡できます。

<span id="standard-plan"></span>
## Pay-As-You-Go Plan インスタンス {#standard-plan}

Pay-As-You-Go Plan は有料プランです。このプランを使用するには、インスタンスを作成してプランに割り当てる必要があります。このプランを使用して QPU に送信されたすべてのワークロードに料金が発生します。

あなたが実行したジョブと、同じインスタンス内の他のコラボレーターが実行したジョブは、そのインスタンスの報告された使用量に計上されます。インスタンスの使用状況は[Instances](https://quantum.cloud.ibm.com/instances) ページで確認できます。また、適切な権限を持つユーザーは[Analytics](https://quantum.cloud.ibm.com/analytics) ページでも確認できます。なお、これらのページで表示される使用量の数値が異なる場合があります。Instances ページは現在のタイムスタンプから直前の 28 日間のローリングウィンドウで使用量を表示するのに対し、Analytics ページは 1 日単位の境界を使用するためです。

管理者によってインスタンスが「limited」としてマークされている場合（使用量エリアに「Remaining」列が表示されます）、28 日間のローリングウィンドウ内でインスタンスが割り当て（管理者が定義）を超えると、アクティブなワークロードは引き続き実行されます（セッションを含む）が、保留中のワークロードは時間が利用可能になるまでキューに残ります。インスタンスに制限が設定されておらず、割り当てを超えた場合、そのインスタンスで実行されるジョブは優先度が低くなり、キュー待ち時間が長くなる可能性があります。

インスタンスの使用量が割り当てを超えると、アラートが表示されます。

<span id="connect-instance"></span>
## コードでインスタンスを指定する {#connect-instance}

CRN またはインスタンス名を使用して、使用するインスタンスを指定できます。指定のタイミングは、サービスの初期化時、または QPU へのワークロード送信時（例：[`QiskitRuntimeServicee.backend()](https://docs.quantum.ibm.com/api/qiskit-ibm-runtime/qiskit-runtime-service#backend)）です。コードでインスタンスを指定する手順については、[アカウントを初期化する](/guides/initialize-account)を参照してください。

インスタンスが渡された場合、そのインスタンスのバックエンドとジョブのみが利用可能になります（Qiskit Runtime v0.40.1 以降で許可）。

サービスの初期化時にインスタンスが指定されない場合、アカウント内のすべてのインスタンスにわたるすべてのバックエンドとジョブが利用可能になります。この場合、バックエンドが指定されると、そのバックエンドにアクセスできるインスタンスが[インスタンスの自動選択](/guides/initialize-account#auto-select)の説明に従って検索・使用されます。

<Admonition type="important" title="必要なバージョン">
以下の機能には Qiskit Runtime v0.40.1 以降が必要です。

    - インスタンス名を使用してインスタンスを指定する
    - インスタンスを渡さない
    - 優先インスタンスタイプとリージョンを設定する

Qiskit Runtime v0.42 以降では、[自動インスタンス選択](/guides/initialize-account#auto-select)が使用される場合、無料プランがデフォルトで優先されます。
</Admonition>

<span id="add-instance"></span>
## インスタンスを作成する {#add-instance}

以下の手順に従って、インスタンスを作成してアカウントに追加します。組織向けに Cloud 上で Qiskit Runtime を設定する必要がある管理者の方は、[組織向け Qiskit Runtime の計画](/guides/considerations-set-up-runtime)を参照してください。

<span id="switcher"></span>

1. アカウントにインスタンスを追加するには、まずヘッダーのアカウントスイッチャーで正しいアカウントとリージョンが選択されていることを確認します。
        * 複数の IBM Cloud アカウントを持つ（またはその一部である）ことができます。期待するインスタンスやジョブが表示されない場合は、別のアカウントを試してください。
        * 同じ API キーを使用してどちらのリージョンにもアクセスできますが、ログインしているリージョンで作成されたインスタンスのみを表示・アクセスできます。
        * Open インスタンスを作成する場合は、us-east リージョンを選択する必要があります。

        ![IBM Quantum Platform のヘッダーが表示されています。アカウントスイッチャーは検索バーのすぐ右にあります。リージョンスイッチャーはアカウントスイッチャーのすぐ右にあります。](/docs/images/guides/cloud-setup/IQP-Header.svg 'IBM Quantum Platform header')

   <span id="regions"></span>

   <Admonition type="note" title="リージョンについて">
   インスタンスは選択されたリージョンに作成されます。このリージョンによって以下の重要な特性が決まります。
     * 利用可能な QPU の種類。
     * コンパイルなど、ジョブのクラシカル計算が実行される場所。
     * ユーザーのワークフローデータが保存される場所。

    ワークフローデータには、入力回路、回路パラメーター、量子計算結果が含まれます。ショット数、回路のサイズ、その他のジョブ統計などのメタデータは含まれません。

   ユーザーのワークフローデータがグローバルな分散型サービス拒否（DDoS）保護レイヤーに公開されないようにするには、[Virtual Private Endpoints](https://cloud.ibm.com/docs/overview?topic=overview-endpoints-support) を使用できます。`qiskit-ibm-runtime` は[`private_endpoint` パラメーター](https://docs.quantum.ibm.com/api/qiskit-ibm-runtime/qiskit-runtime-service)を通じてこれをサポートしています。
   </Admonition>

1. ダッシュボードの Instances ペインから **View all** をクリックするか、メインメニューから [Instances](https://quantum.cloud.ibm.com/instances) ページに移動します。
1. Instances ページから **Create instance** をクリックします。
    * 名前を入力し、オプションでタグを追加します。
    * IBM Cloud&reg; で複数のリソースグループにアクセスできる場合、このインスタンスが属するリソースグループを変更できます。詳細については、[リソースグループの管理](https://cloud.ibm.com/docs/account?topic=account-rgs&interface=ui)および[リソースグループ内のリソースへのアクセス権の付与](https://cloud.ibm.com/docs/account?topic=account-rgs_manage_access)を参照してください。
1. このインスタンスに関連付ける料金プラン（例：Pay-As-You-Go）を選択します。選択した料金プランに応じて、コスト制限やこのインスタンスに割り当てる分数など、他のオプションを指定できます。このインスタンスが無料プランに関連付けられていない場合、ワークロードの実行に使用するとコストが発生することに注意してください。
1. **Next** をクリックします。インスタンスがアクセスできる QPU の一覧が表示されます。
    - インスタンスが一覧表示されたすべての QPU（および将来このプランに追加されるすべての QPU）にアクセスできるようにする場合は、**Next** をクリックします。
    - Open インスタンスを設定していない場合、このインスタンスでアクセスできる QPU をカスタマイズできます。「Customize allocated compute resources」トグルをクリックします。このインスタンスがアクセスできる特定の QPU を選択し、**Next** をクリックします。
        <Admonition type="note">
      割り当てたコンピュートリソースをカスタマイズした場合、親プランに変更が加えられても、利用可能な QPU は自動的に更新されません。ただし、後から手動で QPU を追加または削除することができます。カスタマイズしない場合は、アカウントがそのプランでアクセスできる QPU に常にアクセスできます。そのため、将来プランに新しい QPU が追加された場合、インスタンスは自動的にそれへのアクセス権を取得します。
        </Admonition>
1. Access group ペインが開きます。このインスタンスのための「collaborators」アクセスグループが自動的に作成されます。このアクセスグループに追加されたユーザーは、このインスタンスに割り当てられた時間を使用できます。[IBM Cloud コンソール](https://cloud.ibm.com/iam/groups)を使用して、[追加のアクセスグループを変更または作成](/guides/access-groups)できます。手順については、[アクセスグループの設定](https://cloud.ibm.com/docs/account?topic=account-groups&interface=ui)を参照してください。
1. **Create instance** をクリックします。[Instances](https://quantum.cloud.ibm.com/instances) ページでインスタンスを確認できるようになります。アカウントに複数のプランが設定されている場合、各プランタイプには Instances テーブルに独自のタブがあります。

## インスタンスとその割り当ての編集 {#edit-an-instance-and-its-allocation}

[Instances](https://quantum.cloud.ibm.com/instances) ページでインスタンスを見つけます。インスタンスの行の末尾にあるオーバーフローメニューをクリックし、「Edit details」を選択します。

<Admonition type="note">
インスタンスを編集できるのは管理者とオーナーのみです。アクセス権の割り当てについては、[アクセスポリシーとアクセスグループの作成](/guides/access-groups)ガイドを参照してください。
</Admonition>
