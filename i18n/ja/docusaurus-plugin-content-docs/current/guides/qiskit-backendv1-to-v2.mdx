# `BackendV1` から `BackendV2` への移行 {#migrate-from-backendv1-to-backendv2}

Qiskit の [`BackendV1`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.BackendV1) クラスは非推奨となり、サービスから削除される予定です。この移行ガイドでは、[`BackendV1`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.BackendV1) から [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) にアップグレードしたプロバイダーを使用している場合に必要な小さな調整について説明します。

<Admonition type="note">
 `qiskit_ibm_runtime` と `qiskit_aer` のみを使用している場合、対応は不要です。`qiskit_ibm_runtime` パッケージは常に `BackendV2` を使用しており、`qiskit_aer` はバージョン 0.13 以降 `BackendV2` を使用しています。
 </Admonition>

## `BackendV2` の主な変更点 {#high-level-changes-in-backendv2}

Qiskit の [`Backend`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.Backend) モデルは、SDK の範囲内で量子コンピューターについて推論できる抽象化レイヤーを
Qiskit SDK に提供するために設計されました。モデルの最初のイテレーションは
[`BackendV1`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.BackendV1) クラスで導入されました。このクラスは Backend の情報を一連の
データコンテナ、すなわち [`BackendConfiguration`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.models.BackendConfiguration) クラスと
[`BackendProperties`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.models.BackendProperties) クラスに格納していました。

[`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) クラスは、
ほとんどの Backend プロパティへのユーザーアクセスを再定義し、ネイティブな Qiskit データ構造で動作し、よりフラットな
アクセスパターンを持つようにしました。[`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) モデルの中核は
[`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) クラスであり、Qiskit が実行のために Circuit を最適化するために使用できるトランスパイル制約を含む QPU の表現です。

Qiskit SDK は [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) 入力のみで動作するように更新されており、
ほとんどのプロバイダーは [`BackendV1`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.BackendV1) から [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) にアップグレードしています。既存のプロバイダーは、円滑な移行のために可能な限り古いアクセス方法を非推奨にすることが期待されますが、最終的にはユーザーがコードを調整する必要があります。

[`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) の原則は、
Backend に関する情報のほとんどがその
[`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) オブジェクトに含まれており、Backend の属性は情報を返すために
`BackendV2.target` 属性を照会することが多いということです。ただし、多くの場合、属性は Target が含むことができる情報の
サブセットのみを提供します。例えば、`backend.coupling_map` は
`BackendV2.target` 属性でアクセス可能な
[`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) から構築された [`CouplingMap`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.CouplingMap) を返します。しかし、Target には
2つ以上の Qubit で動作する命令（[`CouplingMap`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.CouplingMap) では表現できません）が含まれている場合や、
Qubit のサブセット（または2 Qubit 命令の場合は2 Qubit リンク）のみで動作する命令が含まれている場合があり、
それらは `BackendV2.coupling_map` が返す完全なカップリングマップには
詳細が記載されません。そのため、ユースケースによっては、
[`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) での同等のアクセスよりも深く調べる必要があるかもしれません。

## `BackendV2` の具体的な変更点 {#specific-changes-in-backendv2}

ほとんどの属性には直接の置き換えがあり、移行作業が簡素化されます。インターフェース間の唯一の不一致点は [`CouplingMap`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.CouplingMap) にあります。

以下は、[`BackendV1`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.BackendV1) でのアクセスパターンの例と、
[`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) での新しい形式の表です。

</Admonition>
