# คู่มือ migration สำหรับ Qiskit v2.0 {#qiskit-v20-migration-guide}

Qiskit v2.0 คือก้าวถัดไปภายใน life cycle ที่มีเสถียรภาพใหม่ของ Qiskit SDK
เวอร์ชัน major นี้ทำให้เข้ากันไม่ได้กับ Qiskit เวอร์ชันก่อนหน้า แต่
ต่างจาก v1.0 ไม่มีการเปลี่ยนแปลงในโครงสร้าง packaging

คู่มือนี้อธิบาย migration paths สำหรับการเปลี่ยนแปลงฟีเจอร์ที่สำคัญที่สุดใน Qiskit v2.0 จัดตาม module
ใช้สารบัญทางด้านขวาเพื่อ navigate ไปยัง module ที่สนใจ

อ่านเพิ่มเติมเกี่ยวกับประโยชน์ของ Qiskit v2.0 ใน [IBM&reg; blog](https://www.ibm.com/quantum/blog/qiskit-2-0-release-summary)

<span id="qiskit.circuit"></span>
## qiskit.circuit {#qiskitcircuit}

<span id="c_if"></span>
### การลบเงื่อนไข `c_if` แบบ legacy {#removal-of-legacy-c_if-conditions}

Classical conditions ใน Qiskit ได้รับการแสดงผลด้วย instruction [`IfElseOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.IfElseOp) เป็นหลักตั้งแต่ก่อน Qiskit v1.0
operations เหล่านี้โดยทั่วไปถูกเพิ่มลงใน [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit) โดยใช้ method [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) เป็น context manager

ซึ่งหมายความด้วยว่า [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) ไม่มี attribute `condition` อีกต่อไป และคุณไม่สามารถ chain `c_if` calls บน output ของ method call ที่สร้าง gate ของ [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit) ได้

instructions เดียวที่อาจมี condition ในตอนนี้คือ [`IfElseOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.IfElseOp) และ [`WhileLoopOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.WhileLoopOp)
วิธีที่ถูกต้องที่สุดในการทดสอบ condition คือใช้การตรวจสอบ `isinstance` เพื่อตรวจสอบว่าคุณมี class ที่ถูกต้อง
หากต้องการรองรับ attribute `Instruction.condition` แบบ legacy ระหว่างการ transition ของ library ไปยัง Qiskit v2.0 และต่อไป คุณสามารถใช้ `getattr(operation, "condition", None)` ซึ่งโดยทั่วไปจะทำให้ `Instruction` instances ทำงานเหมือนใน Qiskit v1.x และก่อนหน้า

วิธีดั้งเดิมที่รองรับ simple register-equality conditions ใน Qiskit คือการใช้ method `c_if` บน individual instructions หรือ instruction sets
วิธีนี้มีฟีเจอร์น้อยกว่า full control-flow form อย่างเห็นได้ชัด และไม่เคยได้รับการรองรับที่ดีบน hardware
วิธี legacy นี้ถูกลบออกใน Qiskit v2.0 ควรใช้ method [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) พร้อม condition ที่เหมาะสม
คุณยังใช้รูปแบบ 2-tuple ของ `(register, integer)` เพื่อแสดง register equality ได้ หรือใช้ [classical computation subsystem](https://docs.quantum.ibm.com/api/qiskit/circuit_classical) ที่มีฟีเจอร์มากกว่าเพื่อแสดง conditions ที่ซับซ้อนกว่า

ตัวอย่างเช่น คุณอาจเคียนว่า:

```python
from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
qc.x(0).c_if(cr, 3)
```

การแปลงที่ตรงที่สุดคือใช้รูปแบบ 2-tuple กับ [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) context manager:

```python
from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
with qc.if_test((cr, 3)):
    qc.x(0)
```

หมายเหตุว่าในรูปแบบนี้ คุณยังสามารถให้ conditional gate เข้าถึง operations หลายตัวได้ (เพียงเพิ่มเข้าไปใน indented block) โดยไม่ต้องประเมิน conditions อีกครั้ง
คุณยังสามารถระบุ statement ``else`` และใช้ conditional logic ที่ซับซ้อนมากขึ้น:

```python
from qiskit.circuit import QuantumCircuit
from qiskit.circuit.classical import expr

qc = QuantumCircuit(2, 2)
qc.h(0)
qc.h(1)
qc.measure(qc.qubits, qc.clbits)

# Apply an X to qubit 0 if both measurements were the same:
with qc.if_test(expr.equal(qc.clbits[0], qc.clbits[1])) as else_:
    qc.x(0)
# ... or apply a Y to both qubits otherwise:
with else_:
    qc.y(0)
    qc.y(1)
```

<span id="instruction_durations"></span>
### การลบ `Instruction.duration` และ `Instruction.unit` {#removal-of-instructionduration-and-instructionunit}

Circuit [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) objects ก่อนหน้ามี attributes ชื่อ `duration` และ `unit`
attributes เหล่านี้ไม่มีอีกต่อไปใน Qiskit v2.0
เนื่องจากฟิลด์นี้โดยทั่วไปเพียงทำซ้ำข้อมูลที่เป็น fixed property ของ hardware และถูกเก็บอย่างชัดเจนใน [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target)
การเก็บข้อมูลแยกกันสำหรับแต่ละ instruction เพิ่ม memory demands ของ Qiskit อย่างมีนัยสำคัญและทำให้ circuit operations ช้าลงแม้กระทั่งที่ไม่จำเป็นต้องแตะข้อมูลนั้นโดยตรง

instruction [`Delay`](https://docs.quantum.ibm.com/api/qiskit/circuit#delay) ยังคงมี fields เหล่านี้ เช่นเดียวกับ [`BoxOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#box) ที่ใหม่ใน v2.0 เนื่องจาก instructions เหล่านี้มี delay _แบบ variable_

คุณสามารถดึง duration ที่ hardware กำหนดของ hardware-supported [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) เฉพาะได้โดย query [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target)
ทำได้โดย index [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) ก่อนด้วยชื่อของ operation แล้วตามด้วย indices ของ hardware qubits เป็น tuple
จะ return ค่า `None` (หาก instruction รองรับแต่ไม่มี properties เฉพาะถูก assign) หรือ instance ของ [`InstructionProperties`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties) ซึ่งฟิลด์ [`duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties#qiskit.transpiler.InstructionProperties.duration) อาจให้ duration เป็นวินาที (หรือ `None` หากไม่ได้ระบุ)

```python
target = "<some Target from a hardware vendor>"

# Query the duration of a CX instruction on hardware qubits (2, 4),
# returning `None` if the duration is not specified.
properties = target["cx"][2, 4]
duration = None if properties is None else properties.duration

# You can also do this in one step, using `getattr`:
duration = getattr(target["cx"][2, 4], "duration", None)
```

โดยทั่วไป เมื่อ classical control-flow กลายเป็นเรื่องปกติมากขึ้นใน circuit executions "duration" ของ circuit จะมีความหมายน้อยลงเรื่อยๆ
แม้ว่า Qiskit v2.x ยังคงมี attribute [`QuantumCircuit.duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.duration) แต่แนะนำให้เลิกใช้ เนื่องจากอาจถูกลบในอนาคต สมมติฐานที่ทำให้ valid จะถูกละเมิดมากขึ้นเรื่อยๆ
แทนที่ คุณสามารถใช้ method [`QuantumCircuit.estimate_duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.estimate_duration) ใหม่ ซึ่งปลอดภัยกว่าและชัดเจนกว่าเกี่ยวกับสมมติฐานที่จำเป็น

## qiskit.transpiler {#qiskittranspiler}

<span id="generate_preset_pass_manager"></span>
### การเปลี่ยนแปลง interface ของ `generate_preset_pass_manager` {#changes-to-generate_preset_pass_manager-interface}

Qiskit v2.0.0 รวมการอัปเดตหลายอย่างสำหรับ preset pass manager interface (คือ
ฟังก์ชัน [`generate_preset_pass_manager`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.generate_preset_pass_manager) และ
[`transpile`](https://docs.quantum.ibm.com/api/qiskit/compiler#transpile)) ที่ส่งผลต่อ
การจัดการ loose constraint inputs เช่น `coupling_map` หรือ `basis_gates` จำนวน
transpilation use cases ที่รับ loose constraints ได้ลดลงเพื่อสนับสนุน
inputs `target` และ `backend` (ที่ `backend` คือ [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) ที่
ประกอบด้วย instance ของ [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target))
ซึ่งช่วยให้สื่อสาร constraints เหล่านี้ได้มีฟีเจอร์มากขึ้น

ตัวอย่างต่อไปนี้แสดงการเปลี่ยนแปลงหลักและ alternatives ที่แนะนำสำหรับ migration หมายเหตุว่า migration tips ทั้งหมดด้านล่างนำไปใช้กับทั้ง `transpile` และ `generate_preset_pass_manager`:

1. input argument `backend_properties` ไม่รองรับอีกต่อไป และ class `BackendProperties` ถูกลบออกแล้ว ควรใช้ arguments `target` หรือ `backend` แทนเพื่อสื่อสาร
   gate durations และ errors มี paths หลายทางในการสร้าง custom [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target):

  a. สร้าง target ตั้งแต่ต้นและเพิ่ม errors และ durations ที่เกี่ยวข้องไปยัง gates ที่สอดคล้องกัน:

    ```python
    from qiskit.transpiler import Target, InstructionProperties, generate_preset_pass_manager
    from qiskit.circuit.library import XGate, CXGate

    target = Target(num_qubits=2)
    target.add_instruction(CXGate(), {(0, 1): InstructionProperties(error=.0001, duration=5e-7)})
    target.add_instruction(
        XGate(),
        {
            (0,): InstructionProperties(error=.00001, duration=5e-8),
            (1,): InstructionProperties(error=.00002, duration=6e-8)
        }
    )

    pm = generate_preset_pass_manager(target=target)
    ```

  b. ใช้ [`GenericBackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.fake_provider.GenericBackendV2) เพื่อสร้าง
     generic target พร้อม errors และ durations ที่กรอกไว้ล่วงหน้าได้อย่างรวดเร็ว ซึ่งสามารถอัปเดตได้หลัง construction:

    ```python
    from qiskit.providers.fake_provider import GenericBackendV2
    from qiskit.transpiler import generate_preset_pass_manager

    target = GenericBackendV2(num_qubits=2, basis_gates=["x", "cx"]).target
    target.update_instruction_properties("x", (0,), InstructionProperties(0.00001))
    target.update_instruction_properties("x", (1,), InstructionProperties(0.00002))
    target.update_instruction_properties("cx", (0, 1), InstructionProperties(0.0001))

    pm = generate_preset_pass_manager(target=target)
    ```

2. input arguments `instruction_durations` และ `timing_constraints` ไม่รองรับอีกต่อไป ควรใช้
  arguments `target` หรือ `backend` แทน alternatives คือการสร้าง custom [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) ตั้งแต่ต้น การสร้าง generic backend และนอกจากนี้ยังสามารถสร้าง custom target ด้วย
  `instruction_durations` และ/หรือ `timing_constraints` โดยใช้ method `Target.from_configuration`:

    ```python
    from qiskit.transpiler import Target, generate_preset_pass_manager
    from qiskit.transpiler.timing_constraints import TimingConstraints
    from qiskit.transpiler.instruction_durations import InstructionDurations

    instruction_durations = InstructionDurations([("x", None, 200), ("cx", None, 1000)], dt=1e-7)

    timing_constraints = TimingConstraints(
            granularity=1, min_length=1, pulse_alignment=1, acquire_alignment=1
        )

    target = Target.from_configuration(num_qubits=2,
                                        basis_gates=["x", "cx"],
                                        instruction_durations=instruction_durations,
                                        timing_constraints=timing_constraints)

    pm = generate_preset_pass_manager(target=target)
    ```

3. การระบุ `coupling_map` และ/หรือ `basis_gates` พร้อมกับ `backend` ไม่แนะนำ และตั้งแต่
  v2.0 จะมีการ raise `UserWarning` ในกรณีเหล่านี้มีแหล่งข้อมูลหลายแหล่ง
  และข้อมูลเช่น gate errors และ durations จาก backend ไม่สามารถแก้ไขได้โดยอิงข้อมูลที่ให้
  alternative ที่แนะนำคือนิยาม custom target ที่รวม constraints ที่เลือกไว้
  ตั้งแต่ต้น โดยใช้ [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#from_configuration) หรือ [`GenericBackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.fake_provider.GenericBackendV2)

4. argument `basis_gates` ไม่รับ custom basis gates อีกต่อไป alternative ที่แนะนำ
  คือ argument `target` ซึ่งสร้างตั้งแต่ต้นหรือผ่าน [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#from_configuration)
  เช่น:

    ```python
    from qiskit.circuit.library import XGate
    from qiskit.transpiler.target import Target

    basis_gates = ["my_x", "cx"]
    custom_name_mapping = {"my_x": XGate()}
    target = Target.from_configuration(
        basis_gates=basis_gates, num_qubits=2, custom_name_mapping=custom_name_mapping
    )
    pm = generate_preset_pass_manager(target=target)
    ```

5. combination input ที่ระบุ custom `coupling_map` พร้อมกับ `backend` หรือ `basis_gates`
  ที่ประกอบด้วย gates สามหรือมากกว่า qubits ไม่ได้รับอนุญาตอีกต่อไป coupling map
  ไม่ให้รายละเอียด connectivity ที่จำเป็นเพื่อกำหนด action ของ gate
  ในกรณีเหล่านี้ migration path ที่แนะนำคือการสร้าง custom target ตั้งแต่ต้น

6. argument `inst_map` ถูกลบออกโดยไม่มี alternative มันใช้เพื่อให้ข้อมูล pulse แก่
  transpiler และ module `qiskit.pulse` ทั้งหมดถูกลบออกแล้ว

<span id="qiskit.pulse"></span>
## qiskit.pulse {#qiskitpulse}

module `qiskit.pulse` ถูกลบออกจาก Qiskit อย่างสมบูรณ์ โดยไม่มีตัวแทน
ซึ่งรวมถึง items ที่เกี่ยวข้อง:

- property `calibrations` ของ [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit), [`DAGCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGCircuit) และ [`DAGDependency`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGDependency)
- methods `has_calibration_for` และ `add_calibration` ของ [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit)
- method `has_calibration_for` ของ [`DAGCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGCircuit)
- transpiler passes `PulseGates`, `RXCalibrationBuilder`, `RZXCalibrationBuilder`, `EchoRZXWeylDecomposition`, `ValidatePulseGates`
- module `qiskit.scheduler` และ contents ทั้งหมด
- methods `instruction_schedule_map`, `drive_channel`, `measure_channel`, `acquire_channel` และ `control_channel` ของ [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2)
- methods `has_calibration`, `get_calibration`, `instruction_schedule_map` และ `update_from_instruction_schedule_map` ของ [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target)
- argument `inst_map` ของ [`generate_preset_pass_manager`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.generate_preset_pass_manager), [`transpile`](https://docs.quantum.ibm.com/api/qiskit/compiler#transpile) และ [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#qiskit.transpiler.Target.from_configuration)
- ฟิลด์ `calibration` ของ [`InstructionProperties`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties)

สิ่งนี้เกิดขึ้นเพราะ IBM Quantum&reg; QPUs หยุดให้การเข้าถึงระดับ pulse และทีม Qiskit ไม่มีทรัพยากรในการดูแล package ที่ใช้น้อยนี้อีกต่อไป
คุณยังสามารถใช้ module `qiskit.pulse` ต่อไปได้ขณะใช้ Qiskit series v1.x ซึ่งมีการรองรับ bug-fix จนถึงกันยายน 2024 และ security support จนถึงมีนาคม 2025 แต่ module นี้ถูกลบออกจาก series v2.x

การใช้งาน `qiskit.pulse` ส่วนใหญ่สามารถแทนได้ด้วยการเข้าถึง fractional gates โดยตรงบน IBM Quantum Heron QPUs (และรุ่นต่อๆ ไป หากมี)
อ่านเพิ่มเติมใน [คู่มือ migration สำหรับการแปลง pulse เป็น fractional gates](/guides/pulse-migration)

คุณไม่สามารถ load QPY files ที่ประกอบด้วย `ScheduleBlock` objects ใน Qiskit v2.0 ได้อีกต่อไปเนื่องจากการลบ Qiskit Pulse
คุณยังสามารถ load files เหล่านี้ใน Qiskit series v1.x ต่อไปได้
