David DiVincenzo นักฟิสิกส์ได้วางข้อกำหนดหลัก 5 ข้อสำหรับการนำคอมพิวเตอร์ควอนตัมไปใช้งานในทุกรูปแบบทางกายภาพ พร้อมเกณฑ์เพิ่มเติมอีก 2 ข้อสำหรับการสื่อสารเชิงควอนตัม ใน Notebook นี้เราจะ**สัมผัสกับเกณฑ์ DiVincenzo แต่ละข้อผ่านการทดลองจริงด้วย Qiskit** แทนที่จะเจาะลึกทฤษฎี แต่ละส่วนจะอธิบายเกณฑ์สั้น ๆ แล้วให้โจทย์โค้ดโดยใช้ Qiskit 2 คุณจะได้รัน Circuit บน Simulator และอุปกรณ์ IBM Quantum จริง เพื่อ**สำรวจแต่ละหลักการด้วยตัวเอง**

**เกณฑ์ 5 ข้อของ DiVincenzo สำหรับการคำนวณเชิงควอนตัม**:

1. **ระบบทางกายภาพที่ขยายขนาดได้พร้อม Qubit ที่มีลักษณะเฉพาะชัดเจน**
2. **ความสามารถในการกำหนดค่าเริ่มต้นของ Qubit** ให้อยู่ในสถานะอ้างอิงง่าย ๆ (เช่น |00…0〉)
3. **เวลา Decoherence ที่ยาวนาน** (ความสอดคล้องของ Qubit นานกว่าเวลาดำเนินการของ Gate มาก)
4. **ชุด Quantum Gate สากล** (สามารถดำเนินการ Unitary ใด ๆ ก็ได้)
5. **ความสามารถในการวัดค่าเฉพาะ Qubit** (อ่านสถานะของแต่ละ Qubit)

## 1. เกณฑ์ 1 – Qubit ที่ขยายขนาดได้และมีลักษณะเฉพาะชัดเจน {#1-criterion-1--scalable-well-characterized-qubits}

**เกณฑ์ 1:** *"ระบบทางกายภาพที่ขยายขนาดได้พร้อม Qubit ที่มีลักษณะเฉพาะชัดเจน"* หมายความว่าเราต้องการแพลตฟอร์มฮาร์ดแวร์ควอนตัมที่ **เพิ่มจำนวน Qubit** ได้และยังควบคุมได้อย่างน่าเชื่อถือ คุณสมบัติของ Qubit แต่ละตัว (ระดับพลังงาน อัตราความผิดพลาด การเชื่อมต่อ ฯลฯ) ควรได้รับการเข้าใจอย่างดี โดยสรุปคือเราต้องการสร้าง Circuit ขนาดใหญ่ขึ้นโดยไม่ให้ระบบพังทลาย ในทางปฏิบัติ เมื่อเราเพิ่มจำนวน Qubit หรือความลึกของ Circuit ข้อผิดพลาดและ Decoherence จะสะสมมากขึ้น ดังนั้นการสาธิต *ความสามารถในการขยายขนาด* ยังหมายถึงการทำความเข้าใจว่าการเพิ่มขนาดส่งผลต่อประสิทธิภาพอย่างไร

**เป้าหมายของการสาธิต:** ใช้ Qiskit เพื่อแสดงผลของการขยายขนาด Circuit (ทั้งจำนวน Qubit และความลึกของ Gate) ต่อความแม่นยำของผลลัพธ์ เราจะจำลองสถานการณ์ในอุดมคติและมีสัญญาณรบกวนเพื่อดูว่าระบบขนาดใหญ่หรือ Circuit ที่ลึกกว่าถูก Decoherence และข้อผิดพลาดโจมตีอย่างไร

ก่อนอื่น มาสร้างสถานะ Entangle ขนาดเล็ก (สถานะ GHZ) บน Qubit 3 ตัว จากนั้นสร้างสถานะที่ใหญ่ขึ้นบน Qubit 5 ตัว เพื่อทดสอบการขยายขนาดอย่างง่าย สถานะ GHZ ของ Qubit จำนวน *n* ตัวคือ $\frac{1}{\sqrt{2}}(|0...0\rangle + |1...1\rangle)$ ในการจำลองในอุดมคติ การวัดสถานะ GHZ แบบ n-Qubit จะให้ผลลัพธ์เพียง 2 แบบ (ทั้งหมด 0 หรือทั้งหมด 1) ด้วยความน่าจะเป็นเท่ากัน เราจะเปรียบเทียบ **ผลลัพธ์ในอุดมคติ** กับ **ผลลัพธ์ที่มีสัญญาณรบกวน** เมื่อเพิ่ม n หรือความลึกของ Circuit

```python
from qiskit import QuantumCircuit
from qiskit_aer import AerSimulator
from qiskit.visualization import plot_histogram
from qiskit.transpiler.preset_passmanagers import generate_preset_pass_manager
from qiskit_ibm_runtime import SamplerV2 as Sampler

# 3-qubit GHZ circuit
qc3 = QuantumCircuit(3, 3)
qc3.h(0)
qc3.cx(0, 1)
qc3.cx(1, 2)
qc3.measure([0, 1, 2], [0, 1, 2])

# 5-qubit GHZ circuit (scaling up the number of qubits)
qc5 = QuantumCircuit(5, 5)
qc5.h(0)
qc5.cx(0, range(1, 5))    # entangle qubit 0 with all others
qc5.measure(range(5), range(5))

# Transpile for a simulator backend
sim_backend = AerSimulator()
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc3 = pm.run(qc3)
isa_qc5 = pm.run(qc5)

# Run ideal simulations (no noise)
sampler = Sampler(mode=sim_backend)

job3 = sampler.run([isa_qc3], shots=1024)
result3 = job3.result()
counts3 = result3[0].data.c.get_counts()

job5 = sampler.run([isa_qc5], shots=1024)
result5 = job5.result()
counts5 = result5[0].data.c.get_counts()

print("3-qubit GHZ counts (ideal):", counts3)
plot_histogram(counts3, legend=['3-qubit ideal'], figsize=(6,4))
```

```python
print("5-qubit GHZ counts (ideal):", counts5)
plot_histogram(counts5, legend=['5-qubit ideal'], figsize=(6,4))
```

**ผลลัพธ์ที่คาดหวัง (กรณีในอุดมคติ):** สถานะ GHZ แบบ 3-Qubit ในอุดมคติจะให้ผล `000` ประมาณ 50% และ `111` ประมาณ 50% สถานะ GHZ แบบ 5-Qubit จะให้ผล `00000` ประมาณ 50% และ `11111` ประมาณ 50% ไม่มี Bit-string อื่นปรากฏ เพราะสถานะนั้น Coherent และ Entangle อย่างสมบูรณ์ในอุดมคติ คุณจะเห็นแท่งสูง 2 แท่งบน Histogram สำหรับแต่ละ Circuit ที่สอดคล้องกับผลลัพธ์ทั้งศูนย์และทั้งหนึ่ง

ต่อไป มาดูว่าเกิดอะไรขึ้นใน**สภาพแวดล้อมที่มีสัญญาณรบกวน** เราจะใช้ความสามารถ Noise Model ของ Qiskit Aer เพื่อจำลองข้อผิดพลาดของอุปกรณ์จริง ตัวอย่างเช่น เราสามารถนำคุณสมบัติของ IBM Backend มาสร้าง Noise Model ที่รวมถึงข้อผิดพลาดของ Gate เวลา Gate จำกัด การคลายตัวของ Qubit (T₁) การสูญเสีย Phase (T₂) และข้อผิดพลาดในการอ่านค่า ที่นี่เราจะใช้ **Fake Backend** ที่แทน IBM Quantum Brisbane เพื่อสร้าง Noise Model แล้วรัน Circuit GHZ ผ่านมันอีกครั้ง

### แบบฝึกหัด 1a: จำลองด้วยสัญญาณรบกวน {#exercise-1a-simulate-with-noise}

เติมโค้ดด้านล่างเพื่อจำลอง Circuit GHZ บน Simulator ที่มีสัญญาณรบกวนโดยอิงจาก Backend `FakeBrisbane` สิ่งนี้จะแสดงให้เห็นว่าประสิทธิภาพลดลงอย่างไรเมื่อระบบขยายขนาดในสภาพแวดล้อมที่มีสัญญาณรบกวนจริง

```python

counts3_noisy = result3_noisy[0].data.meas.get_counts()

result5_noisy = job5.result()

counts5_noisy = result5_noisy[0].data.meas.get_counts()

# --- END YOUR CODE ---
```

**ผลลัพธ์ที่คาดหวัง (มีสัญญาณรบกวน vs ในอุดมคติ):** เมื่อมีสัญญาณรบกวน ไม่ว่าจะเป็นการจำลองหรือบนอุปกรณ์จริง สถานะ GHZ จะ**ไม่สมบูรณ์แบบ** คุณจะเห็นผลลัพธ์เพิ่มเติมนอกเหนือจากทั้งหมด 0 และทั้งหมด 1 สำหรับ Qubit 3 ตัว แทนที่จะได้ 100% ใน `000`/`111` ความน่าจะเป็นบางส่วนจะรั่วไปยัง Bit-string อื่น (เช่น `001`, `010` เป็นต้น) เนื่องจากข้อผิดพลาดของ Gate หรือ Decoherence ที่พลิก Qubit บางตัว สำหรับ Qubit 5 ตัว ผลกระทบยิ่งชัดเจนขึ้น Circuit ที่ใหญ่กว่า (Qubit และ CNOT Gate มากกว่า) สะสมข้อผิดพลาดมากขึ้น จึงทำให้ยอดทั้งหมด 0 และทั้งหมด 1 ต่ำลง และผลลัพธ์อื่น ๆ อีกมากมายปรากฏขึ้น แนวโน้มนี้แสดงให้เห็นถึงความท้าทายของ *ความสามารถในการขยายขนาด*: เมื่อเราขยายใหญ่ขึ้น การรักษาความแม่นยำสูงจะยากขึ้นโดยไม่มีการแก้ไขข้อผิดพลาด

**ข้อสังเกต:** คอมพิวเตอร์ควอนตัมที่ขยายขนาดได้ต้องรักษาความสัมพันธ์เชิงควอนตัมเมื่อระบบเติบโตขึ้น ตัวอย่างของเราแสดงให้เห็นว่าการเพิ่มจำนวน Qubit/ความลึกของ Gate ทำให้ความแม่นยำของผลลัพธ์ลดลงเมื่อมีสัญญาณรบกวน เกณฑ์ที่เหลือจะจัดการกับการทำให้ Qubit เหล่านั้นทำงานได้ดี (ข้อผิดพลาดน้อย กำหนดค่าเริ่มต้นได้ ฯลฯ) เมื่อเราขยายขนาด
## 2. เกณฑ์ 2 – การกำหนดค่าเริ่มต้นของ Qubit {#2-criterion-2--qubit-initialization}

**เกณฑ์ 2:** *"ความสามารถในการกำหนดค่าเริ่มต้นสถานะของ Qubit ให้เป็นสถานะอ้างอิงง่าย ๆ เช่น |000…〉"* Qubit ทั้งหมดควรเริ่มต้นอย่างน่าเชื่อถือในสถานะอ้างอิงที่รู้จัก (โดยทั่วไปคือสถานะพื้นฐาน |0〉 สำหรับแต่ละ Qubit) การกำหนดค่าเริ่มต้นมีความสำคัญเพื่อให้อัลกอริทึมเริ่มต้นบนสถานะที่สะอาด ในทางปฏิบัติ บนอุปกรณ์ IBM Quantum แต่ละ Qubit จะถูกรีเซ็ตเป็น |0〉 โดยอัตโนมัติที่จุดเริ่มต้นของการรัน Circuit แต่ละครั้ง Qiskit ยังมีคำสั่งในการรีเซ็ต Qubit หรือเตรียมสถานะที่กำหนดเองระหว่างการคำนวณด้วย

**เป้าหมายของการสาธิต:** แสดงวิธีกำหนดค่าเริ่มต้นของ Qubit ใน Qiskit ทั้งตอนเริ่มต้นและระหว่าง Circuit เราจะสาธิตการใช้คำสั่ง `reset` และวิธีเตรียมสถานะ

### แบบฝึกหัด 2: เตรียมสถานะที่เฉพาะเจาะจง {#exercise-2-prepare-a-specific-state}

ในบล็อกโค้ดด้านล่าง เติม `QuantumCircuit` เพื่อเตรียมสถานะ $|10\rangle$ ซึ่งหมายความว่า Qubit 0 ควรอยู่ในสถานะ $|0\rangle$ และ Qubit 1 ควรอยู่ในสถานะ $|1\rangle$ ใช้ Gate และคำสั่งที่เหมาะสมเพื่อให้บรรลุสิ่งนี้

```python
from qiskit import QuantumCircuit
from qiskit_aer import AerSimulator

# Create a circuit to initialize qubits to |10> and verify by measurement
qc_init = QuantumCircuit(2, 2)

# --- YOUR CODE HERE ---

# 1. Set qubit 1 to the |1> state
qc_init.x(1)

# 2. Explicitly reset qubit 0 to the |0> state
qc_init.reset(0)

# --- END YOUR CODE ---

qc_init.measure([0, 1], [0, 1])
qc_init.draw('mpl')
```

```python
# Run the circuit and check the outcome
sim_backend = AerSimulator()
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc_init = pm.run(qc_init)

sampler = Sampler(mode=sim_backend)

job = sampler.run([isa_qc_init], shots=1024)
result = job.result()
counts = result[0].data.c.get_counts()

print("Outcome of |10> state measured in Z-basis:", counts)
plot_histogram(counts)
```

คุณควรเห็น `10` (ไบนารีสำหรับ qubit1=1, qubit0=0) ด้วยความน่าจะเป็น 100% จากการจำลอง หมายความว่า Qubit 1 ถูกเตรียมไว้ใน |1〉 และ Qubit 0 ใน |0〉 สำเร็จแล้ว

สำหรับการเตรียมสถานะทั่วไปมากขึ้น Qiskit อนุญาตให้กำหนดค่าเริ่มต้นเป็นสถานะใด ๆ ก็ได้โดยใช้เมธอด `initialize` ตัวอย่างเช่น มาเตรียม Qubit ในสถานะ $|+\rangle = (|0\rangle+|1\rangle)/\sqrt{2}$ ซึ่งเป็นสถานะ Superposition และคู่ Qubit ในสถานะ Bell $(|00\rangle+|11\rangle)/\sqrt{2}$:

```python
import numpy as np

# Initialize a single qubit in |+> state and measure in Z-basis
qc_plus = QuantumCircuit(1, 1)
state_plus = [1/np.sqrt(2), 1/np.sqrt(2)]   # amplitude for |0> and |1>
qc_plus.initialize(state_plus, 0)
qc_plus.measure(0, 0)

# Initialize two qubits in a Bell state manually
qc_bell = QuantumCircuit(2, 2)
bell_state = [1/np.sqrt(2), 0, 0, 1/np.sqrt(2)]  # amplitudes for |00>,|01>,|10>,|11>
qc_bell.initialize(bell_state, [0, 1])
qc_bell.measure([0, 1], [0, 1])

# Transpile and run the initialization circuits
isa_qc_plus = pm.run(qc_plus)
job_plus = sampler.run([isa_qc_plus], shots=1024)
result_plus = job_plus.result()
counts_plus = result_plus[0].data.c.get_counts()

print("Outcome of |+> state measured in Z-basis:", counts_plus)

isa_qc_bell = pm.run(qc_bell)
job_bell = sampler.run([isa_qc_bell], shots=1024)
result_bell = job_bell.result()
counts_bell = result_bell[0].data.c.get_counts()

print("Outcome of Bell state measured in Z-basis:", counts_bell)
```

**ผลลัพธ์ที่คาดหวัง:** สถานะ |+〉 ของ Qubit ตัวเดียว เมื่อวัด จะให้ `0` และ `1` ด้วยความน่าจะเป็นประมาณ 50% ต่อกัน การวัดสถานะ Bell ควรให้ประมาณ 50% `00` และ 50% `11` ถ้าคุณเห็นสิ่งเหล่านี้ แสดงว่าการกำหนดค่าเริ่มต้นเป็นสถานะดังกล่าวสำเร็จแล้ว

**การกำหนดค่าเริ่มต้นระหว่าง Circuit:** `reset` ของ Qiskit สามารถใช้กลางระหว่าง Circuit เพื่อกำหนดค่าเริ่มต้นของ Qubit เป็น |0〉 ได้ทันที ตัวอย่างเช่น ในโค้ดแก้ไขข้อผิดพลาดหรืออัลกอริทึมแบบวนซ้ำ บ่อยครั้งที่เราวัด Qubit แล้วรีเซ็ตเพื่อนำกลับมาใช้ใหม่ การดำเนินการ `reset` เป็นแบบ Deterministic คือมันทิ้งสถานะที่มีอยู่และทำให้ Qubit กลับสู่สถานะพื้นฐาน

**ตัวอย่างอุปกรณ์:** บนฮาร์ดแวร์อย่าง **ibmq_brisbane** (127 Qubit) หรืออุปกรณ์ IBM ใด ๆ Qubit ทั้งหมดเริ่มต้นใน |0〉 โดยค่าเริ่มต้นเมื่อรัน Job ถ้าต้องการสถานะเริ่มต้นที่ต่างออกไป ก็ต้องใช้ Gate ที่ตอนต้น (อย่างที่เราทำกับ X เพื่อได้ |1〉) การกำหนดค่าเริ่มต้นใหม่อย่างต่อเนื่อง (สำหรับการแก้ไขข้อผิดพลาดเชิงควอนตัม) เป็นหัวข้อวิจัยเชิงรุก เพราะการทำได้อย่างรวดเร็วนั้นยาก แต่โชคดีที่สำหรับการใช้งานพื้นฐาน ความสามารถในการเริ่มต้นใหม่ใน |0…0〉 มีให้ใช้ และเราได้สาธิตวิธีบรรลุสถานะเริ่มต้นที่ต้องการอื่น ๆ ด้วยแล้ว
## 3. เกณฑ์ที่ 3 – เวลาโคฮีเรนซ์ที่ยาวนาน (การสูญเสียโคฮีเรนซ์ vs เวลาของ Gate) {#3-criterion-3-long-coherence-time-decoherence-vs-gate-time}

**เกณฑ์ที่ 3:** *"เวลาการสูญเสียโคฮีเรนซ์ที่เกี่ยวข้องต้องยาวนานกว่าเวลาปฏิบัติการของ Gate มาก"* ข้อนี้เน้นว่า Qubit ต้องรักษาสถานะควอนตัมได้นานพอที่จะทำการปฏิบัติการต่าง ๆ ได้ครบ แต่ละ Qubit มี **เวลา T₁** (เวลาการผ่อนคลายพลังงาน คือความเร็วที่ |1〉 สลายกลับเป็น |0〉) และ **เวลา T₂** (เวลาการสูญเสียเฟส คือความเร็วที่โคฮีเรนซ์ของเฟสสัมพัทธ์หายไป) เพื่อให้คอมพิวเตอร์ควอนตัมทำงานได้ ช่วงเวลาเหล่านี้ต้องนานกว่าระยะเวลาของการปฏิบัติการ Gate อย่างมาก

**เป้าหมายของการสาธิต:** ศึกษาโคฮีเรนซ์ของ Qubit ใน Qiskit โดยแสดงให้เห็นว่าการสูญเสียโคฮีเรนซ์ส่งผลต่อผลลัพธ์ของ Circuit อย่างไรเมื่อความยาวของการประมวลผลเพิ่มขึ้น เราจะใช้ fake backend ที่มีค่า T1/T2 ที่ทราบแล้วมาจำลองผลกระทบนี้

เพื่อ **แสดงผลกระทบของโคฮีเรนซ์ที่จำกัด** เราจะจำลองการทดลองการสลายตัวแบบ T1 โดยจะเตรียม Qubit ในสถานะ |1〉 แล้วรอด้วยคำสั่ง `delay` สักช่วงหนึ่ง แล้วจึงวัด เราคาดว่าความน่าจะเป็นของการวัดได้ |1〉 จะลดลงเมื่อเวลา delay เพิ่มขึ้น

```python
# This part is done for you. We are creating a list of circuits,
# each with a different delay time.

time_delays_ns = [0, 50000, 100000, 150000, 200000, 250000, 300000]  # delay durations in ns

decay_expts = []
for delay in time_delays_ns:
    qc = QuantumCircuit(1, 1)
    qc.x(0)  # initialize qubit to |1>
    if delay > 0:
        qc.delay(delay, 0, unit='ns')  # wait 'delay' nanoseconds
    qc.measure(0, 0)
    decay_expts.append(qc)

decay_expts[1].draw('mpl') # Visualize one of the circuits
```

## 4. เกณฑ์ที่ 4 – ชุด Gate ควอนตัมแบบสากล {#4-criterion-4-universal-set-of-quantum-gates}

**เกณฑ์ที่ 4:** *"ชุด Gate ควอนตัมแบบ 'สากล'"* นั่นหมายความว่าฮาร์ดแวร์ของเราต้องรองรับการทำ *การคำนวณควอนตัมใด ๆ ก็ได้* โดยการประกอบชุด Gate พื้นฐานจำนวนจำกัด ในการคำนวณแบบคลาสสิก NAND เป็น Gate สากล ส่วนในควอนตัมมีตัวเลือกหลายชุด เช่น \{H, T, CNOT\} หรือ Gate ที่เครื่องนั้น ๆ รองรับ อุปกรณ์ IBM ตัวอย่างเช่น มีชุดการปฏิบัติการพื้นฐานอย่างการหมุน single-qubit ตามอำเภอใจและ CNOT ระหว่าง Qubit บางคู่ ซึ่งรวมกันแล้วเป็นสากล งานของ Qiskit มักเป็นการ **คอมไพล์ Gate ระดับสูงให้เป็น basis gate เหล่านี้**

**เป้าหมายของการสาธิต:** แสดงความเป็นสากลของ Gate โดยดูว่า Qiskit แยกย่อย Gate อย่างไร เราจะนำ Gate ที่ไม่ใช่ native (เช่น Toffoli Gate 3 qubit หรือ CCX) มาดูว่ามันแยกย่อยลงไปเป็น basis gate ของอุปกรณ์อย่างไร ซึ่งแสดงให้เห็นว่าชุด Gate ที่มีอยู่นั้นเป็น *สากล* จริง – มันสามารถสร้างการปฏิบัติการที่ซับซ้อนกว่าได้

ก่อนอื่น ดูกันก่อนว่า basis gate ของ IBM backend ทั่วไปคืออะไร เราจะดึงข้อมูลคอนฟิกของอุปกรณ์ (หรือ fake version ของมัน) ตัวอย่างเช่น basis gate ของ ibmq_brisbane:
ควรสังเกตว่าความน่าจะเป็น `P(qubit=1)` ลดลงเมื่อเวลา delay เพิ่มขึ้น ตามรูปแบบการสลายตัวแบบเอกซ์โพเนนเชียลที่เป็นลักษณะเฉพาะของการผ่อนคลาย T1 สิ่งนี้แสดงให้เห็นโดยตรงว่าเวลาโคฮีเรนซ์ที่จำกัดนำไปสู่ข้อผิดพลาดในการคำนวณเมื่อ Circuit รันนานเกินไป

**ผลกระทบต่ออัลกอริทึม:** ถ้าลองใช้อัลกอริทึมที่ยาวขึ้น (มี Gate ต่อเนื่องหลายตัว) เวลาการประมวลผลรวมอาจเข้าใกล้หรือเกิน T2 ทำให้สถานะสูญเสียโคฮีเรนซ์ก่อนจบ นี่คือสาเหตุที่การปรับปรุงเวลาโคฮีเรนซ์และทำให้ Gate ทำงานเร็วขึ้นเป็นสองเป้าหมายสำคัญที่สุดในการวิจัยฮาร์ดแวร์ควอนตัม

```python
from qiskit_ibm_runtime.fake_provider import FakeBrisbane
fake_brisbane = FakeBrisbane()
print("Basis gates for ibmq_brisbane:", fake_brisbane.configuration().basis_gates)
```

ผลลัพธ์อาจเป็นอะไรทำนองนี้ `['id', 'rz', 'sx', 'x', 'ecr']` นี่คือการปฏิบัติการพื้นฐานที่ฮาร์ดแวร์รองรับโดยตรง (Identity/no-op, การหมุน RZ, Gate sqrt(X), Gate X, และ controlled-X) Gate อื่น ๆ ทั้งหมดต้องประกอบจากชุดนี้ ชุดนี้เป็นที่รู้กันว่าเป็นสากลสำหรับการคำนวณควอนตัม (โดยพื้นฐานแล้ว การหมุน single-qubit บวกกับ Gate เชื่อมโยง two-qubit ก็เป็นชุดสากล)

ทีนี้ ใช้ **Toffoli Gate (CCX)** เป็นกรณีทดสอบ CCX พลิก Qubit เป้าหมายก็ต่อเมื่อ Qubit ควบคุม 2 ตัวเป็น 1 ทั้งคู่ มันไม่ใช่ Gate ที่ฮาร์ดแวร์ IBM รองรับโดยตรง Qiskit มีคำสั่ง `ccx` แต่เบื้องหลังจะแยกย่อยมัน
### แบบฝึกหัดที่ 4: แยกย่อย Toffoli Gate {#exercise-4-decompose-a-toffoli-gate}

เขียนโค้ดด้านล่างให้ครบเพื่อสร้าง Circuit ที่มี Toffoli Gate (CCX) แล้วใช้ Qiskit แยกย่อยมันลงเป็น basis gate ของ `FakeBrisbane` backend

```python
from qiskit import QuantumCircuit
from qiskit_ibm_runtime.fake_provider import FakeBrisbane

# The fake_brisbane backend from the previous cell is reused here.

# --- YOUR CODE HERE ---

# 1. Create a circuit that can accommodate a Toffoli gate
qc_toffoli = QuantumCircuit(3)

# Apply a CCX gate with controls on qubits 0, 1 and target on qubit 2
qc_toffoli.ccx(0, 1, 2)

# 2. Transpile the circuit to the fake Brisbane backend
pm = generate_preset_pass_manager(backend=FakeBrisbane(), optimization_level=1)

isa_qc_toffoli = pm.run(qc_toffoli)

# --- END YOUR CODE ---

print("Toffoli circuit before decomposition:")
print(qc_toffoli)

print("\nToffoli circuit after transpiling to Brisbane basis:")
# The .draw() method will now show the decomposed circuit
print(isa_qc_toffoli.draw(fold=120))
```

ใน output ที่ถูก Transpile แล้ว จะเห็นว่า CCX ถูกแทนที่ด้วยลำดับ Gate พื้นฐานอย่าง `rz`, `sx`, และ `ecr` ซึ่งพิสูจน์ว่า native gate ก็เพียงพอที่จะแสดง Toffoli ได้

**ความเป็นสากลในทางปฏิบัติ:** แบบฝึกหัดข้างต้นแสดงให้เห็นว่า Gate 3 Qubit ที่ซับซ้อนถูกสร้างจาก Gate ที่ง่ายกว่า โดยทั่วไป **ทุก** unitary หลาย Qubit สามารถประกอบจาก Gate 1 และ 2 Qubit ได้ Transpiler เป็นส่วนประกอบสำคัญของ software stack ควอนตัม เพราะมันเป็นสะพานเชื่อมระหว่างอัลกอริทึมที่เป็นนามธรรมกับการปฏิบัติการทางกายภาพที่อุปกรณ์ควอนตัมจำเพาะสามารถทำได้จริง

**Device example:** อุปกรณ์ **ibmq_brisbane** ใช้สถาปัตยกรรม Eagle พร้อม basis gate ที่แสดงไว้ข้างต้น ซึ่งหมายความว่าทุกอัลกอริทึมที่ส่งไปยังเครื่องเหล่านั้นจะถูกแปลงเป็นลำดับของการปฏิบัติการเหล่านั้น เกณฑ์นี้โดยพื้นฐานเกี่ยวกับ **การควบคุมได้** คือเรามีปุ่มควบคุมเพียงพอที่จะทำการปฏิบัติการใด ๆ ที่จำเป็นบน Qubit ของเราได้
## 5. เกณฑ์ที่ 5 – การวัด Qubit {#5-criterion-5-qubit-measurement}

**เกณฑ์ที่ 5:** *"ความสามารถในการวัด Qubit แบบเฉพาะเจาะจง"* ต้องสามารถวัดสถานะของ Qubit แต่ละตัวได้ (โดยทั่วไปในฐาน computational คือ |0〉 หรือ |1〉) กล่าวคือหลังจากรัน Circuit ควอนตัมแล้ว เราต้องอ่านค่า Qubit แต่ละตัวเป็น bit คลาสสิก 0/1 ได้ เกณฑ์นี้เกี่ยวกับการมีตัวตรวจจับที่เชื่อถือได้สำหรับ Qubit แต่ละตัวและสามารถเลือกว่าจะวัด Qubit ตัวไหน

**เป้าหมายของการสาธิต:** แสดงวิธีการวัดใน Qiskit บน simulator และอุปกรณ์จริง พร้อมเน้นความแตกต่าง (เช่น noise จากการวัด) เราจะวัด Qubit บางตัวในสถานะต่าง ๆ และตรวจสอบผลลัพธ์ รวมถึงสาธิตว่า readout error อาจปรากฏอย่างไรโดยเปรียบเทียบผลลัพธ์จาก simulator กับฮาร์ดแวร์

ก่อนอื่น ตัวอย่างการวัดง่าย ๆ:

```python
qc_measure = QuantumCircuit(2, 2)
qc_measure.x(0)              # qubit 0 -> |1>, qubit 1 stays |0>
qc_measure.measure([0, 1], [0, 1])
qc_measure.draw('mpl')
```

```python
sim_backend = AerSimulator()
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc_measure = pm.run(qc_measure)
job = sampler.run([isa_qc_measure], shots=1000)
result = job.result()
counts = result[0].data.c.get_counts()

print("Simulator measurement counts:", counts)
```

เราคาดว่าจะได้ 1000 counts ของ `01` บน simulator ทีนี้มาดู **measurement error** กัน โดยจำลองมัน เราสามารถเพิ่ม readout error ใน Aer simulator ของเราได้ Qiskit Aer ให้เราสามารถนิยาม `ReadoutError` และผูกมันกับ Qubit ใน noise model ได้
### แบบฝึกหัดที่ 5: จำลอง Readout Error {#exercise-5-simulate-readout-error}

เขียนโค้ดให้ครบเพื่อนิยาม readout error model แบบง่ายที่แต่ละ Qubit มีโอกาส 2% ที่จะถูกวัดผิด (0 ถูกอ่านเป็น 1 หรือ 1 ถูกอ่านเป็น 0) แล้วรัน measurement circuit พร้อม noise model นี้

```python
from qiskit_aer.noise import NoiseModel, ReadoutError

# --- YOUR CODE HERE ---

# 1. Define a 2% readout error for each single qubit.
# [[P(0|0), P(1|0)], [P(0|1), P(1|1)]]
ro_error = ReadoutError([[0.98, 0.02], [0.02, 0.98]])

# 2. Create a new noise model
noise_model_ro = NoiseModel()

# 3. Add the readout error to all qubits in the noise model
noise_model_ro.add_all_qubit_readout_error(ro_error)

# --- END YOUR CODE ---

sim_backend.set_options(noise_model=noise_model_ro)
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc_measure = pm.run(qc_measure)

# Run the measurement circuit with readout noise
sampler = Sampler(mode=sim_backend)

job = sampler.run([isa_qc_measure], shots=1024)
result = job.result()
counts = result[0].data.c.get_counts()

print("Simulation with 2% readout error:", counts)
```

output ที่จำลองนี้จะแสดง counts ที่ผิดพลาดบ้าง (เช่น `11`, `00`, `10`) คล้ายกับที่ฮาร์ดแวร์จริงอาจสร้างขึ้น ซึ่งแสดงผลกระทบของการวัดที่ไม่สมบูรณ์

**Device example:** บนอุปกรณ์จริงอย่าง **ibmq_brisbane** รัน Circuit เดียวกันก็จะเห็น counts ที่ไม่ใช่ศูนย์สำหรับผลลัพธ์ที่ผิดเช่นกัน ข้อมูลการคาลิเบรท (calibration) ของอุปกรณ์จะระบุ readout error ของแต่ละ Qubit การสามารถเลือกเป้าหมายและอ่านค่า Qubit เฉพาะตัวได้นั้นสำคัญมาก และการเข้าใจลักษณะ error ของมันเป็นกุญแจสู่การได้ผลลัพธ์ที่มีความหมาย การรันบนอุปกรณ์จริงได้แสดงไว้แล้วใน **แบบฝึกหัด 1b: รันบนคอมพิวเตอร์ IBM Quantum จริง**
## เกณฑ์การสื่อสารควอนตัม (Flying Qubit) {#quantum-communication-criteria-flying-qubits}

DiVincenzo ยังระบุเกณฑ์อีก 2 ข้อที่เฉพาะสำหรับการสื่อสารควอนตัม ซึ่งสำคัญในการสร้างคอมพิวเตอร์ควอนตัมแบบเครือข่าย:

6. **ความสามารถในการแปลงสภาพระหว่าง stationary qubit และ flying qubit** (เช่น แมป Qubit ในโปรเซสเซอร์ไปเป็นโฟตอนที่เดินทางได้)
7. **ความสามารถในการส่ง flying qubit ระหว่างตำแหน่งต่าง ๆ อย่างซื่อสัตย์** (เช่น ส่งโฟตอน Qubit ผ่านไฟเบอร์โดยไม่สูญเสียข้อมูลควอนตัม)

สิ่งเหล่านี้อยู่นอกเหนือการใช้งาน Qiskit มาตรฐาน เพราะ Qiskit หลักใช้กับ stationary qubit บนชิป อย่างไรก็ตาม เราสามารถอธิบาย *แนวคิด* ของเกณฑ์เหล่านี้ด้วยตัวอย่างง่าย ๆ: **quantum teleportation** การ Teleportation แสดงการแปลงสถานะของ stationary qubit เป็นข้อมูลที่ส่งผ่านคู่ที่พัวพันกัน (ส่วน "flying") และการสื่อสารแบบคลาสสิก ซึ่งนำไปใช้สร้างสถานะใหม่บน stationary qubit อีกตัวที่อื่น
### กิจกรรมแนะนำ: ทำ Module *Quantum Teleportation* ของ Qiskit in Classrooms {#recommended-activity-take-the-qiskit-in-classrooms-quantum-teleportation-module}

Module [Quantum Teleportation](https://quantum.cloud.ibm.com/learning/en/modules/computer-science/quantum-teleportation) ของ Qiskit in Classrooms โดย Dr. Katie McCormick จะพาคุณสำรวจหนึ่งในโปรโตคอลที่น่าตื่นเต้นที่สุดในสารสนเทศควอนตัม: quantum teleportation ที่สถานะควอนตัม (Qubit) ถูกส่งจาก Alice ไปยัง Bob โดยใช้ entanglement และ classical bit เพียง 2 บิต คุณจะเรียนรู้ขั้นตอน teleportation ทั้งหมดทีละขั้น — วิธีเตรียมคู่ Bell ที่พัวพันกัน ทำการวัดในฐาน Bell ฝั่ง Alice ส่งผลลัพธ์คลาสสิก และใช้ Gate ควอนตัมที่ถูกต้องกับ Qubit ของ Bob เพื่อกู้คืนสถานะต้นฉบับ ระหว่างทางคุณจะสำรวจว่าทำไมการ teleport ข้อมูลของ Qubit ถึงไม่ละเมิดทฤษฎีบทห้ามโคลนหรือไม่ได้เร็วกว่าแสง ผ่านแบบฝึกหัดจริงด้วยฮาร์ดแวร์ IBM Quantum หรือ simulator คุณจะเข้าใจเชิงปฏิบัติเกี่ยวกับการวัด, entanglement, และการควบคุมแบบ feed-forward

การเชี่ยวชาญ quantum teleportation จะทำให้เข้าใจวิธีการเข้ารหัส ส่ง และกู้คืนสารสนเทศควอนตัมระหว่างโหนดที่แตกต่างกัน — ซึ่งเป็นรากฐานสำหรับเครือข่ายควอนตัม, ระบบ repeater, แผนการสื่อสารที่ปลอดภัย, และการคำนวณควอนตัมแบบโมดูลาร์ที่ขยายได้
**ความสัมพันธ์กับเกณฑ์ 6 และ 7:** ในเครือข่ายควอนตัมจริง คู่ entangle ที่ใช้ร่วมกันจะสร้างขึ้นโดยการกระจาย "flying" qubit (เช่น โฟตอน) ระหว่างตำแหน่งของ Alice และ Bob (เกณฑ์ 7: การส่งที่ซื่อสัตย์) จากนั้นโปรโตคอล teleportation เองจะเป็นวิธีการแมปสถานะ stationary qubit ของ Alice ไปยังครึ่งหนึ่งของคู่ entangle ของเธอ เพื่อ 'ส่ง' ไปยัง Bob (เกณฑ์ 6: การแปลงสภาพ) Qiskit ช่วยให้เราจำลองตรรกะของโปรโตคอลได้อย่างสมบูรณ์แบบ ทำให้มีแบบจำลองแนวคิดว่าเกณฑ์เหล่านี้ถูกบรรลุในสถาปัตยกรรมการสื่อสารอย่างไร
## บทสรุป {#conclusion-summary}

เราได้ออกแบบชุดแบบฝึกหัดที่เน้นโค้ดเพื่ออธิบายเกณฑ์ DiVincenzo โดยใช้ Qiskit ผ่านตัวอย่างจริงเหล่านี้ คุณได้สำรวจว่าแพลตฟอร์มการคำนวณควอนตัมจริงบรรลุแต่ละข้อกำหนดอย่างไร:

- **ความสามารถในการขยาย**: สร้าง Circuit บน Qubit จำนวนมากขึ้นและเข้าใจการขยายของ noise
- **การเริ่มต้น**: ใช้ reset และการเตรียมสถานะเพื่อเริ่มต้นการคำนวณในสถานะที่รู้จักอย่างน่าเชื่อถือ
- **Gate สากล**: Transpile การปฏิบัติการที่ซับซ้อนให้เป็น basis gate ของเครื่อง ซึ่งพิสูจน์ว่าเราสามารถทำการคำนวณใด ๆ ได้
- **การวัด**: อ่านค่า Qubit และจัดการกับ readout error ที่สมจริง
- **โคฮีเรนซ์**: เห็นผลกระทบของ T₁ และ T₂ ที่จำกัดต่อความเที่ยงตรงของอัลกอริทึม และความจำเป็นที่การปฏิบัติการต้องเร็วกว่าการสูญเสียโคฮีเรนซ์
