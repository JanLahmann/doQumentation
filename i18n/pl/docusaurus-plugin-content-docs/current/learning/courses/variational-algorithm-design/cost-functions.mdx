## Prymitywy {#primitives}

Wszystkie układy fizyczne, zarówno klasyczne, jak i kwantowe, mogą istnieć w różnych stanach. Na przykład samochód na drodze może mieć określoną masę, położenie, prędkość lub przyspieszenie, które charakteryzują jego stan. Podobnie układy kwantowe również mogą mieć różne konfiguracje lub stany, ale różnią się one od układów klasycznych tym, jak traktujemy pomiary i ewolucję stanu. Prowadzi to do unikalnych właściwości, takich jak *superpozycja* i *splątanie*, które są wyłączne dla mechaniki kwantowej. Podobnie jak możemy opisać stan samochodu przy użyciu właściwości fizycznych, takich jak prędkość czy przyspieszenie, tak samo możemy opisać stan układu kwantowego za pomocą *obserwabli*, które są obiektami matematycznymi.

W mechanice kwantowej stany są reprezentowane przez znormalizowane zespolone wektory kolumnowe, czyli *kety* ($|\psi\rangle$), a obserwable są hermitowskimi operatorami liniowymi ($\hat{H}=\hat{H}^{\dagger}$), które działają na ketach. Wektor własny ($|\lambda\rangle$) obserwabli jest nazywany *stanem własnym*. Pomiar obserwabli dla jednego z jej stanów własnych ($|\lambda\rangle$) da nam odpowiadającą wartość własną ($\lambda$) jako odczyt.

Jeśli zastanawiasz się, jak mierzyć układ kwantowy i co można mierzyć, Qiskit oferuje dwa <DefinitionTooltip definition="Podstawowe i fundamentalne operacje lub typy danych. Qiskit posiada prymitywy Sampler i Estimator, które służą jako bloki konstrukcyjne do łatwego budowania złożonych obciążeń algorytmicznych.">prymitywy</DefinitionTooltip>, które mogą pomóc:

* `Sampler`: Mając dany stan kwantowy $|\psi\rangle$, ten prymityw uzyskuje prawdopodobieństwo każdego możliwego stanu bazy obliczeniowej.
* `Estimator`: Mając daną obserwablę kwantową $\hat{H}$ i stan $|\psi\rangle$, ten prymityw oblicza wartość oczekiwaną $\hat{H}$.
### Prymityw Sampler {#the-sampler-primitive}

Prymityw `Sampler` oblicza prawdopodobieństwo uzyskania każdego możliwego stanu $|k\rangle$ z bazy obliczeniowej, mając dany obwód kwantowy, który przygotowuje stan $|\psi\rangle$. Oblicza on

$$
p_k = |\langle k | \psi \rangle|^2 \quad \forall k \in \mathbb{Z}_2^n \equiv \{0,1,\cdots,2^n-1\},
$$

Gdzie $n$ to liczba kubitów, a $k$ to całkowita reprezentacja dowolnego możliwego wyjściowego ciągu binarnego $\{0,1\}^n$ (czyli liczby całkowite w bazie $2$).

Qiskit Runtime [`Sampler`](https://docs.quantum.ibm.com/api/qiskit-ibm-runtime/sampler-v2) wykonuje obwód wielokrotnie na urządzeniu kwantowym, przeprowadzając pomiary przy każdym uruchomieniu, i rekonstruuje rozkład prawdopodobieństwa z odzyskanych ciągów bitów. Im więcej uruchomień (lub *strzałów*) wykonuje, tym dokładniejsze będą wyniki, ale wymaga to więcej czasu i zasobów kwantowych.

Jednakże, ponieważ liczba możliwych wyników rośnie wykładniczo wraz z liczbą kubitów $n$ (czyli $2^n$), liczba strzałów również będzie musiała rosnąć wykładniczo, aby uchwycić _gęsty_ rozkład prawdopodobieństwa. Dlatego `Sampler` jest wydajny tylko dla *rzadkich* rozkładów prawdopodobieństwa; gdzie stan docelowy $|\psi\rangle$ musi być wyrażalny jako kombinacja liniowa stanów bazy obliczeniowej, przy czym liczba składników rośnie co najwyżej wielomianowo wraz z liczbą kubitów:

$$
|\psi\rangle = \sum^{\text{Poly}(n)}_k w_k |k\rangle.
$$

`Sampler` można również skonfigurować tak, aby pobierał prawdopodobieństwa z podsekcji obwodu, reprezentując podzbiór wszystkich możliwych stanów.
### Prymityw Estimator {#the-estimator-primitive}

Prymityw `Estimator` oblicza wartość oczekiwaną obserwabli $\hat{H}$ dla stanu kwantowego $|\psi\rangle$; gdzie prawdopodobieństwa obserwabli można wyrazić jako $p_\lambda = |\langle\lambda|\psi\rangle|^2$, przy czym $|\lambda\rangle$ to stany własne obserwabli $\hat{H}$. Wartość oczekiwana jest zatem zdefiniowana jako średnia wszystkich możliwych wyników $\lambda$ (czyli wartości własnych obserwabli) pomiaru stanu $|\psi\rangle$, ważona odpowiednimi prawdopodobieństwami:

$$
\langle\hat{H}\rangle_\psi := \sum_\lambda p_\lambda \lambda = \langle \psi | \hat{H} | \psi \rangle
$$

Jednak obliczenie wartości oczekiwanej obserwabli nie zawsze jest możliwe, ponieważ często nie znamy jej bazy własnej. Qiskit Runtime [`Estimator`](https://docs.quantum.ibm.com/api/qiskit-ibm-runtime/estimator-v2) wykorzystuje złożony proces algebraiczny do oszacowania wartości oczekiwanej na rzeczywistym urządzeniu kwantowym, rozkładając obserwablę na kombinację innych obserwabli, których bazę własną znamy.

Mówiąc prościej, `Estimator` rozkłada każdą obserwablę, której nie wie, jak zmierzyć, na prostsze, mierzalne obserwable zwane <DefinitionTooltip definition="Zbiór macierzy powszechnie używanych w obliczeniach kwantowych do reprezentowania i manipulowania stanami kwantowymi, składający się z macierzy identyczności i trzech macierzy Pauliego (X, Y i Z).">operatorami Pauli</DefinitionTooltip>.
Każdy operator można wyrazić jako kombinację $4^n$ operatorów Pauli.

$$
\hat{P}_k :=
\sigma_{k_{n-1}}\otimes \cdots \otimes \sigma_{k_0} \quad
\forall k \in \mathbb{Z}_4^n \equiv \{0,1,\cdots,4^n-1\}, \\
$$

tak, że

$$
\hat{H} = \sum^{4^n-1}_{k=0} w_k \hat{P}_k
$$

gdzie $n$ to liczba kubitów, $k \equiv k_{n-1} \cdots k_0$ dla $k_l \in \mathbb{Z}_4 \equiv \{0, 1, 2, 3\}$ (czyli liczby całkowite w bazie $4$) oraz $(\sigma_0, \sigma_1, \sigma_2, \sigma_3) := (I, X, Y, Z)$.

Po wykonaniu tej dekompozycji, `Estimator` wyprowadza nowy obwód $V_k|\psi\rangle$ dla każdej obserwabli $\hat{P}_k$ (z oryginalnego obwodu), aby skutecznie *zdiagonalizować* obserwablę Pauli w bazie obliczeniowej i zmierzyć ją. Możemy łatwo mierzyć obserwable Pauli, ponieważ znamy $V_k$ z góry, co nie jest generalnie prawdziwe dla innych obserwabli.

Dla każdego $\hat{P}_{k}$, `Estimator` uruchamia odpowiedni obwód na urządzeniu kwantowym wielokrotnie, mierzy stan wyjściowy w bazie obliczeniowej i oblicza prawdopodobieństwo $p_{kj}$ uzyskania każdego możliwego wyjścia $j$. Następnie szuka wartości własnej $\lambda_{kj}$ $P_k$ odpowiadającej każdemu wyjściu $j$, mnoży przez $w_k$ i sumuje wszystkie wyniki, aby otrzymać wartość oczekiwaną obserwabli $\hat{H}$ dla danego stanu $|\psi\rangle$.

$$
\langle\hat{H}\rangle_\psi =
\sum_{k=0}^{4^n-1} w_k \sum_{j=0}^{2^n-1}p_{kj} \lambda_{kj},
$$
Ponieważ obliczanie wartości oczekiwanej $4^n$ operatorów Pauli jest niepraktyczne (czyli rośnie wykładniczo), `Estimator` może być wydajny tylko wtedy, gdy duża liczba $w_k$ wynosi zero (czyli *rzadka* dekompozycja Pauli zamiast *gęstej*). Formalnie mówimy, że aby to obliczenie było *efektywnie rozwiązywalne*, liczba niezerowych składników musi rosnąć co najwyżej wielomianowo wraz z liczbą kubitów $n$: $\hat{H} = \sum^{\text{Poly}(n)}_k w_k \hat{P}_k.$

Czytelnik może zauważyć niejawne założenie, że <DefinitionTooltip definition="Proces wykonywania kilku pomiarów jednej lub kilku rzeczy.">próbkowanie</DefinitionTooltip> prawdopodobieństwa również musi być wydajne, jak wyjaśniono dla `Sampler`, co oznacza

$$
\langle\hat{H}\rangle_\psi =
\sum_{k}^{\text{Poly}(n)} w_k \sum_{j}^{\text{Poly}(n)}p_{kj} \lambda_{kj}.
$$
### Przewodnik na przykładzie obliczania wartości oczekiwanych {#guided-example-to-calculate-expectation-values}

Załóżmy stan jednokubitowy $|+\rangle := H|0\rangle = \frac{1}{\sqrt{2}}(|0\rangle + |1\rangle)$ oraz obserwablę

$$
\begin{aligned}
\hat{H}
& = \begin{pmatrix}
-1 & 2 \\
2 & 1 \\
\end{pmatrix}\\[1mm]
& = 2X - Z

\end{aligned}
$$

o następującej teoretycznej wartości oczekiwanej $\langle\hat{H}\rangle_+ = \langle+|\hat{H}|+\rangle = 2.$

Ponieważ nie wiemy, jak zmierzyć tę obserwablę, nie możemy obliczyć jej wartości oczekiwanej bezpośrednio i musimy wyrazić ją ponownie jako $\langle\hat{H}\rangle_+ = 2\langle X \rangle_+ - \langle Z \rangle_+ $. Można pokazać, że daje ona ten sam wynik, zauważając, że $\langle+|X|+\rangle = 1$ oraz $\langle+|Z|+\rangle = 0$.

Zobaczmy, jak bezpośrednio obliczyć $\langle X \rangle_+$ oraz $\langle Z \rangle_+$. Ponieważ $X$ i $Z$ nie komutują (czyli nie dzielą tej samej bazy własnej), nie można ich mierzyć jednocześnie, dlatego potrzebujemy obwodów pomocniczych:

```python
from qiskit import QuantumCircuit
from qiskit.quantum_info import SparsePauliOp

# The following code will work for any other initial single-qubit state and observable
original_circuit = QuantumCircuit(1)
original_circuit.h(0)

H = SparsePauliOp(["X", "Z"], [2, -1])

aux_circuits = []
for pauli in H.paulis:
    aux_circ = original_circuit.copy()
    aux_circ.barrier()
    if str(pauli) == "X":
        aux_circ.h(0)
    elif str(pauli) == "Y":
        aux_circ.sdg(0)
        aux_circ.h(0)
    else:
        aux_circ.id(0)
    aux_circ.measure_all()
    aux_circuits.append(aux_circ)

original_circuit.draw("mpl")
```

Możemy teraz ręcznie przeprowadzić obliczenia za pomocą `Sampler` i sprawdzić wyniki na `Estimator`:

```python
from qiskit.primitives import StatevectorSampler, StatevectorEstimator
from qiskit.result import QuasiDistribution
import numpy as np

## SAMPLER
shots = 10000
sampler = StatevectorSampler()
job = sampler.run(aux_circuits, shots=shots)

# Run the sampler job and step through results
expvals = []
for index, pauli in enumerate(H.paulis):
    data_pub = job.result()[index].data
    bitstrings = data_pub.meas.get_bitstrings()
    counts = data_pub.meas.get_counts()
    quasi_dist = QuasiDistribution(
        {outcome: freq / shots for outcome, freq in counts.items()}
    )

    # Use the probabilities and known eigenvalues of Pauli operators to estimate the expectation value.
    val = 0

    if str(pauli) == "X":
        val += -1 * quasi_dist.get(1, 0)
        val += 1 * quasi_dist.get(0, 0)

    if str(pauli) == "Y":
        val += -1 * quasi_dist.get(1, 0)
        val += 1 * quasi_dist.get(0, 0)

    if str(pauli) == "Z":
        val += 1 * quasi_dist.get(0, 0)
        val += -1 * quasi_dist.get(1, 0)

    expvals.append(val)

# Print expectation values

print("Sampler results:")
for pauli, expval in zip(H.paulis, expvals):
    print(f"  >> Expected value of {str(pauli)}: {expval:.5f}")

total_expval = np.sum(H.coeffs * expvals).real
print(f"  >> Total expected value: {total_expval:.5f}")

# Use estimator for comparison
observables = [
    *H.paulis,
    H,
]  # Note: run for individual Paulis as well as full observable H

estimator = StatevectorEstimator()
job = estimator.run([(original_circuit, observables)])
estimator_expvals = job.result()[0].data.evs

# Print results
print("Estimator results:")
for obs, expval in zip(observables, estimator_expvals):
    if obs is not H:
        print(f"  >> Expected value of {str(obs)}: {expval:.5f}")
    else:
        print(f"  >> Total expected value: {expval:.5f}")
```

### Rygor matematyczny (opcjonalnie) {#mathematical-rigor-optional}

Wyrażając $|\psi\rangle$ względem bazy stanów własnych operatora $\hat{H}$, $|\psi\rangle = \sum_\lambda a_\lambda |\lambda\rangle$, otrzymujemy:

$$
\begin{aligned}
\langle \psi | \hat{H} | \psi \rangle
& = \bigg(\sum_{\lambda'}a^*_{\lambda'} \langle \lambda'|\bigg) \hat{H}
  \bigg(\sum_{\lambda} a_\lambda | \lambda\rangle\bigg)\\[1mm]

& = \sum_{\lambda}\sum_{\lambda'} a^*_{\lambda'}a_{\lambda}
  \langle \lambda'|\hat{H}| \lambda\rangle\\[1mm]

& = \sum_{\lambda}\sum_{\lambda'} a^*_{\lambda'}a_{\lambda} \lambda
\langle \lambda'| \lambda\rangle\\[1mm]

& = \sum_{\lambda}\sum_{\lambda'} a^*_{\lambda'}a_{\lambda} \lambda
\cdot \delta_{\lambda, \lambda'}\\[1mm]

& = \sum_\lambda |a_\lambda|^2 \lambda\\[1mm]

& = \sum_\lambda p_\lambda \lambda\\[1mm]

\end{aligned}
$$
Ponieważ nie znamy wartości własnych ani stanów własnych docelowej obserwabli $\hat{H}$, musimy najpierw rozważyć jej diagonalizację. Biorąc pod uwagę, że $\hat{H}$ jest <DefinitionTooltip definition="Macierz hermitowska to macierz kwadratowa równa swojemu sprzężeniu hermitowskiemu, lub operator liniowy, który jest samosprzężony.">hermitowski</DefinitionTooltip>, istnieje przekształcenie unitarne $V$ takie, że $\hat{H}=V^\dagger \Lambda V,$ gdzie $\Lambda$ jest diagonalną macierzą wartości własnych, zatem $\langle j | \Lambda | k \rangle = 0$ jeśli $j\neq k$ oraz $\langle j | \Lambda | j \rangle = \lambda_j$.

Oznacza to, że wartość oczekiwaną można przepisać jako:

$$
\begin{aligned}
\langle\psi|\hat{H}|\psi\rangle
& = \langle\psi|V^\dagger \Lambda V|\psi\rangle\\[1mm]

& = \langle\psi|V^\dagger \bigg(\sum_{j=0}^{2^n-1} |j\rangle
\langle j|\bigg) \Lambda \bigg(\sum_{k=0}^{2^n-1} |k\rangle \langle k|\bigg) V|\psi\rangle\\[1mm]

& = \sum_{j=0}^{2^n-1} \sum_{k=0}^{2^n-1}\langle\psi|V^\dagger |j\rangle
\langle j| \Lambda  |k\rangle \langle k| V|\psi\rangle\\[1mm]

& = \sum_{j=0}^{2^n-1}\langle\psi|V^\dagger |j\rangle
\langle j| \Lambda  |j\rangle \langle j| V|\psi\rangle\\[1mm]

& = \sum_{j=0}^{2^n-1}|\langle j| V|\psi\rangle|^2 \lambda_j\\[1mm]

\end{aligned}
$$

Biorąc pod uwagę, że jeśli układ znajduje się w stanie $|\phi\rangle = V |\psi\rangle$, to prawdopodobieństwo pomiaru $| j\rangle$ wynosi $p_j = |\langle j|\phi \rangle|^2$, powyższą wartość oczekiwaną można wyrazić jako:

$$
\langle\psi|\hat{H}|\psi\rangle =
\sum_{j=0}^{2^n-1} p_j \lambda_j.
$$

Bardzo ważne jest, aby zauważyć, że prawdopodobieństwa są brane ze stanu $V |\psi\rangle$, a nie $|\psi\rangle$. Dlatego macierz $V$ jest absolutnie niezbędna.
Być może zastanawiasz się, jak uzyskać macierz $V$ i wartości własne $\Lambda$. Gdybyś już miał wartości własne, nie byłoby potrzeby używania komputera kwantowego, ponieważ celem algorytmów wariacyjnych jest znalezienie tych wartości własnych operatora $\hat{H}$.

Na szczęście istnieje sposób na obejście tego: dowolna macierz $2^n \times 2^n$ może być zapisana jako kombinacja liniowa $4^n$ iloczynów tensorowych $n$ macierzy Pauliego i identyczności, z których wszystkie są zarówno hermitowskie, jak i unitarne, ze znanymi $V$ i $\Lambda$. Tak właśnie działa wewnętrznie `Estimator` Runtime, dekomponując dowolny obiekt [`Operator`](https://docs.quantum.ibm.com/api/qiskit/qiskit.quantum_info.Operator) na [`SparsePauliOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.quantum_info.SparsePauliOp).

Oto operatory, których można użyć:

$$
\begin{array}{c|c|c|c}
  \text{Operator} & \sigma & V & \Lambda \\[1mm]
  \hline
  I & \sigma_0 = \begin{pmatrix} 1 & 0 \\ 0 & 1 \end{pmatrix} & V_0 = I & \Lambda_0 = I = \begin{pmatrix} 1 & 0 \\ 0 & 1 \end{pmatrix} \\[4mm]

  X & \sigma_1 = \begin{pmatrix} 0 & 1 \\ 1 & 0 \end{pmatrix} & V_1 = H =\frac{1}{\sqrt{2}} \begin{pmatrix} 1 & 1 \\ 1 & -1 \end{pmatrix} & \Lambda_1 = \sigma_3 = \begin{pmatrix} 1 & 0 \\ 0 & -1 \end{pmatrix} \\[4mm]

  Y & \sigma_2 = \begin{pmatrix} 0 & -i \\ i & 0 \end{pmatrix} & V_2 = HS^\dagger  =\frac{1}{\sqrt{2}} \begin{pmatrix} 1 & 1 \\ 1 & -1 \end{pmatrix}\cdot  \begin{pmatrix} 1 & 0 \\ 0 & -i \end{pmatrix} = \frac{1}{\sqrt{2}} \begin{pmatrix} 1 & -i \\ 1 & i \end{pmatrix}\quad & \Lambda_2 = \sigma_3 = \begin{pmatrix} 1 & 0 \\ 0 & -1 \end{pmatrix} \\[4mm]

  Z & \sigma_3 = \begin{pmatrix} 1 & 0 \\ 0 & -1 \end{pmatrix} & V_3 = I & \Lambda_3 = \sigma_3 = \begin{pmatrix} 1 & 0 \\ 0 & -1 \end{pmatrix}
\end{array}
$$
Przepiszmy zatem $\hat{H}$ względem macierzy Pauliego i identyczności:

$$
\hat{H} =
\sum_{k_{n-1}=0}^3...
\sum_{k_0=0}^3 w_{k_{n-1}...k_0}
\sigma_{k_{n-1}}\otimes ... \otimes \sigma_{k_0} = \sum_{k=0}^{4^n-1} w_k \hat{P}_k,
$$

gdzie $k = \sum_{l=0}^{n-1} 4^l k_l \equiv k_{n-1}...k_0$ dla $k_{n-1},...,k_0\in \{0,1,2,3\}$ (czyli w bazie $4$), a $\hat{P}_{k} := \sigma_{k_{n-1}}\otimes ... \otimes \sigma_{k_0}$:

$$
\begin{aligned}
\langle\psi|\hat{H}|\psi\rangle
& = \sum_{k=0}^{4^n-1} w_k
\sum_{j=0}^{2^n-1}|\langle j| V_k|\psi\rangle|^2 \langle j| \Lambda_k |j\rangle \\[1mm]

& = \sum_{k=0}^{4^n-1} w_k \sum_{j=0}^{2^n-1}p_{kj} \lambda_{kj}, \\[1mm]

\end{aligned}
$$

gdzie $V_k := V_{k_{n-1}}\otimes ... \otimes V_{k_0}$ oraz $\Lambda_k := \Lambda_{k_{n-1}}\otimes ... \otimes \Lambda_{k_0}$, takie że: $\hat{P_k}=V_k^\dagger \Lambda_k V_k.$
## Funkcje kosztu {#cost-functions}

Ogólnie rzecz biorąc, funkcje kosztu służą do opisania celu problemu oraz tego, jak dobrze stan próbny spełnia ten cel. Tę definicję można stosować w różnych przykładach z chemii, uczenia maszynowego, finansów, optymalizacji itd.

Rozważmy prosty przykład znajdowania stanu podstawowego układu. Naszym celem jest zminimalizowanie wartości oczekiwanej obserwabli reprezentującej energię (Hamiltonian $\hat{\mathcal{H}}$):

$$
\min_{\vec\theta} \langle\psi(\vec\theta)|\hat{\mathcal{H}}|\psi(\vec\theta)\rangle
$$

Możemy użyć `Estimator` do wyznaczenia wartości oczekiwanej i przekazać tę wartość do optymalizatora w celu minimalizacji. Jeśli optymalizacja zakończy się sukcesem, zwróci ona zestaw optymalnych wartości parametrów $\vec\theta^*$, z których będziemy w stanie skonstruować proponowany stan rozwiązania $|\psi(\vec\theta^*)\rangle$ i obliczyć obserwowaną wartość oczekiwaną jako $C(\vec\theta^*)$.

Zauważ, że będziemy w stanie minimalizować funkcję kosztu jedynie dla ograniczonego zbioru stanów, które rozważamy. Prowadzi to do dwóch odrębnych możliwości:

- **Nasz ansatz nie definiuje stanu rozwiązania w całej przestrzeni poszukiwań**: Jeśli tak jest, nasz optymalizator nigdy nie znajdzie rozwiązania i musimy eksperymentować z innymi ansatzami, które mogą dokładniej reprezentować naszą przestrzeń poszukiwań.
- **Nasz optymalizator nie jest w stanie znaleźć tego prawidłowego rozwiązania**: Optymalizacja może być zdefiniowana globalnie i lokalnie. Zgłębimy znaczenie tych pojęć w dalszej części.

Podsumowując, będziemy wykonywać klasyczną pętlę optymalizacyjną, ale polegając na ewaluacji funkcji kosztu na komputerze kwantowym. Z tej perspektywy można myśleć o optymalizacji jako o czysto klasycznym przedsięwzięciu, w którym wywołujemy pewną <DefinitionTooltip definition="Hipotetyczne urządzenie lub komponent oprogramowania, który wykonuje określoną funkcję, ale wewnętrzne działanie jest nieznane. Użytkownik zna tylko wejścia i wyjścia czarnej skrzynki, nie wiedząc, w jaki sposób czarna skrzynka przetwarza wejścia w wyjścia.">wyrocznię kwantową typu czarna skrzynka</DefinitionTooltip> za każdym razem, gdy optymalizator musi obliczyć funkcję kosztu.

```python
def cost_func_vqe(params, circuit, hamiltonian, estimator):
    """Return estimate of energy from estimator

    Parameters:
        params (ndarray): Array of ansatz parameters
        ansatz (QuantumCircuit): Parameterized ansatz circuit
        hamiltonian (SparsePauliOp): Operator representation of Hamiltonian
        estimator (Estimator): Estimator primitive instance

    Returns:
        float: Energy estimate
    """
    pub = (circuit, hamiltonian, params)
    cost = estimator.run([pub]).result()[0].data.evs
    return cost
```

### Przykład odwzorowania na układy niefizyczne {#example-mapping-to-non-physical-systems}

Problem maksymalnego cięcia (Max-Cut) to problem optymalizacji kombinatorycznej, który polega na podziale wierzchołków grafu na dwa rozłączne zbiory w taki sposób, aby liczba krawędzi między tymi dwoma zbiorami była maksymalna. Bardziej formalnie, mając nieskierowany graf $G=(V,E)$, gdzie $V$ jest zbiorem wierzchołków, a $E$ jest zbiorem krawędzi, problem Max-Cut polega na podziale wierzchołków na dwa rozłączne podzbiory $S$ i $T$ w taki sposób, aby liczba krawędzi, których jeden koniec znajduje się w $S$, a drugi w $T$, była maksymalna.

Możemy zastosować Max-Cut do rozwiązania różnych problemów, w tym: klastrowania, projektowania sieci, przejść fazowych itp. Zaczniemy od utworzenia grafu problemu:

![Wynik poprzedniej komórki kodu](/learning/images/courses/variational-algorithm-design/cost-functions/extracted-outputs/739ca7a0-fae3-4389-a8de-7ae93c13b9e1-0.avif)

Problem ten można wyrazić jako problem optymalizacji binarnej. Dla każdego węzła $0 \leq i < n$, gdzie $n$ jest liczbą węzłów grafu (w tym przypadku $n=4$), rozważymy zmienną binarną $x_i$. Zmienna ta będzie miała wartość $1$, jeśli węzeł $i$ należy do jednej z grup, którą oznaczymy $1$, oraz $0$, jeśli należy do drugiej grupy, którą oznaczymy jako $0$. Oznaczymy również jako $w_{ij}$ (element $(i,j)$ macierzy sąsiedztwa $w$) wagę krawędzi biegnącej od węzła $i$ do węzła $j$. Ponieważ graf jest nieskierowany, $w_{ij}=w_{ji}$. Wówczas nasz problem możemy sformułować jako maksymalizację następującej funkcji kosztu:

$$
\begin{aligned}
C(\vec{x})
& =\sum_{i,j=0}^n w_{ij} x_i(1-x_j)\\[1mm]

& = \sum_{i,j=0}^n w_{ij} x_i - \sum_{i,j=0}^n w_{ij} x_ix_j\\[1mm]

& = \sum_{i,j=0}^n w_{ij} x_i - \sum_{i=0}^n \sum_{j=0}^i 2w_{ij} x_ix_j
\end{aligned}
$$

Aby rozwiązać ten problem za pomocą komputera kwantowego, wyrazimy funkcję kosztu jako wartość oczekiwaną pewnej obserwabli. Jednakże obserwable, które Qiskit natywnie obsługuje, składają się z operatorów Pauli, które mają wartości własne $1$ i $-1$ zamiast $0$ i $1$. Dlatego dokonamy następującej zamiany zmiennych:

Gdzie $\vec{x}=(x_0,x_1,\cdots ,x_{n-1})$. Możemy wykorzystać macierz sąsiedztwa $w$, aby wygodnie uzyskać dostęp do wag wszystkich krawędzi. Zostanie to użyte do uzyskania naszej funkcji kosztu:

$$
z_i = 1-2x_i \rightarrow x_i = \frac{1-z_i}{2}
$$

Oznacza to, że:

$$
\begin{array}{lcl} x_i=0 & \rightarrow & z_i=1 \\ x_i=1 & \rightarrow & z_i=-1.\end{array}
$$

Zatem nowa funkcja kosztu, którą chcemy zmaksymalizować, to:

$$
\begin{aligned}
C(\vec{z})
& = \sum_{i,j=0}^n w_{ij} \bigg(\frac{1-z_i}{2}\bigg)\bigg(1-\frac{1-z_j}{2}\bigg)\\[1mm]

& = \sum_{i,j=0}^n \frac{w_{ij}}{4} - \sum_{i,j=0}^n \frac{w_{ij}}{4} z_iz_j\\[1mm]

& = \sum_{i=0}^n \sum_{j=0}^i \frac{w_{ij}}{2} -  \sum_{i=0}^n \sum_{j=0}^i \frac{w_{ij}}{2} z_iz_j
\end{aligned}
$$

Co więcej, naturalną tendencją komputera kwantowego jest znajdowanie minimów (zwykle najniższej energii) zamiast maksimów, więc zamiast maksymalizować $C(\vec{z})$, będziemy minimalizować:

$$
-C(\vec{z}) =  \sum_{i=0}^n \sum_{j=0}^i \frac{w_{ij}}{2} z_iz_j -  \sum_{i=0}^n \sum_{j=0}^i \frac{w_{ij}}{2}
$$

Teraz, gdy mamy funkcję kosztu do minimalizacji, której zmienne mogą przyjmować wartości $-1$ i $1$, możemy wykonać następującą analogię z Pauli $Z$:

$$
z_i \equiv Z_i = \overbrace{I}^{n-1}\otimes ... \otimes \overbrace{Z}^{i} \otimes ... \otimes \overbrace{I}^{0}
$$

Innymi słowy, zmienna $z_i$ będzie odpowiadać bramce $Z$ działającej na kubit $i$. Ponadto:

$$
Z_i|x_{n-1}\cdots x_0\rangle = z_i|x_{n-1}\cdots x_0\rangle \rightarrow \langle x_{n-1}\cdots x_0 |Z_i|x_{n-1}\cdots x_0\rangle = z_i
$$

Wtedy obserwablą, którą rozważymy, jest:

$$
\hat{H} = \sum_{i=0}^n \sum_{j=0}^i \frac{w_{ij}}{2} Z_iZ_j
$$

do której będziemy musieli później dodać niezależny człon:

$$
\texttt{offset} = - \sum_{i=0}^n \sum_{j=0}^i \frac{w_{ij}}{2}
$$
Operator jest liniową kombinacją członów z operatorami Z na węzłach połączonych krawędzią (przypomnijmy, że kubit 0 znajduje się najbardziej na prawo): $IIZZ + IZIZ + IZZI + ZIIZ + ZZII$. Po skonstruowaniu operatora ansatz dla algorytmu QAOA można łatwo zbudować, używając obwodu `QAOAAnsatz` z biblioteki obwodów Qiskit.

### Tłumienie błędów {#error-suppression}

[Tłumienie błędów](/guides/configure-error-suppression) odnosi się do technik stosowanych do optymalizacji i transformacji obwodu podczas kompilacji w celu zminimalizowania błędów. Jest to podstawowa technika obsługi błędów, która zwykle skutkuje pewnym klasycznym <DefinitionTooltip definition="Dodatkowe koszty wprowadzane przez nowe techniki w stosunku do implementacji bazowej.">narzutem</DefinitionTooltip> przetwarzania wstępnego do całkowitego czasu wykonania. Narzut ten obejmuje transpilację obwodów do uruchomienia na sprzęcie kwantowym poprzez:

- Wyrażenie obwodu za pomocą natywnych bramek dostępnych w systemie kwantowym
- Odwzorowanie wirtualnych kubitów na fizyczne kubity
- Dodawanie operacji SWAP na podstawie wymagań łączności
- Optymalizację bramek 1Q i 2Q
- Dodawanie dynamicznego rozprzęgania do bezczynnych kubitów, aby zapobiec efektom dekoherencji.

Prymitywy umożliwiają korzystanie z technik tłumienia błędów poprzez ustawienie opcji `optimization_level` i wybór zaawansowanych opcji transpilacji. W późniejszym kursie zagłębimy się w różne metody konstrukcji obwodów w celu poprawy wyników, ale w większości przypadków zalecamy ustawienie `optimization_level=3`.

```python
exp_values_with_opt_es = cost[0]
```

Na koniec możemy wykreślić wyniki i widzimy, że precyzja obliczeń była dość dobra nawet bez optymalizacji, ale zdecydowanie poprawiła się po zwiększeniu optymalizacji do poziomu 3. Zauważ, że w głębszych, bardziej skomplikowanych obwodach różnica między poziomami optymalizacji 0 i 3 będzie prawdopodobnie bardziej znacząca. Jest to bardzo prosty obwód użyty jako zabawkowy model.

```python
import matplotlib.pyplot as plt

plt.plot(phases, noisy_exp_values, "o", label="opt=0")
plt.plot(phases, exp_values_with_opt_es, "o", label="opt=3")
plt.plot(phases, 2 * np.sin(phases / 2) ** 2 - 1, label="ideal")
plt.ylabel("Expectation")
plt.legend()
plt.show()
```

### Łagodzenie błędów {#error-mitigation}

[Łagodzenie błędów](/guides/configure-error-mitigation) odnosi się do technik, które pozwalają użytkownikom redukować błędy obwodu poprzez modelowanie szumu urządzenia w czasie wykonania. Zazwyczaj skutkuje to kwantowym narzutem przetwarzania wstępnego związanym z trenowaniem modelu oraz klasycznym narzutem przetwarzania końcowego w celu złagodzenia błędów w surowych wynikach za pomocą wygenerowanego modelu.

Opcja `resilience_level` prymitywu Qiskit Runtime określa poziom odporności budowanej przeciwko błędom. Wyższe poziomy generują dokładniejsze wyniki kosztem dłuższych czasów przetwarzania z powodu narzutu próbkowania kwantowego. Poziomy odporności mogą być używane do konfiguracji kompromisu między kosztem a dokładnością podczas stosowania łagodzenia błędów w zapytaniach prymitywów.

Wdrażając jakąkolwiek technikę łagodzenia błędów, oczekujemy, że <DefinitionTooltip definition="Systematyczny dryf w mierzonych wielkościach, zwykle spowodowany błędami.">obciążenie</DefinitionTooltip> w naszych wynikach zostanie zmniejszone w stosunku do poprzedniego, niezłagodzonego obciążenia. W niektórych przypadkach obciążenie może nawet zniknąć. Wiąże się to jednak z kosztem. Gdy zmniejszamy obciążenie w naszych szacowanych wielkościach, zmienność statystyczna wzrasta (czyli wariancja), co możemy uwzględnić, dalej zwiększając liczbę pomiarów na obwód w naszym procesie próbkowania. Wprowadzi to narzut poza tym potrzebnym do zmniejszenia obciążenia, więc nie jest to robione domyślnie. Możemy łatwo włączyć to zachowanie, dostosowując liczbę pomiarów na obwód w options.executions.shots, jak pokazano w przykładzie poniżej.

![Diagram pokazujący szersze lub węższe rozkłady, jak w kompromisie obciążenie/wariancja.](/learning/images/courses/variational-algorithm-design/cost-functions/cost-function-bias-variance-trade-off.svg)

W tym kursie zbadamy te modele łagodzenia błędów na wysokim poziomie, aby zilustrować łagodzenie błędów, które prymitywy Qiskit Runtime mogą wykonać bez konieczności znajomości pełnych szczegółów implementacji.

### Twirled readout error extinction (T-REx) {#twirled-readout-error-extinction-t-rex}

Twirled readout error extinction (T-REx) wykorzystuje technikę znaną jako twirling Pauliego w celu zmniejszenia szumu wprowadzanego podczas procesu pomiaru kwantowego. Technika ta nie zakłada konkretnej postaci szumu, co czyni ją bardzo ogólną i skuteczną.

Ogólny przepływ pracy:

1. Pozyskaj dane dla stanu zerowego z losowymi odwróceniami bitów (Pauli X przed pomiarem)
2. Pozyskaj dane dla pożądanego (zaszumionego) stanu z losowymi odwróceniami bitów (Pauli X przed pomiarem)
3. Oblicz specjalną funkcję dla każdego zestawu danych i podziel.
&nbsp;

![Diagram przedstawiający obwody pomiarowe i kalibracyjne dla T-REX.](/learning/images/courses/variational-algorithm-design/cost-functions/cost-function-trex-data-collection.svg)

Możemy ustawić to za pomocą `options.resilience_level = 1`, co pokazano w przykładzie poniżej.

### Ekstrapolacja do zerowego szumu {#zero-noise-extrapolation}

Ekstrapolacja do zerowego szumu (ZNE, zero noise extrapolation) działa poprzez wstępne wzmocnienie szumu w obwodzie przygotowującym żądany stan kwantowy, uzyskanie pomiarów dla kilku różnych poziomów szumu, a następnie wykorzystanie tych pomiarów do wnioskowania o wyniku pozbawionym szumu.

Ogólny przebieg pracy:

1. Wzmocnij szum obwodu dla kilku współczynników szumu
2. Uruchom każdy obwód ze wzmocnionym szumem
3. Ekstrapoluj z powrotem do granicy zerowego szumu

&nbsp;

![Diagram przedstawiający etapy ZNE. Szum jest sztucznie wzmacniany o różne współczynniki. Następnie wartości są ekstrapolowane do tego, jakie powinny być przy zerowym szumie.](/learning/images/courses/variational-algorithm-design/cost-functions/cost-function-zne-stages.svg)

Możemy to ustawić za pomocą `options.resilience_level = 2`. Możemy dodatkowo zoptymalizować to, eksplorując różne `noise_factors`, `noise_amplifiers` i `extrapolators`, ale wykracza to poza zakres tego kursu. Zachęcamy do eksperymentowania z tymi [opcjami opisanymi tutaj](/guides/configure-error-mitigation).

Każda metoda wiąże się z własnym narzutem: kompromisem między liczbą wymaganych obliczeń kwantowych (czas) a dokładnością naszych wyników:

$$
\begin{array}{c|c|c|c}
  \text{Methods} & R=1 \text{, T-REx} & R=2 \text{, ZNE} \\[1mm]
  \hline
  \text{Assumptions} & \text{None} & \text{Ability to scale noise} \\[1mm]
  \text{Qubit overhead} & 1 & 1 \\[1mm]
  \text{Sampling overhead} & 2 & N_{\text{noise-factors}} \\[1mm]
  \text{Bias} & 0 & \mathcal{O}(\lambda^{N_{\text{noise-factors}}}) \\[1mm]
\end{array}
$$
### Korzystanie z opcji mitigacji i tłumienia w Qiskit Runtime {#using-qiskit-runtimes-mitigation-and-suppression-options}

Oto jak obliczyć wartość oczekiwaną przy jednoczesnym użyciu mitigacji i tłumienia błędów w Qiskit Runtime. Możemy wykorzystać dokładnie ten sam obwód i obserwablę co poprzednio, ale tym razem utrzymując poziom optymalizacji na poziomie 2, a teraz dostrajając _odporność_ lub stosowane techniki mitigacji błędów. Ten proces mitigacji błędów odbywa się wielokrotnie w całej pętli optymalizacji.

Tę część wykonujemy na rzeczywistym sprzęcie, ponieważ mitigacja błędów nie jest dostępna w symulatorach.

```python
# Estimated usage: 8 minutes, benchmarked on an Eagle processor, 5-23-24

from qiskit_ibm_runtime import QiskitRuntimeService
from qiskit_ibm_runtime import (
    Session,
    EstimatorOptions,
    EstimatorV2 as Estimator,
)

# We select the least busy backend

# Select the least busy backend
# backend = service.least_busy(
#    operational=True, min_num_qubits=ansatz.num_qubits, simulator=False
# )

# Or use a specific backend
backend = service.backend("ibm_brisbane")

# Initialize some variables to save the results from different runs:

exp_values_with_em0_es = []
exp_values_with_em1_es = []
exp_values_with_em2_es = []

# Use a pass manager to optimize the circuit and observables for the backend chosen:

pm = generate_preset_pass_manager(backend=backend, optimization_level=2)
isa_circuit = pm.run(circuit)
isa_observables = observables.apply_layout(layout=isa_circuit.layout)

# Open a session and run with no error mitigation:

estimator_options = EstimatorOptions(resilience_level=0, default_shots=10_000)

with Session(backend=backend) as session:
    estimator = Estimator(mode=session, options=estimator_options)

    pub = (isa_circuit, isa_observables, [individual_phases])
    cost = estimator.run([pub]).result()[0].data.evs

session.close()

exp_values_with_em0_es = cost[0]

# Open a session and run with resilience = 1:

estimator_options = EstimatorOptions(resilience_level=1, default_shots=10_000)

with Session(backend=backend) as session:
    estimator = Estimator(mode=session, options=estimator_options)

    pub = (isa_circuit, isa_observables, [individual_phases])
    cost = estimator.run([pub]).result()[0].data.evs

session.close()

exp_values_with_em1_es = cost[0]

# Open a session and run with resilience = 2:

estimator_options = EstimatorOptions(resilience_level=2, default_shots=10_000)

with Session(backend=backend) as session:
    estimator = Estimator(mode=session, options=estimator_options)

    pub = (isa_circuit, isa_observables, [individual_phases])
    cost = estimator.run([pub]).result()[0].data.evs
```
