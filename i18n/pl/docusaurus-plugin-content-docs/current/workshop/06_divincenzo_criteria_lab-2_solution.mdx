## 1. Kryterium 1 – Skalowalne, dobrze scharakteryzowane Qubity {#kryterium-1-skalowalne-dobrze-scharakteryzowane-qubity}

**Kryterium 1:** *„Skalowalny system fizyczny z dobrze scharakteryzowanymi Qubitami."* Oznacza to, że potrzebujemy platformy sprzętowej, w której możemy **zwiększać liczbę Qubitów** i nadal sterować nimi niezawodnie. Właściwości każdego Qubitu (poziomy energetyczne, współczynniki błędów, łączność itp.) powinny być dobrze rozumiane. W skrócie: chcemy budować większe Circuit bez degradacji systemu. W praktyce, gdy zwiększamy liczbę Qubitów lub głębokość Circuit, narastają błędy i dekoherencja, więc demonstrowanie *skalowalności* oznacza też rozumienie, jak wzrost rozmiaru wpływa na wydajność.

**Cel demonstracji:** Użyj Qiskit, aby pokazać wpływ skalowania Circuit (pod względem liczby Qubitów lub głębokości Gate'ów) na wierność wyników. Zasymulujemy scenariusz idealny i zaszumiony, aby zobaczyć, jak większy system lub głębszy Circuit ulega dekoherencji i błędom.

Najpierw zbudujemy mały stan splątany (stan GHZ) na 3 Qubitach, a następnie większy na 5 Qubitach, jako prosty test skalowalności. Stan GHZ dla *n* Qubitów to $\frac{1}{\sqrt{2}}(|0...0\rangle + |1...1\rangle)$. W idealnej symulacji pomiar n-Qubitowego stanu GHZ daje tylko dwa wyniki (same 0 lub same 1) z równym prawdopodobieństwem. Porównamy **wynik idealny** z **wynikiem zaszumionym** przy wzrastającym n lub głębokości Circuit.

```python
from qiskit import QuantumCircuit
from qiskit_aer import AerSimulator
from qiskit.visualization import plot_histogram
from qiskit.transpiler.preset_passmanagers import generate_preset_pass_manager
from qiskit_ibm_runtime import SamplerV2 as Sampler

# 3-qubit GHZ circuit
qc3 = QuantumCircuit(3, 3)
qc3.h(0)
qc3.cx(0, 1)
qc3.cx(1, 2)
qc3.measure([0, 1, 2], [0, 1, 2])

# 5-qubit GHZ circuit (scaling up the number of qubits)
qc5 = QuantumCircuit(5, 5)
qc5.h(0)
qc5.cx(0, range(1, 5))    # entangle qubit 0 with all others
qc5.measure(range(5), range(5))

# Transpile for a simulator backend
sim_backend = AerSimulator()
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc3 = pm.run(qc3)
isa_qc5 = pm.run(qc5)

# Run ideal simulations (no noise)
sampler = Sampler(mode=sim_backend)

job3 = sampler.run([isa_qc3], shots=1024)
result3 = job3.result()
counts3 = result3[0].data.c.get_counts()

job5 = sampler.run([isa_qc5], shots=1024)
result5 = job5.result()
counts5 = result5[0].data.c.get_counts()

print("3-qubit GHZ counts (ideal):", counts3)
plot_histogram(counts3, legend=['3-qubit ideal'], figsize=(6,4))
```

```python
print("5-qubit GHZ counts (ideal):", counts5)
plot_histogram(counts5, legend=['5-qubit ideal'], figsize=(6,4))
```

**Oczekiwany wynik (przypadek idealny):** 3-Qubitowy stan GHZ idealnie daje około 50% `000` i 50% `111` w zliczeniach. 5-Qubitowy GHZ daje ~50% `00000` i 50% `11111`. Żadne inne ciągi bitów się nie pojawiają, ponieważ stan jest idealnie w pełni koherentny i splątany. Na histogramie powinieneś widzieć dwa wysokie słupki dla każdego Circuit, odpowiadające wynikom ze wszystkimi zerami i wszystkimi jedynkami.

Zobaczmy teraz, co dzieje się w **środowisku zaszumionym**. Użyjemy możliwości modelu szumu Qiskit Aer, aby naśladować błędy prawdziwego urządzenia. Możemy na przykład pobrać właściwości backendu IBM, aby stworzyć model szumu uwzględniający błędy Gate'ów, skończone czasy Gate'ów, relaksację Qubitów (T₁), depfazowanie (T₂) i błędy odczytu. Użyjemy tu **fałszywego backendu** reprezentującego urządzenie IBM Quantum Brisbane, aby wygenerować model szumu i ponownie uruchomić Circuit ze stanem GHZ.

## 2. Kryterium 2 – Inicjalizacja Qubitów {#2-criterion-2-qubit-initialization}

**Kryterium 2:** *„Możliwość zainicjalizowania stanu qubitów do prostego stanu fiducjalnego, np. |000…〉."* Wszystkie qubity powinny niezawodnie zaczynać w znanym stanie referencyjnym (zazwyczaj stanie podstawowym |0〉 każdego qubitu). Inicjalizacja jest niezbędna, aby algorytmy rozpoczynały się od czystego stanu. W praktyce, na urządzeniach kwantowych IBM, każdy qubit jest automatycznie resetowany do |0〉 na początku każdego wykonania Circuit. Qiskit udostępnia również instrukcje pozwalające resetować qubity lub przygotowywać własne stany w trakcie obliczeń.

**Cel demonstracji:** Pokazać, jak inicjalizować qubity w Qiskit – zarówno na początku, jak i w środku Circuit. Zademonstrujemy użycie instrukcji `reset` oraz metod przygotowania stanu.
### Ćwiczenie 2: Przygotowanie konkretnego stanu {#exercise-2-prepare-a-specific-state}

W bloku kodu poniżej uzupełnij `QuantumCircuit`, aby przygotować stan $|10\rangle$. Oznacza to, że qubit 0 powinien być w stanie $|0\rangle$, a qubit 1 w stanie $|1\rangle$. Użyj odpowiedniej bramki i instrukcji, aby to osiągnąć.

```python
from qiskit import QuantumCircuit
from qiskit_aer import AerSimulator

# Create a circuit to initialize qubits to |10> and verify by measurement
qc_init = QuantumCircuit(2, 2)

# --- YOUR CODE HERE ---

# 1. Set qubit 1 to the |1> state
qc_init.x(1)

# 2. Explicitly reset qubit 0 to the |0> state
qc_init.reset(0)

# --- END YOUR CODE ---

qc_init.measure([0, 1], [0, 1])
qc_init.draw('mpl')
```

```python
# Run the circuit and check the outcome
sim_backend = AerSimulator()
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc_init = pm.run(qc_init)

sampler = Sampler(mode=sim_backend)

job = sampler.run([isa_qc_init], shots=1024)
result = job.result()
counts = result[0].data.c.get_counts()

print("Outcome of |10> state measured in Z-basis:", counts)
plot_histogram(counts)
```

Powinieneś zobaczyć `10` (binarnie: qubit1=1, qubit0=0) z prawdopodobieństwem 100% w symulacji, co oznacza, że qubit 1 został pomyślnie przygotowany w stanie |1〉, a qubit 0 w stanie |0〉.

Qiskit umożliwia przygotowanie bardziej ogólnych stanów do dowolnych stanów za pomocą metody `initialize`. Na przykład przygotujmy qubit w stanie $|+\rangle = (|0\rangle+|1\rangle)/\sqrt{2}$, będącym stanem superpozycji, oraz parę qubitów w stanie Bella $(|00\rangle+|11\rangle)/\sqrt{2}$:

```python
import numpy as np

# Initialize a single qubit in |+> state and measure in Z-basis
qc_plus = QuantumCircuit(1, 1)
state_plus = [1/np.sqrt(2), 1/np.sqrt(2)]   # amplitude for |0> and |1>
qc_plus.initialize(state_plus, 0)
qc_plus.measure(0, 0)

# Initialize two qubits in a Bell state manually
qc_bell = QuantumCircuit(2, 2)
bell_state = [1/np.sqrt(2), 0, 0, 1/np.sqrt(2)]  # amplitudes for |00>,|01>,|10>,|11>
qc_bell.initialize(bell_state, [0, 1])
qc_bell.measure([0, 1], [0, 1])

# Transpile and run the initialization circuits
isa_qc_plus = pm.run(qc_plus)
job_plus = sampler.run([isa_qc_plus], shots=1024)
result_plus = job_plus.result()
counts_plus = result_plus[0].data.c.get_counts()

print("Outcome of |+> state measured in Z-basis:", counts_plus)

isa_qc_bell = pm.run(qc_bell)
job_bell = sampler.run([isa_qc_bell], shots=1024)
result_bell = job_bell.result()
counts_bell = result_bell[0].data.c.get_counts()

print("Outcome of Bell state measured in Z-basis:", counts_bell)
```

**Oczekiwane wyniki:** Pomiar stanu |+〉 pojedynczego qubitu da w przybliżeniu po 50% szans na wynik `0` i `1`. Pomiar stanu Bella powinien dać około 50% `00` i 50% `11`. Jeśli to widzisz, potwierdza to, że inicjalizacja do tych stanów przebiegła pomyślnie.

**Inicjalizacja w środku Circuit:** Instrukcja `reset` w Qiskit może być użyta w środku Circuit, aby ponownie zainicjalizować qubit do stanu |0〉 w locie. Na przykład w kodach korekcji błędów lub algorytmach iteracyjnych często mierzy się qubit, a następnie resetuje go do ponownego użycia. Operacja `reset` jest deterministyczna – usuwa istniejący stan i schładza qubit do stanu podstawowego.

**Przykład na urządzeniu:** Na sprzęcie takim jak **ibmq_brisbane** (127 qubitów) lub dowolnym urządzeniu IBM, wszystkie qubity zaczynają w stanie |0〉 domyślnie przy uruchamianiu zadania. Jeśli potrzebujesz innego stanu startowego, stosujesz bramki na początku (tak jak użyliśmy X, aby uzyskać |1〉). Ciągła re-inicjalizacja (na potrzeby kwantowej korekcji błędów) jest aktywnym tematem badań, ponieważ szybkie jej wykonywanie jest wymagające. Na szczęście do podstawowego użytku możliwość zaczynania od nowa w stanie |0…0〉 jest dostępna, a my pokazaliśmy też, jak osiągnąć inne pożądane stany startowe.
## 3. Kryterium 3 – Długi Czas Koherencji (Dekoherencja a Czas Bramki) {#3-criterion-3-long-coherence-time-decoherence-vs-gate-time}

**Kryterium 3:** *„Długie czasy dekoherencji, znacznie dłuższe niż czas operacji bramkowej."* Chodzi o konieczność utrzymywania stanu kwantowego qubitów wystarczająco długo, aby wykonać niezbędne operacje. Każdy qubit ma **czas T₁** (czas relaksacji energetycznej, jak szybko |1〉 zanika do |0〉) i **czas T₂** (czas defazowania, jak szybko traci się spójność fazy względnej). Aby komputer kwantowy działał, te skale czasowe powinny znacznie przekraczać czas trwania operacji bramkowych.

**Cel demonstracji:** Zbadać koherencję qubitów w Qiskit, pokazując jak dekoherencja wpływa na wyniki Circuit w miarę wydłużania się czasu wykonania. Użyjemy fałszywego Backend z known T1/T2 times, aby zasymulować ten efekt.

Aby **zademonstrować wpływ skończonej koherencji**, zasymulujemy eksperyment zaniku T1. Przygotujemy qubit w stanie |1〉, poczekamy pewien czas przy użyciu instrukcji `delay`, a następnie dokonamy pomiaru. Oczekujemy, że prawdopodobieństwo zmierzenia |1〉 będzie maleć wraz ze wzrostem opóźnienia.

```python
# This part is done for you. We are creating a list of circuits,
# each with a different delay time.

time_delays_ns = [0, 50000, 100000, 150000, 200000, 250000, 300000]  # delay durations in ns

decay_expts = []
for delay in time_delays_ns:
    qc = QuantumCircuit(1, 1)
    qc.x(0)  # initialize qubit to |1>
    if delay > 0:
        qc.delay(delay, 0, unit='ns')  # wait 'delay' nanoseconds
    qc.measure(0, 0)
    decay_expts.append(qc)

decay_expts[1].draw('mpl') # Visualize one of the circuits
```

### Ćwiczenie 3: Symulacja eksperymentu zaniku T1 {#exercise-3-simulate-a-t1-decay-experiment}

Teraz użyj zaszumionego symulatora opartego na `FakeVigo` (który ma czasy T1 rzędu ~50–100 µs), aby uruchomić te Circuit. Symulator automatycznie zastosuje błędy T1/T2 podczas instrukcji `delay`. Transpiluj Circuit dla tego Backend i uruchom je.

```python
from qiskit_ibm_runtime.fake_provider import FakeVigoV2 as FakeVigo
from qiskit_aer import AerSimulator

# --- YOUR CODE HERE ---

# 1. Create a noisy simulator from the FakeVigo backend
sim_vigo = AerSimulator.from_backend(FakeVigo())

# 2. Transpile the list of circuits for this simulator
pm = generate_preset_pass_manager(backend=sim_vigo, optimization_level=1)

isa_decay_expts = pm.run(decay_expts)

# 3. Use the Sampler to run all the transpiled circuits in a single job
sampler = Sampler(mode=sim_vigo)

job = sampler.run(isa_decay_expts, shots=1000)

result = job.result()

# --- END YOUR CODE ---

# This part is done for you to analyze and print the results.
for idx, (delay, qc) in enumerate(zip(time_delays_ns, isa_decay_expts)):
    counts = result[idx].data.c.get_counts()
    p1 = counts.get('1', 0) / 1000  # Assuming 1000 shots
    print(f"Delay {delay} ns: P(qubit=1) = {p1:.3f}")
```
## 4. Kryterium 4 – Uniwersalny zestaw bramek kwantowych {#4-criterion-4-universal-set-of-quantum-gates}

**Kryterium 4:** *„'Uniwersalny' zestaw bramek kwantowych."* Oznacza to, że sprzęt musi umożliwiać wykonanie *dowolnych* obliczeń kwantowych poprzez składanie skończonego zestawu podstawowych bramek. W klasycznych obliczeniach bramka NAND jest uniwersalna; w kwantowych istnieje wiele wyborów uniwersalnych zestawów bramek (np. \{H, T, CNOT\} lub bramki natywne konkretnej maszyny). Urządzenia IBM dysponują na przykład zestawem operacji natywnych, takich jak dowolne obroty jednoqubitowe i bramki CNOT pomiędzy określonymi Qubitami, które razem tworzą zestaw uniwersalny. Zadaniem Transpilatora Qiskit jest często **kompilowanie bramek wysokiego poziomu do tych bramek bazowych**.

**Cel demonstracji:** Zilustrowanie uniwersalności bramek poprzez pokazanie, jak Qiskit rozkłada bramki. Weźmiemy bramkę niennatywną (np. trzycubitową bramkę Toffoli, CCX) i zobaczymy, jak rozkłada się ona na bramki bazowe urządzenia. Demonstracja ta pokazuje, że dostarczony zestaw bramek jest rzeczywiście *uniwersalny* — można za jego pomocą uzyskać bardziej złożone operacje.

Najpierw sprawdźmy, jakie są bramki bazowe typowego backendu IBM. Odpytamy konfigurację urządzenia (lub jego wersji fake). Przykładowo bramki bazowe ibmq_brisbane:
Powinieneś zauważyć, że prawdopodobieństwo `P(qubit=1)` maleje wraz ze wzrostem czasu opóźnienia, podążając za krzywą zaniku wykładniczego charakterystyczną dla relaksacji T1. Bezpośrednio demonstruje to, jak skończony czas koherencji prowadzi do błędów obliczeniowych, gdy Circuit działa zbyt długo.

**Wpływ na algorytmy:** Jeśli spróbujesz uruchomić dłuższy algorytm (z wieloma kolejnymi bramkami), całkowity czas wykonania może zbliżyć się do T2 lub go przekroczyć, powodując utratę koherencji stanu przed zakończeniem obliczeń. Dlatego poprawa czasów koherencji i przyspieszanie bramek to jedne z najważniejszych celów w badaniach nad sprzętem kwantowym.

```python
from qiskit_ibm_runtime.fake_provider import FakeBrisbane
fake_brisbane = FakeBrisbane()
print("Basis gates for ibmq_brisbane:", fake_brisbane.configuration().basis_gates)
```

Wynik może wyglądać mniej więcej tak: `['id', 'rz', 'sx', 'x', 'ecr']`. Są to prymitywne operacje natywnie obsługiwane przez sprzęt (Identity/no-op, obrót RZ, bramka sqrt(X), bramka X i controlled-X). Każda inna bramka musi być z nich złożona. Ten zestaw jest znany jako uniwersalny dla obliczeń kwantowych (zasadniczo obroty jednokubitowe plus dwukubitowa bramka splątująca tworzą zestaw uniwersalny).

Teraz weź **bramkę Toffoli (CCX)** jako przypadek testowy. CCX odwraca docelowy Qubit tylko wtedy, gdy oba Qubity kontrolne mają wartość 1. Nie jest to bramka natywna w sprzęcie IBM. Qiskit udostępnia instrukcję `ccx`, ale pod spodem ją rozkłada.

### Ćwiczenie 4: Rozkład bramki Toffoli {#exercise-4-decompose-a-toffoli-gate}

Uzupełnij poniższy kod, aby zbudować Circuit z bramką Toffoli (CCX), a następnie użyj Qiskit do rozbicia jej na natywne bramki bazowe backendu `FakeBrisbane`.

```python
from qiskit import QuantumCircuit
from qiskit_ibm_runtime.fake_provider import FakeBrisbane

# The fake_brisbane backend from the previous cell is reused here.

# --- YOUR CODE HERE ---

# 1. Create a circuit that can accommodate a Toffoli gate
qc_toffoli = QuantumCircuit(3)

# Apply a CCX gate with controls on qubits 0, 1 and target on qubit 2
qc_toffoli.ccx(0, 1, 2)

# 2. Transpile the circuit to the fake Brisbane backend
pm = generate_preset_pass_manager(backend=FakeBrisbane(), optimization_level=1)

isa_qc_toffoli = pm.run(qc_toffoli)

# --- END YOUR CODE ---
```

W wynikach transpilacji powinieneś zobaczyć CCX zastąpione sekwencją bardziej podstawowych bramek, takich jak `rz`, `sx` i `ecr`. Dowodzi to, że bramki natywne są wystarczające do wyrażenia bramki Toffoli.

**Uniwersalność w praktyce:** Powyższe ćwiczenie pokazuje, że złożona bramka trzycubitowa została zbudowana z prostszych. Ogólnie rzecz biorąc, **dowolna** wielokubitowa jedność może być złożona z bramek 1- i 2-kubitowych. Transpiler jest kluczowym elementem każdego stosu oprogramowania kwantowego, ponieważ wypełnia lukę między abstrakcyjnymi algorytmami, które chcemy uruchomić, a fizycznymi operacjami, które konkretne urządzenie kwantowe może faktycznie wykonywać.

**Przykład urządzenia:** Urządzenie **ibmq_brisbane** używa architektury Eagle z bramkami bazowymi pokazanymi powyżej. Oznacza to, że każdy algorytm wysyłany do tych maszyn zostanie przekształcony w sekwencje tych operacji. To kryterium dotyczy w istocie **sterowalności**; mamy wystarczająco dużo elementów kontrolnych, by wykonać dowolną potrzebną operację na naszych Qubitach.

## 5. Kryterium 5 – Pomiar Qubitów {#5-criterion-5-qubit-measurement}

**Kryterium 5:** *„Zdolność do pomiaru specyficznego dla Qubita."* Stan każdego Qubita musi być mierzalny (zazwyczaj w bazie obliczeniowej, |0〉 lub |1〉). Innymi słowy, po uruchomieniu Circuit kwantowego musimy odczytać każdy Qubit jako klasyczny bit 0/1. Kryterium to dotyczy posiadania wiarygodnych detektorów dla każdego Qubita i możliwości wyboru, które Qubity mierzyć.

**Cel demonstracji:** Pokazanie, jak wykonywać pomiary w Qiskit na symulatorach i prawdziwych urządzeniach, oraz podkreślenie różnic (np. szumów pomiarowych). Zmierzymy kilka Qubitów w różnych stanach i przeanalizujemy wyniki. Zademonstrujemy również, jak mogą pojawiać się błędy odczytu, porównując wyniki symulatora z wynikami sprzętu.

Najpierw prosty przykład pomiaru:

```python
qc_measure = QuantumCircuit(2, 2)
qc_measure.x(0)              # qubit 0 -> |1>, qubit 1 stays |0>
qc_measure.measure([0, 1], [0, 1])
qc_measure.draw('mpl')
```

```python
sim_backend = AerSimulator()
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc_measure = pm.run(qc_measure)
job = sampler.run([isa_qc_measure], shots=1000)
result = job.result()
counts = result[0].data.c.get_counts()

print("Simulator measurement counts:", counts)
```

Na symulatorze oczekujemy 1000 zliczeń `01`. Teraz zobaczmy **błąd pomiarowy** w działaniu, symulując go. Możemy dodać błąd odczytu do naszego symulatora Aer. Qiskit Aer pozwala nam zdefiniować `ReadoutError` i dołączyć go do Qubitów w modelu szumu.

### Ćwiczenie 5: Symulacja błędu odczytu {#exercise-5-simulate-readout-error}

Uzupełnij kod, aby zdefiniować prosty model błędu odczytu, w którym każdy Qubit ma 2% szansy na błędny pomiar (0 odczytane jako 1 lub 1 odczytane jako 0). Następnie uruchom Circuit pomiarowy z tym modelem szumu.

```python
from qiskit_aer.noise import NoiseModel, ReadoutError

# --- YOUR CODE HERE ---

# 1. Define a 2% readout error for each single qubit.
# [[P(0|0), P(1|0)], [P(0|1), P(1|1)]]
ro_error = ReadoutError([[0.98, 0.02], [0.02, 0.98]])

# 2. Create a new noise model
noise_model_ro = NoiseModel()

# 3. Add the readout error to all qubits in the noise model
noise_model_ro.add_all_qubit_readout_error(ro_error)

# --- END YOUR CODE ---

sim_backend.set_options(noise_model=noise_model_ro)
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc_measure = pm.run(qc_measure)

# Run the measurement circuit with readout noise
sampler = Sampler(mode=sim_backend)

job = sampler.run([isa_qc_measure], shots=1024)
result = job.result()
counts = result[0].data.c.get_counts()

print("Simulation with 2% readout error:", counts)
```

Symulowane dane wyjściowe pokażą pewne błędne zliczenia (np. `11`, `00`, `10`) podobne do tych, które może produkować prawdziwy sprzęt, demonstrując wpływ niedoskonałego pomiaru.

**Przykład urządzenia:** Na prawdziwym urządzeniu, takim jak **ibmq_brisbane**, można by uruchomić ten sam Circuit i prawdopodobnie zobaczyć podobne, niezerowe zliczenia dla niepoprawnych wyników. Dane kalibracyjne urządzenia zawierają błąd odczytu dla każdego Qubita. Zdolność do celowania i odczytu konkretnych Qubitów jest kluczowa, a zrozumienie ich charakterystyk błędów jest niezbędne do uzyskania sensownych wyników. Uruchamianie na prawdziwym urządzeniu zostało zademonstrowane w **Ćwiczeniu 1b: Uruchomienie na prawdziwym komputerze IBM Quantum**.

## Kryteria komunikacji kwantowej (Qubity latające) {#quantum-communication-criteria-flying-qubits}

DiVincenzo wymienił również dwa kryteria specyficzne dla komunikacji kwantowej, ważne przy budowie sieciowego komputera kwantowego:

6. **Zdolność do wzajemnej konwersji Qubitów stacjonarnych i „latających".** (Np. odwzorowanie Qubita w procesorze na foton, który może podróżować.)

Teleportacja pokazuje, jak zamienić stan stacjonarnego Qubita w informację przenoszoną przez splątaną parę (część „latającą") i komunikację klasyczną, która jest następnie używana do odtworzenia stanu na innym stacjonarnym Qubicie w innym miejscu.

### Zalecana aktywność: Moduł *Quantum Teleportation* z Qiskit in Classrooms {#recommended-activity-take-the-qiskit-in-classrooms-quantum-teleportation-module}

Qiskit pozwala nam perfekcyjnie symulować logikę protokołu, dostarczając konceptualnego modelu tego, jak te kryteria są spełniane w architekturach komunikacyjnych.
