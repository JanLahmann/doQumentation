## סקירה כללית {#overview}

עם Iskay Quantum Optimizer של Kipu Quantum, תוכל להתמודד עם בעיות אופטימיזציה מורכבות באמצעות מחשבי קוונטום של IBM&reg;. פותר זה מנצל את אלגוריתם [bf-DCQO](https://doi.org/10.48550/arXiv.2409.04477) החדשני של Kipu, הדורש רק את פונקציית המטרה כקלט ומספק פתרונות לבעיות באופן אוטומטי. הוא יכול לטפל בבעיות אופטימיזציה הכוללות עד 156 Qubit, ומאפשר שימוש בכל ה-Qubit של מכשירי IBM קוונטום. ה-Optimizer משתמש במיפוי של 1-ל-1 בין משתנים קלאסיים ל-Qubit, מה שמאפשר לך להתמודד עם בעיות אופטימיזציה עם עד 156 משתנים בינאריים.

ה-Optimizer מאפשר פתרון בעיות אופטימיזציה בינארית ללא אילוצים. בנוסף לניסוח QUBO (Quadratic Unconstrained Binary Optimization) הנפוץ, הוא תומך גם בבעיות אופטימיזציה מסדר גבוה יותר (HUBO). הפותר משתמש באלגוריתם קוונטי לא-וריאציוני, המבצע את רוב החישובים על מכשירים קוונטומיים.

להלן מידע נוסף על האלגוריתם בו נעשה שימוש, מדריך קצר כיצד להשתמש בפונקציה, וכן תוצאות בנצ'מרק על מופעי בעיות שונים בגדלים ומורכבויות שונים.

## תיאור {#description}

ה-Optimizer הוא מימוש מוכן לשימוש של אלגוריתמי אופטימיזציה קוונטיים חדשניים. הוא פותר בעיות אופטימיזציה על ידי הרצת מעגלים קוונטיים דחוסים מאוד על חומרה קוונטית. דחיסה זו מושגת על ידי הכנסת איברים counterdiabatic לתוך אבולוציית הזמן הבסיסית של המערכת הקוונטית. האלגוריתם מבצע מספר איטרציות של ריצות חומרה לקבלת הפתרונות הסופיים ומשלב זאת עם עיבוד לאחר. שלבים אלה משולבים בצורה חלקה בתהליך העבודה של ה-Optimizer ומבוצעים באופן אוטומטי.

### כיצד עובד ה-Quantum Optimizer? {#how-does-the-quantum-optimizer-work}

סעיף זה מתאר את היסודות של אלגוריתם bf-DCQO המיושם. ניתן למצוא מבוא לאלגוריתם גם [בערוץ YouTube של Qiskit.](https://www.youtube.com/watch?v=33QmsXhIlpU&t=1223s)

האלגוריתם מבוסס על אבולוציית הזמן של מערכת קוונטית המשתנה לאורך זמן, כאשר פתרון הבעיה מקודד במצב היסוד של המערכת הקוונטית בסוף האבולוציה. על פי [משפט האדיאבטי](https://en.wikipedia.org/wiki/Adiabatic_theorem), אבולוציה זו חייבת להיות איטית כדי להבטיח שהמערכת תישאר במצב היסוד שלה. דיגיטציה של אבולוציה זו היא הבסיס לחישוב אדיאבטי קוונטי מדויק (DQA) ולאלגוריתם QAOA המפורסם. עם זאת, האבולוציה האיטית הנדרשת אינה ישימה עבור גדלי בעיות גדלים יותר, מכיוון שהיא מביאה לעומק מעגל הולך וגדל. על ידי שימוש בפרוטוקולים counterdiabatic, ניתן לדכא עירורים לא רצויים המתרחשים במהלך זמני אבולוציה קצרים תוך כדי הישארות במצב היסוד. כאן, דיגיטציה של זמן אבולוציה קצר יותר זה מביאה למעגלים קוונטיים עם עומק קצר יותר ופחות שערי הסתבכות.

המעגלים של אלגוריתמי bf-DCQO משתמשים בדרך כלל בפחות עד עשר פעמים שערי הסתבכות מאשר DQA, ובשלוש עד ארבע פעמים פחות שערי הסתבכות מאשר מימושי QAOA סטנדרטיים. בגלל מספר השערים הקטן יותר, מתרחשות פחות שגיאות במהלך ביצוע המעגל על החומרה. לכן, ה-optimizer אינו דורש שימוש בטכניקות כמו דיכוי שגיאות או הפחתת שגיאות. יישומן בגרסאות עתידיות יכול לשפר עוד יותר את איכות הפתרון.

למרות שאלגוריתם bf-DCQO משתמש באיטרציות, הוא אינו וריאציוני. לאחר כל איטרציה של האלגוריתם, מצב ההתפלגות נמדד. ההתפלגות המתקבלת משמשת לחישוב מה שנקרא שדה-הטיה. שדה-ההטיה מאפשר להתחיל את האיטרציה הבאה ממצב אנרגיה קרוב לפתרון שנמצא קודם לכן. בדרך זו, האלגוריתם מתקדם עם כל איטרציה לפתרונות של אנרגיה נמוכה יותר. בדרך כלל, כעשר איטרציות מספיקות להתכנסות לפתרון, ובסך הכול דורשות מספר נמוך בהרבה של איטרציות מאשר אלגוריתמים וריאציוניים, שהוא בסדר גודל של כ-100 איטרציות.

ה-optimizer משלב את אלגוריתם bf-DCQO עם עיבוד לאחר קלאסי. לאחר מדידת התפלגות המצבים, מבוצע חיפוש מקומי. במהלך החיפוש המקומי, הביטים של הפתרון הנמדד הופכים באופן אקראי. לאחר ההיפוך, נבדקת האנרגיה של מחרוזת הביטים החדשה. אם האנרגיה נמוכה יותר, מחרוזת הביטים נשמרת כפתרון החדש. החיפוש המקומי גדל בצורה ליניארית בלבד עם מספר ה-Qubit; לכן, הוא זול מבחינה חישובית. מכיוון שהעיבוד לאחר מתקן היפוכי ביטים מקומיים, הוא מפצה על שגיאות היפוך-ביטים שהן לרוב תוצאה של אי-שלמות חומרה ושגיאות קריאה.

מדדי הבנצ'מרק שלהלן מראים שה-Optimizer מטפל ביעילות בבעיות הכוללות עד 156 Qubit, ומציעים סקירה כללית של דיוק ה-optimizer ויכולת ההרחבה שלו על פני סוגי בעיות שונים. שים לב שמדדי הביצועים בפועל עשויים להשתנות בהתאם למאפיינים הספציפיים של הבעיה, כגון מספר המשתנים, הצפיפות והמקומיות של האיברים בפונקציית המטרה, והסדר הפולינומי.

הטבלה הבאה כוללת את יחס הקירוב (AR), מדד המוגדר כדלקמן:
$$
AR = \frac{C^{*} - C_\textrm{max}}{C_{\textrm{min}} - C_{\textrm{max}}},
$$
כאשר $C$ היא פונקציית המטרה, $C_{\textrm{min}}$, $C_{\textrm{max}}$ הם הערכים המינימלי והמקסימלי שלה, ו-$C^{*}$ הוא עלות הפתרון הטוב ביותר שנמצא, בהתאמה. לכן, AR=100\% פירושו שהושג מצב היסוד של הבעיה.

#### פורמטים מקובלים לבעיה {#accepted-problem-formats}

הארגומנטים `problem` ו-`problem_type` מקודדים בעיית אופטימיזציה בצורה

$$
\begin{align}
\min_{(x_1, x_2, \ldots, x_n) \in D} C(x_1, x_2, \ldots, x_n) \nonumber
\end{align}
$$
כאשר

$$
C(x_1, ... , x_n) = a + \sum_{i} b_i x_i + \sum_{i, j} c_{i, j} x_i x_j + ... + \sum_{k_1, ..., k_m} g_{k_1, ..., k_m} x_{k_1} ... x_{k_m}
$$

- על ידי בחירת `problem_type = "binary"`, אתה מציין שפונקציית העלות היא בפורמט `binary`, כלומר $D = \{0,  1\}^{n}$, כלומר פונקציית העלות כתובה בניסוח QUBO/HUBO.
- מצד שני, על ידי בחירת `problem_type = "spin"`, פונקציית העלות כתובה בניסוח Ising, כאשר $D = \{-1, 1\}^{n}$.

המקדמים של הבעיה צריכים להיות מקודדים במילון כדלקמן:
$$
\begin{align} \nonumber
&\texttt{\{} \\ \nonumber
&\texttt{"()"}&: \quad &a, \\ \nonumber
&\texttt{"(i,)"}&: \quad &b_i, \\ \nonumber
&\texttt{"(i, j)"}&: \quad &c_{i, j}, \\ \nonumber
&\quad  \vdots \\ \nonumber
&\texttt{"(} k_1, ..., k_m  \texttt{)"} &: \quad &g_{k_1, ..., k_m}, \\ \nonumber
&\texttt{\}}
\end{align}
$$

- שים לב שמפתחות המילון חייבים להיות מחרוזות המכילות tuple תקין של מספרים שלמים שאינם חוזרים על עצמם.

#### אפשרויות {#options}

Iskay מספקת יכולות כיוון עדין דרך פרמטרים אופציונליים. בעוד שברירות המחדל עובדות טוב עבור רוב הבעיות, תוכל להתאים אישית את ההתנהגות לדרישות ספציפיות:

| פרמטר | סוג | ברירת מחדל | תיאור |
|-----------|------|---------|-------------|
| **shots** | `int` | 10000 | מדידות קוונטומיות לכל איטרציה (גבוה יותר = מדויק יותר) |
| **num_iterations** | `int` | 10 | איטרציות של האלגוריתם (יותר איטרציות יכולות לשפר את איכות הפתרון) |
| **use_session** | `bool` | True | שימוש בסשנים של IBM לזמני תור מופחתים |
| **seed_transpiler** | `int` | None | הגדר לקומפילציית מעגל קוונטי ניתנת לשחזור |
| **direct_qubit_mapping** | `bool` | False | מיפוי Qubit וירטואליים ישירות ל-Qubit פיזיים |
| **job_tags** | `List[str]` | None | תגיות מותאמות אישית למעקב אחר משימות |
| **preprocessing_level** | `int` | 0 | עוצמת עיבוד מוקדם של הבעיה (0-3) - ראה פרטים להלן |
| **postprocessing_level** | `int` | 2 | רמת עידון פתרון (0-2) - ראה פרטים להלן |
| **transpilation_level** | `int` | 0 | ניסיונות אופטימיזציה של Transpiler (0-5) - ראה פרטים להלן |
| **transpile_only** | `bool` | False | ניתוח אופטימיזציית מעגל ללא הרצת ביצוע מלא |

**רמות עיבוד מוקדם (0-3)**: חשוב במיוחד לבעיות גדולות יותר שאינן יכולות להתאים כרגע לזמני הקוהרנטיות של החומרה. רמות עיבוד מוקדם גבוהות יותר משיגות עומקי מעגל רדודים יותר על ידי קירובים בהעברת הבעיה ל-Transpiler:
- **רמה 0**: מעגלים מדויקים, ארוכים יותר
- **רמה 1**: איזון טוב בין דיוק לקירוב, גיזום שערים עם זוויות בפרצנטיל העשירי הנמוך ביותר
- **רמה 2**: קירוב מעט גבוה יותר, גיזום שערים עם זוויות בפרצנטיל ה-20 הנמוך ביותר ושימוש ב-`approximation_degree=0.95` בהעברה
- **רמה 3**: רמת קירוב מקסימלית, גיזום שערים בפרצנטיל ה-30 הנמוך ביותר ושימוש ב-`approximation_degree=0.90` בהעברה

**רמות העברה (0-5)**: שליטה בניסיונות אופטימיזציית Transpiler מתקדמים לקומפילציית מעגל קוונטי. זה יכול להוביל לעלייה בעומס הקלאסי, ובמקרים מסוימים ייתכן ולא ישנה את עומק המעגל. ערך ברירת המחדל `2` בדרך כלל מוביל למעגל הקצר ביותר והוא מהיר יחסית.
- **רמה 0**: אופטימיזציה של מעגל DCQO המפורק (פריסה, ניתוב, תזמון)
- **רמה 1**: אופטימיזציה של `PauliEvolutionGate` ולאחר מכן מעגל DCQO המפורק (max_trials=10)
- **רמה 2**: אופטימיזציה של `PauliEvolutionGate` ולאחר מכן מעגל DCQO המפורק (max_trials=15)
- **רמה 3**: אופטימיזציה של `PauliEvolutionGate` ולאחר מכן מעגל DCQO המפורק (max_trials=20)
- **רמה 4**: אופטימיזציה של `PauliEvolutionGate` ולאחר מכן מעגל DCQO המפורק (max_trials=25)
- **רמה 5**: אופטימיזציה של `PauliEvolutionGate` ולאחר מכן מעגל DCQO המפורק (max_trials=50)

#### ביצוע סטנדרטי {#standard-execution}

כאשר הפרמטר האופציונלי הוא `transpile_only=False`:

**מילון solution_info:**
- **"bitstring"** (`str`): ייצוג מחרוזת הביטים הגולמי של הפתרון.
- **"cost"** (`float`): ערך העלות/אנרגיה הקשור לפתרון.
- **"seed_transpiler"** (`int`): ה-seed האקראי ששימש ל-Transpiler שייצר תוצאה זו.
- **"mapping"** (`Dict[int, int]`): מיפוי ה-Qubit-למשתנה המקורי ששימש בחישוב.
- **"qpu_time"** (`float`, אופציונלי): זמן ביצוע ה-QPU בשניות.

**הערות מיפוי משתנים:**
- מילון ה-`solution` מתקבל ממחרוזת הביטים של הפתרון, תוך שימוש באובייקט ה-`mapping` לאינדוקס המשתנים.
- כאשר `problem_type=spin` אנו משתמשים בהקצאה $1 \rightarrow -1, \quad 0 \rightarrow 1$.
- מפתחות במילון הפתרון הם אינדקסי משתנים ממוינים מספרית כמחרוזות.

#### ניתוח העברה {#transpilation-analysis}

כאשר הפרמטר האופציונלי הוא `transpile_only=True`:

**מילון transpilation_info:**
- **"best_seed"** (`int`): ה-seed האופטימלי שנמצא להעברה
- **"transpilation_time_seconds"** (`float`): זמן שנדרש לתהליך ההעברה
- **"transpiled_circuit"** (`Dict`): ניתוח מעגל המכיל:
  - **"depth"** (`int`): עומק המעגל (מספר שכבות)
  - **"gate_count"** (`int`): המספר הכולל של שערים במעגל
  - **"num_qubits"** (`int`): מספר ה-Qubit בשימוש
  - **"width"** (`int`): רוחב המעגל
  - **"operations"** (`Dict[str, int]`): ספירת כל סוג שער בשימוש

**שימוש במצב Transpile-only:**
- זמין למשתמשים שרוצים לנתח אופטימיזציית מעגל מבלי להריץ את ביצוע האלגוריתם הקוונטי המלא.
- שימושי לניתוח מעגלים, מחקרי אופטימיזציית עומק, והבנת השפעות ההעברה לפני התחייבות לביצוע מלא.

## דוגמה 2: MaxCut {#example-2-maxcut}

בעיות גרף רבות כמו MaxCut או Maximum independent set הן בעיות NP-קשות ומועמדות אידיאליות לבדיקת אלגוריתמים קוונטיים וחומרה. דוגמה זו מדגימה פתרון בעיית MaxCut של גרף 3-רגולרי עם ה-Quantum Optimizer.

להרצת דוגמה זו עליך להתקין את חבילת `networkx` בנוסף ל-`qiskit-ibm-catalog`. להתקנתה, הרץ את הפקודה הבאה:

### 1. יצירת פונקציית המטרה {#1-create-the-objective-function-maxcut}

התחל ביצירת גרף 3-רגולרי אקראי. עבור גרף זה, אנו מגדירים את פונקציית המטרה של בעיית MaxCut.

```python
import networkx as nx

# Create a random 3-regular graph
G = nx.random_regular_graph(3, 10, seed=42)

# Create the objective function for MaxCut in Ising formulation
def graph_to_ising_maxcut(G):
    """
    Convert a NetworkX graph to an Ising Hamiltonian for the Max-Cut problem.
    Args:
        G (networkx.Graph): The input graph.
    Returns:
        dict: The objective function of the Ising model
    """
    # Initialize the linear and quadratic coefficients
    objective_func = {}
    # Populate the coefficients
    for i, j in G.edges:
        objective_func[f"({i}, {j})"] = 0.5
    return objective_func

objective_func = graph_to_ising_maxcut(G)
```

### 2. הרצת ה-Optimizer {#2-run-the-optimizer-maxcut}

פתור את הבעיה על ידי הרצת ה-optimizer.

```python
options = {"shots": 5000, "num_iterations": 5, "use_session": True}

arguments = {
    "problem": objective_func,
    "problem_type": "spin",
    "backend_name": backend_name,  # such as "ibm_fez"
    "options": options,
}

job = optimizer.run(**arguments)
```

### 3. אחזור התוצאה {#3-retrieve-the-result-maxcut}

אחזר את התוצאה ומפה את מחרוזת הביטים של הפתרון חזרה לצמתי הגרף המקוריים.

from github import Github
import urllib
import json
import ast

repo = "Kipu-Quantum-GmbH/benchmark-instances"
path = "maxcut/maxcut_32_nodes.json"
gh = Github()
repo = gh.get_repo(repo)
branch = "main"
file = repo.get_contents(urllib.parse.quote(path), ref=branch)

# load json file with benchmark problem
problem_json = json.loads(file.decoded_content)

# convert objective function to compatible format
objective_func = {
    key: ast.literal_eval(value) for key, value in problem_json.items()
}

# Setup configuration to run the optimizer
options = {
    "shots": 5_000,
    "num_iterations": 5,
    "use_session": True,
    "direct_qubit_mapping": False,
}

arguments = {
