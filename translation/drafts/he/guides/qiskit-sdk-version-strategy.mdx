# אסטרטגיית גרסאות Qiskit SDK {#qiskit-sdk-version-strategy}

מספרי גרסאות Qiskit עוקבים אחר [Semantic Versioning](https://semver.org/).
מספר הגרסה מורכב משלושה מרכיבים עיקריים: גרסת המייג'ור, המיינור והפאץ'. לדוגמה, במספר גרסה `X.Y.Z`, `X` היא גרסת המייג'ור, `Y` היא גרסת המיינור ו-`Z` היא גרסת הפאץ'.

שינויים בלתי-תואמים של ה-API שמורים לשחרורי גרסאות מייג'ור. התקופה **המינימלית** בין שחרורי גרסאות מייג'ור היא שנה אחת. גרסאות מיינור מציגות תכונות חדשות ותיקוני באגים מבלי לשבור תאימות API, ומתפרסמות מדי פעם (כיום כל שלושה חודשים) **אך ורק** עבור גרסת המייג'ור הנוכחית. גרסאות פאץ' מספקות תיקונים לבאגים שזוהו בגרסת המיינור האחרונה של כל סדרת שחרורים הנתמכת פעיל (כלומר, גרסת המייג'ור). אנו תומכים לכל היותר בשתי סדרות שחרורים בו-זמנית, דבר המתרחש רק במהלך תקופת החפיפה שלאחר שחרור גרסת מייג'ור חדשה, כמתואר בפירוט להלן.

## לוח שחרורים {#release-schedule}

להלן לוח שחרורים משוער:

![לוח שחרורים משוער של Qiskit](/docs/images/guides/install/release_schedule.avif)

לקבלת לוח שחרורים עדכני, עיין ב[רשימת ה-milestones](https://github.com/Qiskit/qiskit/milestones) של פרויקט Qiskit ב-GitHub, אשר תכיל תמיד את תוכנית השחרור הנוכחית.

עם שחרור גרסת מייג'ור חדשה, גרסת המייג'ור הקודמת נתמכת למשך לפחות שישה חודשים עם תיקוני באגים בלבד, ושנה אחת לתיקוני אבטחה. במהלך תקופה זו מתפרסמות רק גרסאות פאץ' עבור גרסת המייג'ור הזו. גרסת פאץ' סופית מתפרסמת כאשר הסיוע מופסק, ושחרור זה גם מתעד את סיום הסיוע לסדרת גרסאות המייג'ור הזו. נדרשת חלון סיוע ארוך יותר לגרסת המייג'ור הקודמת מכיוון שזה נותן לצרכני Qiskit downstream ולמשתמשיהם הזדמנות להעביר את הקוד שלהם.
ספריות downstream שתלויות ב-Qiskit לא אמורות להעלות את גרסת Qiskit המינימלית הנדרשת לגרסת מייג'ור חדשה מיד לאחר שחרורה, מכיוון שבסיס המשתמשים של הספרייה זקוק לזמן כדי לעבור לשינויי ה-API החדשים. קיומו של חלון סיוע מורחב לגרסת Qiskit המייג'ור הקודמת נותן לפרויקטי downstream זמן להבטיח תאימות עם גרסת המייג'ור הבאה. פרויקטי downstream יכולים לספק תמיכה בשתי סדרות שחרורים בו-זמנית כדי לתת למשתמשיהם נתיב העברה.

לצורך Semantic Versioning, ה-API הציבורי של Qiskit נחשב לכל מודול, מחלקה, פונקציה או שיטה מתועדת שאינה מסומנת כפרטית (עם קידומת קו תחתון `_`). עם זאת, ניתן לעשות חריגים מפורשים עבור APIs ספציפיים מתועדים. במקרים כאלה, APIs אלה יתועדו בבירור כממשקים שאינם נחשבים עדיין לממשקים יציבים, ואזהרה גלויה למשתמש תוצג באופן פעיל בכל שימוש בממשקים בלתי-יציבים אלה. בנוסף, במצבים מסוימים, ממשק המסומן כפרטי נחשב לחלק מה-API הציבורי. בדרך כלל זה מתרחש רק בשני מקרים: או הגדרת ממשק מופשט שבו מחלקות בנות מיועדות לדרוס/לממש שיטה פרטית כחלק מהגדרת מימוש של הממשק, או שיטות ברמה נמוכה לשימוש מתקדם שיש להן ממשקים יציבים אך אינן נחשבות בטוחות לשימוש, מכיוון שהנטל הוא על המשתמש לשמור על invariants המחלקה/הבטיחות בעצמו (הדוגמה הקנונית לכך היא השיטה `QuantumCircuit._append`).

גרסאות Python הנתמכות, גרסת Rust המינימלית הנתמכת (לבנייה של Qiskit מקוד מקור), וכל תלויות חבילות Python (כולל הגרסאות המינימליות הנתמכות של תלויות) בשימוש של Qiskit אינן חלק מערבויות תאימות לאחור ועשויות להשתנות בכל שחרור. רק שחרורי מיינור או מייג'ור יעלו את הדרישות המינימליות לשימוש או לבניית Qiskit (כולל הוספת תלויות חדשות), אך תיקוני פאץ' עשויים לכלול תמיכה בגרסאות חדשות של Python או תלויות אחרות. בדרך כלל הגרסה המינימלית של תלות מועלית רק כאשר גרסאות תלות ישנות יוצאות מסיוע או כאשר לא ניתן לשמור תאימות עם הגרסה האחרונה של התלות והגרסה הישנה.

## אסטרטגיית שדרוג {#upgrade-strategy}

כאשר משתחררת גרסת מייג'ור חדשה, נתיב השדרוג המומלץ הוא תחילה לשדרג לגרסת המיינור האחרונה של גרסת המייג'ור הקודמת. זמן קצר לפני גרסת מייג'ור חדשה, תתפרסם גרסת מיינור סופית. שחרור גרסת המיינור הסופית הזו `X.Y+1.0.0` שווה ערך ל-`X.Y.0` אך עם אזהרות והוצאות משימוש עבור כל שינויי API שנעשו בסדרת גרסאות המייג'ור החדשה.

לדוגמה, מיד לפני שחרור 1.0.0, יתפרסם שחרור 0.46.0. שחרור 0.46.0 יהיה שווה ערך לשחרור 0.45.0 אך עם אזהרות הוצאה משימוש נוספות המתעדות את שינויי ה-API שנעשו כחלק משחרור 1.0.0. דפוס זה ישמש לכל שחרורי גרסאות מייג'ור עתידיים.

משתמשי Qiskit צריכים תחילה לשדרג לגרסת המיינור הסופית הזו כדי לראות אזהרות הוצאה משימוש ולהתאים את השימוש שלהם ב-Qiskit לפני ניסיון שחרור שעלול לשבור תאימות. גרסת המייג'ור הקודמת תיתמך לפחות שישה חודשים כדי לתת זמן מספיק לשדרוג. דפוס טיפוסי לניהול זה הוא לנעץ את הגרסה המקסימלית כדי להימנע מהשימוש בסדרת המייג'ור הבאה עד שאתה בטוח בתאימות. לדוגמה, ציון `qiskit<2` בקובץ requirements כאשר גרסת Qiskit המייג'ור הנוכחית היא 1 מבטיח שאתה משתמש בגרסת Qiskit שאין בה שינויי API בלתי-תואמים.

הגבלת הגרסה לפחות מגרסת המייג'ור הבאה מבטיחה שתראה אזהרות הוצאה משימוש לפני שחרור מייג'ור.
ללא הגבלה, `pip` מתקין את הגרסה הזמינה החדשה ביותר כברירת מחדל.

פורמט הסריאליזציה QPY הוא תואם-לאחור כך שגרסת Qiskit חדשה תמיד יכולה לטעון קובץ QPY שנוצר עם גרסה קודמת של Qiskit. עם זאת, הפורמט אינו תואם-קדמה, כך שבעיקרון לא ניתן לטעון קבצי QPY שנוצרו עם גרסה חדשה יותר של Qiskit באמצעות שחרור ישן יותר. כדי להקל על העברת משתמשים בין שחרורי מייג'ור, הפונקציה [`qiskit.qpy.dump()`](https://docs.quantum.ibm.com/api/qiskit/qpy#dump) תתמוך תמיד לפחות בגרסה חופפת אחת בין השחרור `X.0.0` לשחרור `X-1.Y.0` (כאשר `Y` הוא גרסת המיינור האחרונה של אותה סדרה). הפרמטר `qiskit.qpy.dump(..., version=...)` יאפשר שמירת קבצי פורמט QPY שניתן לטעון על ידי שתי גרסאות המייג'ור מהשחרור החדש יותר. ראה פרטים נוספים ב-[RFC 0020](https://github.com/Qiskit/RFCs/blob/master/0020-release_cycle.md#qpy-support).

## גרסאות מקדימות {#pre-releases}

עבור כל שחרור מיינור ומייג'ור, Qiskit מפרסם גרסאות מקדימות התואמות ל-[PEP440](https://peps.python.org/pep-0440/). בדרך כלל מדובר במועמדי שחרור בצורה `X.Y.0rc1`. שחרורי ה-`rc` יהיו עם משטח API סופי ומשמשים לבדיקת שחרור עתידי.

שים לב שכאשר אחד מסיומות גרסאות-מקדימות של PEP440 (כגון `a`, `b` או `pre`) מתפרסם, אין לו אותן ערבויות כמו שחרור `rc`, והוא שחרור תצוגה מקדימה בלבד. ה-API עשוי להשתנות בין גרסאות מקדימות אלה לשחרור הסופי עם אותו מספר גרסה. לדוגמה, ל-`1.0.0pre1` עשוי להיות API סופי שונה מ-`1.0.0`.

## גרסאות אחרי-שחרור {#post-releases}

אם יש בעיות עם האריזה של שחרור, עשויה להיות מונפקת גרסת אחרי-שחרור לתיקון זה. אלה יעקבו אחר הצורה `X.Y.Z.1` כאשר המספר השלם הרביעי מציין שזהו גרסת אחרי-שחרור ראשונה של שחרור `X.Y.Z`. לדוגמה, בשחרור qiskit-terra (שם החבילה הישן של Qiskit) 0.25.2 הייתה בעיה עם פרסום חבילת sdist, ופורסמה גרסת אחרי-שחרור 0.25.2.1 שתיקנה בעיה זו. הקוד היה זהה, ו-0.25.2.1 רק תיקן את בעיית האריזה עבור השחרור.
