4. **קבוצה אוניברסלית של Gate-ים קוונטיים** (המאפשרת לבצע כל פעולה יוניטרית שרירותית).

כל אחד מהחלקים הבאים מתאים לקריטריון אחד. נשתמש ב-Qiskit כדי להמחיש את המושג בקוד ו**ניסויים אינטראקטיביים** שאפשר לנסות. לדוגמה, נראה כיצד הגדלת מספר ה-Qubit-ים ועומק ה-Circuit משפיעה על התוצאות (קריטריון 1), כיצד לאפס ולהכין מצבי Qubit (קריטריון 2), כיצד למדוד Qubit-ים על סימולטורים מול מכשירים אמיתיים (קריטריון 4), כיצד Qiskit מרכיב Gate-ים אוניברסליים (קריטריון 3), וכיצד קוהרנטיות סופית (T₁, T₂) משפיעה על חישובים (קריטריון 5). בסיום, יהיה לכם אינטואיציה עמוקה יותר למה כל קריטריון של דיווינצ'נצו משמעותי בפועל וכיצד Qiskit מאפשר להתנסות בהם.

## 1. קריטריון 1 – Qubit-ים ניתנים להרחבה ומאופיינים היטב

**קריטריון 1:** *"מערכת פיזית ניתנת להרחבה עם Qubit-ים מאופיינים היטב."* המשמעות היא שאנחנו צריכים פלטפורמת חומרה קוונטית שבה אפשר **להגדיל את מספר ה-Qubit-ים** ועדיין לשלוט בהם בצורה אמינה. המאפיינים של כל Qubit (רמות אנרגיה, שיעורי שגיאות, קישוריות וכו') צריכים להיות מובנים היטב. בעצם, אנחנו רוצים לבנות Circuit-ים גדולים יותר מבלי שהמערכת תתפרק. בפועל, כשאנחנו מגדילים את מספר ה-Qubit-ים או עומק ה-Circuit, שגיאות ודה-קוהרנטיות מצטברות, ולכן הוכחת *ניתנות להרחבה* משמעה גם הבנה כיצד הגדלת הגודל משפיעה על הביצועים.

**מטרת ההדגמה:** להשתמש ב-Qiskit כדי להראות את ההשפעה של הגדלת Circuit (במספר Qubit-ים או עומק Gate) על נאמנות הפלט. נדמה תרחיש אידיאלי מול תרחיש רועש כדי לראות כיצד מערכת גדולה יותר או Circuit עמוק יותר נחלש בגלל דה-קוהרנטיות ושגיאות.

ראשית, בואו נבנה מצב שזור קטן (מצב GHZ) על 3 Qubit-ים, ולאחר מכן מצב גדול יותר על 5 Qubit-ים, כמבחן הרחבה פשוט. מצב GHZ של *n* Qubit-ים הוא $\frac{1}{\sqrt{2}}(|0...0\rangle + |1...1\rangle)$. בסימולציה אידיאלית, מדידת GHZ בן n Qubit-ים מניבה רק שני תוצאות אפשריות (כולם 0 או כולם 1) בהסתברות שווה. נשווה את **הפלט האידיאלי** ל**פלט הרועש** כשאנחנו מגדילים n או עומק ה-Circuit.

```python
from qiskit import QuantumCircuit
from qiskit_aer import AerSimulator
from qiskit.visualization import plot_histogram
from qiskit.transpiler.preset_passmanagers import generate_preset_pass_manager
from qiskit_ibm_runtime import SamplerV2 as Sampler

# 3-qubit GHZ circuit
qc3 = QuantumCircuit(3, 3)
qc3.h(0)
qc3.cx(0, 1)
qc3.cx(1, 2)
qc3.measure([0, 1, 2], [0, 1, 2])

# 5-qubit GHZ circuit (scaling up the number of qubits)
qc5 = QuantumCircuit(5, 5)
qc5.h(0)
qc5.cx(0, range(1, 5))    # entangle qubit 0 with all others
qc5.measure(range(5), range(5))

# Transpile for a simulator backend
sim_backend = AerSimulator()
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc3 = pm.run(qc3)
isa_qc5 = pm.run(qc5)

# Run ideal simulations (no noise)
sampler = Sampler(mode=sim_backend)

job3 = sampler.run([isa_qc3], shots=1024)
result3 = job3.result()
counts3 = result3[0].data.c.get_counts()

job5 = sampler.run([isa_qc5], shots=1024)
result5 = job5.result()
counts5 = result5[0].data.c.get_counts()

print("3-qubit GHZ counts (ideal):", counts3)
plot_histogram(counts3, legend=['3-qubit ideal'], figsize=(6,4))
```

```python
print("5-qubit GHZ counts (ideal):", counts5)
plot_histogram(counts5, legend=['5-qubit ideal'], figsize=(6,4))
```

**תוצאה צפויה (מקרה אידיאלי):** GHZ בן 3 Qubit-ים מניב בתנאים אידיאליים בערך 50% `000` ו-50% `111` בספירות. GHZ בן 5 Qubit-ים מניב כ-50% `00000` ו-50% `11111`. אין מחרוזות סיביות אחרות מכיוון שהמצב בתנאים אידיאליים שזור לחלוטין ובעל קוהרנטיות מלאה. אמורים להופיע שני עמודים גבוהים בהיסטוגרמה עבור כל Circuit, המתאימים לתוצאות של כולם-אפסים וכולם-אחדות.

לאחר מכן, בואו נראה מה קורה ב**סביבה רועשת**. נשתמש ביכולות מודל הרעש של Qiskit Aer כדי לחקות שגיאות של מכשיר אמיתי. לדוגמה, נוכל לקחת את מאפייני Backend של IBM כדי ליצור מודל רעש הכולל שגיאות Gate, זמני Gate סופיים, רגיעת Qubit (T₁), דה-פאזינג (T₂) ושגיאות קריאה. כאן נשתמש ב**Backend מדומה** המייצג את מכשיר IBM Quantum Brisbane כדי ליצור מודל רעש, ונריץ מחדש את Circuit-י GHZ דרכו.
### תרגיל 1א: סימולציה עם רעש
השלם את הקוד להלן כדי לדמות את Circuit-י GHZ על סימולטור רועש המבוסס על Backend ה-`FakeBrisbane`. זה יראה לך כיצד הביצועים מתדרדרים עם הגדלת המערכת בסביבת רעש ריאליסטית.

```python
from qiskit_ibm_runtime.fake_provider import FakeBrisbane

# We will reuse the ideal circuits qc3 and qc5 and their results from the previous cell.

# 1. Create a fake backend for IBM Quantum Brisbane
brisbane_backend = FakeBrisbane()

# 2. Create a noisy AerSimulator from the fake backend's properties
noisy_sim = AerSimulator.from_backend(brisbane_backend)

# 3. Transpile the circuits for the noisy simulator
pm = generate_preset_pass_manager(backend=noisy_sim, optimization_level=1)

isa_qc3_noisy = pm.run(qc3)

isa_qc5_noisy = pm.run(qc5)

# 4. Run the noisy simulations using the Sampler and get the counts
sampler = Sampler(mode=noisy_sim)

job3 = sampler.run([isa_qc3_noisy], shots=1024)

result3_noisy = job3.result()

counts3_noisy = result3_noisy[0].data.meas.get_counts()

job5 = sampler.run([isa_qc5_noisy], shots=1024)

result5_noisy = job5.result()

counts5_noisy = result5_noisy[0].data.meas.get_counts()

# --- END YOUR CODE ---

# This part is done for you to visualize the results.
print("3-qubit GHZ counts (noisy):", counts3_noisy)
plot_histogram(counts3_noisy, legend=['3-qubit noisy'], figsize=(6,4))
```

**תוצאה צפויה (רועש מול אידיאלי):** עם רעש, בין אם בסימולציה ובין אם על מכשיר אמיתי, מצב GHZ הוא **פחות מושלם**. יופיעו תוצאות נוספות מעבר לכולם-0 וכולם-1. עבור 3 Qubit-ים, במקום 100% ב-`000`/`111`, חלק מההסתברות זולגת לסיביות אחרות (כמו `001`, `010` וכו') בגלל שגיאות Gate או דה-קוהרנטיות שהופכת Qubit-ים מסוימים. עבור 5 Qubit-ים, ההשפעה בולטת עוד יותר; ה-Circuit הגדול יותר (יותר Qubit-ים ושערי CNOT) צובר יותר שגיאות, כך שפסגות ה-0 וה-1 נמוכות יותר, ומופיעות תוצאות רבות אחרות. מגמה זו מדגימה את האתגר של *ניתנות להרחבה*: ככל שמרחיבים, שמירה על נאמנות גבוהה נהיית קשה יותר ללא תיקון שגיאות.

**תובנה:** מחשב קוונטי ניתן להרחבה צריך לשמר מתאמים קוונטיים עם גידול המערכת. הדוגמאות שלנו מראות כיצד הגדלת מספר ה-Qubit-ים/עומק ה-Gate גורמת לירידה בנאמנות התוצאות בנוכחות רעש. הקריטריונים הנותרים יעסקו בשמירה על Qubit-ים תקינים (שגיאות נמוכות, ניתנים לאתחול וכו') עם הגדלת הגודל.
