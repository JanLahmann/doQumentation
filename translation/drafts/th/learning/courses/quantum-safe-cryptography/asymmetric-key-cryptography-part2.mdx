## การทำลาย RSA {#breaking-rsa}

ประโยชน์และความปลอดภัยของอัลกอริทึม <DefinitionTooltip definition="ระบบ cryptosystem แบบคีย์สาธารณะที่ตั้งชื่อตาม Ron Rivest, Adi Shamir และ Leonard Adleman พัฒนาขึ้นในปี 1977">RSA</DefinitionTooltip> ที่อธิบายข้างต้นขึ้นอยู่กับสมมติฐานทางคณิตศาสตร์สองข้อ:

1. การหา <DefinitionTooltip definition="สำหรับจำนวนเต็ม $a$ และ modulus $m$ เศษจาก ax/m คือ 1 ซึ่งเป็นเฉพาะในกรณีของ coprime">modular multiplicative inverse</DefinitionTooltip> $d$ โดยมีเพียงสิทธิ์เข้าถึง $(e, n)$ เท่านั้นนั้นไม่สามารถทำได้ในเชิงการคำนวณ

2. ในการตั้งค่า RSA การดำเนินการ modular exponentiation มีพฤติกรรมเหมือน <DefinitionTooltip definition="ฟังก์ชัน one-way ที่คำนวณ output จาก inputs ได้ง่าย แต่ยากที่จะระบุ inputs จาก output">one-way trapdoor function</DefinitionTooltip> เมื่อใช้สำหรับการเข้ารหัส จะให้ ciphertext ที่อ่านไม่ออก และหากไม่มีสิทธิ์เข้าถึงคีย์ส่วนตัว การกลับการดำเนินการเพื่อกู้คืน plain text จาก ciphertext นั้นไม่สามารถทำได้ อย่างไรก็ตาม ด้วยสิทธิ์เข้าถึงคีย์ส่วนตัวซึ่งทำหน้าที่เป็น trapdoor การถอดรหัส ciphertext ทำได้ง่าย

การโจมตีที่โดดเด่นที่สุดต่ออัลกอริทึม <DefinitionTooltip definition="ระบบ cryptosystem แบบคีย์สาธารณะที่ตั้งชื่อตาม Ron Rivest, Adi Shamir และ Leonard Adleman พัฒนาขึ้นในปี 1977">RSA</DefinitionTooltip> มุ่งเป้าไปที่การบ่อนทำลายสมมติฐานที่ 1 โดยการกู้คืนหมายเลขคีย์ส่วนตัว $d$ อย่างมีประสิทธิภาพผ่านการแยกตัวประกอบ <DefinitionTooltip definition="modulus คือเศษที่เหลือหลังจากหารด้วยค่าที่ระบุ modulo คือการดำเนินการนี้">modulus</DefinitionTooltip> $n$ ดังที่จะแสดงด้านล่าง การกู้คืน $d$ นั้นง่ายหากมีสิทธิ์เข้าถึง <DefinitionTooltip definition="ตัวประกอบเฉพาะของจำนวน n คือจำนวนเฉพาะเหล่านั้นที่คูณกันแล้วได้ n">ตัวประกอบเฉพาะ</DefinitionTooltip> $p$ และ $q$ ของ $n$ หรือ totient <DefinitionTooltip definition="สัญลักษณ์กรีก phi ที่มักใช้แทน Euler's totient">$φ$</DefinitionTooltip>$(n)$ ขอให้จำว่า $p$, $q$ และ <DefinitionTooltip definition="สัญลักษณ์กรีก phi ที่มักใช้แทน Euler's totient">$φ$</DefinitionTooltip>$(n)$ ถูกเก็บเป็นความลับระหว่างการสร้างคีย์และถูกทิ้งหลังจากนั้น บุคคลที่สามที่กู้คืนข้อมูลนี้โดยใช้คอมพิวเตอร์คลาสสิกหรือควอนตัมจะเปิดเผยคีย์ส่วนตัวได้จริง ทำลาย RSA ดังนั้น การแยกตัวประกอบเฉพาะจึงเป็น computational primitive หลักที่จำเป็นสำหรับการทำลาย RSA
### การคำนวณแบบคลาสสิกและ RSA {#classical-computing-and-rsa}

การแยกตัวประกอบเฉพาะของจำนวนเต็มขนาดใหญ่เป็นที่ทราบกันว่ามี scaling แบบ <DefinitionTooltip definition="เติบโตเร็วกว่า polynomial เช่น ฟังก์ชัน exponential">super-polynomial</DefinitionTooltip> หรือ <DefinitionTooltip definition="Scale ช้ากว่า exponential เช่น polynomial และ linear scaling">subexponential</DefinitionTooltip> บนคอมพิวเตอร์คลาสสิก อัลกอริทึมคลาสสิกที่ดีที่สุดในการแยกตัวประกอบจำนวนเต็มที่ใหญ่กว่า $10^{100}$ คือ <DefinitionTooltip definition="General Number Field Sieve - อัลกอริทึมคลาสสิกที่มีประสิทธิภาพสำหรับการแยกตัวประกอบจำนวนเต็มขนาดใหญ่">*general number field sieve*</DefinitionTooltip> [(GNFS)](https://en.wikipedia.org/w/index.php?title=General_number_field_sieve&oldid=1151781536)

<details>

<summary>
รายละเอียดทางคณิตศาสตร์
</summary>

$L_n[\frac{1}{3}, (\frac{64}{9})^{(\frac{1}{3})}] = e^{[((\frac{64}{9})^{(\frac{1}{3})} + o(1)) (log(n))^{\frac{1}{3}}(log(log(n))^{\frac{2}{3}})]}$

เป็นฟังก์ชันของจำนวนเต็ม $n$ ที่ต้องแยกตัวประกอบ

Scaling นี้เป็น super-polynomial ในจำนวนบิตที่จำเป็นในการแทน $n$

ดังนั้น การแยกตัวประกอบเฉพาะจึงถือว่าไม่มีประสิทธิภาพบนคอมพิวเตอร์คลาสสิก

</details>

ในปัจจุบัน [จำนวนเต็มที่ใหญ่ที่สุดที่แยกตัวประกอบได้บนฮาร์ดแวร์คลาสสิก](https://en.wikipedia.org/w/index.php?title=RSA_Factoring_Challenge&oldid=1163481797) อยู่ที่ประมาณ 829 บิต หรือ 250 หลักทศนิยม เมื่อพิจารณาจากการเติบโตแบบ exponential ของพลังคอมพิวเตอร์คลาสสิกที่เกิดขึ้นในช่วงหลายทศวรรษที่ผ่านมา RSA ขนาด 1024 บิตไม่ถือว่าปลอดภัยในระยะใกล้อีกต่อไปและถูกเลิกใช้แล้ว อย่างไรก็ตาม ในอนาคตอันใกล้ การแยกตัวประกอบจำนวนเต็ม 2048 บิตที่มีขนาดอยู่ในช่วง $10^{617}$ ยังคงถือว่าไม่สามารถทำได้บนระบบคลาสสิก แสดงให้เห็นถึงความยังคงใช้งานได้ต่อไป อย่างไรก็ตาม การมาถึงของคอมพิวเตอร์ควอนตัมทำให้การประเมินนี้ไม่ถูกต้องอีกต่อไป
### อัลกอริทึมควอนตัม Shor และ RSA {#shors-quantum-algorithm-and-rsa}

อัลกอริทึมควอนตัมที่รู้จักกันดีที่สุดในปัจจุบันอาจเป็น <DefinitionTooltip definition="อัลกอริทึมควอนตัมที่สามารถหาตัวประกอบเฉพาะได้อย่างมีประสิทธิภาพ พัฒนาโดยนักคณิตศาสตร์ Peter Shor ในปี 1994">อัลกอริทึม Shor</DefinitionTooltip> สำหรับการหาตัวประกอบเฉพาะของจำนวนเต็ม เมื่อ [Peter Shor แนะนำในปี 1994](https://doi.org/10.1109/SFCS.1994.365700) มันถูกยอมรับว่าเป็นอัลกอริทึมควอนตัมตัวแรกที่ให้ <DefinitionTooltip definition="เติบโตเร็วกว่า polynomial เช่น ฟังก์ชัน exponential">super-polynomial speedup</DefinitionTooltip> เมื่อเทียบกับอัลกอริทึมคลาสสิกในปัญหาที่มีความสำคัญในทางปฏิบัติอย่างยิ่ง นั่นคือการแยกตัวประกอบเฉพาะ

อัลกอริทึม Shor สามารถแยกตัวประกอบเฉพาะด้วย $O(n^2)$ โดยที่ $n$ คือจำนวนบิต
<details>
<summary>
คำอธิบายทางคณิตศาสตร์ของอัลกอริทึม Shor
</summary>

ในบริบทของ <DefinitionTooltip definition="ระบบ cryptosystem แบบคีย์สาธารณะที่ตั้งชื่อตาม Ron Rivest, Adi Shamir และ Leonard Adleman พัฒนาขึ้นในปี 1977">RSA</DefinitionTooltip>, <DefinitionTooltip definition="อัลกอริทึมควอนตัมที่สามารถหาตัวประกอบเฉพาะได้อย่างมีประสิทธิภาพ พัฒนาโดยนักคณิตศาสตร์ Peter Shor ในปี 1994">อัลกอริทึม Shor</DefinitionTooltip> ทำงานโดยใช้ประโยชน์จาก <DefinitionTooltip definition="ช่วงเวลาที่ฟังก์ชันทำซ้ำตัวเอง">ความเป็นคาบ</DefinitionTooltip> ของ <DefinitionTooltip definition="Modular exponentiation คือเศษที่ได้จากการยกจำนวนเต็มขึ้นกำลัง (exponent) แล้วหารด้วยตัวเลขหนึ่ง (modulus) เช่น $x^e mod m$">ฟังก์ชัน modular exponential</DefinitionTooltip> $f(x) = a^x (mod~n)$ และให้ <DefinitionTooltip definition="ช่วงเวลาที่ฟังก์ชันทำซ้ำตัวเอง">period</DefinitionTooltip>-finding primitive แบบควอนตัมที่ช่วยให้แยกตัวประกอบเฉพาะของ <DefinitionTooltip definition="modulus คือเศษที่เหลือหลังจากหารด้วยค่าที่ระบุ modulo คือการดำเนินการนี้">modulus</DefinitionTooltip> $n$ ได้อย่างมีประสิทธิภาพ

โครงร่างระดับสูงที่ง่ายขึ้นของแผนการโดยรวมของ Shor ในการทำลาย RSA มีดังนี้:

1. กำหนด modulus $n$ ที่ถูกประกาศเป็นส่วนหนึ่งของคีย์สาธารณะ เลือกตัวเลข $a$ ที่เป็น coprime กับ $n$ กล่าวคือ $gcd(a,n) = 1$ เนื่องจากเราทราบว่า $n = p*q$ มีตัวประกอบเฉพาะสองตัว $(p, q)$ เกือบทุกจำนวนที่น้อยกว่า $n$ ที่เราเลือกแบบสุ่มมีแนวโน้มจะเป็น coprime กับ $n$

2. เมื่อเลือก $a$ แล้ว ให้หา exponent $r$ โดยที่ $a^r \equiv 1 (mod~n)$ สิ่งนี้หมายความว่า $a^r - 1 \equiv 0 (mod~n)$ การมีอยู่ของ <DefinitionTooltip definition="จำนวนครั้งที่ตัวเลขถูกคูณด้วยตัวเอง">exponent</DefinitionTooltip> $r$ โดยที่ <DefinitionTooltip definition="ความสอดคล้องหรือความเทียบเท่า กล่าวคือ ทุกอย่างทำงานในลักษณะเดียวกัน">congruence</DefinitionTooltip> ข้างต้นเป็นจริง ได้รับการรับประกันจาก <DefinitionTooltip definition="ช่วงเวลาที่ฟังก์ชันทำซ้ำตัวเอง">คุณสมบัติความเป็นคาบ</DefinitionTooltip> ของ <DefinitionTooltip definition="Modular exponentiation คือเศษที่ได้จากการยกจำนวนเต็มขึ้นกำลัง (exponent) แล้วหารด้วยตัวเลขหนึ่ง (modulus) เช่น $x^e mod m$">modular exponentiation</DefinitionTooltip>

3. ถ้า $r$ เป็นเลขคู่ $a^r - 1 \equiv 0 (mod~n) \implies (a^{r/2} - 1) (a^{r/2} + 1) =  \gamma*n$ สำหรับจำนวนเต็ม $\gamma$ บางตัว ด้านซ้ายของความเท่ากันนี้ต้องมี $p$ และ $q$ เป็นสองในตัวประกอบเฉพาะ เนื่องจากด้านขวามีตัวประกอบเหล่านั้น ถ้า $r$ เป็นเลขคี่ ให้กลับไปขั้นตอนที่ 1 และลองเลือก $a$ อื่น

4. ใช้ <DefinitionTooltip definition="วิธีที่มีประสิทธิภาพในการคำนวณ greatest common divisor ของจำนวนเต็มบวกสองตัว ซึ่งปรับปรุงจาก Euclidean algorithm ดั้งเดิม">extended Euclid algorithm</DefinitionTooltip> สำหรับการหา $gcd((a^{r/2} - 1), n)$ หรือ $gcd((a^{r/2} + 1), n)$ <DefinitionTooltip definition="Greatest Common Divisor - จำนวนเต็มบวกที่ใหญ่ที่สุดที่สามารถหารจำนวนเต็มบวกสองตัวหรือมากกว่าได้ลงตัว">GCD</DefinitionTooltip>ที่คำนวณได้มีแนวโน้มสูงมากที่จะระบุตัวประกอบเฉพาะตัวหนึ่ง $p$ หรือ $q$ หาร $n$ ด้วยตัวประกอบเฉพาะที่พบแล้วเพื่อกู้คืนตัวอื่น

5. เมื่อทราบ $p, q$ แล้ว ให้ใช้ขั้นตอนจากอัลกอริทึม RSA ดั้งเดิมเพื่อสร้าง totient $\phi(n)$ ขึ้นมาใหม่ และสร้างหมายเลขคีย์ส่วนตัว $d$ เป็น <DefinitionTooltip definition="สำหรับจำนวนเต็ม $a$ และ modulus $m$ เศษจาก ax/m คือ 1 ซึ่งเป็นเฉพาะในกรณีของ coprime">modular inverse</DefinitionTooltip> ของหมายเลขคีย์สาธารณะ $e$ ที่ทราบอยู่แล้ว

</details>
ในเดือนสิงหาคม 2023 Oded Regev [ได้ประกาศการปรับปรุง](https://doi.org/10.48550/arXiv.2308.06572) จาก Shor ดั้งเดิมโดยใช้แนวทาง multi-dimensional ได้ผลลัพธ์ที่ $O(n^{1.5})$ ยังคงมีการวิจัยต่อเนื่องรวมถึงโดย [Ragavan และ Vaikuntanathan](https://doi.org/10.48550/arXiv.2310.00899) ในพื้นที่นี้ซึ่งอาจปรับปรุงเวลา ต้นทุน หรือจำนวน qubits ที่จำเป็น แม้ว่าเราจะไม่สามารถบอกได้ว่าการรันอัลกอริทึมดังกล่าวกับการเข้ารหัส RSA จริงในโลกจริงจะเป็นไปได้จริงเมื่อใด แต่มันกำลังเข้าใกล้มากขึ้นทุกที
#### ตัวอย่าง Python แสดงการทำลายการเข้ารหัส RSA {#python-example-demonstrating-breaking-rsa-encryption}

ในเซลล์โค้ดต่อไปนี้ เราจะแสดงตัวอย่างการหาคีย์ส่วนตัวโดยมีเพียงคีย์สาธารณะ ซึ่งจะใช้การคำนวณคลาสสิกแบบ brute-force แต่แสดงให้เห็นว่าอัลกอริทึม Shor สามารถนำมาใช้ได้อย่างไร รวมถึงสำหรับคีย์ขนาดใหญ่

<Admonition type="note">
ในส่วนนี้เราจะแสดงการคำนวณทางคณิตศาสตร์โดยละเอียดเป็นส่วนหนึ่งของโค้ด Python
</Admonition>

ในตัวอย่าง เรามีคีย์สาธารณะ $(5, 247)$ และเราจะกู้คืนคีย์ส่วนตัว

**ขั้นตอนที่ 1:** ขั้นตอนแรกคือการเลือกตัวเลขที่เป็น coprime กับ 247 เกือบทุกจำนวนที่เราเดาจะทำงานได้ ลองเลือก 6

```python
n = 247  # the modulus
e = 5  # public key number
a = 6  # an integer coprime to n
assert gcd(a, n) == 1
print(f"Checked {n} and {a} are coprime")
```

**ขั้นตอนที่ 2:** ต่อมาเราต้องหา <DefinitionTooltip definition="ช่วงเวลาที่ฟังก์ชันทำซ้ำตัวเอง">period</DefinitionTooltip> $r$ โดยที่ $6^r \equiv 1 (mod~247)$ ในตัวอย่างนี้เราคำนวณ $r$ แบบคลาสสิกโดยใช้ brute force แต่เราก็สามารถใช้อัลกอริทึม Shor บนคอมพิวเตอร์ควอนตัมโดยใช้ <DefinitionTooltip definition="SDK โอเพนซอร์สสำหรับการทำงานกับอัลกอริทึมและคอมพิวเตอร์ควอนตัม พัฒนาโดย IBM">Qiskit</DefinitionTooltip>

```python
r = 0
rem = 100
while rem != 1:
    r += 1
    rem = (a**r) % n

print(f"period r is: {r}")
assert a**r % n == 1

print(f"Checked {a}^{r} mod {n} is 1")
```

**ขั้นตอนที่ 3:** เนื่องจาก <DefinitionTooltip definition="ช่วงเวลาที่ฟังก์ชันทำซ้ำตัวเอง">period</DefinitionTooltip> $r = 36$ เป็นเลขคู่ เราสามารถคำนวณ $f1 = (a^{r/2}-1), f2=(a^{r/2}+1)$

```python
# explicitly use as integer
f1 = int(a ** (r / 2) - 1)
f2 = int(a ** (r / 2) + 1)

print(f"f1 = {f1}")
print(f"f2 = {f2}")
```

**ขั้นตอนที่ 4:** หา <DefinitionTooltip definition="Greatest Common Divisor - จำนวนเต็มบวกที่ใหญ่ที่สุดที่สามารถหารจำนวนเต็มบวกสองตัวหรือมากกว่าได้ลงตัว">GCD</DefinitionTooltip> ของตัวประกอบใดตัวหนึ่งกับ $n$ เพียงหาร $n$ ด้วยตัวประกอบเฉพาะที่พบแล้วเพื่อได้ตัวประกอบเฉพาะตัวที่สอง

```python
q_found = gcd(f1, n)
print(f"One possible prime factor of n ({n}) is: {q_found}")

# explicit int (to avoid floating point)
p_found = int(n / q_found)
print(f"The second prime factor of n ({n}) is: {p_found}")

assert n == p_found * q_found
```

**ขั้นตอนที่ 5:** เมื่อกู้คืนตัวประกอบเฉพาะของ $n = 247$ เป็น $p_{found}=13$ และ $q_{found}=19$ แล้ว เราคำนวณ totient $\phi_{found} = (p_{found}-1)*(q_{found}-1)$

คีย์ส่วนตัวคือ <DefinitionTooltip definition="สำหรับจำนวนเต็ม $a$ และ modulus $m$ เศษจาก ax/m คือ 1 ซึ่งเป็นเฉพาะในกรณีของ coprime">modular inverse</DefinitionTooltip> ของหมายเลขคีย์สาธารณะ $e=5$

```python
# Compute the totient
phi_found = (p_found - 1) * (q_found - 1)
print(f"The totient is: {phi_found}")

# Recover the private key number d_found by satisfying (d_found * e) % phi_found = 1
d_found = 1
while True:
    if (d_found * e) % phi_found == 1:
        break
    else:
        d_found += 1
print("Private Key number:", d_found)
```

ในแผนการข้างต้น ขั้นตอนที่ 2 คือการดำเนินการ <DefinitionTooltip definition="ช่วงเวลาที่ฟังก์ชันทำซ้ำตัวเอง">period</DefinitionTooltip>-finding ที่สำคัญ ซึ่ง [อัลกอริทึม Shor](https://en.wikipedia.org/w/index.php?title=Shor%27s_algorithm&oldid=1164409434) ใช้ primitives ควอนตัมพื้นฐานสองอย่าง ได้แก่ <DefinitionTooltip definition="อัลกอริทึมควอนตัมที่ใช้หา periodic patterns ตั้งชื่อตาม Fourier transform ในฟิสิกส์คลาสสิกที่สามารถแยก signal เป็นชุดของคลื่น sine">*quantum Fourier transform*</DefinitionTooltip> และ <DefinitionTooltip definition="อัลกอริทึมควอนตัมที่ใช้ประเมิน phase ใน qubits ตั้งชื่อตาม phase estimation ในฟิสิกส์คลาสสิก">*quantum phase estimation*</DefinitionTooltip> สำหรับคำอธิบายโดยละเอียดเกี่ยวกับด้านควอนตัมของอัลกอริทึม Shor ดูบทเรียน Phase estimation and factoring ใน [Fundamentals of Quantum Algorithms](/learning/courses/fundamentals-of-quantum-algorithms) ขั้นตอนที่ 1 และ 3 ถึง 5 เกี่ยวข้องกับการดำเนินการที่ค่อนข้างไม่แพงซึ่งสามารถดำเนินการบนคอมพิวเตอร์คลาสสิกได้ง่าย

ตัวเลือก นี่คือคำอธิบายภาพโดยละเอียดของการ implement [อัลกอริทึม Shor](https://www.youtube.com/watch?v=EdJ7RoWcU48&t=325s)

บนคอมพิวเตอร์ควอนตัม อัลกอริทึม Shor สามารถแสดง <DefinitionTooltip definition="อัลกอริทึมที่ scale คล้าย logarithmic ซึ่งมีประสิทธิภาพเมื่อเทียบกับขนาดข้อมูล binary search เป็นตัวอย่างง่ายๆ">polylogarithmic scaling</DefinitionTooltip> ที่ดีเท่ากับ $O((log~n)^2 (log~log~n))$ ในแง่ของ modulus $n$ หรือ <DefinitionTooltip definition="Scale ตาม power เช่น n^3 เติบโตเร็วกว่า linear scaling แต่ช้ากว่า exponential">polynomial scaling</DefinitionTooltip> ในแง่ของจำนวนบิตที่จำเป็นในการแทน $n$ นี่คือ <DefinitionTooltip definition="เติบโตเร็วกว่า polynomial เช่น ฟังก์ชัน exponential">super-polynomial speedup</DefinitionTooltip> เมื่อเทียบกับอัลกอริทึมคลาสสิก <DefinitionTooltip definition="General Number Field Sieve - อัลกอริทึมคลาสสิกที่มีประสิทธิภาพสำหรับการแยกตัวประกอบจำนวนเต็มขนาดใหญ่">GNFS</DefinitionTooltip>

[การประมาณทรัพยากรล่าสุด](https://quantum-journal.org/papers/q-2021-04-15-433/) ชี้ว่าโดยอิงจากสมมติฐานบางอย่างเกี่ยวกับการกำหนดค่าฮาร์ดแวร์ จะต้องใช้ <DefinitionTooltip definition="บิตควอนตัม เทียบเท่ากับบิตในการคำนวณคลาสสิก Qubit มีพฤติกรรมแตกต่างอย่างมากเพราะสามารถอยู่ใน superposition ได้ (นึกภาพการรวมกันของสถานะ) แต่เมื่อวัดแล้ว superposition จะพังทลาย และผลลัพธ์จะเป็น 0 หรือ 1">qubits</DefinitionTooltip> หลายหมื่นถึงหลายล้านตัวเพื่อทำลาย RSA ขนาด 2048 บิตโดยใช้อัลกอริทึม Shor ไม่ใช่เรื่องที่เป็นไปไม่ได้ที่คอมพิวเตอร์ควอนตัมที่มี qubits หลายหมื่นตัวจะพร้อมใช้งานในช่วงหลายปีข้างหน้า ทำให้ขอบล่างของการประมาณทรัพยากรเข้าถึงได้
## การแลกเปลี่ยนคีย์ Diffie-Hellman และ Digital Signature Algorithm {#diffie-hellman-key-exchange-and-the-digital-signature-algorithm}

ในส่วนก่อนหน้า เราได้กล่าวถึง RSA cryptosystem ซึ่งความปลอดภัยขึ้นอยู่กับความยากในการคำนวณของการแยกตัวประกอบเฉพาะ ที่นี่ เราจะกล่าวถึงโปรโตคอลคริปโตกราฟีแบบอสมมาตรยอดนิยมสองโปรโตคอล ได้แก่ [*Diffie-Hellman* key exchange](https://en.wikipedia.org/w/index.php?title=Diffie%E2%80%93Hellman_key_exchange&oldid=1163912552) (DH) และ [*Digital Signature Algorithm*](https://en.wikipedia.org/w/index.php?title=Digital_Signature_Algorithm&oldid=1148950522) (DSA) ซึ่งทั้งคู่ขึ้นอยู่กับปัญหาทางคณิตศาสตร์ที่แตกต่างกัน นั่นคือ discrete logarithm problem (DLP)
### ปัญหา discrete logarithm {#the-discrete-logarithm-problem}

ในสมการต่อไปนี้เราต้องหา $a$ โดยให้เพียง $e$, $M$, $c$ เท่านั้น

$a^e$ $mod$ $M = c$

เชื่อกันว่าสิ่งนี้ยากบนคอมพิวเตอร์คลาสสิกเนื่องจากการใช้เลขคณิต modulo และจึงเป็นพื้นฐานทางคณิตศาสตร์ที่ดีสำหรับอัลกอริทึมการเข้ารหัส

สิ่งนี้เรียกว่า [discrete logarithm problem](https://en.wikipedia.org/w/index.php?title=Discrete_logarithm&oldid=1161792952) (DLP)

### รายละเอียดทางคณิตศาสตร์ของ discrete logarithm problem {#mathematical-details-of-the-discrete-logarithm-problem}

DLP มักถูกกำหนดในบริบทของ <DefinitionTooltip definition="Group ที่ทุกองค์ประกอบถูกสร้างขึ้นโดยการใช้องค์ประกอบเดียวซ้ำๆ ที่รู้จักกันว่า generator กระบวนการนี้สร้างวัฏจักรขององค์ประกอบที่ปิดตัวเอง">cyclic groups</DefinitionTooltip> และระบุดังนี้

พิจารณา [cyclic group](https://en.wikipedia.org/w/index.php?title=Cyclic_group&oldid=1150017831) $G$ ที่ถูกสร้างโดย group element $g \in G$ และกำหนด arbitrary element $h \in G$ หาจำนวนเต็ม $k$ โดยที่ $h = g^{k}$

ที่นี่จำนวนเต็ม $k \equiv log_{g}h$ คือ discrete logarithm คุณสมบัติ cyclic ของ $G$ รับประกันว่าสำหรับทุก $h$ จำนวนเต็ม $k$ ที่ถูกต้องมีอยู่

สำหรับคริปโตกราฟี DLP บน <DefinitionTooltip definition="เซตของจำนวนเต็มแบบ modulo จำนวนเฉพาะ โดย group operation คือ modular multiplication องค์ประกอบใน multiplicative group ถูกเลือกในแบบที่ coprime กับ modulus ทำให้แต่ละองค์ประกอบมี multiplicative inverse ที่ไม่ซ้ำกันภายใน group">multiplicative group</DefinitionTooltip> ของจำนวนเต็ม modulo จำนวนเฉพาะ $p$ ที่แทนด้วย $(\mathbb{Z}_p)^{\times}$ กลายเป็นสิ่งที่มีประโยชน์ องค์ประกอบของ $(\mathbb{Z}_p)^{\times}$ คือ congruence classes ที่ระบุด้วยจำนวนเต็ม modulo $p$ ที่เป็น coprime กับ $p$

ตัวอย่างเช่น:

$(\mathbb{Z}_5)^{\times} = \{[1],[2],[3],[4]\}~\mathrm{and}~(\mathbb{Z}_7)^{\times} = \{[1],[2],[3],[4],[5],[6]\} $

การดำเนินการคูณ $(\times)$ บน groups เหล่านี้เป็นเพียงการคูณจำนวนเต็มธรรมดาตามด้วยการลดค่า modulo $p$ และการยกกำลังด้วยจำนวนเต็ม $k$ เป็นการคูณซ้ำ $k$ ครั้งและลดค่า modulo $p$

มาแสดงตัวอย่างของ DLP บน $(\mathbb{Z}_7)^{\times}$

multiplicative group นี้มี generators สองตัว ${[3],[5]}$ ที่เรียกอีกอย่างว่า primitive roots เราจะใช้ $[5]$ เป็น generator กล่าวคือ สร้างทุกองค์ประกอบของ $(\mathbb{Z}_7)^{\times}$ โดยใช้กำลังจำนวนเต็มที่ต่อเนื่องกันของ 5

```python
#Generate elements of (Z_7)^{x} using generator [5].
g = 5
p = 7
print(f"Using generator {g}")
for k in range(3*p):
    print(f"{g}**{k} mod {p} ≡ {(g**k)%7}")
```

เราเห็นว่าในเลขคณิต modulo 7 การยก 5 ขึ้นกำลังจำนวนเต็มที่ต่อเนื่องกันให้ทุกองค์ประกอบของ $(\mathbb{Z}_7)^{\times}$ ครั้งละหนึ่งครั้งก่อนที่วัฏจักรจะทำซ้ำอย่างไม่มีกำหนดด้วยรอบ $p-1 =6$

ดังนั้น DLP บน $(\mathbb{Z}_7)^{\times}$ ด้วย generator [5] คือ:

$ \mathrm{Given}~h \in (\mathbb{Z}_7)^{\times} \mathrm{, find~} k \mathrm{~such~that~} 5^{k} \equiv h~(mod~7)$

จาก output ของเซลล์ Python ข้างต้นเราเห็นว่า:

$h = 2 \implies k=4 \mathrm{~or~}  4 = log_5(2) (mod~7)$

$h = 6 \implies k=3 \mathrm{~or~}  3 = log_5(6) (mod~7)$

ในเลขคณิตจำนวนจริงธรรมดา การยกกำลังเป็นฟังก์ชัน <DefinitionTooltip definition="พฤติกรรมหรือความสัมพันธ์ที่เคลื่อนที่ในทิศทางเดียวอย่างสม่ำเสมอโดยไม่กลับทิศ เช่น ในฟังก์ชันที่เพิ่มขึ้นแบบ monotonic ค่าจะเพิ่มขึ้นอย่างสม่ำเสมอ">monotonic</DefinitionTooltip> และการหา logarithm ของตัวเลขใดก็ได้ในฐานใดก็ได้นั้นง่ายในเชิงการคำนวณ ในทางตรงกันข้าม ตามที่เห็นได้จากตัวอย่าง $(\mathbb{Z}_7)^{\times}$ ง่ายๆ ข้างต้น modular exponentiation เป็น non-monotonic และแม้ว่ามันจะเป็นคาบด้วยรอบ $p-1$ แต่นอกจากนั้นก็ไม่ใช่เรื่องง่าย ดังนั้น การคำนวณ inverse ของมัน ซึ่งก็คือ discrete logarithm จึงไม่มีประสิทธิภาพสำหรับ $p$ ขนาดใหญ่บนคอมพิวเตอร์คลาสสิก

การสังเกตนี้เป็นพื้นฐานของทั้ง Diffie-Hellman (DH) key exchange และ Digital Signature Algorithm (DSA) ที่จะกล่าวถึงในส่วนถัดไป

DLP สามารถขยายไปยัง cyclic subgroups ได้ดังนี้:

-   พิจารณา $(\mathbb{Z}_p)^{\times}$ ที่กำหนดข้างต้นและ element $g \in (\mathbb{Z}_p)^{\times}$ ที่มี prime order $r$ กล่าวคือ $g^r \equiv 1 (~mod~p)$
-   เซตของกำลังจำนวนเต็มของ $g$: $\{g^k~(mod~p) | 1 \le k \le r\} = \langle g \rangle$ เป็น cyclic subgroup ของ $(\mathbb{Z}_p)^{\times}$ ที่มี group order $r$
-   DLP สามารถระบุบน $\langle g \rangle$ โดยการเลือก $h \in \\langle g \rangle$ และถามหา $1 \le a \le r$ โดยที่ $ g^a~(mod~p) = h$
### การแลกเปลี่ยนคีย์ Diffie-Hellman {#diffie-hellman-key-exchange}

ในปี 1976 [Whitfield Diffie และ Martin Hellman ได้เสนอโปรโตคอลการแลกเปลี่ยนคีย์](https://doi.org/10.1109/TIT.1976.1055638) เพื่อให้สามารถสร้าง shared secret key ผ่านช่องทางการสื่อสารที่ไม่ปลอดภัย secret key นี้สามารถนำไปใช้โดยฝ่ายที่แชร์มันเพื่อการเข้ารหัสแบบสมมาตร อัลกอริทึมนี้พึ่งพา DLP

![รูปที่ 1. การแลกเปลี่ยนคีย์ Diffie-Hellman](/learning/images/courses/quantum-safe-cryptography/asymmetric-key-cryptography/key_exchange_dh.avif)

*รูปที่ 1. การแลกเปลี่ยนคีย์ Diffie-Hellman*

<details>

<summary>
รายละเอียดทางคณิตศาสตร์ของการแลกเปลี่ยนคีย์ Diffie-Hellman
</summary>

โดยมี Alice และ Bob เป็นสองฝ่ายที่สื่อสาร โปรโตคอลทำงานดังนี้:

- ขั้นแรก Alice และ Bob ตกลงกันเรื่องจำนวนเฉพาะขนาดใหญ่ $p$ และ primitive root หรือ generator $a$
- จากนั้น Alice เลือก secret exponent $k_A$ แบบสุ่มโดยที่ $1 \le k_A \le p-2$ และคำนวณ $h_A = a^{k_A}~(mod~p)$ โดย $k_A, h_A$ เป็นคีย์ส่วนตัวและสาธารณะของ Alice ตามลำดับ
- ต่อมา Bob เลือก secret exponent $k_B$ แบบสุ่มโดยที่ $1 \le k_B \le p-2$ และคำนวณ $h_B = a^{k_B}~(mod~p)$ โดย $k_B, h_B$ เป็นคีย์ส่วนตัวและสาธารณะของ Bob ตามลำดับ
- จากนั้น Alice ส่ง $h_A$ ให้ Bob และ Bob ส่ง $h_B$ ให้ Alice ผ่านช่องทางที่เชื่อถือได้แต่ไม่จำเป็นต้องปลอดภัย
- จากนั้น Alice ใช้ $h_B$ ที่ได้รับเพื่อคำนวณ shared secret key $ \kappa = h_B^{k_A}~(mod~p)$
- สุดท้าย Bob ใช้ $h_A$ ที่ได้รับเพื่อคำนวณ shared secret key $ \kappa = h_A^{k_B}~(mod~p)$

ด้วยโปรโตคอลนี้:

- Alice และ Bob ได้รับการรับประกันว่าจะได้ secret key $\kappa$ เดียวกัน เพราะ $h_B^{k_A}~(mod~p) = (a^{k_B})^{k_A}~(mod~p) = a^{k_A k_B}~(mod~p) = (a^{k_A})^{k_B}~(mod~p) = h_A^{k_B}~(mod~p)$
- บุคคลที่สามที่ดักจับ $h_A$ หรือ $h_B$ ไม่สามารถสร้าง secret key $\kappa$ ได้ เพราะพวกเขาไม่มีสิทธิ์เข้าถึง $k_B$ หรือ $k_A$ ตามลำดับ
- การดึง $k_A$ หรือ $k_B$ โดยใช้ข้อมูลสาธารณะ $a$, $p$, $h_A$ และ $h_B$ นั้นยากในเชิงการคำนวณ เพราะเทียบเท่ากับการแก้ DLP บน $(\mathbb{Z}_p)^{\times}$

</details>
#### การแสดงโปรโตคอล Diffie-Hellman ด้วย Python {#illustration-of-the-diffie-hellman-protocol-in-python}

มาดูตัวอย่างง่ายๆ ของโปรโตคอล DH ด้วย Python โดยใช้จำนวนเฉพาะขนาดเล็ก:

<Admonition type="note">
ในส่วนนี้เราจะแสดงการคำนวณทางคณิตศาสตร์โดยละเอียดเป็นส่วนหนึ่งของโค้ด Python
</Admonition>

**ขั้นตอนที่ 1:** Alice และ Bob ตกลงกันเรื่อง prime $p$ และ primitive root $a$ เลือก $p=11, a=7$

```python
# Step 1: Choose a prime `p` and a primitive root `a`
p = 11
a = 7

print(f"prime: {p}")
print(f"primitive root: {a}")
```

**ขั้นตอนที่ 2, 3:** Alice เลือก secret exponent $k_A$ และคำนวณ $h_A = a^{k_A}~(mod~p)$ เช่นเดียวกัน Bob เลือก secret exponent $k_B$ และคำนวณ $h_B = a^{k_B}~(mod~p)$

```python
k_A = 4  # Alice's private key
h_A = (a ** (k_A)) % p  # Alice's public key

print(f"Alice's private key is {k_A} and public key is {h_A}")

k_B = 8  # Bob's private key
h_B = (a ** (k_B)) % p  # Bob's public key

print(f"Bob's private key is {k_B} and public key is {h_B}")
```

**ขั้นตอนที่ 4:** ทั้งสองฝ่ายประกาศคีย์สาธารณะ $h_A$ และ $h_B$

**ขั้นตอนที่ 5, 6:** แต่ละฝ่ายรวมคีย์ส่วนตัวของตนกับคีย์สาธารณะของอีกฝ่ายเพื่อสร้าง shared secret key

```python
secret_key_alice = h_B**k_A % p
secret_key_bob = h_A**k_B % p
assert secret_key_alice == secret_key_bob
print(f"The shared secret key is: {secret_key_bob}")
```

ตอนนี้ Alice และ Bob สามารถใช้ shared secret key สำหรับการเข้ารหัสแบบสมมาตรได้
#### ความปลอดภัยของการแลกเปลี่ยนคีย์ Diffie-Hellman {#security-of-diffie-hellman-key-exchange}

ตามที่กล่าวไว้ข้างต้น ความปลอดภัยของ DH ขึ้นอยู่กับความยากในเชิงการคำนวณของการแก้ DLP ด้วย prime ขนาดใหญ่ $p$ ในการใช้งานทั่วไป NIST แนะนำ prime integer ขนาด 2048- หรือ 3072-บิตสำหรับการแลกเปลี่ยนคีย์ DH ซึ่งถือว่าปลอดภัยเพียงพอต่อความพยายามแก้ DLP โดยใช้คอมพิวเตอร์คลาสสิก

**การโจมตีแบบ Man-in-the-middle (MITM)**: ความจริงที่ว่า DH เป็น scheme แบบ interactive โดย shared secret ขึ้นอยู่กับการรวมคีย์ส่วนตัวของฝ่ายหนึ่งกับคีย์สาธารณะของอีกฝ่าย ทำให้มันเสี่ยงต่อการโจมตีที่เรียกว่า *Man-in-the-middle (MITM)*

<details>

<summary>
รายละเอียดทางคณิตศาสตร์ของความปลอดภัย DH และการโจมตี MITM
</summary>

ในสถานการณ์นี้ บุคคลที่สาม เช่น Eve ดักจับคีย์สาธารณะ $h_A, h_B$ ระหว่างการส่ง และแทนที่ด้วยคีย์สาธารณะของตน $h_E$ สำหรับแต่ละ $h_A$ และ $h_B$ ก่อนที่จะส่งต่อไปยัง Bob และ Alice ตามลำดับ

จากนั้น แทนที่จะใช้ $h_B$ สร้าง shared secret ของตน Alice จะใช้ $h_E$ แม้ว่าเธอคิดว่าเธอกำลังใช้คีย์สาธารณะของ Bob เช่นเดียวกัน แทนที่จะใช้ $h_A$ สร้าง shared secret ของตน Bob จะใช้ $h_E$ แม้ว่าเขาคิดว่าเขากำลังใช้คีย์สาธารณะของ Alice

เนื่องจาก $h_E$ ถูกใช้สร้าง shared secret ของ Alice (Bob) plain text ที่เข้ารหัสโดย Alice (Bob) จึงสามารถถอดรหัสได้โดย Eve

ดังนั้น การแลกเปลี่ยนคีย์ DH มักถูกใช้ร่วมกับอัลกอริทึมลายเซ็นดิจิทัลเพื่อให้แน่ใจว่าแต่ละฝ่ายใช้คีย์สาธารณะที่ได้รับการรับรองตัวตนในการสร้าง shared secret

</details>
### Digital Signature Algorithm (DSA) {#the-digital-signature-algorithm-dsa}

แม้ว่า cryptosystems ทั่วไปเช่น [RSA](https://en.wikipedia.org/w/index.php?title=RSA_(cryptosystem)&oldid=1163962255) ให้ฟังก์ชันลายเซ็นดิจิทัล ในปี 1994 NIST ได้นำ scheme ลายเซ็นเฉพาะทางที่อิงจาก modular exponentiation และ discrete logarithm problem มาใช้เป็นมาตรฐานรัฐบาลกลางสำหรับลายเซ็นดิจิทัล scheme นี้ที่รู้จักกันง่ายๆ ว่า [*Digital Signature Algorithm*](https://en.wikipedia.org/w/index.php?title=Digital_Signature_Algorithm&oldid=1148950522) (DSA) ประกอบด้วยสี่ขั้นตอนที่แตกต่างกัน:
1. **การสร้างคีย์**:

   คีย์ DSA สร้างจาก:
     - จำนวนเฉพาะ 2 ตัวที่ตอบสนองกฎบางอย่าง
       -   $p$ - มักจะ 256 บิต (เราจะเรียกความยาวนี้ว่า $N$)
       -   $q$ - มักจะ 3072 บิต (เราจะเรียกความยาวนี้ว่า $L$)
     - ฟังก์ชัน hash เชิงคริปโตกราฟีที่แปลงจาก strings ความยาว $L$ ไปยัง $N$
     - พารามิเตอร์เพิ่มเติม $g$ (ดูรายละเอียดด้านล่าง)

   จากสิ่งนี้เราเลือกตัวเลขสุ่ม $x$ เป็นคีย์ส่วนตัว และสามารถคำนวณคีย์สาธารณะ $y$ ได้

   <details>
   <summary>
   รายละเอียดทางคณิตศาสตร์ของการสร้างคีย์
   </summary>

   - **การสร้างพารามิเตอร์:** ในเชิงคณิตศาสตร์ DSA เกี่ยวข้องกับ cyclic subgroup ของ $(\mathbb{Z}_p)^{\times}$ ที่สร้างโดย element $g$ ที่มี prime order q < p ขั้นตอนแรกใน DSA จึงคือการเลือกสอง prime p, q เพื่อตั้งค่าโครงสร้างทางคณิตศาสตร์ที่จำเป็น

     - เลือก prime $q$ ขนาด $N$-บิต
     - เลือก prime $p$ ขนาด $L$-บิต โดยที่ $p-1$ เป็นผลคูณของ $q$ การเลือก $p$ ระบุ group $(\mathbb{Z}_p)^{\times}$
     - เลือกจำนวนเต็ม $1 < h < p-1$ แบบสุ่มและคำนวณ $g = h^{(p-1)/q}~mod~p$ ถ้า $g=1$ ซึ่งเกิดขึ้นน้อยมาก ให้ลอง h อื่น
     - ยืนยันว่า $g^q~mod~p = 1$ โดย g เป็น generator ของ cyclic subgroup $\langle g \rangle$ ของ $(\mathbb{Z}_p)^{\times}$ ที่มี prime order q

     พารามิเตอร์ $(q, p, g)$ ระบุ instance ของอัลกอริทึมและเป็นข้อมูลสาธารณะ โดยทั่วไป $ N \sim 256$ และ $L \sim 3072$ ในการใช้งานจริง

     โปรโตคอลยังต้องการฟังก์ชัน hash เชิงคริปโตกราฟี $H : \{0,1\}^L \rightarrow \{0, 1\}^N$ ที่แปลง binary $L$-bit strings ไปยัง $N$-bit strings เช่น SHA-256

   - **การสร้าง key-pair:** ผู้ลงนามต้องสร้าง key pair ในฝั่งของตน

     - เลือก random secret integer $ x \in \{1,2...q-1\}$ โดย $x$ คือคีย์ส่วนตัว
     - สร้าง $ y = g^{x}~mod~p$ โดย $y$ คือคีย์สาธารณะของผู้ลงนาม

   </details>
2. **การกระจายคีย์:**

   ข้อมูลต่อไปนี้จะถูกแชร์กับทุกคนที่ต้องการตรวจสอบลายเซ็น
   - พารามิเตอร์ที่ใช้ $(p,q,g)$ ซึ่งอธิบายอัลกอริทึม
   - ฟังก์ชัน hash $H$
   - คีย์สาธารณะ $y$
1. **การลงนาม:**

   - ผู้ลงนามสามารถลงนามข้อความ $m$ ได้ ลายเซ็นที่ได้คือ $(r,s)$
   - ข้อความและลายเซ็นทั้งสองตอนนี้ถูกส่งไปยังผู้รับ

   <details>
   <summary>
   รายละเอียดทางคณิตศาสตร์ของการลงนามข้อความ
   </summary>

   ผู้ลงนามลงนามข้อความ $m$ ดังนี้:

   - เลือก secret integer k แบบสุ่มจาก $\{1,2...q-1\}$
   - คำนวณ $r = (g^k~mod~p)~mod~q$ ในกรณีที่หายากเมื่อ $r=0$ ให้ลอง random k อื่น
   - คำนวณ $s = (k^{-1} (H(m) + xr))~mod~q$ ในกรณีที่หายากถ้า $s=0$ ให้ลอง random k อื่น
   - ลายเซ็นคือ tuple $(r, s)$
   - ผู้ลงนามส่งข้อความ $m$ รวมถึงลายเซ็น $(r,s)$ ไปยังผู้ตรวจสอบ

   เนื่องจากทั้ง $r, s$ เป็นจำนวนเต็ม modulo $q$ ขนาดลายเซ็นจึงอยู่ในระดับ $N$-บิต ไม่ใช่ $L$-บิตที่ยาวกว่า

   </details>
2. **การตรวจสอบ:**

   ผู้รับต้องการยืนยันความถูกต้องของผู้ส่ง
   พวกเขามีสิทธิ์เข้าถึงข้อมูลสาธารณะ $(q, p, g, H, y, (r,s), m)$
   พวกเขาสามารถดำเนินการคำนวณที่พิสูจน์ว่าผู้ส่งมีสิทธิ์เข้าถึงคีย์ส่วนตัว $x$

   และพยายามยืนยันว่าผู้ลงนามคือบุคคลที่มีสิทธิ์เข้าถึงคีย์ส่วนตัว $x$

   <details>
   <summary>
   รายละเอียดทางคณิตศาสตร์ของ DSA verification scheme
   </summary>

   - ยืนยันว่า $0 \lt r \lt q$ และ $0 \lt s \lt q$ กล่าวคือ $r, s$ เป็น valid modulo~q integers
   - คำนวณ $w = s^{-1}~mod~q$
   - คำนวณ $u_1 = H(m)w~mod~q$
   - คำนวณ $u_2 = rw~mod~q$
   - คำนวณ $v = (g^{u_1}y^{u_2}~mod~p)~mod~q$
   - ลายเซ็นได้รับการยืนยันถ้า $v = r$

   สำหรับลายเซ็นที่ถูกต้อง ความถูกต้องของอัลกอริทึม DSA สามารถแสดงได้ง่ายดังนี้:

   - ในฝั่งผู้ลงนาม: $s = (k^{-1} (H(m) + xr))~mod~q \implies k = ((H(m) + xr)s^{-1})~mod~q $
   - พิจารณาด้านขวาของความเท่ากันหลัง ผู้ตรวจสอบสามารถคำนวณ $s^{-1}, H(m)$ เพราะ $s, q, m, H$ เป็นข้อมูลสาธารณะ
   - ดังนั้น ผู้ตรวจสอบคำนวณ $w = s^{-1}~mod~q$ และ $u_1 = H(m)w~mod~q = H(m)s^{-1}~mod~q$
   - ผู้ตรวจสอบยังคำนวณ $u_2 = rw~mod~q = rs^{-1}~mod~q$ เนื่องจาก $r$ ก็เป็นข้อมูลสาธารณะเช่นกัน
   - โปรดทราบว่า $k = (u_1 + xu_2)~mod~q$
   - อย่างไรก็ตาม ผู้ตรวจสอบไม่มีสิทธิ์เข้าถึง $x$ เพราะมันคือคีย์ส่วนตัวของผู้ลงนาม ดังนั้น $k$ เองจึงไม่สามารถคำนวณโดยตรงได้  - scheme แทนที่จะอาศัยความจริงที่ว่าแม้ผู้ตรวจสอบจะไม่สามารถคำนวณ $k$ ได้ ด้วยผู้ลงนามที่ถูกต้อง ผู้ตรวจสอบควรสามารถคำนวณ $(g^k~mod~p)~mod~q = r$ ขึ้นมาใหม่ได้ด้วยความช่วยเหลือของคีย์สาธารณะ $y = g^{x}~mod~p$ ของผู้ลงนาม
   - ดังนั้น ผู้ตรวจสอบคำนวณ $v = (g^{u_1}y^{u_2}~mod~p)~mod~q = (g^{u_1}g^{xu_2}~mod~p)mod~q = (g^{u_1+xu_2}~mod~p)mod~q = (g^k~mod~p)mod~q$
   - ความเท่ากันสุดท้ายเป็นไปได้เพราะ $g$ มี order $q$ และพิสูจน์ $v = r$ สำหรับลายเซ็นที่ถูกต้อง

   </details>
### การแสดง DSA ด้วย Python {#illustration-of-the-dsa-in-python}
ในตัวอย่างนี้ เราจะใช้จำนวนเฉพาะขนาดเล็กเพื่อแสดงกระบวนการ DSA ในสถานการณ์ที่ Alice ส่งข้อความที่ลงนามแล้วให้ Bob เราจะใช้จำนวนเต็มขนาดเล็กในตัวอย่างนี้ สถานการณ์ในโลกจริงจะใช้จำนวนเฉพาะขนาดใหญ่กว่ามากในระดับ 2048-3072 บิต

<Admonition type="note">
คุณสามารถลองรันตัวอย่างนี้ซ้ำด้วยค่าต่างๆ เพื่อดูว่าอัลกอริทึมทำงานอย่างไร แต่ให้แน่ใจว่าเริ่มต้นรันจากเซลล์แรกในส่วนนี้
</Admonition>

เราเริ่มต้นด้วยการนำเข้าไลบรารีที่จำเป็นและเลือกพารามิเตอร์ของเรา
พารามิเตอร์จำนวนเต็ม $p$, $q$, $g$ เป็นข้อมูลสาธารณะ และต้องตอบสนองกฎต่อไปนี้:

- $p$, $q$ ทั้งคู่เป็นจำนวนเฉพาะ
- $(p-1) \mod\ q = 0$
- $g \ge 2$
- $g \le (p-2)$
- $g^{(p-1)/q} \mod\ p \neq 1$

```python
from random import randint

# parameter generation: select the primes q, p and generator g:
# EXPERIMENT with the values, they must meet certain rules
# this example code does not verify p,q are prime

q = 11
p = 23
g = 4

assert (p - 1) % q == 0
assert g >= 2
assert g <= (p - 2)
assert (pow(g, (p - 1) / q) % p) != 1

print(f"Public information is good: q={p}, p={q}, g={g}")
```

ต่อมาผู้ลงนาม Alice สร้างคีย์ส่วนตัวของตน

คีย์ส่วนตัว k (`alice-private-key` ในโค้ด python) ต้องตอบสนอง:
 - $k \ge 2$
 - $k \le (q-1)$

```python
# Alice chooses an integer randomly from {2..q-1}
# EXPERIMENT with the values

alice_private_key = randint(2, q - 1)
# alice_private_key =

assert alice_private_key >= 2
assert alice_private_key <= (q - 1)

print(f"Alice's private key is {alice_private_key}")
```

Alice เก็บคีย์ส่วนตัวของเธอเป็นความลับ

ต่อมา Alice คำนวณคีย์สาธารณะโดยใช้ modular exponentiation การกลับขั้นตอนนี้เพื่อกู้คืนคีย์ส่วนตัวเป็น instance ของ DLP จึงยากที่จะคำนวณบนคอมพิวเตอร์คลาสสิกเมื่อใช้ prime ขนาดใหญ่

จากคำอธิบายทางคณิตศาสตร์ข้างต้น เราทราบว่าสามารถคำนวณได้โดยใช้สูตร:

$y = g^{x} \mod\ p$

```python
alice_public_key = pow(g, alice_private_key, p)
# Alternatively can use (g ** alice_private_key) % p

print(f"Alice's public key is {alice_public_key}")
```

เหมือนเดิม Alice ทำให้คีย์สาธารณะของเธอสามารถเข้าถึงได้ผ่านช่องทางที่ไม่จำเป็นต้องลับ

ตอนนี้ Alice สามารถลงนามข้อความได้

สำหรับ digital signature scheme เราต้องสร้าง hash $H(m)$ ของข้อความ $m$ ที่ต้องการลงนาม

สมมติว่า Alice และ Bob ตกลงใช้ hashing algorithm เฉพาะที่มีความยาว hash $N$ เท่ากับจำนวนบิตใน $q$ ในตัวอย่างง่ายๆ นี้ เราจะจำกัด outputs ของ mock hash function ของเราด้วย $q$ ฟังก์ชัน hash ที่นี่เป็นแบบ trivial เพียงสร้างจำนวนเต็มสุ่ม

```python
hash_dict = {}

def mock_hash_func(input_message):
    print(input_message)
    if input_message not in hash_dict:
        hash_dict[input_message] = randint(1, q)
    return hash_dict[input_message]

alice_message = "Inspection tomorrow!"
alice_hash = mock_hash_func(alice_message)  # In reality, you'd use a hash function
print(f"Alice's message hash is: {alice_hash}")
```

ต่อมา Alice ต้องสร้าง random per-message secret number $k$ รวมถึง multiplicative inverse ของมัน modulo $q$ เพื่อคำนวณลายเซ็น

สามารถใช้อัลกอริทึม brute-force ง่ายๆ เพื่อคำนวณ modular inverse อย่างไรก็ตาม ดีกว่าที่จะใช้ฟังก์ชัน Python ในตัว `pow` ตามที่แสดงด้านล่าง

```python
# brute-force implementation to find modular inverse
def modular_inverse(k, q):
    for i in range(0, q):
        if (k * i) % q == 1:
            return i
    print(f"error! no inverse found! for {k},{q}")
    return 0

# Let's compare this algorithm with the standard python 'pow' function

n1 = modular_inverse(3, 7)
n2 = modular_inverse(4, 11)
n3 = modular_inverse(7, 5)
m1 = pow(3, -1, 7)
m2 = pow(4, -1, 11)
m3 = pow(7, -1, 5)

assert n1 == m1
assert n2 == m2
# assert(n3==m3)

print(f"modular_inverse(3,7) = {m1}")
print(f"modular_inverse(4,11) = {m2}")
print(f"modular_inverse(7,5) = {m3}")

# Some numbers don't have modular inverses - our function throws an error
n4 = modular_inverse(2, 6)

# The python library will throw an exception, which must be caught
if math.gcd(2, 6) == 1:
    m4 = pow(2, -1, 6)
else:
    print("Exception from pow() - no modular inverse found!")
```

ตอนนี้ Alice สามารถคำนวณลายเซ็นได้

- $r = (g^{k} \mod p) \mod\ q$
- $s = (k^{-1} (H(m) + xr)) \mod q$

โปรดทราบว่ามีเงื่อนไขบางอย่างที่จะต้องให้เราเลือก $k$ ที่แตกต่างในกรณีที่ $r$ หรือ $s$ คำนวณออกมาเป็น $0$ ในกรณีนี้เราจะสร้างค่าใหม่และทำซ้ำ

```python
# Start an infinite loop; we will 'break' out of it once a valid signature is found.
while True:
    k = randint(1, q - 1)  # Should be different for every message
    print("Using random k =", k)

    r = pow(g, k, p) % q
    # If r is 0, the value is invalid. Try again with a new k.
    if r == 0:
        print(f"{k} is not a good random value to use to calculate r. Trying another k")
        continue

    s = (pow(k, -1, q) * (alice_hash + alice_private_key * r)) % q
    # If s is 0, the value is also invalid. Try again with a new k.
    if s == 0:
        print(f"{k} is not a good random value to use to calculate s. Trying another k")
        continue

    # If we've reached this point, both r and s are valid. Break the loop.
    signature = (r, s)
    print(f"Alice's signature is : {(r,s)}")
    break

# After the loop, the valid r and s values can be used here.
# print(f"Generated Signature -> r: {r}, s: {s}")
```

Alice ประกาศข้อความและลายเซ็นของเธอไปยังผู้รับหรือผู้ตรวจสอบซึ่งก็คือ Bob

Bob ได้รับคีย์สาธารณะของ Alice ก่อนหน้านี้และตอนนี้สามารถตรวจสอบลายเซ็นเพื่อรับรองตัวตนข้อความของเธอได้

เพื่อทำสิ่งนี้ เขาทำการตรวจสอบบางอย่าง จากนั้นสร้าง hash ของข้อความที่ Alice ประกาศขึ้นใหม่และคำนวณปริมาณเสริม $w, u_1, u_2$ และ $v$
- $0 < r < q$
- $0 < s < q$
- $w = s^{-1} \mod\ q$
- $u_{1} = H(m) . w \mod\ q$
- $u_{2} = r . w \mod\ q$
- $v = (g^{u1}y^{u2} \mod\ p) \mod\ q$

สุดท้าย Bob ตรวจสอบว่า $v$ เท่ากับ $r$ หรือไม่ ถ้าเท่ากัน ลายเซ็นจะได้รับการยืนยัน

```python
# Bob re-generates message hash using Alice's broadcast message
bob_hash = mock_hash_func(alice_message)

# Bob computes auxiliary quantities w (using modular inverse), u1, u2 and v
w = (pow(s, -1, q)) % q
u1 = (bob_hash * w) % q
u2 = (r * w) % q
v = ((g**u1 * alice_public_key**u2) % p) % q

if v == r:
    print("Signature is valid!")
else:
    print("Signature is invalid!")
```

### ความปลอดภัยของ DSA {#security-of-the-dsa}

ด้วยการคำนวณแบบคลาสสิก ความปลอดภัยของ DSA สามารถได้รับผลกระทบจากหลายปัจจัย:

1. **ขนาดคีย์**: เหมือนเสมอ ขนาดคีย์ให้การป้องกันพื้นฐานต่อการโจมตีแบบ brute force การใช้งานสมัยใหม่ที่ใช้ DSA ใช้ขนาดคีย์อย่างน้อย 2048 บิต

2. **คุณภาพของ $k$**: ใน DSA แต่ละลายเซ็นต้องการค่า $k$ ที่ไม่ซ้ำกัน สุ่ม และเป็นความลับ (ดูข้างต้น) ความเป็นเอกลักษณ์ entropy และความลับของ $k$ มีความสำคัญมาก และจุดอ่อนในด้านใดด้านหนึ่งอาจทำให้คีย์ส่วนตัว $x$ ถูกเปิดเผย ตัวสร้างเลขสุ่มที่ใช้สร้าง $k$ ต้องปลอดภัยด้วยตัวเอง

3. **ความแข็งแกร่งของฟังก์ชัน hash**: เนื่องจาก DSA ใช้ฟังก์ชัน hash เป็นส่วนหนึ่งของกระบวนการสร้างและตรวจสอบลายเซ็น ฟังก์ชัน hash ที่อ่อนแอจึงอาจส่งผลกระทบต่อความปลอดภัยของลายเซ็นดิจิทัล ตัวอย่างเช่น การใช้ SHA-1 กับ DSA ถูกเลิกใช้แล้ว และแนะนำให้ใช้ SHA-2 หรือสูงกว่า

นอกจากนี้ การ implement DSA ที่แข็งแกร่งควรป้องกันการโจมตีประเภทอื่นที่มุ่งเป้าไปที่การเข้ารหัสแบบอสมมาตรตามที่อธิบายไว้ก่อนหน้า
### การแลกเปลี่ยนคีย์ที่รับรองตัวตนด้วย Diffie-Hellman และ DSA {#authenticated-key-exchange-with-diffie-hellman-and-dsa}

โปรโตคอลความปลอดภัยเครือข่ายสมัยใหม่ เช่น [Transport Layer Security (TLS)](https://en.wikipedia.org/w/index.php?title=Transport_Layer_Security&oldid=1155601531) และอื่นๆ อีกมาก เกี่ยวข้องกับการรวมฟังก์ชันการแลกเปลี่ยนคีย์และการรับรองตัวตน ในบริบทของ Diffie-Hellman การรับรองตัวตนจำเป็นเพื่อป้องกันการโจมตีแบบ MITM ในเซลล์โค้ดต่อไปนี้ เราแสดงตัวอย่างที่ Alice และ Bob ทำการแลกเปลี่ยนคีย์ที่รับรองตัวตนโดยการรวมโปรโตคอล Diffie-Hellman และ DSA สำหรับสิ่งนี้ เราจะใช้ไลบรารี Python `cryptography`

![รูปที่ 2. การแลกเปลี่ยนคีย์ที่รับรองตัวตนด้วย Diffie-Hellman และ DSA](/learning/images/courses/quantum-safe-cryptography/asymmetric-key-cryptography/auth_key_exchange.avif)

*รูปที่ 2. การแลกเปลี่ยนคีย์ที่รับรองตัวตนด้วย Diffie-Hellman และ DSA*
ขั้นแรก Alice และ Bob ตกลงกันเรื่องพารามิเตอร์ DH และสร้างชุด DH public-private key pairs

```python
# import necessary library modules
from cryptography.hazmat.primitives import hashes
from cryptography.hazmat.primitives.asymmetric import dh, dsa

# Generate DH parameters
parameters = dh.generate_parameters(generator=2, key_size=2048)

# Generate Alice's DH private key and public key
alice_dh_private_key = parameters.generate_private_key()
alice_dh_public_key = alice_dh_private_key.public_key()

# Generate Bob's DH private key and public key
bob_dh_private_key = parameters.generate_private_key()
bob_dh_public_key = bob_dh_private_key.public_key()

print("Public and private keys generated for Bob and Alice")
```

คีย์สาธารณะ DH จะถูกประกาศ ต่อมา ทั้ง Alice และ Bob ต่างสร้าง key pair แยกกันสำหรับใช้กับ DSA คีย์เหล่านี้จะถูกใช้เพื่อลงนามคีย์สาธารณะ DH ที่จะแลกเปลี่ยน

```python
# Generate DSA keys for Alice and Bob
alice_dsa_private_key = dsa.generate_private_key(key_size=2048)
alice_dsa_public_key = alice_dsa_private_key.public_key()
bob_dsa_private_key = dsa.generate_private_key(key_size=2048)
bob_dsa_public_key = bob_dsa_private_key.public_key()

print("Additional key pair generated for signing")
```

Alice ลงนามคีย์สาธารณะ DH ของเธอโดยใช้คีย์ส่วนตัว DSA ของเธอ

```python
# Alice signs
alice_public_bytes = alice_dh_public_key.public_bytes(
    encoding=serialization.Encoding.PEM,
    format=serialization.PublicFormat.SubjectPublicKeyInfo,
)
alice_signature = alice_dsa_private_key.sign(alice_public_bytes, hashes.SHA256())

print("Alice signed public key")
```

เช่นเดียวกัน Bob ลงนามคีย์สาธารณะ DH ของเขาโดยใช้คีย์ส่วนตัว DSA ของเขา

```python
bob_public_bytes = bob_dh_public_key.public_bytes(
    encoding=serialization.Encoding.PEM,
    format=serialization.PublicFormat.SubjectPublicKeyInfo,
)
bob_signature = bob_dsa_private_key.sign(bob_public_bytes, hashes.SHA256())

print("Bob signed public key")
```

คีย์สาธารณะ DH และลายเซ็นที่สอดคล้องกันตอนนี้ถูกประกาศโดยทั้ง Alice และ Bob เมื่อได้รับคีย์สาธารณะและลายเซ็นของคู่สนทนาแล้ว แต่ละฝ่ายจะตรวจสอบว่าลายเซ็นถูกต้อง วิธีนี้สามารถป้องกันการโจมตีแบบ MITM ได้ เพราะ Alice ยืนยันได้ว่าคีย์สาธารณะ DH ของ Bob ถูกลงนามโดย Bob จริงๆ และในทางกลับกัน

```python
# Alice and Bob verify each other's DH public keys using DSA public keys
# An InvalidSignature exception will occur if they are not valid
alice_dsa_public_key.verify(alice_signature, alice_public_bytes, hashes.SHA256())
bob_dsa_public_key.verify(bob_signature, bob_public_bytes, hashes.SHA256())

print("Signatures are valid")
```

หลังจากตรวจสอบลายเซ็นแล้ว Alice และ Bob สร้าง shared secret ซึ่งทำให้การแลกเปลี่ยนคีย์เสร็จสมบูรณ์

```python
# Perform key exchange
alice_shared_key = alice_dh_private_key.exchange(bob_dh_public_key)
bob_shared_key = bob_dh_private_key.exchange(alice_dh_public_key)

print("Shared secrets generated")
```

ตัวเลือก เพื่อความปลอดภัยเพิ่มเติม Alice และ Bob สามารถใช้ key derivation function เฉพาะทางเช่น *HKDF* เพื่อสร้าง symmetric key ที่ปลอดภัยยิ่งขึ้นจาก shared secret โดยใช้เทคนิค *key stretching*

```python
# Derive a shared symmetric key using key-stretching
def derive_key(shared_key):
    return HKDF(
        algorithm=hashes.SHA256(),
        length=32,
        salt=None,
        info=None,
    ).derive(shared_key)

alice_symmetric_key = derive_key(alice_shared_key)
bob_symmetric_key = derive_key(bob_shared_key)

assert alice_symmetric_key == bob_symmetric_key
print("Keys checked to be the same")
```
