# CSS Codes

## Classical linear codes {#classical-linear-codes}

โค้ดแก้ไขข้อผิดพลาดแบบคลาสสิกได้รับการศึกษาครั้งแรกในทศวรรษ 1940 และปัจจุบันรู้จักโค้ดจำนวนมาก โดยโค้ดที่ศึกษาและใช้กันมากที่สุดอยู่ในหมวดหมู่ที่เรียกว่า *linear codes*
เราจะเห็นในอีกสักครู่ว่าคำว่า "linear" หมายถึงอะไรในบริบทนี้ แต่วิธีง่าย ๆ ในการอธิบาย linear codes ณ จุดนี้คือ พวกมันคือ stabilizer codes ที่บังเอิญเป็นแบบคลาสสิก
CSS codes โดยพื้นฐานแล้วคือ *คู่* ของ classical linear codes ที่รวมเข้าด้วยกันเพื่อสร้างโค้ดแก้ไขข้อผิดพลาดเชิงควอนตัม
ดังนั้นเพื่อการอภิปรายต่อไปนี้ เราต้องเข้าใจพื้นฐานบางอย่างเกี่ยวกับ classical linear codes

ให้ $\Sigma$ เป็น binary alphabet สำหรับการอภิปรายทั้งหมดนี้
เมื่อเราอ้างถึง *classical linear code* เราหมายถึงชุดที่ไม่ว่าง $\mathcal{C}\subseteq\Sigma^n$ ของ binary strings ที่มีความยาว $n$ สำหรับจำนวนเต็มบวกบางตัว $n$ ซึ่งต้องตอบสนองคุณสมบัติพื้นฐานเพียงหนึ่งอย่าง:
ถ้า $u$ และ $v$ เป็น binary strings ใน $\mathcal{C}$ แล้ว string $u\oplus v$ ก็อยู่ใน $\mathcal{C}$ ด้วย
ที่นี่ $u\oplus v$ หมายถึง bitwise exclusive-OR ของ $u$ และ $v$ ดังที่เราพบหลายครั้งใน
คอร์ส "Fundamentals of quantum algorithms"

โดยพื้นฐาน เมื่อเราอ้างถึง classical error correcting code ว่าเป็น *linear* เราคิดถึง binary strings ที่มีความยาว $n$ ในฐานะ vectors $n$ มิติ โดยที่ entries เป็น $0$ หรือ $1$ และต้องการให้โค้ดเองเป็น linear subspace
แทนที่จะเป็น vector addition ธรรมดาบนจำนวนจริงหรือจำนวนเชิงซ้อน เราใช้การบวก modulo $2$ ซึ่งก็คือ exclusive-OR
กล่าวคือ ถ้ามี *codewords* สองตัว $u$ และ $v$ หมายความว่า $u$ และ $v$ เป็น binary strings ใน $\mathcal{C}$ แล้ว $u + v$ modulo 2 ซึ่งก็คือ $u\oplus v$ ต้องเป็น codeword ใน $\mathcal{C}$ ด้วย
สังเกตโดยเฉพาะว่า implication นี้ต้องเป็นจริงแม้ $u = v$
สิ่งนี้หมายความว่า $\mathcal{C}$ ต้องรวม all-zero string $0^n$ เพราะ bitwise exclusive-OR ของ string ใด ๆ กับตัวมันเองคือ all-zero string

### ตัวอย่าง: 3-bit repetition code {#example-the-3-bit-repetition-code}

3-bit repetition code เป็นตัวอย่างของ classical linear code
โดยเฉพาะ เรามี $\mathcal{C} = \{000,111\}$ ดังนั้นเทียบกับเงื่อนไข linearity มีเพียงสองทางเลือกสำหรับ $u$ และสองทางเลือกสำหรับ $v$
เป็นเรื่องง่ายที่จะตรวจสอบสี่คู่ที่เป็นไปได้เพื่อดูว่าเราได้ codeword เสมอเมื่อนำ bitwise exclusive-OR:

$$
000 \oplus 000 = 000, \quad
000 \oplus 111 = 111, \quad
111 \oplus 000 = 111, \quad
111 \oplus 111 = 000.
$$

### ตัวอย่าง: $[7,4,3]$-Hamming code {#example-the-743-hamming-code}

นี่คืออีกตัวอย่างของ classical linear code ที่เรียกว่า $[7,4,3]$-Hamming code
มันเป็นหนึ่งใน classical error correcting codes แรก ๆ ที่เคยค้นพบ และประกอบด้วย binary strings ความยาว 7 จำนวน 16 strings เหล่านี้
(บางครั้ง $[7,4,3]$-Hamming code เข้าใจว่าหมายถึงโค้ดที่มี strings เหล่านี้กลับทิศ แต่เราจะถือว่ามันเป็นโค้ดที่มี strings ที่แสดงที่นี่)

$$
\begin{array}{cccc}
0000000 & 1100001 & 1010010 & 0110011\\[1mm]
0110100 & 1010101 & 1100110 & 0000111\\[1mm]
1111000 & 0011001 & 0101010 & 1001011\\[1mm]
1001100 & 0101101 & 0011110 & 1111111
\end{array}
$$

มีตรรกะง่าย ๆ เบื้องหลังการเลือก strings เหล่านี้ แต่เป็นเรื่องรองจากบท และจะไม่อธิบายที่นี่
สำหรับตอนนี้ เพียงแค่สังเกตว่านี่คือ classical linear code: การ XOR strings สองตัวใด ๆ เข้าด้วยกันจะให้ string อื่นในโค้ดเสมอ

สัญกรณ์ $[7,4,3]$ (ในวงเล็บเหลี่ยมเดี่ยว) หมายความคล้ายกับสัญกรณ์วงเล็บเหลี่ยมคู่สำหรับ stabilizer codes ที่กล่าวถึงในบทก่อนหน้า แต่ที่นี่สำหรับ classical linear codes
โดยเฉพาะ codewords มี $7$ bits เราสามารถ encode $4$ bits โดยใช้โค้ด (เพราะมี $16 = 2^4$ codewords) และบังเอิญเป็น distance $3$ code หมายความว่า codewords ที่ต่างกันสองอันต้องต่างกันในอย่างน้อย $3$ ตำแหน่ง ดังนั้นต้อง flip อย่างน้อย $3$ bits เพื่อเปลี่ยน codeword หนึ่งเป็นอีกอัน
ข้อเท็จจริงที่ว่านี่คือ distance $3$ code หมายความว่าสามารถแก้ไข bit-flip error ได้สูงสุดหนึ่งอย่าง

### การอธิบาย classical linear codes {#describing-classical-linear-codes}

ตัวอย่างที่กล่าวถึงเพิ่งเป็นตัวอย่างง่าย ๆ ของ classical linear codes แต่แม้แต่ $[7,4,3]$-Hamming code ก็ดูลึกลับบ้างเมื่อ codewords ถูกระบุไว้ง่าย ๆ
มีวิธีที่ดีกว่าและมีประสิทธิภาพมากกว่าในการอธิบาย classical linear codes รวมถึงสองวิธีต่อไปนี้

1. *Generators* วิธีหนึ่งในการอธิบาย classical linear code คือด้วยรายการ codewords ขั้นต่ำที่ *generates* โค้ด หมายความว่าโดยการนำ subsets ทั้งหมดที่เป็นไปได้ของ codewords เหล่านี้และ XOR เข้าด้วยกัน เราได้โค้ดทั้งหมด

   โดยละเอียดยิ่งขึ้น strings $u_1,\ldots,u_m\in\Sigma^n$ generates classical linear code $\mathcal{C}$ ถ้า

   $$
   \mathcal{C} = \bigl\{\alpha_1 u_1 \oplus \cdots \oplus \alpha_m u_m\,:\,\alpha_1,\ldots,\alpha_m\in\{0,1\}\bigr\},
   $$

   โดยเข้าใจว่า $\alpha u = u$ เมื่อ $\alpha = 1$ และ $\alpha u = 0^n$ เมื่อ $\alpha = 0$
   และเราบอกว่ารายการนี้ *minimal* ถ้าการลบ string หนึ่งออกสร้างโค้ดที่เล็กลง
   วิธีธรรมชาติในการคิดถึงคำอธิบายดังกล่าวคือ ชุด $\{u_1,\ldots,u_m\}$ เป็น *basis* ของ $\mathcal{C}$ ในฐานะ subspace โดยที่เราคิดถึง strings ในฐานะ vectors ที่มี entries เป็น binary โดยจำไว้ว่าเราทำงานใน vector space ที่คณิตศาสตร์ทำ modulo $2$

2. *Parity checks*
   อีกวิธีธรรมชาติในการอธิบาย classical linear code คือด้วย *parity checks* — หมายความว่ารายการ binary strings ขั้นต่ำที่ strings ในโค้ดคือตัวที่ *binary dot product* กับทุก parity check string เป็นศูนย์ (คล้ายกับ bitwise exclusive-OR binary dot product ปรากฏหลายครั้งใน "Fundamentals of quantum algorithms")

   กล่าวคือ strings $v_1,\ldots,v_r\in\Sigma^n$ เป็น parity check strings สำหรับ classical linear code $\mathcal{C}$ ถ้า

   $$
   \mathcal{C} = \bigl\{ u\in \Sigma^n\,:\, u\cdot v_1 = \cdots = u \cdot v_r = 0 \bigr\},
   $$

   และชุดของ strings นี้ *minimal* ถ้าการลบหนึ่งออกให้โค้ดที่ใหญ่ขึ้น
   เรียกว่า parity check strings เพราะ $u$ มี binary dot product เท่ากับศูนย์กับ $v$ ถ้าและเฉพาะถ้า bits ของ $u$ ในตำแหน่งที่ $v$ มี 1 มี parity คู่
   ดังนั้น เพื่อดูว่า string $u$ อยู่ใน code $\mathcal{C}$ หรือไม่ ก็เพียงพอที่จะตรวจสอบ parity ของ subsets บางอย่างของ bits ของ $u$

   สิ่งสำคัญที่ต้องสังเกตที่นี่คือ binary dot product ไม่ใช่ inner product ในความหมายเป็นทางการ โดยเฉพาะ เมื่อ strings สองตัวมี binary dot product เท่ากับศูนย์ ไม่ได้หมายความว่าพวกมัน orthogonal ในทิศทางที่เรามักคิดถึง orthogonality ตัวอย่างเช่น binary dot product ของ string $11$ กับตัวมันเองเป็นศูนย์ ดังนั้นเป็นไปได้ที่ parity check string สำหรับ classical linear code อยู่ในโค้ดเอง

Classical linear codes บน binary alphabet มักรวม strings จำนวนหนึ่งที่เป็น power ของ $2$ — และสำหรับ classical linear code เดี่ยวที่อธิบายด้วยสองวิธีที่เพิ่งอธิบาย มักเป็นจริงว่า $n = m + r$
โดยเฉพาะ ถ้ามี generators ขั้นต่ำ $m$ ตัว โค้ดจะ encode $m$ bits และจะมี $2^m$ codewords
และถ้ามี parity check strings ขั้นต่ำ $r$ ตัว จะมี $2^{n-r}$ codewords
ดังนั้น generator แต่ละตัวทำให้ code space เป็นสองเท่า ขณะที่ parity check string แต่ละตัวทำให้ครึ่งหนึ่ง

ตัวอย่างเช่น 3-bit repetition code เป็น linear code ดังนั้นสามารถอธิบายได้ทั้งสองวิธีนี้
โดยเฉพาะ มีเพียงหนึ่งทางเลือกสำหรับ generator ที่ใช้ได้: $111$
เราสามารถอธิบายโค้ดด้วย parity check strings สองตัวแทน เช่น $110$ และ $011$ — ซึ่งน่าจะดูคุ้นเคยจากการอภิปรายก่อนหน้าของโค้ดนี้ — หรือเราอาจนำ parity check strings เป็น $110$ และ $101$ หรือ $101$ และ $011$ แทนก็ได้ (Generators และ parity check strings โดยทั่วไปไม่ unique สำหรับ classical linear code ที่กำหนด)

สำหรับตัวอย่างที่สอง พิจารณา $[7,4,3]$-Hamming code
นี่คือทางเลือกหนึ่งสำหรับรายการ generators ที่ใช้ได้

$$
\begin{array}{c}
1111000\\[1mm]
0110100\\[1mm]
1010010\\[1mm]
1100001
\end{array}
$$

และนี่คือทางเลือกสำหรับรายการ parity checks สำหรับโค้ดนี้

$$
\begin{array}{c}
1111000\\[1mm]
1100110\\[1mm]
1010101
\end{array}
$$

ที่นี่ นอกจากนี้ เราเห็นว่า *ทั้งหมด* ของ parity check strings ของเรานั้นอยู่ในโค้ดเอง

ข้อสังเกตสุดท้ายเกี่ยวกับ classical linear codes ซึ่งเชื่อมโยงพวกมันกับ stabilizer formalism คือ parity check strings เทียบเท่ากับ stabilizer generators ที่ประกอบด้วยเฉพาะ $Z$ และ identity Pauli matrices
ตัวอย่างเช่น parity check strings $110$ และ $011$ สำหรับ 3-bit repetition code สอดคล้องกับ stabilizer generators $Z\otimes Z\otimes \mathbb{I}$ และ $\mathbb{I}\otimes Z\otimes Z$ พอดี ซึ่งสอดคล้องกับการอภิปรายของ Pauli observables จากบทก่อนหน้า

## นิยามของ CSS codes {#definition-of-css-codes}

CSS codes คือ stabilizer codes ที่ได้จากการรวม *คู่* ของ classical linear codes บางอย่างเข้าด้วยกัน
สิ่งนี้ใช้ไม่ได้กับ classical linear codes สองตัวใด ๆ — โค้ดสองตัวต้องมีความสัมพันธ์บางอย่าง
อย่างไรก็ตาม construction นี้เปิดความเป็นไปได้มากมายสำหรับโค้ดแก้ไขข้อผิดพลาดเชิงควอนตัม อาศัยส่วนหนึ่งจาก classical coding theory กว่า 75 ปี

ใน stabilizer formalism stabilizer generators ที่มีเฉพาะ $Z$ และ identity Pauli matrices เทียบเท่ากับ parity checks ดังที่เราเพิ่งสังเกตสำหรับ 3-bit repetition code
สำหรับตัวอย่างอื่น พิจารณา parity check strings ต่อไปนี้สำหรับ $[7,4,3]$-Hamming code

$$
\begin{array}{c}
1111000\\[1mm]
1100110\\[1mm]
1010101
\end{array}
$$

parity check strings เหล่านี้สอดคล้องกับ stabilizer generators ต่อไปนี้ (เขียนโดยไม่มีสัญลักษณ์ tensor product) ซึ่งเราได้โดยแทน $1$ ด้วย $Z$ และ $0$ ด้วย $\mathbb{I}$
นี่คือสามในหก stabilizer generators สำหรับ 7-qubit Steane code

$$
\begin{array}{ccccccc}
Z & Z & Z & Z & \mathbb{I} & \mathbb{I} & \mathbb{I} \\[1mm]
Z & Z & \mathbb{I} & \mathbb{I} & Z & Z & \mathbb{I} \\[1mm]
Z & \mathbb{I} & Z & \mathbb{I} & Z & \mathbb{I} & Z
\end{array}
$$

เราให้ชื่อ *$Z$ stabilizer generators* กับ stabilizer generators แบบนี้ หมายความว่ามีเฉพาะ Pauli $Z$ และ identity tensor factors — ดังนั้น $X$ และ $Y$ Pauli matrices ไม่เกิดขึ้นใน $Z$ stabilizer generators

เราสามารถพิจารณา stabilizer generators ที่มีเฉพาะ $X$ และ identity Pauli matrices เป็น tensor factors ด้วย
Stabilizer generators แบบนี้มองได้ว่าคล้ายกับ $Z$-stabilizer generators ยกเว้นว่าอธิบาย parity checks ใน $\{\vert+\rangle,\vert-\rangle\}$ basis แทน standard basis
Stabilizer generators ของรูปแบบนี้เรียกว่า *$X$ stabilizer generators* — ดังนั้นไม่อนุญาตให้มี $Y$ หรือ $Z$ Pauli matrices

ตัวอย่างเช่น พิจารณา stabilizer generators สามตัวที่เหลือจาก 7-qubit Steane code

$$
\begin{array}{ccccccc}
X & X & X & X & \mathbb{I} & \mathbb{I} & \mathbb{I} \\[1mm]
X & X & \mathbb{I} & \mathbb{I} & X & X & \mathbb{I} \\[1mm]
X & \mathbb{I} & X & \mathbb{I} & X & \mathbb{I} & X
\end{array}
$$

พวกมันทำตามรูปแบบเดียวกันจาก $[7,4,3]$-Hamming code กับ $Z$-stabilizer generators ยกเว้นคราวนี้แทน $X$ สำหรับ $1$ แทนที่จะเป็น $Z$
สิ่งที่เราได้จาก stabilizer generators สามตัวนี้เท่านั้นคือโค้ดที่รวม 16 states ที่แสดงที่นี่ ซึ่งเราได้โดยนำ Hadamard operations ไปใช้กับ standard basis states ที่สอดคล้องกับ strings ใน $[7,4,3]$-Hamming code
(แน่นอน code space สำหรับโค้ดนี้ยังรวม linear combinations ของ states เหล่านี้)

$$
\begin{array}{cccc}
  \vert {+++++++} \rangle \quad & \vert {--++++-} \rangle \quad & \vert {-+-++-+} \rangle \quad & \vert {+--++--} \rangle \\
  \vert {+--+-++} \rangle \quad & \vert {-+-+-+-} \rangle \quad & \vert {--++--+} \rangle \quad & \vert {++++---} \rangle \\
  \vert {----+++} \rangle \quad & \vert {++--++-} \rangle \quad & \vert {+-+-+-+} \rangle \quad & \vert {-++-+--} \rangle \\
  \vert {-++--++} \rangle \quad & \vert {+-+--+-} \rangle \quad & \vert {++----+} \rangle \quad & \vert {-------} \rangle
\end{array}
$$

ตอนนี้เราสามารถนิยาม CSS codes ด้วยคำพูดง่าย ๆ

<Figure title="นิยาม">
CSS code คือ stabilizer code ที่สามารถแสดงได้โดยใช้เฉพาะ $X$ และ $Z$ stabilizer generators
</Figure>

กล่าวคือ CSS codes คือ stabilizer codes ที่เรามี stabilizer generators ที่ไม่มี Pauli $Y$ matrices และ $X$ และ $Z$ ไม่ปรากฏใน *stabilizer generator เดียวกัน*

เพื่อให้ชัดเจน ตามนิยามนี้ CSS code คือโค้ดที่ *เป็นไปได้* ที่จะเลือกแค่ $X$ และ $Z$ stabilizer generators — แต่เราต้องจำไว้ว่ามีอิสระในการเลือก stabilizer generators สำหรับ stabilizer codes
ดังนั้น โดยทั่วไปจะมีทางเลือกต่าง ๆ สำหรับ stabilizer generators ของ CSS code ที่บังเอิญไม่เป็น $X$ หรือ $Z$ stabilizer generators (นอกเหนือจากอย่างน้อยหนึ่งทางเลือกที่เป็น)

นี่คือตัวอย่างง่าย ๆ มากของ CSS code ที่รวมทั้ง $Z$ stabilizer generator และ $X$ stabilizer generator:

$$
\begin{array}{cc}
Z & Z \\[1mm]
X & X
\end{array}
$$

ชัดเจนว่านี่คือ CSS code เพราะ stabilizer generator แรกเป็น $Z$ stabilizer generator และตัวที่สองเป็น $X$ stabilizer generator
แน่นอน CSS code ต้องเป็น valid stabilizer code ด้วย — หมายความว่า stabilizer generators ต้อง commute กัน เป็น minimal generating set และ fix quantum state vector อย่างน้อยหนึ่งตัว
ข้อกำหนดเหล่านี้ง่ายต่อการสังเกตสำหรับโค้ดนี้
ดังที่เราสังเกตในบทก่อนหน้า code space สำหรับโค้ดนี้คือ one-dimensional space ที่ span ด้วย $\vert\phi^+\rangle$ Bell state
ข้อเท็จจริงที่ทั้งสอง stabilizer generators fix state นี้เห็นได้ชัดจากการพิจารณาสอง expressions ต่อไปนี้ของ e-bit พร้อมกับการตีความ stabilizer generators เหล่านี้เป็น parity checks ใน $\{\vert  0\rangle, \vert 1\rangle\}$ และ
$\{\vert  +\rangle, \vert -\rangle\}$ bases

$$
\vert\phi^+\rangle
= \frac{\vert 0\rangle\vert 0\rangle + \vert 1\rangle\vert 1\rangle}{\sqrt{2}}
= \frac{\vert +\rangle\vert +\rangle + \vert -\rangle\vert -\rangle}{\sqrt{2}}
$$

7-qubit Steane code เป็นอีกตัวอย่างของ CSS code

$$
\begin{array}{ccccccc}
Z & Z & Z & Z & \mathbb{I} & \mathbb{I} & \mathbb{I} \\[1mm]
Z & Z & \mathbb{I} & \mathbb{I} & Z & Z & \mathbb{I} \\[1mm]
Z & \mathbb{I} & Z & \mathbb{I} & Z & \mathbb{I} & Z \\[1mm]
X & X & X & X & \mathbb{I} & \mathbb{I} & \mathbb{I} \\[1mm]
X & X & \mathbb{I} & \mathbb{I} & X & X & \mathbb{I} \\[1mm]
X & \mathbb{I} & X & \mathbb{I} & X & \mathbb{I} & X
\end{array}
$$

ที่นี่เรามี $Z$ stabilizer generators สามตัวและ $X$ stabilizer generators สามตัว และเราได้ตรวจสอบแล้วว่านี่คือ valid stabilizer code

และ 9-qubit Shor code เป็นอีกตัวอย่าง

$$
\begin{array}{ccccccccc}
Z & Z & \mathbb{I} &
\mathbb{I} & \mathbb{I} & \mathbb{I} &
\mathbb{I} & \mathbb{I} & \mathbb{I}\\[1mm]
\mathbb{I} & Z & Z &
\mathbb{I} & \mathbb{I} & \mathbb{I} &
\mathbb{I} & \mathbb{I} & \mathbb{I}\\[1mm]
\mathbb{I} & \mathbb{I} & \mathbb{I} &
Z & Z & \mathbb{I} &
\mathbb{I} & \mathbb{I} & \mathbb{I}\\[1mm]
\mathbb{I} & \mathbb{I} & \mathbb{I} &
\mathbb{I} & Z & Z &
\mathbb{I} & \mathbb{I} & \mathbb{I}\\[1mm]
\mathbb{I} & \mathbb{I} & \mathbb{I} &
\mathbb{I} & \mathbb{I} & \mathbb{I} &
Z & Z & \mathbb{I}\\[1mm]
\mathbb{I} & \mathbb{I} & \mathbb{I} &
\mathbb{I} & \mathbb{I} & \mathbb{I} &
\mathbb{I} & Z & Z\\[1mm]
X & X & X & X & X & X &
\mathbb{I} & \mathbb{I} & \mathbb{I}\\[1mm]
\mathbb{I} & \mathbb{I} & \mathbb{I}&

\end{array}
$$

คราวนี้เรามี $Z$ stabilizer generators หกตัวและ $X$ stabilizer generators เพียงสองตัว
ไม่เป็นไร ไม่จำเป็นต้องมีความสมดุลหรือสมมาตรระหว่างสองประเภทของ generators (แม้ว่าบ่อยครั้งจะมี)

อีกครั้ง สิ่งสำคัญคือ CSS codes ต้องเป็น valid stabilizer codes และโดยเฉพาะ $Z$ stabilizer generator แต่ละตัวต้อง commute กับ $X$ stabilizer generator แต่ละตัว
ดังนั้น ไม่ใช่ทุกชุดของ $X$ และ $Z$ stabilizer generators ที่กำหนด valid CSS code
