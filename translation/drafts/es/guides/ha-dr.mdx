# Entender la alta disponibilidad y la recuperación ante desastres en Qiskit Runtime {#understanding-high-availability-and-disaster-recovery-for-qiskit-runtime}

La alta disponibilidad (HA, por sus siglas en inglés) es la capacidad de un servicio de mantenerse operativo y accesible ante fallos inesperados. La recuperación ante desastres es el proceso de restablecer la instancia del servicio a un estado funcional. Los clientes de Qiskit Runtime no son responsables de planificar ni ejecutar acciones de recuperación ante desastres.

Qiskit Runtime es un servicio regional que cumple con los [Objetivos de Nivel de Servicio (SLO)](https://cloud.ibm.com/docs/resiliency?topic=resiliency-slo) definidos en cualquier plan de pago. Para más información, consulta el [Acuerdo de Nivel de Servicio (SLA)](https://www.ibm.com/support/customer/csol/terms/?id=i126-9268&lc=en) de IBM Cloud&reg;. Para más información sobre las regiones y centros de datos de IBM Cloud disponibles para Qiskit Runtime, consulta [Disponibilidad de servicios e infraestructura por ubicación.](https://cloud.ibm.com/docs/overview?topic=overview-services_region)

## Arquitectura de alta disponibilidad {#high-availability-architecture}

### Características de alta disponibilidad {#high-availability-features}

Este servicio se aprovisiona en tres zonas dentro de una región multi-zona, sin ningún punto único de fallo. Las solicitudes de API se enrutan a través de un balanceador de carga hacia los nodos de instancia de HA, cada uno en una zona diferente.

Si un nodo de instancia de HA o una zona falla, el servicio sigue procesando las solicitudes de API a través de los nodos de instancia de HA restantes. No se produce ninguna degradación perceptible del servicio durante un fallo zonal.

Está disponible en las siguientes regiones:

- Washington DC (us-east)
- Frankfurt (eu-de)
