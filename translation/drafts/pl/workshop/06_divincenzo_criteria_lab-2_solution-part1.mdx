## 1. Kryterium 1 – Skalowalne, dobrze scharakteryzowane Qubity {#kryterium-1-skalowalne-dobrze-scharakteryzowane-qubity}

**Kryterium 1:** *„Skalowalny system fizyczny z dobrze scharakteryzowanymi Qubitami."* Oznacza to, że potrzebujemy platformy sprzętowej, w której możemy **zwiększać liczbę Qubitów** i nadal sterować nimi niezawodnie. Właściwości każdego Qubitu (poziomy energetyczne, współczynniki błędów, łączność itp.) powinny być dobrze rozumiane. W skrócie: chcemy budować większe Circuit bez degradacji systemu. W praktyce, gdy zwiększamy liczbę Qubitów lub głębokość Circuit, narastają błędy i dekoherencja, więc demonstrowanie *skalowalności* oznacza też rozumienie, jak wzrost rozmiaru wpływa na wydajność.

**Cel demonstracji:** Użyj Qiskit, aby pokazać wpływ skalowania Circuit (pod względem liczby Qubitów lub głębokości Gate'ów) na wierność wyników. Zasymulujemy scenariusz idealny i zaszumiony, aby zobaczyć, jak większy system lub głębszy Circuit ulega dekoherencji i błędom.

Najpierw zbudujemy mały stan splątany (stan GHZ) na 3 Qubitach, a następnie większy na 5 Qubitach, jako prosty test skalowalności. Stan GHZ dla *n* Qubitów to $\frac{1}{\sqrt{2}}(|0...0\rangle + |1...1\rangle)$. W idealnej symulacji pomiar n-Qubitowego stanu GHZ daje tylko dwa wyniki (same 0 lub same 1) z równym prawdopodobieństwem. Porównamy **wynik idealny** z **wynikiem zaszumionym** przy wzrastającym n lub głębokości Circuit.

```python
from qiskit import QuantumCircuit
from qiskit_aer import AerSimulator
from qiskit.visualization import plot_histogram
from qiskit.transpiler.preset_passmanagers import generate_preset_pass_manager
from qiskit_ibm_runtime import SamplerV2 as Sampler

# 3-qubit GHZ circuit
qc3 = QuantumCircuit(3, 3)
qc3.h(0)
qc3.cx(0, 1)
qc3.cx(1, 2)
qc3.measure([0, 1, 2], [0, 1, 2])

# 5-qubit GHZ circuit (scaling up the number of qubits)
qc5 = QuantumCircuit(5, 5)
qc5.h(0)
qc5.cx(0, range(1, 5))    # entangle qubit 0 with all others
qc5.measure(range(5), range(5))

# Transpile for a simulator backend
sim_backend = AerSimulator()
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc3 = pm.run(qc3)
isa_qc5 = pm.run(qc5)

# Run ideal simulations (no noise)
sampler = Sampler(mode=sim_backend)

job3 = sampler.run([isa_qc3], shots=1024)
result3 = job3.result()
counts3 = result3[0].data.c.get_counts()

job5 = sampler.run([isa_qc5], shots=1024)
result5 = job5.result()
counts5 = result5[0].data.c.get_counts()

print("3-qubit GHZ counts (ideal):", counts3)
plot_histogram(counts3, legend=['3-qubit ideal'], figsize=(6,4))
```

```python
print("5-qubit GHZ counts (ideal):", counts5)
plot_histogram(counts5, legend=['5-qubit ideal'], figsize=(6,4))
```

**Oczekiwany wynik (przypadek idealny):** 3-Qubitowy stan GHZ idealnie daje około 50% `000` i 50% `111` w zliczeniach. 5-Qubitowy GHZ daje ~50% `00000` i 50% `11111`. Żadne inne ciągi bitów się nie pojawiają, ponieważ stan jest idealnie w pełni koherentny i splątany. Na histogramie powinieneś widzieć dwa wysokie słupki dla każdego Circuit, odpowiadające wynikom ze wszystkimi zerami i wszystkimi jedynkami.

Zobaczmy teraz, co dzieje się w **środowisku zaszumionym**. Użyjemy możliwości modelu szumu Qiskit Aer, aby naśladować błędy prawdziwego urządzenia. Możemy na przykład pobrać właściwości backendu IBM, aby stworzyć model szumu uwzględniający błędy Gate'ów, skończone czasy Gate'ów, relaksację Qubitów (T₁), depfazowanie (T₂) i błędy odczytu. Użyjemy tu **fałszywego backendu** reprezentującego urządzenie IBM Quantum Brisbane, aby wygenerować model szumu i ponownie uruchomić Circuit ze stanem GHZ.

### Ćwiczenie 1a: Symulacja z szumem {#cwiczenie-1a-symulacja-z-szumem}

Uzupełnij poniższy kod, aby zasymulować Circuit ze stanem GHZ na zaszumionym symulatorze opartym na backendzie `FakeBrisbane`. Zobaczysz, jak wydajność spada wraz ze skalowaniem systemu w realistycznym środowisku szumu.

```python
from qiskit_ibm_runtime.fake_provider import FakeBrisbane

# We will reuse the ideal circuits qc3 and qc5 and their results from the previous cell.

# 1. Create a fake backend for IBM Quantum Brisbane
brisbane_backend = FakeBrisbane()

# 2. Create a noisy AerSimulator from the fake backend's properties
noisy_sim = AerSimulator.from_backend(brisbane_backend)

# 3. Transpile the circuits for the noisy simulator
pm = generate_preset_pass_manager(backend=noisy_sim, optimization_level=1)

isa_qc3_noisy = pm.run(qc3)

isa_qc5_noisy = pm.run(qc5)

# 4. Run the noisy simulations using the Sampler and get the counts
sampler = Sampler(mode=noisy_sim)

job3 = sampler.run([isa_qc3_noisy], shots=1024)

result3_noisy = job3.result()

counts3_noisy = result3_noisy[0].data.meas.get_counts()

job5 = sampler.run([isa_qc5_noisy], shots=1024)

result5_noisy = job5.result()

counts5_noisy = result5_noisy[0].data.meas.get_counts()
```

**Oczekiwany wynik (zaszumiony vs idealny):** Przy szumie – czy to symulowanym, czy na prawdziwym urządzeniu – stan GHZ jest **mniej doskonały**. Zobaczysz dodatkowe wyniki poza wszystkimi zerami i jedynkami. Dla 3 Qubitów, zamiast 100% w `000`/`111`, część prawdopodobieństwa „wycieka" do innych ciągów bitów (np. `001`, `010` itd.) z powodu błędów Gate'ów lub dekoherencji odwracającej niektóre Qubity. Dla 5 Qubitów efekt jest jeszcze wyraźniejszy; większy Circuit (więcej Qubitów i bramek CNOT) kumuluje więcej błędów, więc szczyty dla wszystkich zer i wszystkich jedynek są niższe, a pojawia się wiele innych wyników. Ten trend ilustruje wyzwanie *skalowalności*: w miarę skalowania utrzymanie wysokiej wierności staje się trudniejsze bez korekcji błędów.

**Wniosek:** Skalowalny komputer kwantowy musi zachowywać korelacje kwantowe wraz ze wzrostem systemu. Nasze przykłady pokazują, jak zwiększanie liczby Qubitów i głębokości Gate'ów powoduje spadek wierności wyników w obecności szumu. Pozostałe kryteria dotyczą utrzymywania tych Qubitów w dobrym stanie (niski poziom błędów, możliwość inicjalizacji itd.) podczas skalowania.
