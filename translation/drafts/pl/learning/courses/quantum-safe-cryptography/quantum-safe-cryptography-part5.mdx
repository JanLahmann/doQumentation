for i in range(N):
    sum_ai = sum_ai + r[i] * alice_public_key[i][0]
    sum_bi = sum_bi + r[i] * alice_public_key[i][1]
sum_ai = [x % q for x in sum_ai]
# sum_bi = sum_bi
ciphertext = (sum_ai, (bob_message_bit * int(np.floor(q / 2)) + sum_bi) % q)
print(f"ciphertext is: {ciphertext}")
```

Na koniec Bob rozgłasza szyfrogram, który Alice może odszyfrować przy użyciu swojego klucza prywatnego.

```python
# Decryption
adots = np.dot(ciphertext[0], alice_private_key) % q
b_adots = (ciphertext[1] - adots) % q

decrypted_message_bit = round((2 * b_adots) / q) % 2

print(
    f"original message bit={bob_message_bit}, decrypted message bit={decrypted_message_bit}"
)

assert bob_message_bit == decrypted_message_bit
```

Ponieważ ten protokół działa bit po bicie przy szyfrowaniu dłuższych ciągów bitów, po prostu powtarzamy operacje w pętli. Poniżej przedstawiono scenariusz, w którym Bob chce przesłać 16 zaszyfrowanych bitów.

```python
bob_message_bits = np.random.randint(0, 2, 16)
print(f"Bob's message bits are : {bob_message_bits}")
decrypted_bits = []

for ib in range(len(bob_message_bits)):
    bob_message_bit = bob_message_bits[ib]

    r = np.random.randint(0, 2, N)

    sum_ai = np.zeros(n, dtype=int)
    sum_bi = 0
    for i in range(N):
        sum_ai = sum_ai + r[i] * alice_public_key[i][0]
        sum_bi = sum_bi + r[i] * alice_public_key[i][1]
    sum_ai = [x % q for x in sum_ai]

    ciphertext = (sum_ai, (bob_message_bit * int(np.floor(q / 2)) + sum_bi) % q)

    adots = np.dot(ciphertext[0], alice_private_key) % q
    b_adots = (ciphertext[1] - adots) % q

    decrypted_message_bit = round((2 * b_adots) / q) % 2
    assert decrypted_message_bit == bob_message_bit

    decrypted_bits.append(decrypted_message_bit)

print(f"Decrypted message bits = {np.array(decrypted_bits)}")
```

## Module-LWE i pakiet CRYSTALS {#module-lwe-and-the-crystals-suite}

Problem uczenia się z błędami (LWE), wprowadzony powyżej w uproszczonej formie i ogólnie obowiązujący na dowolnych kratach, został rozszerzony na algebraiczne *pierścienie* w ramach tak zwanego frameworku [**Ring-LWE**](https://en.wikipedia.org/w/index.php?title=Ring_learning_with_errors&oldid=1148731729) głównie w celu poprawy efektywności wynikających z niego kryptosystemów. Jednakże dodatkowa struktura algebraiczna Ring-LWE może być potencjalnie wykorzystywana, mimo że obecnie nie są znane żadne takie sposoby ataku.

Dwóch z czterech finalistów procesu standaryzacji QSC organizowanego przez NIST — a mianowicie mechanizm enkapsulacji klucza (KEM) [**CRYSTALS-Kyber**](https://pq-crystals.org/kyber/index.shtml) oraz protokół podpisu cyfrowego [**CRYSTALS-Dilithium**](https://pq-crystals.org/dilithium/index.shtml) — opiera się na strukturach znanych jako <DefinitionTooltip definition="Collection of all submodules of a given module. A submodule is a subset of the module that itself forms a module under the same operations. The set of all submodules forms a lattice, which is a partially ordered set where any two elements have a unique least upper bound and greatest lower bound.">*kraty modułowe*</DefinitionTooltip> oraz powiązanym [**Module-LWE**](https://eprint.iacr.org/2012/090).

Zainteresowanych czytelników odsyłamy do specjalistycznych kursów dotyczących tych zagadnień w celu zapoznania się ze szczegółami matematycznymi, ale zwracamy tu uwagę, że Module-LWE jest postrzegane jako kompromis pomiędzy LWE a Ring-LWE. Zapewnia ono większą efektywność niż LWE (choć mniejszą niż Ring-LWE), ale daje również wyższy poziom gwarancji bezpieczeństwa niż Ring-LWE, ponieważ nie polega tak silnie na strukturze algebraicznej problemu.

### Mechanizmy enkapsulacji klucza i CRYSTALS-Kyber {#key-encapsulation-mechanisms-and-crystals-kyber}
Tradycyjne kryptosystemy z kluczem asymetrycznym są najszerzej wdrażane ze względu na funkcjonalności *wymiany klucza* oraz *podpisu cyfrowego*, dlatego też proces standaryzacji NIST miał na celu opracowanie kwantowo-bezpiecznych alternatyw dla tych dwóch funkcjonalności. Protokół [**CRYSTALS-Kyber**](https://pq-crystals.org/kyber/index.shtml) został zatem zaprojektowany jako dedykowany [**Mechanizm enkapsulacji klucza**](https://en.wikipedia.org/w/index.php?title=Key_encapsulation_mechanism&oldid=1110687965) (KEM), a nie jako schemat szyfrowania ogólnego przeznaczenia, taki jak RSA.
