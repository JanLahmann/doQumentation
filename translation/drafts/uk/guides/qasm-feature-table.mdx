# Таблиця функцій OpenQASM 3

Нижче наведено список мовних функцій `OpenQASM 3`.

Докладніші відомості про ці можливості дивись у [живій специфікації OpenQASM 3.X](https://openqasm.com/).

Ключ:

*   ❌ Не підтримується
*   🟡 Часткова підтримка
*   ✅ Підтримується

Значення повної позначки «підтримується» залежить від стовпця:

* *Qiskit SDK*: функція може бути розібрана [`qiskit.qasm3.loads`](https://docs.quantum.ibm.com/api/qiskit/qasm3) (за допомогою розширення `qiskit-qasm3-import`), представлена у [`QuantumCircuit`,](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit)
  та експортована до OpenQASM 3 через [`qiskit.qasm3.dumps`.](https://docs.quantum.ibm.com/api/qiskit/qasm3#dumps)

* *IBM Qiskit Runtime*: схема, що містить відповідну функцію Qiskit, може бути успішно виконана на апаратному забезпеченні через IBM&reg; Qiskit Runtime.

Значення «часткової підтримки» зазвичай залежить від наведених приміток за посиланням.

<Admonition type="note">
Найпоширенішим способом надсилання схем до IBM Qiskit Runtime є створення схеми через інтерфейс Python для Qiskit SDK.
Схеми, побудовані та надіслані таким чином, не потребують завантаження з файлів OpenQASM 3 до Qiskit SDK.

Якщо ти не використовуєш OpenQASM 3 безпосередньо, можна безпечно застосовувати функції, які підтримуються для представлення в Qiskit SDK, експорту до OpenQASM 3 і надсилання до IBM Qiskit Runtime.
Це стосується й функцій, які не можуть бути завантажені Qiskit SDK з OpenQASM 3.
</Admonition>

## Примітки {#notes}

<span id="1"></span>
1.  Ці функції програми OpenQASM 3 не впливають на виконання, і Qiskit видаляє їх під час розбору файлів. Файли, що їх використовують, можна надсилати, але вони не матимуть жодного ефекту. Для файлів `include` `stdgates.inc` наразі підтримується як вхідні дані для Qiskit, а виконання на бекенді завжди вимагає, щоб схеми були скомпільовані до архітектури набору інструкцій (ISA) бекенду, де файли `include` є нерелевантними.

<span id="2"></span>
2. Qiskit SDK підтримує розбір і запис файлів OpenQASM 3 з будь-якими оголошеннями ``qubit``. Для виконання на апаратному забезпеченні дійсними є лише схеми, визначені в термінах апаратних кубітів (наприклад, `$0`). Qiskit SDK автоматично виводить OpenQASM 3 у термінах підтримуваних ідентифікаторів апаратних кубітів, якщо схема транспільована для бекенду з інформацією про розміщення.

<span id="3"></span>
3. Оголошення змінних типу `bit` та `bit[n]` у Qiskit SDK відповідають оголошенням `Clbit` і `ClassicalRegister`.

<span id="4"></span>
4. Станом на липень 2025 року Qiskit SDK може представляти локальні змінні обмеженого набору типів, підтримувати багато операцій виконання над цими об'єктами та виводити їх до OpenQASM 3. Однак Qiskit SDK (через `qiskit-qasm3-import` v0.6.0) не підтримує розбір файлів OpenQASM 3, що містять оголошення змінних, і має дуже обмежену підтримку розбору виразів змінних. Загалом, більшість того, що Qiskit може представити у своїй системі виразів, може виконуватися на відповідному апаратному забезпеченні динамічних схем, навіть якщо вираз ще не може бути розібраний Qiskit SDK. Актуальну інформацію дивись у [документації Qiskit модуля `qiskit.circuit.classical`](https://docs.quantum.ibm.com/api/qiskit/circuit_classical).

<span id="5"></span>
5. Qiskit SDK може представляти аліасинг регістрів як для квантових, так і для класичних регістрів, але використання аліасингу класичних регістрів настійно не рекомендується. Більшість виразів над класичними регістрами не працює з псевдонімами, а аліасовані класичні регістри не підтримуються для виконання на апаратному забезпеченні. Парсер OpenQASM 3 Qiskit може розв'язувати оператори псевдонімів `let`, що прив'язують результат конкатенації регістрів.

<span id="6"></span>
6. Qiskit SDK підтримує явні затримки через `QuantumCircuit.delay`, а блоки схем (`QuantumCircuit.box`) також можуть мати явні тривалості. Ці тривалості можуть включати класичні вирази змінних `stretch`. Qiskit SDK (станом на липень 2025 року через `qiskit-qasm3-import` v0.6.0) не підтримує розбір оголошень типу `duration` або `stretch` з файлів OpenQASM 3. Апаратне забезпечення має обмежену підтримку тривалостей, що включають `stretch`.

<span id="7"></span>
7. Схеми мають бути транспільовані до ISA бекенду для запуску на апаратному забезпеченні IBM. Це унеможливлює використання власних визначень `gate` і конструкцій вищого рівня, таких як модифікатори вентилів (наприклад, `inv @`), безпосередньо для виконання на апаратному забезпеченні, однак процес `transpile` перетворює їх на дійсні ISA-схеми. Qiskit SDK (станом на липень 2025 року, через `qiskit-qasm3-import` v0.6.0) активно обчислює модифікатори вентилів під час розбору, тому вони не будуть присутні в результуючому `QuantumCircuit`, що потенційно пов'язано з витратами на виконання.

<span id="8"></span>
8. Qiskit SDK може представляти структурований потік керування та експортувати його до OpenQASM 3. Оператори `continue` і `break` технічно можуть бути представлені Qiskit, але погано підтримуються навіть у самому Qiskit SDK. Цикли `for` у Qiskit v2.1.0 погано підтримуються. Вкладений потік керування (наприклад, `if` всередині іншого `if` або оператор `else if`) не може виконуватися на апаратному забезпеченні.

<span id="9"></span>
9. Qiskit SDK підтримує оголошення будь-якого підтримуваного класичного типу як змінної `input` у схемі. Такі змінні наразі не можуть виконуватися на апаратному забезпеченні та не можуть бути завантажені імпортером OpenQASM 3 Qiskit. Незв'язані об'єкти `Parameter` у `QuantumCircuit` експортуються як змінні `input float[64]`. Певні параметри конфігурації виконання можуть дозволити виконання таких схем на деяких бекендах.
