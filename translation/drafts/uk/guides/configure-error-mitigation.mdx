Техніки пом'якшення помилок дозволяють користувачам зменшувати помилки Circuit, моделюючи шуми пристрою на момент виконання. Зазвичай це призводить до накладних витрат на квантову попередню обробку, пов'язаних з навчанням моделі, та накладних витрат на класичну постобробку для пом'якшення помилок у «сирих» результатах за допомогою згенерованої моделі.

Примітив Estimator підтримує кілька технік пом'якшення помилок, зокрема [TREX](https://docs.quantum.ibm.com/api/qiskit-ibm-runtime/options-resilience-options-v2#measure_mitigation), [ZNE](https://docs.quantum.ibm.com/api/qiskit-ibm-runtime/options-resilience-options-v2#zne), [PEC](https://docs.quantum.ibm.com/api/qiskit-ibm-runtime/options-resilience-options-v2#pec) та [PEA](https://docs.quantum.ibm.com/api/qiskit-ibm-runtime/options-zne-options). Дивись [Техніки пом'якшення та придушення помилок](error-mitigation-and-suppression-techniques) для пояснення кожної з них. Під час використання примітивів можна вмикати або вимикати окремі методи. Подробиці дивись у розділі [Власні налаштування помилок](#advanced-error).

<Admonition type="note">

Sampler не підтримує пом'якшення помилок, але ти можеш використовувати пакет [mthree](https://qiskit.github.io/qiskit-addon-mthree/) (matrix-free measurement mitigation) для локального пом'якшення помилок.

</Admonition>

Estimator також підтримує `resilience_level`. Рівень стійкості визначає, наскільки захистити результати від помилок. Вищі рівні дають точніші результати, але потребують більше часу на обробку. Рівні стійкості можна використовувати для налаштування балансу між витратами та точністю при застосуванні пом'якшення помилок до запиту примітиву. Пом'якшення помилок зменшує похибки (зміщення) результатів, обробляючи виходи з колекції або ансамблю пов'язаних Circuit. Ступінь зменшення похибки залежить від застосованого методу. Рівень стійкості абстрагує детальний вибір методу пом'якшення помилок, щоб дати користувачам змогу думати про компроміс між витратами та точністю, що підходить для їхнього застосування.

З огляду на це, кожен рівень відповідає методу або методам зі зростаючими накладними витратами на квантову вибірку, що дає змогу експериментувати з різними компромісами між часом і точністю. У наступній таблиці показано, які рівні та відповідні методи доступні для кожного з примітивів.

<Admonition type="info" title="Увага">
Пом'якшення помилок є специфічним для конкретного завдання, тому техніки, які можна застосувати, відрізняються залежно від того, чи ти виконуєш вибірку розподілу або обчислюєш очікувані значення.
</Admonition>

<span id="resilience-table"></span>

Estimator підтримує наведені нижче рівні стійкості. Sampler не підтримує рівні стійкості.

| Рівень стійкості | Визначення                                                                                                                              | Техніка                                                                          |
|------------------|-----------------------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------|
| 0                | Без пом'якшення                                                                                                                         | Немає                                                                            |
| 1 [За замовчуванням] | Мінімальні витрати на пом'якшення: усунення похибок, пов'язаних з помилками зчитування                                             | Twirled Readout Error eXtinction (TREX) із застосуванням вихрового зчитування    |
| 2                | Середні витрати на пом'якшення. Зазвичай зменшує зміщення в оцінювачах, але не гарантує нульового зміщення. | Рівень 1 + Zero Noise Extrapolation (ZNE) та вихрове застосування Gate

<Admonition type="info" title="Увага">

Рівні стійкості наразі перебувають у бета-версії, тому накладні витрати на вибірку та якість рішень варіюватимуться від Circuit до Circuit. Нові функції, розширені параметри та інструменти управління випускатимуться поетапно. Конкретні методи пом'якшення помилок не гарантується застосовувати на кожному рівні стійкості.
</Admonition>

## Налаштування Estimator з рівнями стійкості {#configure-estimator-with-resilience-levels}

Ти можеш використовувати рівні стійкості для вказання технік пом'якшення помилок або встановлювати власні техніки окремо, як описано в розділі [Власні налаштування помилок.](#advanced-error)

<details>
<summary>Рівень стійкості 0</summary>

До програми користувача не застосовується жодне пом'якшення помилок.

</details>

<details>
<summary>Рівень стійкості 1</summary>

Рівень 1 застосовує **пом'якшення помилок зчитування** та **вихрове вимірювання** за допомогою безмодельної техніки відомої як Twirled Readout Error eXtinction (TREX). Вона зменшує похибку вимірювання шляхом діагоналізації каналу шуму, пов'язаного з вимірюванням, за допомогою випадкового перекидання Qubit через X Gate безпосередньо перед вимірюванням. Масштабний коефіцієнт діагонального шумового каналу визначається шляхом еталонування випадкових Circuit, ініціалізованих у нульовому стані. Це дозволяє сервісу усувати зміщення з очікуваних значень, що виникають через шум зчитування. Цей підхід детальніше описано в [Model-free readout-error mitigation for quantum expectation values](https://arxiv.org/abs/2012.09738).

</details>

<details>
<summary>Рівень стійкості 2</summary>

Рівень 2 застосовує **техніки пом'якшення помилок, що входять до рівня 1**, а також додає **вихрове застосування Gate** та використовує **метод Zero Noise Extrapolation (ZNE)**. ZNE обчислює очікуване значення спостережуваної величини для різних коефіцієнтів шуму (стадія підсилення), а потім використовує отримані очікувані значення для визначення ідеального очікуваного значення при нульовому шумі (стадія екстраполяції). Цей підхід, як правило, зменшує похибки в очікуваних значеннях, але не гарантує отримання незміщеного результату.

![На цьому зображенні показано графік. Вісь X позначена як «Коефіцієнт підсилення шуму». Вісь Y позначена як «Очікуване значення». Лінія з нахилом вгору позначена «Пом'якшене значення». Точки поблизу лінії є значеннями з підсиленим шумом. Горизонтальна лінія трохи вище осі X позначена «Точне значення».](/docs/images/guides/configure-error-mitigation/resilience-2.svg "Ілюстрація методу ZNE")

Накладні витрати цього методу масштабуються відповідно до кількості коефіцієнтів шуму. Налаштування за замовчуванням вибирають очікуване значення при трьох коефіцієнтах шуму, що призводить до приблизно 3-кратних накладних витрат при використанні цього рівня стійкості.

На рівні 2 метод TREX випадково перекидає Qubit через X Gate безпосередньо перед вимірюванням і перекидає відповідний виміряний біт, якщо було застосовано X Gate. Цей підхід детальніше описано в [Model-free readout-error mitigation for quantum expectation values](https://arxiv.org/abs/2012.09738).

</details>

### Приклад {#example}

Інтерфейс `EstimatorV2` дозволяє користувачам безперешкодно працювати з різноманітними методами пом'якшення помилок для зменшення похибок в очікуваних значеннях спостережуваних величин. Наступний код використовує Zero Noise Extrapolation та пом'якшення помилок зчитування, просто встановивши `resilience_level 2`.

```python
# Added by doQumentation — required packages for this notebook
!pip install -q qiskit-ibm-runtime
```

```python
from qiskit_ibm_runtime import QiskitRuntimeService
from qiskit_ibm_runtime import EstimatorV2 as Estimator

service = QiskitRuntimeService()
backend = service.least_busy(operational=True, simulator=False)

# Setting options during primitive initialization
estimator = Estimator(backend, options={"resilience_level": 2})
```

## Власні налаштування помилок {#custom-error-settings}

    ```python
from qiskit_ibm_runtime import QiskitRuntimeService
from qiskit_ibm_runtime import EstimatorV2 as Estimator

service = QiskitRuntimeService()
backend = service.least_busy(operational=True, simulator=False)

estimator = Estimator(backend)
options = estimator.options
# Turn on gate twirling.
options.twirling.enable_gates = True
# Turn on measurement error mitigation.
options.resilience.measure_mitigation = True

print(f">>> gate twirling is turned on: {estimator.options.twirling.enable_gates}")
print(f">>> measurement error mitigation is turned on: {estimator.options.resilience.measure_mitigation}")
```
