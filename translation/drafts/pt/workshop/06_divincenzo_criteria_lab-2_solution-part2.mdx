## 3. Critério 3 – Longo Tempo de Coerência (Decoerência vs Tempo de Gate)

**Critério 3:** *"Longos tempos de decoerência relevantes, muito mais longos do que o tempo de operação do Gate."* Isso trata da necessidade de os Qubits manterem seu estado quântico por tempo suficiente para realizar as operações necessárias. Cada Qubit tem um **tempo T₁** (tempo de relaxamento de energia, com qué velocidade |1〉 decai para |0〉) e um **tempo T₂** (tempo de defasagem, com qué velocidade a coerência de fase relativa é perdida). Para que um computador quântico funcione, essas escalas de tempo devem ser muito maiores do que a duração das operações de Gate.

**Objetivo da Demonstração:** Investigar a coerência dos Qubits no Qiskit mostrando como a decoerência impacta os resultados do Circuit à medida que o comprimento da execução aumenta. Vamos usar um backend falso com tempos T1/T2 conhecidos para simular esse efeito.

Para **demonstrar o impacto da coerência finita**, vamos simular um experimento de decaimento T1. Vamos preparar um Qubit no estado |1〉, aguardar algum tempo usando uma instrução `delay`, e depois medir. Esperamos que a probabilidade de medir |1〉 diminua à medida que o atraso aumenta.

```python
# This part is done for you. We are creating a list of circuits,
# each with a different delay time.

time_delays_ns = [0, 50000, 100000, 150000, 200000, 250000, 300000]  # delay durations in ns

decay_expts = []
for delay in time_delays_ns:
    qc = QuantumCircuit(1, 1)
    qc.x(0)  # initialize qubit to |1>
    if delay > 0:
        qc.delay(delay, 0, unit='ns')  # wait 'delay' nanoseconds
    qc.measure(0, 0)
    decay_expts.append(qc)

decay_expts[1].draw('mpl') # Visualize one of the circuits
```

## 4. Critério 4 – Conjunto Universal de Gates Quânticos

**Critério 4:** *"Um conjunto 'universal' de Gates quânticos."* Isso significa que nosso hardware deve nos permitir realizar *qualquer* computação quântica compondo um conjunto finito de Gates básicos. Na computação clássica, NAND é universal; no quântico, há muitas escolhas de conjuntos de Gates universais (por exemplo, \{H, T, CNOT\} ou os Gates nativos de uma determinada máquina). Os dispositivos IBM, por exemplo, têm um conjunto de operações nativas como rotações de Qubit único arbitrárias e CNOTs entre certos Qubits, que juntos são universais. O trabalho do Qiskit muitas vezes é **compilar Gates de alto nível nesses Gates de base**.

**Objetivo da Demonstração:** Ilustrar a universalidade dos Gates mostrando como o Qiskit decompõe os Gates. Vamos pegar um Gate não-nativo (como um Gate de Toffoli de 3 Qubits, CCX) e ver como ele se divide nos Gates de base do dispositivo. Isso demonstra que o conjunto de Gates fornecido é de fato *universal* – ele pode produzir a operação mais complexa.

Primeiro, vamos ver quais são os Gates de base para um backend IBM típico. Vamos consultar a configuração de um dispositivo (ou sua versão falsa). Por exemplo, os Gates de base do ibmq_brisbane:
Você deve observar a probabilidade `P(qubit=1)` diminuindo à medida que o tempo de atraso aumenta, seguindo uma curva de decaimento exponencial característica do relaxamento T1. Isso demonstra diretamente como o tempo de coerência finito leva a erros computacionais se o Circuit for executado por muito tempo.

**Impacto nos algoritmos:** Se você tentar um algoritmo mais longo (com muitos Gates sequenciais), o tempo total de execução pode se aproximar ou exceder T2, fazendo com que o estado perca coerência antes do fim. É por isso que melhorar os tempos de coerência e tornar os Gates mais rápidos são dois dos objetivos mais críticos na pesquisa de hardware quântico.

Isso pode produzir algo como `['id', 'rz', 'sx', 'x', 'ecr']`. Essas são as operações primitivas que o hardware suporta nativamente (Identidade/sem operação, rotação RZ, Gate sqrt(X), Gate X e X controlado). Qualquer outro Gate deve ser composto a partir desses. Esse conjunto é reconhecidamente universal para a computação quântica (essencialmente rotações de Qubit único mais um Gate de dois Qubits entrelaçante formam um conjunto universal).

## 5. Critério 5 – Medição de Qubit

**Critério 5:** *"Uma capacidade de medição específica por Qubit."* O estado de cada Qubit deve ser mensurável (tipicamente na base computacional, |0〉 ou |1〉). Em outras palavras, após executar um Circuit quântico, precisamos ler cada Qubit como um bit clássico 0/1. Esse critério trata de ter detectores confiáveis para cada Qubit e de ser capaz de selecionar quais Qubits medir.

**Objetivo da Demonstração:** Mostrar como realizar medições no Qiskit em simuladores e dispositivos reais, e destacar as diferenças (como o ruído de medição). Vamos medir alguns Qubits em vários estados e examinar os resultados. Também vamos demonstrar como erros de leitura podem aparecer comparando resultados do simulador com os do hardware.

Primeiro, um exemplo simples de medição:

```python
qc_measure = QuantumCircuit(2, 2)
qc_measure.x(0)              # qubit 0 -> |1>, qubit 1 stays |0>
qc_measure.measure([0, 1], [0, 1])
qc_measure.draw('mpl')
```

```python
sim_backend = AerSimulator()
```

### Exercício 5: Simular Erro de Leitura
Complete o código para definir um modelo simples de erro de leitura onde cada Qubit tem 2% de chance de ser medido incorretamente (um 0 é lido como 1, ou um 1 é lido como 0). Em seguida, execute o Circuit de medição com esse modelo de ruído.

```python
from qiskit_aer.noise import NoiseModel, ReadoutError

# --- YOUR CODE HERE ---

# 1. Define a 2% readout error for each single qubit.
# [[P(0|0), P(1|0)], [P(0|1), P(1|1)]]
ro_error = ReadoutError([[0.98, 0.02], [0.02, 0.98]])

# 2. Create a new noise model
noise_model_ro = NoiseModel()

# 3. Add the readout error to all qubits in the noise model
noise_model_ro.add_all_qubit_readout_error(ro_error)

# --- END YOUR CODE ---

sim_backend.set_options(noise_model=noise_model_ro)
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc_measure = pm.run(qc_measure)

# Run the measurement circuit with readout noise
sampler = Sampler(mode=sim_backend)

job = sampler.run([isa_qc_measure], shots=1024)
result = job.result()
counts = result[0].data.c.get_counts()

print("Simulation with 2% readout error:", counts)
```

Essa saída simulada vai mostrar algumas contagens errôneas (como `11`, `00`, `10`) semelhantes ao que o hardware real pode produzir, demonstrando o impacto de uma medição imperfeita.

**Exemplo de dispositivo:** Em um dispositivo real como o **ibmq_brisbane**, você poderia executar o mesmo Circuit e provavelmente veria contagens semelhantes, não nulas, para os resultados incorretos. Os dados de calibração do dispositivo listam um erro de leitura para cada Qubit. Ser capaz de selecionar e ler Qubits específicos é crucial, e entender suas características de erro é fundamental para obter resultados significativos. ## Critérios de Comunicação Quântica (Qubits Voadores)

DiVincenzo também listou dois critérios específicos para comunicação quântica, importantes na construção de um computador quântico em rede:

6. **Capacidade de interconverter Qubits estacionários e voadores.** (Por exemplo, mapear um Qubit em um processador para um fóton que pode viajar.)
7. **Capacidade de transmitir fielmente Qubits voadores entre localizações.** (Por exemplo, enviar um Qubit de fóton por uma fibra sem perder informação quântica.)

Esses estão além do uso padrão do Qiskit porque o Qiskit lida principalmente com Qubits estacionários em um chip. No entanto, podemos ilustrar o *conceito* desses critérios com um exemplo simples: **teleportação quântica**. A teleportação mostra a conversão do estado de um Qubit estacionário em informação carregada por um par entrelaçado (a parte "voadora") e comunicação clássica, que é então usada para reconstruir o estado em outro Qubit estacionário em outro lugar.
