# Mag-migrate mula sa `BackendV1` patungong `BackendV2`

Ang Qiskit [`BackendV1`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.BackendV1) class ay na-deprecate na at aalisin sa serbisyo. Inilalarawan ng migration guide na ito ang maliliit na pagbabagong kailangan mong gawin kung gumagamit ka ng provider na nag-upgrade mula sa [`BackendV1`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.BackendV1) patungong [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2).

<Admonition type="note">
 Kung eksklusibo kang gumagamit ng `qiskit_ibm_runtime` at `qiskit_aer`, walang kailangang gawin. Ang `qiskit_ibm_runtime` package ay palaging gumagamit ng `BackendV2`, at ang `qiskit_aer` ay gumagamit ng `BackendV2` mula pa sa bersyon 0.13.
 </Admonition>

## Mga mataas na antas na pagbabago sa `BackendV2` {#high-level-changes-in-backendv2}

Ang Qiskit [`Backend`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.Backend) model ay idinisenyo para bigyan ang Qiskit SDK ng abstraction layer na nagbibigay-daan sa pag-iisip tungkol sa mga quantum computer sa loob ng saklaw ng SDK. Ang unang iterasyon ng modelo ay ipinakilala gamit ang [`BackendV1`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.BackendV1) class. Inimbak ng class na ito ang impormasyon ng backend sa isang serye ng mga data container, partikular ang [`BackendConfiguration`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.models.BackendConfiguration) at [`BackendProperties`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.models.BackendProperties) classes.

Binago ng [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) class ang paraan ng pag-access ng user sa karamihan ng backend properties para gumana ang mga ito kasama ang native na Qiskit data structures at magkaroon ng mas simpleng access patterns. Ang pangunahing bahagi ng [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) model ay ang [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) class, isang representasyon ng QPU na naglalaman ng mga transpilation constraints na magagamit ng Qiskit para i-optimize ang mga Circuit para sa execution.

Na-update na ang Qiskit SDK para eksklusibong gumana kasama ang mga [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) input, at karamihan sa mga provider ay nag-upgrade na mula sa [`BackendV1`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.BackendV1) patungong [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2). Inaasahan na ang mga kasalukuyang provider ay mag-deprecate ng lumang access kung posible para magbigay ng maayos na migration, ngunit sa huli ay kailangang i-adjust ng mga gumagamit ang kanilang code.

Ang prinsipyo sa likod ng [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) ay ang karamihan sa impormasyon tungkol sa isang backend ay nakapaloob sa loob ng [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) object nito, at ang mga attribute ng backend ay madalas na nagtatanong sa `BackendV2.target` attribute nito para magbalik ng impormasyon. Gayunpaman, sa maraming kaso ang mga attribute ay nagbibigay lamang ng subset ng impormasyon na maaaring nakapaloob sa target. Halimbawa, ang `backend.coupling_map` ay nagbabalik ng [`CouplingMap`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.CouplingMap) object na itinayo mula sa [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) na accessible sa `BackendV2.target` attribute. Gayunpaman, maaaring naglalaman ang target ng mga instruction na gumagana sa higit sa dalawang Qubit (na hindi maaaring irepresenta sa isang [`CouplingMap`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.CouplingMap)) o maaaring mayroon itong mga instruction na gumagana lamang sa isang subset ng mga Qubit (o dalawang Qubit link, para sa dalawang Qubit na instruction), na hindi magiging detalyado sa buong coupling map na ibinalik ng `BackendV2.coupling_map`. Kaya depende sa iyong use case, maaaring kinakailangan na mag-look deeper kaysa sa katumbas na access gamit ang [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2).

## Mga tiyak na pagbabago sa `BackendV2` {#specific-changes-in-backendv2}

Karamihan sa mga attribute ay may direktang kapalit, na pinapasimple ang mga pagsisikap sa migration. Ang tanging punto ng hindi pagkakatugma sa pagitan ng mga interface ay nasa [`CouplingMap`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.CouplingMap).

Sumusunod ay isang talahanayan ng mga halimbawa ng access pattern sa [`BackendV1`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.BackendV1) at ang bagong form gamit ang [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2).
