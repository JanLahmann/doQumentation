# Gabay sa migration ng Qiskit v2.0 {#qiskit-v20-migration-guide}

Ang Qiskit v2.0 ang susunod na hakbang sa bagong na-stabilize na life cycle ng Qiskit SDK.
Ang major version na ito ay nakaka-break ng compatibility sa mga nakaraang bersyon ng Qiskit, pero,
kumpara sa v1.0 release, walang mga pagbabago sa packaging structure.

Inilalarawan ng gabay na ito ang mga migration path para sa pinakamahalagang pagbabago sa features ng Qiskit v2.0, nakaayos ayon sa module.
Gamitin ang table of contents sa kanang bahagi para pumunta sa module na interesado ka.

Basahin pa ang mga benepisyo ng Qiskit v2.0 sa [IBM&reg; blog](https://www.ibm.com/quantum/blog/qiskit-2-0-release-summary).

## qiskit.circuit

<span id="c_if"></span>
### Pagtanggal ng legacy na `c_if` conditions {#removal-of-legacy-c_if-conditions}

Ang mga classical condition sa Qiskit ay kinakatawan na ng instruction na [`IfElseOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.IfElseOp) bago pa ang Qiskit v1.0.
Ang mga operasyong ito ay karaniwang idinadagdag sa isang [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit) gamit ang paraan na [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) bilang isang context manager.

Ibig sabihin din nito na ang [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) ay wala nang `condition` attribute, at hindi mo na maaaring i-chain ang mga `c_if` na tawag sa output ng isang method call na lumilikha ng gate sa [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit).

Ang mga tanging instruction na maaaring may condition ngayon ay [`IfElseOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.IfElseOp) at [`WhileLoopOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.WhileLoopOp) lamang.
Ang pinakatamang paraan para suriin ang isang condition ay gumamit ng `isinstance` check para matiyak na tama ang klase.
Kung gusto mong suportahan ang legacy na `Instruction.condition` attribute habang nag-e-transition ang iyong library sa Qiskit v2.0 at mas bago, maaari kang gumamit ng `getattr(operation, "condition", None)`, na kadalasan ay magpapaliwanag ng `Instruction` instances na parang nasa Qiskit v1.x at mas lumang bersyon.

Ang orihinal, legacy na paraan ng pag-suporta sa simpleng register-equality conditions sa Qiskit ay ang paggamit ng `c_if` method sa mga indibidwal na instruction o instruction sets.
Ito ay mahigpit na kulang sa features kumpara sa buong control-flow form, at hindi kailanman maayos na sinuportahan sa hardware.
Ang legacy na pamamaraang ito ay inalis na sa Qiskit v2.0; dapat kang gumamit ng paraan na [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) na may tamang condition.
Maaari kang patuloy na gumamit ng 2-tuple form na `(register, integer)` para kumatawan ng register equality, o maaari kang gumamit ng mas maliwanag na [classical computation subsystem](https://docs.quantum.ibm.com/api/qiskit/circuit_classical) para kumatawan ng mas kumplikadong conditions.

Halimbawa, maaaring nasulat mo ito:

```python
from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
qc.x(0).c_if(cr, 3)
```

Ang pinakadirektang pagsasalin nito ay ang paggamit ng 2-tuple form kasama ang [`if_test`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.if_test) context manager:

```python
from qiskit.circuit import QuantumCircuit, QuantumRegister, ClassicalRegister

qr = QuantumRegister(2, "q")
cr = ClassicalRegister(2, "c")
qc = QuantumCircuit(qr, cr)
qc.h(0)
qc.h(1)
qc.measure(qr, cr)

# Apply an X to qubit 0 if both measurements were 1.
with qc.if_test((cr, 3)):
    qc.x(0)
```

Pansinin na sa form na ito, maaari ka ring magbigay ng access sa conditional gate sa maraming operasyon (idagdag lang ang mga ito sa indented block) nang hindi muling sinusuri ang mga condition.
Maaari ka ring magbigay ng ``else`` statement, at gumamit ng mas kumplikadong conditional logic:

```python
from qiskit.circuit import QuantumCircuit
from qiskit.circuit.classical import expr

qc = QuantumCircuit(2, 2)
qc.h(0)
qc.h(1)
qc.measure(qc.qubits, qc.clbits)

# Apply an X to qubit 0 if both measurements were the same:
with qc.if_test(expr.equal(qc.clbits[0], qc.clbits[1])) as else_:
    qc.x(0)
# ... or apply a Y to both qubits otherwise:
with else_:
    qc.y(0)
    qc.y(1)
```

<span id="instruction_durations"></span>
### Pagtanggal ng `Instruction.duration` at `Instruction.unit` {#removal-of-instructionduration-and-instructionunit}

Ang mga Circuit [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) objects ay dati'y may mga attribute na tinatawag na `duration` at `unit`.
Wala na ang mga ito sa Qiskit v2.0.
Ito ay dahil ang field sa pangkalahatan ay nagdo-duplicate lamang ng impormasyon na isang fixed property ng hardware, at tahasan itong nakaimbak sa [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target).
Ang pag-iimbak ng impormasyon nang hiwalay para sa bawat indibidwal na instruction ay malaki ang dagdag sa memory requirements ng Qiskit at nagpabagal pa kahit sa mga circuit na operasyon na hindi direktang kailangan ng impormasyon.

Ang [`Delay`](https://docs.quantum.ibm.com/api/qiskit/circuit#delay) instruction ay nagtataglay pa rin ng mga field na ito, gayundin ang bagong [`BoxOp`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#box) na idinagdag sa v2.0, dahil ang mga instruction na ito ay may _variable_ na delay.

Maaari mong makuha ang hardware-specified na duration ng isang partikular na hardware-supported na [`Instruction`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.Instruction) sa pamamagitan ng pag-query sa [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target).
Para gawin ito, i-index muna ang [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) gamit ang pangalan ng operasyon, pagkatapos ay ang mga index ng hardware qubits bilang isang tuple.
Ibabalik nito ang `None` (kung sinusuportahan ang instruction, pero walang partikular na properties na itinalaga dito), o isang [`InstructionProperties`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties) instance, kung saan maaaring magbigay ng duration sa segundo ang field na [`duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties#qiskit.transpiler.InstructionProperties.duration) (o `None`, kung hindi ibinigay).

```python
target = "<some Target from a hardware vendor>"

# Query the duration of a CX instruction on hardware qubits (2, 4),
# returning `None` if the duration is not specified.
properties = target["cx"][2, 4]
duration = None if properties is None else properties.duration

# You can also do this in one step, using `getattr`:
duration = getattr(target["cx"][2, 4], "duration", None)
```

Sa pangkalahatan, habang nagiging mas karaniwan ang classical control-flow sa loob ng mga circuit execution, ang "duration" ng isang circuit ay magiging lalong hindi malinaw ang kahulugan.
Bagama't kasama pa rin sa Qiskit v2.x ang attribute na [`QuantumCircuit.duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.duration), inirerekomenda na lumayo na rito, dahil malamang na aalisin ito sa hinaharap; ang mga pagpapalagay kung saan ito valid ay mabibigo nang mas madalas at madalas.
Sa halip, maaari mong gamitin ang bagong paraan na [`QuantumCircuit.estimate_duration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit#qiskit.circuit.QuantumCircuit.estimate_duration), na mas ligtas at mas malinaw tungkol sa mga kinakailangang pagpapalagay.

## qiskit.transpiler

<span id="generate_preset_pass_manager"></span>
### Mga pagbabago sa interface ng `generate_preset_pass_manager` {#changes-to-generate_preset_pass_manager-interface}

Ang Qiskit v2.0.0 ay naglalaman ng serye ng mga update sa preset pass manager interface (iyon ay, ang mga function na
[`generate_preset_pass_manager`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.generate_preset_pass_manager) at
[`transpile`](https://docs.quantum.ibm.com/api/qiskit/compiler#transpile)) na nakakaapekto sa
paghawak ng loose constraint inputs, tulad ng `coupling_map` o `basis_gates`. Nabawasan ang bilang ng
mga transpilation use case kung saan tinatanggap ang loose constraints pabor sa
`target` at `backend` inputs (kung saan ang `backend` ay isang [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) na
naglalaman ng isang [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) instance),
na nagbibigay-daan sa mas maliwanag na komunikasyon ng mga constraint na ito.

Ipinapakita ng mga sumusunod na halimbawa ang mga pangunahing pagbabago at mga mungkahing alternatibo para sa migration. Pansinin na lahat ng
mga tip sa migration sa ibaba ay naaangkop sa parehong `transpile` at `generate_preset_pass_manager`:

1. Ang input argument na `backend_properties` ay hindi na sinusuportahan, at inalis na ang klase na `BackendProperties`.
   Dapat mong gamitin ang mga argument na `target` o `backend` sa halip para ipahayag ang
   mga gate duration at error. Maraming paraan para bumuo ng custom na [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target):

  a. Bumuo ng target mula sa simula at idagdag ang mga kaugnay na error at duration sa mga katumbas na gate:

    ```python
    from qiskit.transpiler import Target, InstructionProperties, generate_preset_pass_manager
    from qiskit.circuit.library import XGate, CXGate

    target = Target(num_qubits=2)
    target.add_instruction(CXGate(), {(0, 1): InstructionProperties(error=.0001, duration=5e-7)})
    target.add_instruction(
        XGate(),
        {
            (0,): InstructionProperties(error=.00001, duration=5e-8),
            (1,): InstructionProperties(error=.00002, duration=6e-8)
        }
    )

    pm = generate_preset_pass_manager(target=target)
    ```

  b. Gamitin ang [`GenericBackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.fake_provider.GenericBackendV2) para mabilis na bumuo ng
     generic target na may pre-filled na mga error at duration. Maaari rin itong i-update pagkatapos ng construction:

    ```python
    from qiskit.providers.fake_provider import GenericBackendV2
    from qiskit.transpiler import generate_preset_pass_manager

    target = GenericBackendV2(num_qubits=2, basis_gates=["x", "cx"]).target
    target.update_instruction_properties("x", (0,), InstructionProperties(0.00001))
    target.update_instruction_properties("x", (1,), InstructionProperties(0.00002))
    target.update_instruction_properties("cx", (0, 1), InstructionProperties(0.0001))

    pm = generate_preset_pass_manager(target=target)
    ```

2. Ang mga input argument na `instruction_durations` at `timing_constraints` ay hindi na sinusuportahan. Dapat mong gamitin ang
  mga argument na `target` o `backend` sa halip. Ang mga alternatibo ay ang pagbuo ng custom na [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) mula sa
  simula, pagbuo ng generic backend, at bilang karagdagan sa mga ito, posible ring bumuo ng custom target gamit ang
  `instruction_durations` at/o `timing_constraints` gamit ang paraan na `Target.from_configuration`:

    ```python
    from qiskit.transpiler import Target, generate_preset_pass_manager
    from qiskit.transpiler.timing_constraints import TimingConstraints
    from qiskit.transpiler.instruction_durations import InstructionDurations

    instruction_durations = InstructionDurations([("x", None, 200), ("cx", None, 1000)], dt=1e-7)

    timing_constraints = TimingConstraints(
            granularity=1, min_length=1, pulse_alignment=1, acquire_alignment=1
        )

    target = Target.from_configuration(num_qubits=2,
                                        basis_gates=["x", "cx"],
                                        instruction_durations=instruction_durations,
                                        timing_constraints=timing_constraints)

    pm = generate_preset_pass_manager(target=target)
    ```

3. Ang pagbibigay ng `coupling_map` at/o `basis_gates` kasama ang isang `backend` ay hindi inirerekomenda, at mula sa
  v2.0 ay magtataas ng `UserWarning`. Sa mga kasong ito ay maraming pinagkukunan ng katotohanan,
  at ang impormasyon tulad ng mga gate error at duration mula sa backend ay hindi malulutas batay sa
  impormasyong ibinigay.
  Ang mungkahing alternatibo ay ang pagtukoy ng custom target na pinagsama ang mga piniling constraint alinman
  mula sa simula, gamit ang [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#from_configuration) o [`GenericBackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.fake_provider.GenericBackendV2).

4. Ang argument na `basis_gates` ay hindi na tumatanggap ng mga custom basis gate. Ang mungkahing alternatibo
  ay ang argument na `target`, na maaaring buuin mula sa simula o sa pamamagitan ng [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#from_configuration)
  tulad ng:

    ```python
    from qiskit.circuit.library import XGate
    from qiskit.transpiler.target import Target

    basis_gates = ["my_x", "cx"]
    custom_name_mapping = {"my_x": XGate()}
    target = Target.from_configuration(
        basis_gates=basis_gates, num_qubits=2, custom_name_mapping=custom_name_mapping
    )
    pm = generate_preset_pass_manager(target=target)
    ```

5. Ang input combination kung saan ibinibigay ang custom na `coupling_map` kasama ang `backend` o `basis_gates`
  na naglalaman ng mga gate na may tatlo o higit pang Qubit ay hindi na pinapayagan. Ang coupling map
  ay hindi nagbibigay ng kinakailangang connectivity details para matukoy ang aksyon ng gate.
  Sa mga kasong ito, ang inirerekomendang migration path ay ang paglikha ng custom target mula sa simula.

6. Ang argument na `inst_map` ay inalis nang walang alternatibo. Ginamit ito para magbigay ng pulse information
  sa transpiler, at ang buong module na `qiskit.pulse` ay inalis na.

## qiskit.pulse

Ang module na `qiskit.pulse` ay ganap na inalis sa Qiskit, nang walang kapalit.
Kasama rito ang mga kaugnay na aytem:

- ang property na `calibrations` ng [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit), [`DAGCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGCircuit) at [`DAGDependency`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGDependency).
- ang mga paraan ng [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit) na `has_calibration_for` at `add_calibration`.
- ang paraan ng [`DAGCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.dagcircuit.DAGCircuit) na `has_calibration_for`.
- ang mga transpiler pass na `PulseGates`, `RXCalibrationBuilder`, `RZXCalibrationBuilder`, `EchoRZXWeylDecomposition`, `ValidatePulseGates`.
- ang module na `qiskit.scheduler` at lahat ng nilalaman nito.
- ang mga paraan ng [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) na `instruction_schedule_map`, `drive_channel`, `measure_channel`, `acquire_channel` at `control_channel`.
- ang mga paraan ng [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) na `has_calibration`, `get_calibration`, `instruction_schedule_map` at `update_from_instruction_schedule_map`.
- ang argument na `inst_map` ng [`generate_preset_pass_manager`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.generate_preset_pass_manager), [`transpile`](https://docs.quantum.ibm.com/api/qiskit/compiler#transpile), at [`Target.from_configuration`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target#qiskit.transpiler.Target.from_configuration).
- ang field na `calibration` ng [`InstructionProperties`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.InstructionProperties).

Nangyari ito dahil ang IBM Quantum&reg; QPUs ay tumigil sa pag-aalok ng pulse-level access, at ang Qiskit team ay wala nang mapagkukunan para mapanatili ang maliit na ginagamit na package.
Maaari kang patuloy na gumamit ng module na `qiskit.pulse` habang ginagamit ang v1.x series ng Qiskit, na may bug-fix support hanggang Setyembre 2024 at security support hanggang Marso 2025, pero ang module ay inalis na sa v2.x series.

Karamihan sa mga paggamit ng `qiskit.pulse` ay maaaring palitan ng direktang access sa mga fractional gate sa IBM Quantum Heron QPUs (at mga susunod na henerasyon, kung naaangkop).
Maaari kang magbasa pa sa [gabay sa migration para sa na-convert na pulse usage papunta sa fractional gates](/guides/pulse-migration).

Hindi mo na maaaring mag-load ng mga QPY file na naglalaman ng mga `ScheduleBlock` object sa Qiskit v2.0 dahil sa pag-aalis ng Qiskit Pulse.
Maaari kang patuloy na mag-load ng mga file na ito sa Qiskit v1.x series.
