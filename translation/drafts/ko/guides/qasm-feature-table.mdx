# OpenQASM 3 기능 표 {#openqasm-3-feature-table}

아래는 `OpenQASM 3` 언어 기능 목록입니다.

이러한 기능에 대한 자세한 내용은 [OpenQASM 3.X 라이브 사양](https://openqasm.com/)을 참조하세요.

범례:

*   ❌ 지원되지 않음
*   🟡 부분 지원
*   ✅ 지원됨

완전히 "지원됨" 표시의 의미는 열에 따라 다릅니다:

* *Qiskit SDK*: [`qiskit.qasm3.loads`](https://docs.quantum.ibm.com/api/qiskit/qasm3)로 파싱하고 (`qiskit-qasm3-import` 확장을 사용하여), [`QuantumCircuit`](https://docs.quantum.ibm.com/api/qiskit/qiskit.circuit.QuantumCircuit)으로 표현하며,
  [`qiskit.qasm3.dumps`](https://docs.quantum.ibm.com/api/qiskit/qasm3#dumps)로 OpenQASM 3으로 내보낼 수 있는 기능입니다.

* *IBM Qiskit Runtime*: 해당 Qiskit 기능을 포함하는 Circuit을 IBM&reg; Qiskit Runtime을 통해 하드웨어에서 성공적으로 실행할 수 있습니다.

"부분 지원"의 의미는 일반적으로 링크된 주석에 따라 다릅니다.

<Admonition type="note">
IBM Qiskit Runtime에 Circuit을 제출하는 가장 일반적인 방법은 Qiskit SDK의 Python 공간 인터페이스에서 Circuit을 생성하는 것입니다.
이러한 방식으로 구성하고 제출한 Circuit은 OpenQASM 3 파일에서 Qiskit SDK로 로드할 필요가 없습니다.

OpenQASM 3을 직접 사용하지 않는다면, Qiskit SDK에서 표현하고 OpenQASM 3으로 내보내며 IBM Qiskit Runtime에 제출하도록 지원되는 기능을 안전하게 사용할 수 있습니다.
여기에는 Qiskit SDK가 OpenQASM 3에서 로드할 수 없는 기능도 포함됩니다.
</Admonition>

## 주석 {#notes}

<span id="1"></span>
1.  이러한 OpenQASM 3 프로그램 기능은 실행에 영향을 주지 않으며, Qiskit은 파일을 파싱하는 과정에서 이를 제거합니다. 이를 사용하는 파일은 제출할 수 있지만 아무런 효과가 없습니다. `include` 파일의 경우, `stdgates.inc`는 현재 Qiskit에 대한 입력으로 지원되며, Backend 실행은 항상 Backend 명령어 집합 아키텍처(ISA)로 컴파일된 Circuit을 필요로 하므로 `include` 파일은 관련이 없습니다.

<span id="2"></span>
2. Qiskit SDK는 모든 `qubit` 선언이 있는 OpenQASM 3 파일의 파싱 및 덤프를 지원합니다. 하드웨어에서 실행하려면 하드웨어 Qubit(예: `$0`) 기준으로 정의된 Circuit만 유효합니다. Circuit이 레이아웃 정보가 있는 Backend에 맞게 Transpiler 처리된 경우, Qiskit SDK는 지원되는 하드웨어 Qubit 식별자를 기준으로 OpenQASM 3을 자동으로 출력합니다.

<span id="3"></span>
3. Qiskit SDK의 `bit` 및 `bit[n]` 타입 변수 선언은 `Clbit` 및 `ClassicalRegister` 선언에 해당합니다.

<span id="4"></span>
4. 2025년 7월 기준으로, Qiskit SDK는 제한된 타입 집합의 로컬 변수를 표현하고, 이러한 객체에 대한 많은 런타임 작업을 표현하며, OpenQASM 3으로 출력하는 것을 지원합니다. 그러나 Qiskit SDK(`qiskit-qasm3-import` v0.6.0 기준)는 변수 선언이 포함된 OpenQASM 3 파일의 파싱을 지원하지 않으며, 변수 표현식 파싱에 대한 지원이 매우 제한적입니다. 일반적으로 Qiskit이 표현식 시스템에서 표현할 수 있는 대부분은 Qiskit SDK가 아직 파싱할 수 없더라도 적합한 동적 Circuit 하드웨어에서 실행할 수 있습니다. 최신 정보는 [`qiskit.circuit.classical` 모듈의 Qiskit 문서](https://docs.quantum.ibm.com/api/qiskit/circuit_classical)를 참조하세요.

<span id="5"></span>
5. Qiskit SDK는 양자 및 클래식 레지스터 모두에 대한 레지스터 별칭을 표현할 수 있지만, 클래식 레지스터의 별칭 사용은 강력히 권장하지 않습니다. 클래식 레지스터에 대한 대부분의 표현식은 별칭과 작동하지 않으며, 별칭 처리된 클래식 레지스터는 하드웨어 실행에 지원되지 않습니다. Qiskit OpenQASM 3 파서는 레지스터 연결 결과를 바인딩하는 `let` 별칭 문을 해석할 수 있습니다.

<span id="6"></span>
6. Qiskit SDK는 `QuantumCircuit.delay`를 통한 명시적 지연과 Circuit 박스(`QuantumCircuit.box`)에 명시적 지속 시간을 지원합니다. 이러한 지속 시간에는 `stretch` 변수의 클래식 표현식이 포함될 수 있습니다. Qiskit SDK(2025년 7월 기준 `qiskit-qasm3-import` v0.6.0)는 OpenQASM 3 파일에서 `duration` 또는 `stretch` 타입 선언의 파싱을 지원하지 않습니다. 하드웨어는 `stretch`를 포함한 지속 시간에 대해 제한적인 지원을 제공합니다.

<span id="7"></span>
7. IBM 하드웨어에서 실행하려면 Circuit을 Backend ISA로 Transpiler 처리해야 합니다. 이로 인해 사용자 정의 `gate` 정의 및 Gate 수정자(예: `inv @`)와 같은 고급 구문은 하드웨어에서 그대로 실행하기에 유효하지 않지만, `transpile` 프로세스가 이를 유효한 ISA Circuit으로 변환합니다. Qiskit SDK(2025년 7월 기준 `qiskit-qasm3-import` v0.6.0)는 파싱 중에 Gate 수정자를 즉시 평가하므로, 이는 결과 `QuantumCircuit`에 나타나지 않을 수 있으며 런타임 비용이 발생할 수 있습니다.

<span id="8"></span>
8. Qiskit SDK는 구조화된 제어 흐름을 표현하고 이를 OpenQASM 3으로 내보낼 수 있습니다. `continue` 및 `break` 문은 기술적으로 Qiskit으로 표현할 수 있지만, Qiskit SDK 내에서도 충분히 지원되지 않습니다. Qiskit v2.1.0의 `for` 루프는 충분히 지원되지 않습니다. 중첩된 제어 흐름(예: 다른 `if` 내의 `if` 또는 `else if` 문)은 하드웨어 실행에 적합하지 않습니다.

<span id="9"></span>
9. Qiskit SDK는 지원되는 모든 클래식 타입을 Circuit의 `input` 변수로 선언하는 것을 지원합니다. 이러한 변수는 현재 하드웨어 실행에 적합하지 않으며, Qiskit OpenQASM 3 임포터로 로드할 수 없습니다. `QuantumCircuit`에 있는 언바운드 `Parameter` 객체는 `input float[64]` 변수로 내보내집니다. 특정 런타임 구성 옵션을 사용하면 일부 Backend에서 이러한 Circuit을 실행할 수 있습니다.
