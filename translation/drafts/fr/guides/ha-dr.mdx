# Comprendre la haute disponibilité et la reprise après sinistre pour Qiskit Runtime {#understanding-high-availability-and-disaster-recovery-for-qiskit-runtime}

La haute disponibilité (HA) désigne la capacité d'un service à rester opérationnel et accessible en cas de défaillances imprévues. La reprise après sinistre est le processus de restauration de l'instance de service dans un état fonctionnel. Les clients de Qiskit Runtime ne sont pas responsables de la planification ni des actions liées à la reprise après sinistre.

Qiskit Runtime est un service régional qui respecte les [objectifs de niveau de service (SLO)](https://cloud.ibm.com/docs/resiliency?topic=resiliency-slo) définis avec tout plan payant. Pour plus d'informations, consulte le [contrat de niveau de service (SLA)](https://www.ibm.com/support/customer/csol/terms/?id=i126-9268&lc=en) pour IBM Cloud&reg;. Pour plus d'informations sur les régions et centres de données IBM Cloud disponibles pour Qiskit Runtime, consulte [Disponibilité des services et de l'infrastructure par région.](https://cloud.ibm.com/docs/overview?topic=overview-services_region)

## Architecture de haute disponibilité {#high-availability-architecture}

### Fonctionnalités de haute disponibilité {#high-availability-features}

Ce service est déployé dans trois zones d'une région multi-zones, sans point de défaillance unique. Les requêtes API sont acheminées via un équilibreur de charge vers les nœuds d'instance HA, chacun situé dans une zone différente.

Si un nœud ou une zone d'instance HA tombe en panne, le service continue de traiter les requêtes API via les nœuds d'instance HA restants. Aucune dégradation perceptible du service ne se produit lors d'une défaillance zonale.

Il est disponible dans les régions suivantes :

- Washington DC (us-east)
- Francfort (eu-de)
