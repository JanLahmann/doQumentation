# Migrer de `BackendV1` vers `BackendV2`

La classe Qiskit [`BackendV1`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.BackendV1) est dépréciée et sera supprimée du service. Ce guide de migration décrit les petits ajustements à effectuer si tu utilises un fournisseur qui est passé de [`BackendV1`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.BackendV1) à [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2).

<Admonition type="note">
 Si tu utilises exclusivement `qiskit_ibm_runtime` et `qiskit_aer`, aucune action n'est nécessaire. Le paquet `qiskit_ibm_runtime` a toujours utilisé `BackendV2`, et `qiskit_aer` utilise `BackendV2` depuis la version 0.13.
 </Admonition>

## Changements de haut niveau dans `BackendV2` {#high-level-changes-in-backendv2}

Le modèle Qiskit [`Backend`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.Backend) a été conçu pour fournir au SDK Qiskit une couche d'abstraction permettant de raisonner sur les ordinateurs quantiques dans le périmètre du SDK. La première itération du modèle a été introduite avec la classe [`BackendV1`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.BackendV1). Cette classe stockait les informations du Backend dans une série de conteneurs de données, à savoir les classes [`BackendConfiguration`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.models.BackendConfiguration) et [`BackendProperties`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.models.BackendProperties).

La classe [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) a redéfini l'accès utilisateur à la plupart des propriétés du Backend afin de les faire fonctionner avec les structures de données natives de Qiskit et de proposer des schémas d'accès plus plats. Le cœur du modèle [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) est la classe [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target), une représentation du QPU qui contient les contraintes de transpilation que Qiskit peut utiliser pour optimiser les circuits en vue de leur exécution.

Le SDK Qiskit a été mis à jour pour fonctionner exclusivement avec des entrées [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2), et la plupart des fournisseurs sont passés de [`BackendV1`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.BackendV1) à [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2). Il est attendu que les fournisseurs existants déprecient l'ancien accès dans la mesure du possible afin de proposer une migration progressive, mais les utilisateurs devront à terme adapter leur code.

Le principe derrière [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2) est que la plupart des informations sur un Backend sont contenues dans son objet [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target), et que les attributs du Backend interrogent souvent son attribut `BackendV2.target` pour retourner des informations. Cependant, dans de nombreux cas, les attributs ne fournissent qu'un sous-ensemble des informations que le target peut contenir. Par exemple, `backend.coupling_map` retourne un objet [`CouplingMap`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.CouplingMap) construit à partir du [`Target`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.Target) accessible via l'attribut `BackendV2.target`. Cependant, le target peut contenir des instructions opérant sur plus de deux qubits (qui ne peuvent pas être représentées dans un [`CouplingMap`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.CouplingMap)), ou il peut avoir des instructions qui n'opèrent que sur un sous-ensemble de qubits (ou de liens à deux qubits, pour une instruction à deux qubits), ce qui ne sera pas détaillé dans la coupling map complète retournée par `BackendV2.coupling_map`. Selon ton cas d'usage, il peut donc être nécessaire d'aller plus loin que le simple accès équivalent avec [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2).

## Changements spécifiques dans `BackendV2` {#specific-changes-in-backendv2}

La plupart des attributs ont un remplacement direct, ce qui simplifie les efforts de migration. Le seul point de divergence entre les interfaces concerne le [`CouplingMap`](https://docs.quantum.ibm.com/api/qiskit/qiskit.transpiler.CouplingMap).

Voici un tableau des exemples de schémas d'accès dans [`BackendV1`](https://docs.quantum.ibm.com/api/qiskit/1.4/qiskit.providers.BackendV1) et leur nouvelle forme avec [`BackendV2`](https://docs.quantum.ibm.com/api/qiskit/qiskit.providers.BackendV2).
