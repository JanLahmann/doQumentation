## 3. 基準3 – 長いコヒーレンス時間（デコヒーレンスとゲート時間）

**基準3：** *「ゲート操作時間よりもはるかに長い、適切なデコヒーレンス時間。」* これは、必要な操作を実行するのに十分な時間、Qubitが量子状態を維持する必要性に対処するものです。各Qubitには **T₁時間**（エネルギー緩和時間：|1〉が|0〉に崩壊する速さ）と**T₂時間**（デファイジング時間：相対的な位相コヒーレンスが失われる速さ）があります。量子コンピュータが正しく動作するためには、これらの時間スケールがゲート操作の持続時間をはるかに上回る必要があります。

**デモの目標：** 実行時間が長くなるにつれてデコヒーレンスがCircuitの結果にどう影響するかを示すことで、Qiskit内のQubitコヒーレンスを調査します。既知のT1/T2時間を持つフェイクBackendを使用して、この効果をシミュレートします。

**有限コヒーレンスの影響を実証する**ために、T1崩壊実験をシミュレートします。Qubitを|1〉状態に準備し、`delay`命令を使用してしばらく待機してから、測定します。遅延時間が増加するにつれて、|1〉を測定する確率が低下することが期待されます。

```python
# This part is done for you. We are creating a list of circuits,
# each with a different delay time.

time_delays_ns = [0, 50000, 100000, 150000, 200000, 250000, 300000]  # delay durations in ns

decay_expts = []
for delay in time_delays_ns:
    qc = QuantumCircuit(1, 1)
    qc.x(0)  # initialize qubit to |1>
    if delay > 0:
        qc.delay(delay, 0, unit='ns')  # wait 'delay' nanoseconds
    qc.measure(0, 0)
    decay_expts.append(qc)

decay_expts[1].draw('mpl') # Visualize one of the circuits
```

### 演習3：T1崩壊実験のシミュレート

次に、`FakeVigo`（T1時間が約50〜100 µs）に基づくノイズありシミュレータを使用して、これらのCircuitを実行します。シミュレータは`delay`命令の実行中にT1/T2エラーを自動的に適用します。このBackend向けにCircuitをトランスパイルして実行してください。

```python
from qiskit_ibm_runtime.fake_provider import FakeVigoV2 as FakeVigo
from qiskit_aer import AerSimulator

# --- YOUR CODE HERE ---

# 1. Create a noisy simulator from the FakeVigo backend
sim_vigo = AerSimulator.from_backend(FakeVigo())

# 2. Transpile the list of circuits for this simulator
pm = generate_preset_pass_manager(backend=sim_vigo, optimization_level=1)

isa_decay_expts = pm.run(decay_expts)

# 3. Use the Sampler to run all the transpiled circuits in a single job
sampler = Sampler(mode=sim_vigo)

job = sampler.run(isa_decay_expts, shots=1000)

# --- END YOUR CODE ---

# This part is done for you to analyze and print the results.
for idx, (delay, qc) in enumerate(zip(time_delays_ns, isa_decay_expts)):
    counts = result[idx].data.c.get_counts()
    p1 = counts.get('1', 0) / 1000  # Assuming 1000 shots
    print(f"Delay {delay} ns: P(qubit=1) = {p1:.3f}")
```

## 4. 基準4 – 普遍的な量子Gateのセット

**基準4：** *「量子Gateの『普遍的な』セット。」* これは、有限の基本Gateのセットを組み合わせることで、ハードウェアが*任意の*量子計算を実行できる必要があることを意味します。古典コンピュータではNANDが普遍的ですが、量子においては普遍的なGateセットの選択肢が多数あります（例：\{H, T, CNOT\}または特定のマシンのネイティブGate）。たとえば、IBMデバイスには特定のQubit間の任意の単一Qubit回転とCNOTなどのネイティブ操作のセットがあり、これらを組み合わせることで普遍的になります。Qiskitの役割はしばしば、**高レベルのGateをこれらの基底Gateにコンパイルする**ことです。

**デモの目標：** GateがQiskitによってどのように分解されるかを示すことで、Gateの普遍性を説明します。ネイティブでないGate（3-Qubit Toffoli Gate、CCXなど）を取り上げ、それがデバイスの基底Gateにどのように分解されるかを見ます。これにより、提供されたGateセットが確かに*普遍的*であること、つまりより複雑な操作を生成できることが示されます。

まず、典型的なIBM BackendのベースGateを確認しましょう。デバイス（またはそのフェイクバージョン）の設定をクエリします。たとえば、ibmq_brisbaneの基底Gateは以下の通りです：
`P(qubit=1)`の確率が遅延時間の増加とともに低下し、T1緩和を特徴とする指数的崩壊曲線をたどることが確認できるはずです。これは、Circuitの実行時間が長すぎる場合に有限コヒーレンス時間がいかに計算エラーを引き起こすかを直接示しています。

**アルゴリズムへの影響：** 長いアルゴリズム（多数の連続したGateを含む）を試みると、総実行時間がT2に近づくかそれを超え、終了前に状態がコヒーレンスを失う可能性があります。そのため、コヒーレンス時間の改善とGateの高速化が、量子ハードウェア研究における最も重要な目標の二つです。

```python
from qiskit_ibm_runtime.fake_provider import FakeBrisbane
fake_brisbane = FakeBrisbane()
print("Basis gates for ibmq_brisbane:", fake_brisbane.configuration().basis_gates)
```

これにより、`['id', 'rz', 'sx', 'x', 'ecr']`のような出力が得られる場合があります。これらはハードウェアがネイティブにサポートするプリミティブ操作です（Identity/no-op、RZ回転、sqrt(X) Gate、X Gate、そしてcontrolled-X）。他のGateはこれらから構成する必要があります。このセットは量子計算において普遍的であることが知られています（本質的に単一Qubit回転とエンタングルする2-Qubit Gateが普遍的なセットを形成します）。

次に、テストケースとして**Toffoli（CCX）Gate**を取り上げます。CCXは、2つの制御Qubitが両方とも1である場合にのみターゲットQubitを反転させます。これはIBMハードウェアのネイティブGateではありません。Qiskitは`ccx`命令を提供していますが、内部的にはそれを分解します。
### 演習4：Toffoli Gateの分解

以下のコードを完成させて、Toffoli（CCX）Gateを含むCircuitを構築し、Qiskitを使用して`FakeBrisbane` BackendのネイティブベースGateに分解してください。

```python
from qiskit import QuantumCircuit
from qiskit_ibm_runtime.fake_provider import FakeBrisbane

# The fake_brisbane backend from the previous cell is reused here.

# --- YOUR CODE HERE ---

# 1. Create a circuit that can accommodate a Toffoli gate
qc_toffoli = QuantumCircuit(3)

# Apply a CCX gate with controls on qubits 0, 1 and target on qubit 2
qc_toffoli.ccx(0, 1, 2)

# 2. Transpile the circuit to the fake Brisbane backend
pm = generate_preset_pass_manager(backend=FakeBrisbane(), optimization_level=1)

isa_qc_toffoli = pm.run(qc_toffoli)

# --- END YOUR CODE ---

print("Toffoli circuit before decomposition:")
print(qc_toffoli)

print("\nToffoli circuit after transpiling to Brisbane basis:")
# The .draw() method will now show the decomposed circuit
print(isa_qc_toffoli.draw(fold=120))
```

トランスパイル後の出力では、CCXが`rz`、`sx`、`ecr`などのより基本的なGateのシーケンスに置き換えられているはずです。これにより、ネイティブGateがToffoliを表現するのに十分であることが証明されます。

**実践における普遍性：** 上記の演習は、複雑な3-Qubit Gateがよりシンプルなものから構築されたことを示しています。一般に、**任意の**多Qubitユニタリは1-Qubitおよび2-Qubit Gateから構成できます。Transpilerは量子ソフトウェアスタックの重要なコンポーネントであり、実行したい抽象的なアルゴリズムと特定の量子デバイスが実際に実行できる物理的な操作との間のギャップを埋めます。

**デバイスの例：** **ibmq_brisbane** デバイスはEagleアーキテクチャを使用し、上記に示した基底Gateを持ちます。つまり、それらのマシンに送られるアルゴリズムはすべて、それらの操作のシーケンスに変換されます。この基準は本質的に**制御可能性**に関するもので、Qubitに対して必要な操作を実行するための十分な制御手段を持っていることを意味します。
## 5. 基準5 – Qubitの測定

**基準5：** *「Qubit固有の測定能力。」* 各Qubitの状態は測定可能でなければなりません（通常は計算基底、|0〉または|1〉で）。言い換えると、量子Circuitを実行した後、各Qubitを0/1の古典的なビットとして読み出す必要があります。この基準は、各Qubitに対して信頼性の高い検出器を持ち、測定するQubitを選択できることに関するものです。

**デモの目標：** Qiskitでシミュレータおよび実際のデバイスで測定を実行する方法を示し、違い（測定ノイズなど）を強調します。さまざまな状態にあるいくつかのQubitを測定し、結果を検討します。また、シミュレータとハードウェアの結果を比較することで、読み出しエラーがどのように現れるかを示します。

まず、シンプルな測定の例を見てみましょう：

```python
qc_measure = QuantumCircuit(2, 2)
qc_measure.x(0)              # qubit 0 -> |1>, qubit 1 stays |0>
qc_measure.measure([0, 1], [0, 1])
qc_measure.draw('mpl')
```

```python
sim_backend = AerSimulator()
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc_measure = pm.run(qc_measure)
job = sampler.run([isa_qc_measure], shots=1000)
result = job.result()
counts = result[0].data.c.get_counts()

print("Simulator measurement counts:", counts)
```

シミュレータでは`01`の1000カウントが期待されます。次に、これをシミュレートすることで**測定エラー**の影響を見てみましょう。AerシミュレータにReadoutErrorを追加することができます。Qiskit Aerでは、`ReadoutError`を定義してノイズモデルのQubitに付加することができます。
### 演習5：読み出しエラーのシミュレート

各Qubitが2%の確率で誤って測定される（0が1として読み取られる、または1が0として読み取られる）シンプルな読み出しエラーモデルを定義するコードを完成させてください。次に、このノイズモデルで測定Circuitを実行してください。

```python
from qiskit_aer.noise import NoiseModel, ReadoutError

# --- YOUR CODE HERE ---

# 1. Define a 2% readout error for each single qubit.
# [[P(0|0), P(1|0)], [P(0|1), P(1|1)]]
ro_error = ReadoutError([[0.98, 0.02], [0.02, 0.98]])

# 2. Create a new noise model
noise_model_ro = NoiseModel()

# 3. Add the readout error to all qubits in the noise model
noise_model_ro.add_all_qubit_readout_error(ro_error)

# --- END YOUR CODE ---

sim_backend.set_options(noise_model=noise_model_ro)
pm = generate_preset_pass_manager(backend=sim_backend, optimization_level=1)
isa_qc_measure = pm.run(qc_measure)

# Run the measurement circuit with readout noise
sampler = Sampler(mode=sim_backend)

job = sampler.run([isa_qc_measure], shots=1024)
result = job.result()
counts = result[0].data.c.get_counts()

print("Simulation with 2% readout error:", counts)
```

このシミュレートされた出力は、実際のハードウェアが生成する可能性のあるものと同様に、誤ったカウント（`11`、`00`、`10`など）を示し、不完全な測定の影響を実証します。

**デバイスの例：** **ibmq_brisbane**などの実際のデバイスでは、同じCircuitを実行すると、不正確な結果に対して同様のゼロでないカウントが見られる可能性が高いです。デバイスのキャリブレーションデータには、各Qubitの読み出しエラーが記載されています。特定のQubitをターゲットにして読み取る能力は非常に重要であり、そのエラー特性を理解することが意味のある結果を得るための鍵です。実際のデバイスでの実行については、**演習1b：実際のIBM Quantum コンピュータでの実行**で示しました。
## 量子通信基準（フライング・Qubit）

DiVincenzoは、ネットワーク接続された量子コンピュータを構築する場合に重要な、量子通信に特化した2つの基準も挙げています：

6. **静止Qubitとフライング・Qubitを相互変換する能力。**（例：プロセッサ内のQubitを移動できる光子にマッピングする。）
7. **フライング・Qubitを場所間で忠実に伝送する能力。**（例：量子情報を失わずに光ファイバーを通じて光子Qubitを送信する。）

Qiskitは主にチップ上の静止Qubitを扱うため、これらは標準的なQiskitの使用範囲を超えています。ただし、**量子テレポーテーション**というシンプルな例でこれらの基準の*概念*を示すことができます。テレポーテーションは、静止Qubitの状態をエンタングルしたペア（「フライング」部分）によって運ばれる情報と古典的な通信に変換し、それを使用して別の場所にある別の静止Qubitで状態を再構成することを示しています。
### 推奨アクティビティ：Qiskit in Classrooms の*量子テレポーテーション*モジュールを受講する

Katie McCormick博士によるQiskit in Classrooms [量子テレポーテーション](https://quantum.cloud.ibm.com/learning/en/modules/computer-science/quantum-teleportation)モジュールは、量子情報において最も魅力的なプロトコルの一つである量子テレポーテーション（エンタングルメントとわずか2つの古典ビットを使用して、AliceからBobに量子状態（Qubit）を送信する）を順を追って案内します。エンタングルしたBellペアの準備方法、Aliceの側でBell基底測定を実行する方法、古典的な結果を送信する方法、そしてBobのQubitに正しい量子Gateを適用して元の状態を完全に回復する方法など、テレポーテーション手順全体をステップバイステップで学びます。その過程で、Qubitをテレポートさせる情報が不可複製定理に違反したり光速を超えたりしない理由も探求します。IBM Quantumハードウェアまたはシミュレータを使用したハンズオン演習を通じて、実際の測定、エンタングルメント、フィードフォワード制御の実用的な理解を深めます。

量子テレポーテーションをマスターすることで、異なるノード間で量子情報をエンコード、送信、回復する方法を理解し、量子ネットワーク、リピータシステム、安全な通信スキーム、およびスケーラブルなモジュール型量子コンピューティングの基盤を築くことができます。
**これが基準6と7にどう関連するか：** 実際の量子ネットワークでは、共有されたエンタングルペアは「フライング」Qubit（光子など）をAliceとBobの場所間に分散させることで生成されます（基準7：忠実な伝送）。テレポーテーションプロトコル自体は、AliceのQubit状態をエンタングルペアの彼女の半分にマッピングし、効果的にBobに「送信」する方法として機能します（基準6：相互変換）。Qiskitはプロトコルのロジックを完全にシミュレートでき、これらの基準が通信アーキテクチャでどのように満たされるかの概念モデルを提供します。
