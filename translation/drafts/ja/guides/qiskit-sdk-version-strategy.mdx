# Qiskit SDKバージョン戦略 {#qiskit-sdk-version-strategy}

Qiskitのバージョン番号は[セマンティックバージョニング](https://semver.org/)に従っています。
バージョン番号は、メジャー、マイナー、パッチという3つの主要なコンポーネントで構成されています。例えば、バージョン番号 `X.Y.Z` において、`X` はメジャーバージョン、`Y` はマイナーバージョン、`Z` はパッチバージョンです。

破壊的なAPI変更はメジャーバージョンのリリースのみに予約されています。メジャーバージョンのリリース間隔の**最短**期間は1年です。マイナーバージョンはAPIの互換性を壊すことなく新機能とバグ修正を導入し、**現在の**メジャーバージョンに対してのみ定期的（現在は3ヶ月ごと）に公開されます。パッチバージョンは、各アクティブにサポートされているリリースシリーズ（つまりメジャーバージョン）の最新マイナーバージョンで識別されたバグの修正を提供します。同時にサポートするリリースシリーズは最大2つであり、これは新しいメジャーバージョンのリリース後の重複期間にのみ発生します。詳細は以下に記載します。

## リリーススケジュール {#release-schedule}

暫定的なリリーススケジュールを以下に示します：

![Tentative Qiskit release schedule](/docs/images/guides/install/release_schedule.avif)

最新のリリーススケジュールについては、常に現在のリリース計画が含まれているQiskit GitHubプロジェクトの[マイルストーンリスト](https://github.com/Qiskit/qiskit/milestones)をご参照ください。

新しいメジャーバージョンのリリースにより、以前のメジャーバージョンはバグ修正のみで少なくとも6ヶ月間、セキュリティ修正については1年間サポートされます。この期間中、このメジャーバージョンに対してはパッチリリースのみが公開されます。サポートが終了する際には最終パッチバージョンが公開され、そのリリースにはそのメジャーバージョンシリーズのサポート終了も記録されます。以前のメジャーバージョンに対してより長いサポートウィンドウが必要なのは、Qiskitのダウンストリームコンシューマーとそのユーザーがコードを移行する機会を与えるためです。Qiskitに依存するダウンストリームライブラリは、新しいメジャーバージョンのリリース直後に必要なQiskitの最小バージョンを新しいメジャーバージョンに引き上げるべきではありません。なぜなら、ライブラリのユーザーベースが新しいAPIの変更に移行するための時間が必要だからです。以前のQiskitメジャーバージョンに対して拡張されたサポートウィンドウを持つことで、ダウンストリームプロジェクトは次のメジャーバージョンとの互換性を確保するための時間を得られます。ダウンストリームプロジェクトは、ユーザーに移行パスを提供するために、一度に2つのリリースシリーズのサポートを提供できます。

セマンティックバージョニングの目的上、QiskitのパブリックAPIはプライベートとしてマークされていない（アンダースコア `_` プレフィックスが付いていない）、文書化されたモジュール、クラス、関数、またはメソッドと見なされます。ただし、特定の文書化されたAPIに対して明示的な例外が設けられる場合があります。そのような場合、これらのAPIはまだ安定したインターフェースとは見なされないことが明確に文書化され、これらの不安定なインターフェースを使用する際にはユーザーに見えるウォーニングが積極的に発行されます。また、状況によっては、プライベートとしてマークされたインターフェースがパブリックAPIの一部と見なされることがあります。これは通常、2つのケースでのみ発生します：インターフェースの実装を定義する一部として、サブクラスがプライベートメソッドをオーバーライド/実装することが意図されている抽象インターフェース定義、または安定したインターフェースを持つが、ユーザー自身がクラス/安全性の不変条件を維持する負担があるため安全に使用するとは見なされない高度な低レベルメソッド（この典型的な例が `QuantumCircuit._append` メソッドです）です。

サポートされているPythonのバージョン、サポートされている最小Rustバージョン（ソースからQiskitをビルドするため）、およびQiskitが使用するPythonパッケージの依存関係（依存関係のサポートされている最小バージョンを含む）は後方互換性の保証の対象外であり、どのリリースでも変更される可能性があります。Qiskitの使用またはビルドの最小要件を引き上げるのは（新しい依存関係の追加を含む）マイナーまたはメジャーバージョンのリリースのみですが、パッチ修正にはPythonや他の依存関係の新しいバージョンのサポートが含まれる場合があります。通常、依存関係の最小バージョンが引き上げられるのは、古い依存関係のバージョンがサポート終了になった場合、または依存関係の最新リリースと古いバージョンとの互換性を維持することが不可能な場合に限られます。

## アップグレード戦略 {#upgrade-strategy}

新しいメジャーバージョンがリリースされた場合、推奨されるアップグレードパスは、まず以前のメジャーバージョンの最新マイナーバージョンにアップグレードすることです。新しいメジャーバージョンの直前に、最終マイナーバージョンが公開されます。この最終マイナーバージョンリリース `X.Y+1.0.0` は `X.Y.0` と同等ですが、新しいメジャーバージョンシリーズで行われるAPI変更に対するウォーニングと非推奨通知が追加されています。

例えば、1.0.0のリリース直前に、0.46.0リリースが公開されます。0.46.0リリースは0.45.0リリースと同等ですが、1.0.0リリースの一部として行われたAPI変更を文書化する追加の非推奨ウォーニングが含まれています。このパターンは将来のメジャーバージョンリリースにも使用されます。

Qiskitのユーザーはまずこの最終マイナーバージョンにアップグレードして非推奨ウォーニングを確認し、潜在的に破壊的なリリースを試みる前にQiskitの使用方法を調整する必要があります。以前のメジャーバージョンはアップグレードに十分な時間を与えるために少なくとも6ヶ月間サポートされます。これを管理する典型的なパターンは、互換性を確認するまで次のメジャーリリースシリーズの使用を避けるために最大バージョンをピン留めすることです。例えば、現在のQiskitメジャーバージョンが1の場合、requirementsファイルに `qiskit<2` を指定することで、破壊的なAPI変更を含まないバージョンのQiskitを使用していることが保証されます。

次のメジャーバージョン未満にバージョンを制限することで、メジャーバージョンリリースの前に非推奨ウォーニングを確認できます。制限がない場合、`pip` はデフォルトで利用可能な最新バージョンをインストールします。

QPYシリアライゼーション形式は後方互換性があり、新しいQiskitリリースは常に以前のQiskitリリースで生成されたQPYファイルを読み込むことができます。ただし、この形式は前方互換性がないため、原則として新しいバージョンのQiskitで生成されたQPYファイルを古いリリースで読み込むことはできません。メジャーバージョンリリース間のユーザー移行を容易にするため、[`qiskit.qpy.dump()`](https://docs.quantum.ibm.com/api/qiskit/qpy#dump) 関数は常に `X.0.0` リリースと `X-1.Y.0` リリース（`Y` はそのシリーズの最後のマイナーバージョン）の間で少なくとも1つの重複バージョンをサポートします。パラメータ `qiskit.qpy.dump(..., version=...)` を使用することで、新しいリリースから両方のメジャーバージョンで読み込み可能なQPY形式ファイルを保存できます。詳細は[RFC 0020](https://github.com/Qiskit/RFCs/blob/master/0020-release_cycle.md#qpy-support)をご覧ください。

## プレリリース {#pre-releases}

各マイナーおよびメジャーバージョンのリリースに対して、Qiskitは[PEP440](https://peps.python.org/pep-0440/)と互換性のあるプレリリースを公開します。通常、これらは `X.Y.0rc1` の形式のリリース候補です。`rc` リリースはファイナライズされたAPIサーフェスを持ち、予定されているリリースのテストに使用されます。

PEP440のプレリリースサフィックス（`a`、`b`、`pre` など）が公開される場合、`rc` リリースと同じ保証はなく、プレビューリリースに過ぎないことに注意してください。これらのプレリリースと同じバージョン番号の最終リリース間でAPIが変更される可能性があります。例えば、`1.0.0pre1` は `1.0.0` とは異なる最終APIを持つ可能性があります。

## ポストリリース {#post-releases}

リリースのパッケージングに問題がある場合、これを修正するためにポストリリースが発行される可能性があります。これらは `X.Y.Z.1` の形式に従い、4番目の整数が `X.Y.Z` リリースの最初のポストリリースであることを示します。例えば、qiskit-terra（Qiskitの旧パッケージ名）の0.25.2リリースはsdistパッケージの公開に問題があり、この問題を修正したポストリリース0.25.2.1が公開されました。コードは同一であり、0.25.2.1はリリースのパッケージング問題のみを修正しました。

## コントリビューターが非推奨をマークする方法 {#how-contributors-can-mark-deprecations}

ソースコードに非推奨を追加する方法については、Qiskit SDKリポジトリの[非推奨ガイド](https://github.com/Qiskit/qiskit/blob/main/DEPRECATION.md)をご参照ください。
